# vision: clearer guidance on human approval blocks

## the outcome world

### a day-in-the-life

a clone runs `rhx route.stone.set --stone 3.3.blueprint.v1 --as passed`. the stone has a guard that requires human approval. the judge fails and emits clear guidance. the clone sees in stderr:

```
🔎 judge 1
   ├─
   │
   │  passed: false
   │  reason: wait for human approval
   │
   │  ✋ halted, human approval required
   │     ├─ please ask your human to
   │     │  └─ rhx route.stone.set --stone 3.3.blueprint.v1 --as approved
   │     │
   │     └─ after human approves, run
   │        └─ rhx route.stone.set --stone 3.3.blueprint.v1 --as passed
   │
   └─
```

the clone reads this, tells the human exactly what to do, and the human runs the approval command. done.

### before/after contrast

**before** — the judge output was minimal:
```
passed: false
reason: wait for human approval
```

the clone doesn't know:
- what command the human should run
- what command the clone should run after approval
- had to read help text, search files, or guess

**after** — the judge emits full guidance with tree structure:
```
passed: false
reason: wait for human approval

✋ halted, human approval required
   ├─ please ask your human to
   │  └─ rhx route.stone.set --stone 3.3.blueprint.v1 --as approved
   │
   └─ after human approves, run
      └─ rhx route.stone.set --stone 3.3.blueprint.v1 --as passed
```

### the "aha" moment

the clone says to the human: "i need you to run this command" and pastes the exact command from the output. no search, no guess, no help text.

## design: judges emit their own guidance

the key insight: **each judge emits its own next-step guidance**. the judge artifact content flows into stderr automatically via `parseJudgeArtifactForStderr`.

### why this works

1. **no special case** — `setStoneAsPassed` doesn't need to detect "approval" vs "review" failures
2. **accurate commands** — judge has access to stone name, so commands are always correct
3. **composable** — each judge type (`approved?`, `reviewed?`, custom) can emit its own guidance
4. **consistent** — uses same tree structure as `route.drive --mode hook`
5. **testable** — snapshots verify exact output

### the `approved?` judge output

```
passed: false
reason: wait for human approval

✋ halted, human approval required
   ├─ please ask your human to
   │  └─ rhx route.stone.set --stone <stone> --as approved
   │
   └─ after human approves, run
      └─ rhx route.stone.set --stone <stone> --as passed
```

### the `reviewed?` judge output (when blockers exceed threshold)

```
passed: false
reason: blockers exceed threshold (2 > 0)

🔧 fix the blockers, then run
   └─ rhx route.stone.set --stone <stone> --as passed
```

## user experience

### usecases & goals

| actor | goal | what they do |
|-------|------|--------------|
| clone | pass a guarded stone | runs `--as passed`, sees clear next-step commands |
| human | approve agent work | runs the `--as approved` command shown in output |
| clone | continue after approval | runs `--as passed` again (command shown in output) |

### timeline

1. clone runs `--as passed`
2. guard judges run, `approved?` judge fails
3. judge emits guidance with tree structure
4. clone tells human: "please run: `rhx route.stone.set --stone X --as approved`"
5. human runs approval command
6. clone runs `--as passed` again
7. stone passes

## mental model

### how would users describe this to a friend?

> "when the agent gets blocked on my approval, it tells me exactly what command to run. no guess work."

### analogies & metaphors

- like a gatekeeper who hands you the key and says "come back with this"
- like a CI system that says "click here to approve deployment" not "deployment blocked, see logs"

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| clone knows next step | yes | judge emits command |
| human knows what to run | yes | judge emits command |
| no search or guess | yes | all info in judge output |
| regression protection | yes | snapshots + assertions |

### pros

- zero cognitive load for clone or human
- consistent tree structure with emoji
- copy-pasteable commands
- each judge owns its guidance

### cons

- slightly longer output when blocked (but clearer)

### edgecases & pit of success

| edgecase | how we handle it |
|----------|------------------|
| clone runs `--as passed` before human approves | judge emits same clear message |
| human runs `--as approved` twice | idempotent, no harm |
| stone has multiple guards | each failed judge emits its own guidance |
| stone has no guard | passes directly, no message needed |
