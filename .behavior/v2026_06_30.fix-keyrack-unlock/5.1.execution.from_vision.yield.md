## design notes

- **why subprocess, not `keyrack.source` or raw `unlockKeyrackKeys`:** `keyrack.source` does a
  keyrack *get* (fails if locked — doesn't unlock); the raw `unlockKeyrackKeys` API needs a full
  `ContextKeyrack` rebuilt via `genContextKeyrack` + host-manifest load. the `rhx keyrack unlock`
  subprocess is the only path that actually unlocks and reuses the shipped CLI.
- **CI-safe:** keyrack uses `process.env` (wisher-confirmed); the temp-dir acceptance test proved
  the unlock succeeds from a fixture repo via the global host manifest for `ehmpath`.
- **verb choice:** `setKeyrackUnlocked` — `set` (mutate/upsert session state) per get-set-gen verbs.
