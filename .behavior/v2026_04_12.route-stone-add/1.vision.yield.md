# vision: route.stone.add

## the outcome world

### before

a driver is mid-route, working on a behavior. the wisher says: "hey, add a research stone to investigate X before we proceed."

the driver has no way to do this. they must:
1. ask the wisher to manually create the stone file
2. or exit the route, manually create the file, then resume
3. or skip the research entirely

the route is rigid — stones are declared at init time, not discovered during the journey.

### after

```sh
# driver adds a research stone on the fly
rhx route.stone.add --stone 3.1.6.research.custom.x --from @stdin <<'EOF'
investigate X:
- what is X?
- how does X relate to our wish?
- what are the tradeoffs of X?

emit findings to $route/3.1.6.research.custom.x.yield.md
EOF
```

the stone appears in the route. the driver continues their journey with the new checkpoint.

### the "aha" moment

the route is no longer a fixed path — it's a living map that adapts to discoveries along the way. just like a real road trip, sometimes you need to add a detour to explore something interesting.

## user experience

### usecases

| goal | command |
|------|---------|
| add research stone from stdin | `rhx route.stone.add --stone 3.1.6.research.custom --from @stdin` |
| add stone from template | `rhx route.stone.add --stone 3.1.6.research.adhoc --from template($behavior/refs/.research.adhoc.template.stone)` |
| add stone with inline content | `rhx route.stone.add --stone 2.3.criteria.edge --from 'define edge cases for the api'` |
| preview before add | `rhx route.stone.add --stone 3.1.6.research.custom --from @stdin --mode plan` |

### contract

```
rhx route.stone.add [options]

options:
  --stone <name>     stone name to create (required)
                     must have numeric prefix + at least one alpha segment
                     e.g., 3.1.6.research.custom, 5.2.evaluation.adhoc
  --from <source>    content source (required):
                     - @stdin: read content from stdin
                     - template(<path>): use template file, e.g., template($behavior/refs/.research.adhoc.template.stone)
                     - 'literal text...': use inline text as content
  --route <path>     path to route directory (optional)
                     defaults to current bound route; failfast if unbound
  --mode <plan|apply> plan = preview (default), apply = create stone
```

### timeline

1. driver is bound to a route, working on stone N
2. wisher requests additional research or checkpoint
3. driver runs `rhx route.stone.add --stone X --from @stdin`
4. driver pipes the stone content
5. stone file appears in route
6. driver continues journey, eventually reaching stone X

### outputs

- plan mode: shows what would be created (stone name, content preview)
- apply mode: creates the stone file, confirms success

### note on stdin and plan mode

when `--from @stdin` is used with plan mode (default), stdin content is consumed for preview. to apply, the driver must pipe content again or use `--mode apply` directly.

```sh
# plan mode consumes stdin — must re-pipe to apply
cat prompt.md | rhx route.stone.add --stone 3.1.6.research --from @stdin       # preview
cat prompt.md | rhx route.stone.add --stone 3.1.6.research --from @stdin --mode apply  # apply

# or: skip plan mode when stdin is involved
cat prompt.md | rhx route.stone.add --stone 3.1.6.research --from @stdin --mode apply
```

## mental model

### how users describe it

> "i can add a checkpoint to my route on the fly"
> "it's like adding a todo item to your journey"
> "the route grows as you discover what's needed"

### analogies

| analogy | maps to |
|---------|---------|
| road trip detour | adding a research stone mid-route |
| todo list append | adding any stone |
| git commit | each stone is a checkpoint |

### terminology

| user says | we say |
|-----------|--------|
| "add a step" | `route.stone.add` |
| "insert a checkpoint" | `--stone 3.1.6.research.custom` |
| "from this file" | `--from @stdin` |
| "use the standard template" | `--from template($behavior/refs/.research.adhoc.template.stone)` |

## evaluation

### pros

- flexible routes that adapt to discoveries
- matches extant `route.stone.*` conventions
- simple, familiar cli pattern
- plan/apply safety pattern

### cons

- could lead to route bloat if overused
- stone order must be understood (numeric prefixes)

### edgecases and pit-of-success

| edgecase | handling |
|----------|----------|
| no bound route | failfast with clear error: "no route bound; use `rhx route.bind.set` first" |
| stone already exists | failfast: "stone already exists; use different name or `route.stone.del` first" |
| invalid stone name | failfast: "stone name must have numeric prefix followed by at least one alpha segment (e.g., 3.1.6.research.custom)" |
| empty stdin | failfast: "no content provided via stdin" |
| template file not found | failfast: "template file not found: $path" |
| unbound `--from` | require explicit source, no default |

## open questions & assumptions

### assumptions

1. stone names follow numeric prefix convention (1.x, 2.x, 3.x, etc.)
2. templates are path-based, referenced via `template($behavior/refs/...)` syntax
3. `--route` defaults to bound route (consistent with how `route.drive` works)
4. plan mode is default (consistent with `route.stone.del`)
5. `$behavior` variable in template paths resolves to the current behavior route

### questions for wisher

1. [answered] should `--route` be auto-detected only, or also accept explicit path?
   - answer: follow extant pattern — auto-detect via `resolveRouteFromBind()`, accept explicit `--route` override
   - all `route.stone.*` commands use this pattern already
   - note: extant shell docs incorrectly say `--route` is required; should be fixed to say "optional, uses bound route if absent"
2. [answered] should guards be addable too?
   - answer: no, stones only for now
   - can add guard support later if needed
3. [answered] should we validate stone position relative to current stone?
   - answer: no validation — let driver pick position
   - rationale: driver knows their journey; don't gatekeep

## what is awkward?

### naming tension

the wish uses `--where` and `--how`, but extant conventions use:
- `--stone` for the stone identifier
- no extant pattern for content source

proposal: use `--stone` (matches family) and `--from` (clear semantics)

| wish | proposal | rationale |
|------|----------|-----------|
| `--where` | `--stone` | matches route.stone.get/set/del |
| `--how` | `--from` | clear "from this source" semantics |

### template vs stdin vs literal

three content sources:
- `--from @stdin` — pipe content
- `--from template($path)` — use template file with `$behavior` variable support
- `--from 'text...'` — inline literal

single `--from` with pragmas is simpler and extensible than separate flags.

### ordering complexity

stone names have numeric prefixes (3.1.6, 5.2, etc.). users must:
1. understand the numbering scheme
2. pick an appropriate position
3. avoid collisions

this is inherent to the stone model, not specific to `route.stone.add`. the skill should validate format but not enforce position logic.

---

## summary

`route.stone.add` enables drivers to dynamically expand their route by adding stones on the fly. it follows extant `route.stone.*` conventions, uses `--stone` and `--from` flags, and defaults to the bound route with failfast if unbound.

the key insight: routes should be living maps, not fixed paths. discoveries during the journey should be capturable as new stones.
