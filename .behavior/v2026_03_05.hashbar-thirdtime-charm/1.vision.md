# vision: 30s cooldown, self-trust after one change (hashbar: 1, threshold: 30s)

## the outcome world

### day-in-the-life with this in place

a clone works on a stone artifact. they complete their work and attempt to pass:

```
🗿 route.stone.set
   ├─ stone = 1.vision
   └─ passage = blocked (self-review required)

🌕 lets reflect
   │
   ├─ review.self 1/2
   │  ├─ slug = all-done
   │  └─ ...
   │
   └─ promise its done? if so, run
      └─ rhx route.stone.set --stone 1.vision --as promised --that all-done
```

the clone does the self-review. they wait 30s. they attempt to promise again:

```
🗿 route.stone.set
   └─ stone = 1.vision
   └─ passage = progressed (self-review 1/2 promised)
```

done. 30s cooldown was enough.

if they had edited the artifact (hash change), they'd wait 30s again. after one edit, self-trust kicks in — further edits don't reset the timer.

### before/after contrast

| before (hashbar: 3, 90s) | after (hashbar: 1, 30s) |
|--------------------------|-------------------------|
| 3 hash changes before self-trust | 1 hash change to self-trust |
| 90s cooldown per attempt | 30s cooldown per attempt |
| ~5+ minutes if editing | ~30s if no edit, ~1min if one edit |
| clones game by inflating counter | simple: wait 30s, edit = wait again |

### the "aha" moment

the value clicks when a clone realizes the system enforces a **cooldown pattern**:

1. **first attempt** → challenged, must wait 30s
2. **if no change** → promise after 30s
3. **if change** → wait 30s again

after one hash change (hashbar reached), the trigger becomes hashless. further edits are **self-trust** — clone just waits 30s again, no timer reset on hash change.

the clone can't rush. they must wait at least 30s before promise is accepted. if they change the artifact, they wait again. simple.

## user experience

### usecases and goals

1. **quick but honest feedback loop** — clone catches gaps, fixes them, one iteration is enough
2. **hashless triggers faster** — timer persists after one change instead of three
3. **less gaming incentive** — no need to make trivial changes just to bump the counter
4. **configurable rigor** — route authors can still set higher hashbar if they want more iterations

### guard config (unchanged syntax)

```yaml
reviews:
  self:
    - slug: all-done
      hashbar: 1           # now the default (was 3)
      say: |
        did you complete all that was requested?

    - slug: tests-pass
      hashbar: 3           # route author can still require more iterations
      say: |
        do all tests pass?
```

`hashbar` = how many hash changes before the trigger becomes hashless.
- `hashbar: 1` (new default) → after 1 change, trigger is hashless (timer persists)
- `hashbar: 3` → prior default, more iterations before timer persists
- `hashbar: 0` → trigger always hashless (timer never resets on hash change)

`threshold` = cooldown time before promise can be accepted.
- `30s` (new default) → quicker feedback loop
- `90s` → prior default

### timeline (internal mechanics, invisible to clone)

```
t0: clone attempts --as passed
    → self-review triggered for "all-done"
    → creates: $stone.guard.selfreview.all-done.$hash1.triggered
    → glob count = 1, challenged

t1a (no edit): clone waits 30s, attempts --as promised
    → same hash, mtime > 30s → promise accepted

t1b (with edit): clone edits artifact, waits 30s, attempts --as promised
    → hash changed to $hash2
    → creates: $stone.guard.selfreview.all-done.$hash2.triggered
    → glob count = 2 > $hashbar (1) → trigger is hashless
    → check mtime of newest triggered file > 30s? yes → promise accepted

t2: further edits = self-trust
    → trigger is hashless, hash changes don't reset timer
    → clone waits 30s, promises → accepted
```

the clone sees: challenged → wait 30s → promised. if they edit, wait 30s again. after one edit, self-trust.

## mental model

### how users describe this to a friend

> "you just wait 30 seconds. if you edit, wait 30 seconds again. simple."

### analogies that fit

- **cooldown timer**: like a spell cooldown — use it, wait, use again
- **elevator door**: press the button, wait, doors open
- **microwave**: set time, wait, done — edit, wait again

### terms users vs we use

| user says | we say |
|-----------|--------|
| "just wait 30 seconds" | threshold: 30s cooldown |
| "if i edit, wait again" | hash change resets timer (until hashbar reached) |
| "then it trusts me" | count > hashbar, trigger is hashless, self-trust |

## evaluation

### how well does it solve the goals?

| goal | how well? |
|------|-----------|
| reduce unnecessary iteration | ✅ one change is enough |
| maintain review quality | ✅ 30s cooldown still enforced |
| reduce gaming incentive | ✅ no need to inflate counter |
| still configurable | ✅ route authors can set hashbar: 3+ |
| backwards compatible | ✅ just a default change |

### pros

1. **faster feedback loop** — clones get unblocked sooner
2. **less gaming** — one real change beats three trivial ones
3. **simpler mental model** — "fix once, move on"
4. **still rigorous** — 30s cooldown ensures actual review time
5. **configurable** — route authors can raise hashbar if needed

### cons

1. **less forced iteration** — some clones might benefit from more cycles
2. **route authors may need to explicitly raise hashbar** — if they want the old behavior

### edgecases and pit of success

| edgecase | how contract keeps users safe |
|----------|-------------------------------|
| clone tries quick fix without review | 30s cooldown still enforced — must wait |
| route author wants more rigor | can set hashbar: 3 or higher explicitly |
| extant routes with no hashbar specified | get new default (1) — simpler |
| clone makes many quick changes | still need to wait 30s after count > hashbar |

### what's awkward

1. **migration**: extant routes get new default — but this is intentional (simpler behavior)
2. **"third time's the charm" becomes "second time's the charm"** — the name "hashbar-thirdtime-charm" is ironic now

## summary

the vision is:
- **30s cooldown** — clone must wait before promise accepted
- **self-trust after one change** — hashbar: 1 means trigger is hashless after first edit
- **hashbar: 3 → 1** — one hash change to reach self-trust instead of three
- **threshold: 90s → 30s** — quicker cooldown
- **route authors can configure higher values** — flexibility preserved
- **mechanism stays invisible** — clone just sees "blocked" or "progressed"

clone can't rush. wait 30s, promise. if you edit, wait 30s again. after one edit, self-trust kicks in.
