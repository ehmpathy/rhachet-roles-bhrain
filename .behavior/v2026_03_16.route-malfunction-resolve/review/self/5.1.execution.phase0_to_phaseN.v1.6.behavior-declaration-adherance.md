### verification 2: runStoneGuardReviews.ts

opened `src/domain.operations/route/guard/runStoneGuardReviews.ts`

#### the blueprint says (fix 2):

```ts
// before
const cached = priorReviews.find(r => r.hash === artifactHash);

// after
const cached = priorReviews.find(r =>
  r.hash === artifactHash &&
  r.exitClass === 'passed'
);
```

#### what the code actually does:

```ts
// lines 134-140 in runStoneGuardReviews.ts
const cachedReviews = priorArtifacts.reviews.filter(
  (r) => r.exitClass !== 'malfunction',
);
```

#### adherance analysis:

the blueprint suggests filter by `exitClass === 'passed'` but the actual code uses filter `exitClass !== 'malfunction'`.

**is this a deviation?**

the vision says:
> broken reviewers don't cache
> only successes get cached

the criteria says:
> given: reviewer malfunctions
> then: fresh review runs (not cached)
> given: reviewer succeeds
> then: cached review is reused

**analysis:**

| filter | keeps 'passed' | keeps 'constraint' | keeps 'malfunction' |
|--------|----------------|--------------------|--------------------|
| `=== 'passed'` | YES | NO | NO |
| `!== 'malfunction'` | YES | YES | NO |

the actual implementation keeps 'constraint' reviews. is this correct?

a 'constraint' review means: reviewer ran successfully, found blockers, exited normally.
- the reviewer did not crash
- the review is valid
- blockers were correctly identified

the vision says "broken reviewers don't cache" — constraint is NOT broken.
the criteria says "reviewer succeeds" — constraint IS a success (reviewer worked).

**conclusion:** `!== 'malfunction'` is MORE CORRECT than `=== 'passed'`. the blueprint was too restrictive. the implementation is correct.

**adherance: implementation is BETTER than blueprint** ✓
