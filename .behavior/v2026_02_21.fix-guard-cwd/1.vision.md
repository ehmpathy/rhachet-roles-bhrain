# vision: $route variable works intuitively in guard judge commands

## the outcome world

### before

a behavior author writes a guard file with a judge command:

```yaml
judges:
  - npx rhachet run --repo bhrain --skill route.stone.judge --mechanism approved? --stone $stone --route $route
```

when the guard executes:
1. the `runStoneGuardJudges` function sets `cwd` to the route directory
2. `$route` is substituted with the same route path
3. the judge receives a route path relative to the route directory
4. the judge looks for stones at `<route>/<route>` — a doubled, nonexistent path
5. the guard fails with a cryptic "stone not found" error

the author is confused. they try various workarounds. they eventually learn they must use `--route .` instead of `--route $route`. this feels wrong — why does `$route` exist if it fails?

### after

the same guard file works as expected:

```yaml
judges:
  - npx rhachet run --repo bhrain --skill route.stone.judge --mechanism approved? --stone $stone --route $route
```

when the guard executes:
1. the command runs from repo root (no cwd override)
2. `$route` is substituted with the route path (e.g., `.behavior/v2026_02_21.my-feature`)
3. the judge receives the correct path, finds the stone
4. the guard passes

the author writes guards intuitively. `$route` does what they expect.

### the "aha" moment

> "oh, `$route` just works now. i can write guards without concern for work directories."

## user experience

### usecase: behavior author writes a guard

**goal**: gate stone progression on approval artifacts

**inputs**:
- guard template with `$route` and `$stone` variables
- judge command that validates artifacts exist

**outputs**:
- guard passes when artifacts present
- guard fails with clear error when artifacts absent

**timeline**:
1. author creates `.behavior/.../2.guard.jsonc`
2. author adds judge: `npx rhachet run ... --route $route`
3. behavior progresses through stones
4. guards execute as expected — no path errors

### usecase: bhrain developer tests route mechanics

**goal**: ensure route.drive respects guards

**inputs**:
- test fixture with route, stones, guards
- integration test that drives route progression

**outputs**:
- tests pass when guards are satisfied
- tests fail fast with helpful errors when guards block

**timeline**:
1. developer writes integration test
2. test invokes `route.drive`
3. route respects guards, judge commands work
4. test asserts expected outcomes

## mental model

### how users describe it

> "guards are checkpoints. judges are the checks. $route tells the judge where to look."

### analogies

- `$route` is like `$PWD` in shell — it's the path to where we are
- change cwd while also pass the path = give directions twice, overlap them
- remove cwd override = run from home base, pass clear directions

### terms

| user term | our term | definition |
|-----------|----------|------------|
| guard | `stoneGuard` | checkpoint that blocks progression |
| judge | `judgeCmd` | command that evaluates the checkpoint |
| route | `input.route` | path to the behavior route directory |

## evaluation

### how well does it solve the goals?

✅ `$route` works intuitively — no workarounds needed
✅ guards execute from repo root — consistent with user expectations
✅ artifact paths resolve correctly — no doubled paths
✅ backwards compatible — prior `--route .` workarounds still work

### pros

- simpler mental model: commands run from repo root, paths are relative to repo root
- no special cwd logic = fewer surprises
- aligns with how other rhachet commands work

### cons

- if someone relied on cwd set to the route dir, their guards break
  - mitigation: this is unlikely since the prior behavior was broken for `$route`
  - mitigation: any guard that used `$route` was already broken, so they used `--route .`

### edgecases & pit of success

| edgecase | behavior | pit of success |
|----------|----------|----------------|
| `$route` in judge command | substituted with route path from repo root | ✅ works |
| `--route .` workaround | works when cwd was route; fails when cwd is repo root | ⚠️ breaks — but this was the workaround, not intended usage |
| absolute route path | still works | ✅ works |
| nested behavior dirs | path stays relative to repo root | ✅ works |

**awkward discovery**: prior guards that use `--route .` will break after the fix. these guards will need to change to `--route $route`. however, this is the correct usage — the workaround becomes unnecessary.

**mitigation**: release notes should document this change and the migration path.
