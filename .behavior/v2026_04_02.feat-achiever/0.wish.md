wish =

we'd like to add a new role to the bhrain repo, called 'achiever'

---

this role's purpose is to enable achivement of goals

---

sofar, we already have a driver, which enables brains to follow thought routes

but there's another, independent dimension that we've realized both human and robot brains need and often lack for success

the 'achiever' role dimension

---

specifically

1. the ability to discern distinct goals 
    - from communications (e.g., when a human talks to a robot brain, that input)
    - from internalizations (e.g., from self talk, and own thoughts)

we'd like to start out with the skill to detect and persist distinct goals

---

eventually, we'll build on this with a hook system to ensure that an achiever never forgets to accomplish their goals

and further, we'll build on this to enable the achiever to prioritize the goals theyve detected

and we'll likely need to go through multiple interations on how to tune deduplication and decomposition of goals

but for now, lets start with just the ability to detect and remember goals

---

specifically,

we want to start with the absolute root primitive of how to persist goals and the fundamental shape of goals

probably the following skills

goal.memory.set
goal.memory.get 
goal.infer 

---

in support of those

we'd like to formalize the shape of a goal. a goal has the key components of 
- ask (what was said)
- task (what to do)
- gate (how to verify)
- root (why it was asked)

---

 - what,

 - how
}

---

and formalize how and where to persist them

1. into the $route/.goals/ dir, if within a route
2. into the reporoot/.goals/ dir, if not within a route

---

and if its not already clear, `goal.infer.triage` operates on the queue of ALL the user content. so we need a way to guarantee that the brain has processed each of the user feedback and associated it to a distinct goal.

so, `goal.infer.triage` should be backed by an `onTalk` hook which accumulates literlaly every distinct input that is sent from the talker

saves it to a jsonl list in order of receipt

and then guarantees that each item on that list gets explicitly tied to a specific goal. (i.e., the 'ask' should get appended on that goal further and further, and in order to complete `goal.infer.triage` so that the `onStop` hook stops halting the brain, each and every ask that was accumulated needs a distinct goal tied to it.

the other benefit here is, we'll be able to detect that whenever the hash of the goal dir changes, we need to recheck which asks are covered and which are not

so that if a goalset is recomposed (e.g., split or reorganized), we can be sure that no asks are left behind
