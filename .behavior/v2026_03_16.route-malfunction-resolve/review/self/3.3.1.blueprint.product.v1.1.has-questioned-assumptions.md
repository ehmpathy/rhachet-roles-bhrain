**what we assume:** entries are only appended, never edited or deleted.

**what if the opposite were true?** if entries could be edited, we'd need to track versions or timestamps. but the vision explicitly states "passage.jsonl is a log" with append-only semantics.

**evidence:** code uses `fs.appendFile` for passage writes. this is verified in research stone.

**verdict:** valid assumption — based on evidence.

### assumption 2: last entry per stone = current state

**what we assume:** the most recent entry for a stone represents its current status.

**what if the opposite were true?** we'd need a different mechanism to track "official" status. but the wisher explicitly said "if the last status is passed, then the malfunction was fixed."

**evidence:** the wisher's words in the wish. log semantics are standard (last entry wins).

**verdict:** valid assumption — explicitly stated by wisher.

**what we assume:** reviews with exitClass 'passed' are successful and cacheable.

**evidence:** judges already filter by `&& passed` in cache logic. this pattern is extant.

**verdict:** valid assumption — follows extant pattern.

### assumption 4: malfunctions should not be cached

**what we assume:** only successful reviews should be cached.

**what if the opposite were true?** cached malfunctions would block retry of fixed reviewers forever.

**evidence:** wisher explicitly said "only successes get cached."

**verdict:** valid assumption — explicitly stated by wisher.

### assumption 5: one review per hash is sufficient

**what we assume:** if hash matches and review passed, we can reuse it.

**what if the opposite were true?** we'd need to re-run reviews even when content hasn't changed. wasteful.

**evidence:** this is how the cache already works. we only add a stricter filter.

**verdict:** valid assumption — follows extant pattern.
