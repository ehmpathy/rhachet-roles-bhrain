# vision: achiever role

## the outcome world

### day-in-the-life

a human opens claude code and says:

> "hey, can you fix the flaky test in auth.test.ts, and also update the readme to mention the new env var, and when you're done ping me on slack"

today: the brain dives into the first thing it understood, forgets the readme, never pings slack. the human has to re-ask.

with achiever: the brain immediately detects three distinct goals, persists them, and works through each — checking them off as complete. no goal falls through the cracks.

### before/after contrast

| before | after |
|--------|-------|
| goals blur together | goals are discrete, named, tracked |
| brain forgets mid-task | goals persist across context |
| human re-asks frequently | human trusts goals are remembered |
| no visibility into what's pending | `.goals/` shows all outstanding asks |

### the "aha" moment

the human glances at `.goals/` and sees:

```
# repo-scoped goals (branch-aware, main forbidden)
.goals/feat-achiever/
├── asks.inventory.jsonl                                    # all peer input, in order
├── asks.coverage.jsonl                                # askId → goalSlug map
├── 0000000.fix-auth-test.goal.yaml
├── 0000000.fix-auth-test.status=fulfilled.flag  # empty marker
├── 0000003.update-readme-env.goal.yaml
├── 0000003.update-readme-env.status=inflight.flag
├── 0086400.notify-slack-on-done.goal.yaml
└── 0086400.notify-slack-on-done.status=enqueued.flag

# route-scoped goals
$route/.goals/
├── asks.inventory.jsonl
├── asks.coverage.jsonl
├── 0000000.define-shape.goal.yaml
├── 0000000.define-shape.status=fulfilled.flag
├── 0003600.write-tests.goal.yaml
└── 0003600.write-tests.status=blocked.flag      # when.goal = define-shape
```

**file pattern:**
- `asks.inventory.jsonl` — all peer input, in order, each with content hash
- `asks.coverage.jsonl` — checklist: which hash is covered by which goalSlug
- `$offset.$slug.goal.yaml` — source of truth for a goal (full content)
- `$offset.$slug.status=$choice.flag` — status marker (empty file, status from filename)

**why flag files?** — status.choice visible from filename alone. no file parse needed. `glob('**/*.status=inflight.flag')` finds all active goals in O(1) per file. full status.reason persisted in `.goal.yaml` for complete record.

note: prefix is seconds offset from parent dir mtime (`.goals/$branch/` for repo-scoped, `$route/.goals/` for route-scoped). 7-digit leftpad supports weeks of offset. goals on main branch are forbidden. repo-scoped `.goals/` findserts a `.gitignore`.

the brain didn't forget. the human didn't have to repeat themselves. that's the aha.

## user experience

### usecases

1. **multi-part requests** — human asks for 3 things, all 3 get tracked
2. **long-running work** — goals survive context compression
3. **self-generated goals** — brain notices it should also update the changelog, adds that goal
4. **goal review** — human checks `.goals/` to see what's pending
5. **goal completion** — brain marks goal done with verification evidence

### contracts

**goal.infer.triage**

*usecase:* triage input into goals — create new, update extant, or link to extant

*why triage, not just infer:* the direct brain has full context (conversation history, extant goals). a nested brain call would lose that context. triage requires the brain itself to assess with everything it knows.

*hooks:*
- `goal.infer.triage --from peer --when hook.onTalk`
  - fires when peer sends message
  - **does NOT halt** brain if inflight — just reminds: "if this impacts your goals, think through it first"
  - accumulates ask to `.goals/$branch/asks.inventory.jsonl` (with content hash)

- `goal.infer.triage --from peer --when hook.onStop`
  - fires on session end
  - **halts until triage complete** — guarantees all asks get processed
  - brain must cover each ask via `goal.memory.set --covers $hash,...`
  - coverage entries appended to `.goals/$branch/asks.coverage.jsonl`, then session may continue
  - if `route.drive` is also loaded, triage takes precedence (route.drive yields)

*ask accumulation:*
- every peer input saved to `.goals/$branch/asks.inventory.jsonl` in order of receipt
- each ask identified by content hash (deterministic, no counter needed)
- preserves context to diagnose abandoned or changed-mind asks later

*coverage map:*
- `.goals/$branch/asks.coverage.jsonl` maps each ask hash to a goalSlug
- format: `{ hash, goalSlug, coveredAt }`
- coverage recorded via `goal.memory.set --covers $hash,...`
- when goal dir hash changes, recheck which asks are covered
- ensures no asks left behind if goals are recomposed (split, reorganized)
- session may end only when all asks have coverage entries

*self-reflection (background):*
- `goal.infer.triage get --from self` — subconscious brain runs in background
- assesses internal thoughts for goal candidates
- drops `.goals/$branch/needs-triage.self.flag` with thought content
- main brain picks up flag and triages when ready

*briefs:*
- seed the brain with habit: "always run goal.memory.set when you detect a goal"
- seed the brain with habit: "ask follow-up questions to disambiguate vague asks before you set a goal"
- explain peer vs self sources
- make goal triage core behavior, not afterthought

*triage flow:*
```
peer input arrives (hook.onTalk)
  → append to asks.inventory.jsonl with content hash
  → remind brain: "consider if this impacts goals"

session ends (hook.onStop)
  → brain runs goal.infer.triage, then for each uncovered ask:
    1. read asks.inventory.jsonl (all accumulated peer input)
    2. read asks.coverage.jsonl (which asks are already covered)
    3. goal.memory.get (fetch extant goals)
    4. triage each uncovered ask with full context:
       - new goal? → goal.memory.set with full schema + --covers $hash
       - update extant? → goal.memory.set --slug $slug --covers $hash
       - already covered by extant? → goal.memory.set --slug $slug --covers $hash
    5. repeat until all asks covered
    6. session may end
```

*contracts (directional, not prescribed):*

**goal.infer.triage**
```
input:
  - from?: 'peer' | 'self'
  - context?: { route?: string }

output:
  - asks: Ask[] (uncovered asks, each with hash)
  - goals: Goal[] (extant goals for reference)
  - coverage: Coverage[] (which asks are already covered)
```

**why explicit coverage?** — forces the brain to cross off asks like a checklist. each `--covers $hash` is a deliberate act that links asks to goals. this helps the brain focus: triage asks, cover them, move on. no ask left behind.

**why the nested Goal shape matters:**

the brain can't shortcut. "fix the test" must become:
- why.ask: what was literally said
- why.purpose: the motivation (ci pass)
- why.benefit: what success enables (team ships)
- what.outcome: the end state (test passes)
- how.task: the approach and next action (diagnose, stabilize, run in isolation)
- how.gate: verification criteria (10 consecutive passes)

structure unlocks clarity.

**goal.memory.set**
```
input:
  - goal: Goal (full schema for new, or --slug for update)
  - covers: string[] (ask hashes this goal covers)
  - context?: { route?: string }

output:
  - path: string (where goal persisted)
  - covered: string[] (hashes appended to asks.coverage.jsonl)
```

**goal.memory.get**
```
input:
  - filter?: { status?: { choice?: 'blocked' | 'enqueued' | 'inflight' | 'fulfilled' } }
  - context?: { route?: string }

output:
  - goals: Goal[]
```

**Goal shape**
```ts
interface Goal {
  slug: string;         // unique identifier
  why: {
    ask: string;        // what was said (verbatim or summarized)
    purpose: string;    // the motivation behind the ask
    benefit: string;    // what success enables
  };
  what: {
    outcome: string;    // the desired end state
  };
  how: {
    task: string;       // actionable next step
    gate: string;       // how to verify completion
  };
  status: {
    choice: 'blocked' | 'enqueued' | 'inflight' | 'fulfilled';  // from flag filename
    reason: string;  // from flag content
  };
  when?: PickOne<{
    goal: string;    // blocked until this goal is fulfilled
    event: string;   // blocked on external event
  }>;
  source: 'peer:human' | 'peer:robot' | 'self';  // who originated it
  createdAt: string;
  updatedAt: string;
}
```

**why nested?** — forces the brain to think through all dimensions. each field is a distinct thought that must be articulated. shallow inference ("just fix it") becomes impossible when the shape demands purpose, benefit, outcome, task, and gate. same for status: can't mark `blocked` without a reason.

### what it looks like

human says: "fix the test and update the readme"

brain runs `goal.infer` and reasons through each field. example `.goal.yaml`:

```yaml
# 0000000.fix-auth-test.goal.yaml
slug: fix-auth-test
why:
  ask: fix the test
  purpose: human wants ci to pass before merge
  benefit: unblocks the pr, team can ship
what:
  outcome: auth.test.ts passes reliably
how:
  task: run test in isolation, identify flake source, apply fix
  gate: test passes 10 consecutive runs
status:
  reason: |
    ran `npm run test:unit -- auth.test.ts` 10 times
    all passed after mock stabilization
source: peer:human
createdAt: "2026-04-02"
updatedAt: "2026-04-02"
```

brain persists via `goal.memory.set`, works on goals, marks fulfilled with evidence in `status.reason`.

### timelines

1. **immediate** — goal.infer runs on human input, goals persisted
2. **during work** — brain checks goals, works toward them
3. **on completion** — brain updates status to fulfilled, adds evidence in status.reason
4. **on context switch** — goals survive, brain can resume

## evaluation

### how well does it solve the goals?

**primary goal: don't forget multi-part requests** — solved. goals are persisted, enumerable, checkable.

**secondary goal: transparency** — solved. human can inspect `.goals/` anytime.

**tertiary goal: self-generated goals** — enabled. brain can add goals from its own observations.

### pros

- simple, file-based persistence (git-friendly, inspectable)
- clear shape: ask/task/gate/root covers the essential dimensions
- location-aware: route-scoped vs repo-scoped
- source tracking: know if human asked or brain inferred

### cons

- inference quality depends on llm — may miss implicit goals
- deduplication not yet solved — same goal asked twice?
- prioritization not yet solved — which goal first?
- no cross-repo goals — scoped to current repo

### edge cases & pit of success

| edge case | pit of success |
|-----------|----------------|
| vague ask ("make it better") | gate forces specificity — brain must articulate verification |
| duplicate ask | dedup by task similarity (future iteration) |
| conflicting goals | surface conflict, ask human to clarify |
| goal becomes irrelevant | human or brain can mark 'blocked' with reason |

## open questions & assumptions

### assumptions

1. **file-based persistence is sufficient** — we assume `.goals/` in filesystem is adequate; no db needed
2. **ask/task/gate/root captures the essential shape** — we may discover absent dimensions
3. **goals are repo-scoped** — no cross-repo goal track (yet)
4. **llm can reliably infer goals** — inference quality untested
5. **[answered] completed goals are kept, not deleted** — auditability requires history; archive to `.goals/.archive/` if clutter becomes a problem
6. **[answered] source distinction is valuable** — wish distinguishes "from communications" vs "from internalizations"; track origin for filter/audit
7. **[answered] gates are descriptive for v1** — executable gates exceed "root primitive" scope; plain text verification for now

### questions to validate with wisher

1. **[wisher]** should goals have priority/urgency? (wish defers to future, but confirm scope)
2. **[wisher]** should goals have dependencies (goal B blocked by goal A)?
3. **[wisher]** can goals span multiple routes? (current design: route-scoped or repo-scoped)

### external research needed

1. **[research]** how do other ai systems track goals? (autogpt, babyagi, etc.)
2. **[research]** what goal taxonomies exist in cognitive science?
3. **[research]** how do humans naturally express goals vs tasks vs wishes?

## what is awkward? (triaged)

### naming: ask vs task — [answered: keep "ask"]

"ask" implies human-originated, but self-generated goals don't have an "ask" in the same sense.

**decision for v1**: keep "ask". the term is simple and the wish uses it ("what was said"). brain can still articulate what prompted a self-generated goal. revisit if user feedback indicates confusion.

### gate specificity — [answered: require gates]

some goals are fuzzy ("improve code quality") and hard to gate.

**decision for v1**: require gates. forcing a gate promotes clarity — if brain can't articulate how to verify, the goal may be too vague. "improve code quality" becomes "reduce lint errors to zero" or "pass all tests". fuzzy goals are a smell — articulate verification or decompose into clearer sub-goals.

### persistence location — [partially answered]

route-scoped vs repo-scoped feels right, but what about:
- goals that span multiple routes? → **[wisher]** added to questions
- goals that exist before any route is created? → **[answered]** repo-scoped `.goals/` handles this per wish line 63

**decision for v1**: repo-scoped by default, route-scoped when in a route. multi-route goals deferred to wisher confirmation.

### goal lifecycle — [answered: brain marks done]

when is a goal truly "done"?

**decision for v1**: brain marks done with evidence. this matches "root primitive" scope — simple state transition. human can inspect `.goals/` and challenge if needed. human confirmation or automated gate verification can be added later.
