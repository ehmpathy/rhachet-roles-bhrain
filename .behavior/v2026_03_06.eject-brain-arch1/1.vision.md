# vision: rhachet-brains-bhrain

> the bhrain supplier of brains 🐢🧠

---

## the outcome world

### what does a day-in-the-life look like?

**before (current state):**
- brain-arch1 is embedded in rhachet-roles-bhrain
- to use the brain.repl, you install the whole roles package
- the brain role and the reviewer/thinker roles are coupled
- updating the brain requires releasing the entire roles package
- briefs about brain architecture live alongside role-specific briefs

**after (ejected state):**
- rhachet-brains-bhrain is a standalone package that supplies brains
- any repo can `npm install rhachet-brains-bhrain` to get the brain.repl
- rhachet-roles-bhrain depends on rhachet-brains-bhrain for brain capabilities
- brain updates ship independently from role updates
- architect briefs about brains live in the brain package

```
# before
roles-bhrain
├── brain role (skills, briefs)
├── reviewer role
├── architect role (brain briefs)
├── domain.operations/brain.replic.arch1/
└── domain.objects/BrainArch1/

# after
brains-bhrain
├── .agent/repo=.this/role=architect/
│   └── briefs/ (brain architecture knowledge)
├── domain.operations/arch1/ (the repl implementation)
├── domain.objects/ (BrainArch1* types)
└── access/sdks/ (anthropic, openai, qwen)

roles-bhrain
├── reviewer role
└── thinker role
```

### what's the "aha" moment?

the aha is: **"decoupled test lifecycles = massive cost and time savings"**

the core insight:
- **bhrain roles change frequently** — reviewer, thinker, behaver roles evolve rapidly
- **brain arch1 changes rarely** — the repl implementation is stable
- **test suites are expensive** — integration tests cost time (ci minutes) and money (api calls)

with the eject:
- brain arch1 tests only run when brain arch1 changes (rarely)
- bhrain role tests run independently (frequently, but cheaper without brain tests)
- no more spend on expensive brain integration tests for every role change

**secondary benefit: ecosystem reuse**
- rhachet-roles-ehmpathy could use rhachet-brains-bhrain
- any rhachet package could leverage the same brain.repl
- brain improvements benefit everyone

## user experience

### what usecases do folks fulfill?

| usecase | what they do | what they get |
|---------|--------------|---------------|
| **invoke brain.repl** | `await invokeBrainArch1({ actor, userInput }, context)` | provider-agnostic agentic loop |
| **register as brain supplier** | add to `rhachet.use.ts: getBrainRepls()` | brain appears in `rhx act`, `rhx ask` |
| **customize toolboxes** | compose toolboxFiles, toolboxBash, toolboxWeb | fine-grained capability control |
| **guard permissions** | use readOnly, promptUser, or custom guard | safety by default |
| **swap atoms** | use anthropic, openai, or qwen | provider flexibility |

### what contract inputs & outputs do they leverage?

**inputs:**
```typescript
// the actor defines the brain's configuration
const actor = new BrainArch1Actor({
  atom: genAtomAnthropic({ model: 'claude-3-5-sonnet-latest' }),
  toolboxes: [toolboxFiles, toolboxBash],
  memory: null,
  permission: permissionGuardReadOnly,
  constraints: {
    maxIterations: 100,
    maxTokens: 8192,
  },
  role: {
    systemPrompt: null,
  },
});

// invocation
const result = await invokeBrainArch1(
  { actor, userInput: 'read the file at /tmp/test.txt' },
  context,
);
```

**outputs:**
```typescript
interface BrainArch1LoopResult {
  terminationReason: 'NATURAL_COMPLETION' | 'MAX_ITERATIONS' | 'ERROR';
  finalResponse: string;
  finalMessage: BrainArch1SessionMessage;
  messages: BrainArch1SessionMessage[];
  iterations: BrainArch1LoopIteration[];
  iterationCount: number;
  totalTokenUsage: BrainArch1MemoryTokenUsage;
  error: Error | null;
}
```

### what would it look like to leverage them?

```bash
# from rhachet-roles-bhrain (unchanged cli)
npx rhachet skills run act -t ./output.md "your task here"

# programmatically (now from dedicated package)
import { invokeBrainArch1 } from 'rhachet-brains-bhrain';
import { genAtomAnthropic, toolboxFiles } from 'rhachet-brains-bhrain';

const result = await invokeBrainArch1({
  actor: new BrainArch1Actor({ ... }),
  userInput: 'analyze this codebase',
}, context);
```

## mental model

### how would users describe this to a friend?

> "it's like having claude-code as a library. you configure an actor with tools and permissions, give it a task, and it loops until done. the brain handles tool calls, memory, and termination — you just ask and get answers."

### what analogies or metaphors fit?

| metaphor | explanation |
|----------|-------------|
| **brain as plug-in** | like a graphics card you slot in — same interface, different capabilities |
| **actor as character sheet** | defines what the brain can do (tools), how cautious it is (permissions), and its limits (constraints) |
| **loop as conversation** | user asks → brain thinks → brain acts → loop until satisfied |
| **toolbox as capability kit** | files, bash, web — compose what the brain can touch |

### what terms would they use vs what terms would we use?

| user term | our term | mapping |
|-----------|----------|---------|
| "the AI" | brain.repl | BrainArch1 |
| "model" | atom | BrainArch1Atom |
| "tools" | toolbox | BrainArch1Toolbox |
| "conversation" | session | BrainArch1Session |
| "response" | loop result | BrainArch1LoopResult |
| "turn" | iteration | BrainArch1LoopIteration |

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| decouple test lifecycles | ✓ | brain tests run separately from role tests |
| reduce ci costs | ✓ | expensive brain tests don't run on role changes |
| reduce ci time | ✓ | focused test runs per package |
| decouple brain from roles | ✓ | separate npm package |
| enable reuse across repos | ✓ | any rhachet package can depend on it |
| architect knowledge stays with brain | ✓ | briefs move to .agent/repo=.this/role=architect |

### what are the pros?

1. **decoupled test lifecycles** — brain arch1 tests run separately from role tests
2. **cost reduction** — expensive integration tests don't run on unrelated changes
3. **time savings** — ci pipelines only run what's relevant to the change
4. **single responsibility** — brains-bhrain does one thing: supply brains
5. **reusability** — any package can use the brain without the roles
6. **smaller install footprint** — roles package becomes lighter

### what are the cons?

1. **dependency management** — roles-bhrain now depends on brains-bhrain
2. **coordination overhead** — breaking changes require coordinated releases
3. **split context** — debugging may span two repos
4. **migration work** — extant imports need updating

### what edgecases exist and how do our contracts keep users in a pit of success?

| edgecase | pit of success |
|----------|----------------|
| missing api key | clear error: "no anthropic api key found" |
| input exceeds context | BadRequestError before loop starts |
| max iterations hit | terminationReason: 'MAX_ITERATIONS' |
| permission denied | permissionGuard returns verdict before execution |
| invalid toolbox | type errors at compile time |

## open questions & assumptions

### what assumptions have we made?

1. **BrainArch1 is the only brain impl** — currently true, but room for arch2, arch3
2. **briefs split cleanly** — architect briefs about brains vs roles
3. **no circular deps** — roles depends on brains, not vice versa
4. **sdk wrappers stay in brains** — anthropic, openai, qwen sdks move with brain

### what questions have been answered?

1. **where does the brain role go?** → moves to brains-bhrain
   - wisher clarified: brain role drops from this repo after ejection
   - brain role and brain implementation stay together

2. **how do we handle rhachet's BrainRepl interface?** → arch1 implements it
   - wish explicitly says: "BrainArch1 is actually exactly a BrainRepl implementation"
   - registration via rhachet.use.ts in consumer repo

3. **what about the test utilities?** → split based on dependencies
   - genMockBrainArch1Context → moves (imports BrainArch1Context)
   - genTestBrainContext → stays (imports from rhachet-brains-* packages)

4. **version strategy?** → start at 0.1.0
   - npm convention: new package = new version sequence

5. **repo name** → `rhachet-brains-bhrain` (wish explicit)

6. **architect briefs** → all of `brains.replic/` moves (wish explicit)

7. **thinker briefs** → stay (verified: no BrainArch1 references)

8. **reviewer briefs** → stay (verified: no BrainArch1 references)

### what must we validate with the wisher before we proceed?

**none** — all questions were answerable via wish, code inspection, or logic.

## what is awkward?

### what feels off or forced?

1. **the name "arch1"** — implies there will be arch2. is this intentional?
   - if yes: good, version scheme is explicit
   - if no: maybe rename to just "repl" or "loop"

2. **briefs split** — resolved but feels forced
   - architect briefs: move (brain arch concepts belong with brain)
   - thinker briefs: stay (general brain terminology, no arch1 refs)
   - reviewer briefs: stay (about brain selection, not arch1 internals)
   - awkward: the split feels arbitrary but is correct by dependency analysis

3. **access/sdk vs access/sdks** — two directories for the same concept
   - sdk/sdkOpenAi.ts
   - sdks/anthropic/sdkAnthropic.ts
   - sdks/qwen/sdkQwen.ts

### where does the design fight the user's mental model?

1. **BrainArch1 vs BrainRepl** — users might expect a single "Brain" class
   - arch1 is implementation, repl is interface
   - documentation needs to clarify this

2. **actor vs config** — "actor" is theatrical, might confuse users
   - but it's intentional: the actor plays a role with capabilities

### what tradeoffs feel uncomfortable?

1. **two packages to maintain** — more overhead, but cleaner separation
2. **import path changes** — extant code needs updates
3. **reviewer findings about brains** → decided: stay with reviewer
   - verified: no BrainArch1 references in reviewer briefs
   - findings are about brain selection for review tasks, not arch1 internals

## summary

the ejection creates `rhachet-brains-bhrain` as the canonical supplier of brain.repl implementations. BrainArch1 implements rhachet's BrainRepl interface, enabling any rhachet package to leverage agentic capabilities.

the split follows single responsibility: brains supply brains, roles supply roles.

**files to eject:**
- `src/domain.objects/BrainArch1/*`
- `src/domain.operations/brain.replic.arch1/*`
- `src/access/sdk/` and `src/access/sdks/`
- `src/domain.roles/architect/briefs/brains.replic/*`
- `src/domain.roles/brain/` — brain role moves to brains-bhrain
- `src/.test/genMockBrainArch1Context.ts`

**files that stay (decided via code inspection):**
- `src/domain.roles/thinker/briefs/term=brain.*` — stays (no arch1 refs)
- `src/domain.roles/reviewer/briefs/lessons/lesson.brain-*` — stays (no arch1 refs)
- `src/.test/genTestBrainContext.ts` — stays (imports from rhachet-brains-* packages)
