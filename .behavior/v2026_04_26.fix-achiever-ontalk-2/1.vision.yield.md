# vision: fix achiever onTalk hook trigger for other hooks

---

## ⛔ BLOCKER: upstream dependency

**this vision is blocked by Claude Code bug #16952.**

the `UserPromptSubmit` stdin JSON lacks a `source` field to distinguish user prompts from hook/subagent output. the schema is:

```json
{ "session_id", "cwd", "hook_event_name", "prompt" }  // no source field
```

**options to unblock:**
1. file upstream feature request for `source` field
2. proceed with fragile content heuristics
3. implement internal state track (rhachet remembers "just ran onStop")

**see:** `1.vision.blocker.md` for full analysis

## the outcome world

### what does a day-in-the-life look like with this in place?

when the fix is in place:

1. human sends message → onTalk triggers → ask is accumulated → reminder shown
2. human continues work → tools run → onTalk does NOT trigger
3. human attempts to stop → onStop triggers → check for uncovered asks
4. human stops session cleanly

the key difference: **onTalk only fires for actual human prompts**, never for hook outputs or system events.

### what's the before/after contrast?

**before (current):**
```
human: "fix the bug"
  → onTalk fires ✓ (ask accumulated)

[onStop runs, outputs reminder]
  → onTalk fires ✗ (incorrectly accumulates hook output as ask!)
  → spurious asks pollute asks.inventory.jsonl
```

**after (fixed):**
```
human: "fix the bug"
  → onTalk fires ✓ (ask accumulated)

[onStop runs, outputs reminder]
  → onTalk does NOT fire (hook output is not a human prompt)
  → asks.inventory.jsonl contains only real human asks
```

### what's the "aha" moment where the value clicks?

the "aha" moment: run `rhx goal.triage.infer` and see only real human asks in the uncovered list — not spurious entries from hook outputs. the triage flow becomes trustworthy again.

## what is awkward?

### what feels off or forced?

1. **source discrimination is absent:** the current contract has no way to distinguish user prompts from other sources. adding this requires either upstream change or heuristics.

2. **hook filter limitation:** `BrainHookFilter` only supports `what` (tool pattern) and `when` (before/after) — no event source filter for onTalk hooks.

3. **reproduction confirmed:** the mechanism is Claude Code bug #16952 — `UserPromptSubmit` fires on subagent/Task completion (like onStop), not just user prompts. the hook output appears as the `prompt` field in stdin.

### where does the design fight the user's mental model?

users expect "asks" to be "things i asked the clone to do". when hook output appears as asks, it violates this expectation completely. the mental model is correct — the implementation is wrong.

### what tradeoffs feel uncomfortable?

| tradeoff | discomfort |
|----------|------------|
| upstream dependency | if fix requires Claude Code changes, we can't ship it independently |
| heuristics | if we filter by content pattern, it may miss edge cases |
| added complexity | any filter adds code that must be maintained and tested |

the least uncomfortable path: if Claude Code already distinguishes user prompts at the event level, we just need to expose that information. if not, we may need to propose an upstream contract change or accept a more fragile local solution.
