# self-review: has-play-test-convention (iteration 3)

## deeper review: test file convention

a breath. fresh eyes. question the convention choice.

### the guide asks

use `.play.test.ts` suffix for journey tests:
- `feature.play.test.ts`
- `feature.play.integration.test.ts`
- `feature.play.acceptance.test.ts`

### what I previously concluded

use `.acceptance.test.ts` because that's the repo convention.

### re-examine that conclusion

**is my test really an acceptance test?**

yes. it tests the blackbox behavior via skill invocation:
- invokes `route.drive --mode hook` via shell
- invokes `route.stone.set --as passed` via shell
- asserts on exit codes and file contents

this matches the extant acceptance tests in `blackbox/`:
- same pattern: `given` → `when` → `then`
- same invocation: `invokeRouteSkill()`
- same assertions: exit codes, file contents

**would `.play.acceptance.test.ts` be better?**

the guide says to use `.play.acceptance.test.ts` if repo requires acceptance runner. this repo does require it (`npm run test:acceptance`).

however, the extant tests don't use `.play.` prefix:
- `driver.route.drive.acceptance.test.ts` — not `.play.`
- `driver.route.bind.acceptance.test.ts` — not `.play.`

a `.play.` prefix would be inconsistent with extant tests.

### final decision

**use `driver.route.malfunction.acceptance.test.ts`**

**why it holds:**
1. matches extant convention in this repo
2. works with `npm run test:acceptance` runner
3. avoids inconsistency with other route tests
4. the test is an acceptance test — name is accurate

### lesson

when repo convention differs from guide recommendation, follow repo convention for consistency. document the deviation.

no issues — convention choice is justified.
