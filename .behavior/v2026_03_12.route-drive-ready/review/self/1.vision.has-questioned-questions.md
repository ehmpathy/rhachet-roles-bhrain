## question #1: is "done? run this to proceed" clear enough?

**can i answer via code?** yes — let me trace the extant pattern.

**i went to the code.** in `stepRouteDrive.ts:278`:
```typescript
lines.push(`   ├─ are we there yet? if so, run`);
lines.push(`   │  └─ ${passCmd}`);
```

the current pattern: question + conditional + command.

**i examined refs/example.case1.md.** the stuck clone saw this text 21+ times. it responded by saying "vision artifact ready. pending passage." it never ran the command.

**what did the clone interpret?**
- "are we there yet?" → a question for someone else to answer
- "if so, run" → a conditional the clone didn't feel it could evaluate

**will "done? run this to proceed" fix this?**

let me trace the semantic shift:
- "done?" → still a question, but simpler
- "run this to proceed" → no conditional, direct imperative

but: the example clone's issue was *who runs*, not *what to run*. "done? run this" still doesn't say *you* run this.

**the nudge says:** "YOU run this command, don't wait"

this is the explicit addressee. the normal guidance still lacks it.

**answer via logic:** the normal text change improves clarity but doesn't fully fix the agency issue. the nudge is the real fix. the normal change is incremental improvement.

**escalate to wisher?** no — the wisher asked for the nudge specifically. normal text is bonus improvement. can ship as-is.

## question #3: should nudge escalate further?

**can i answer via logic?** yes.

**analysis:**

the vision asks: should nudge become more insistent at 10, 15?

**i checked code patterns.** extant escalation model:
- hooks 1-20: clone keeps work
- hook 21+: human notified

**if we add nudge tiers:**
- hook 5: gentle nudge
- hook 10: stronger nudge
- hook 15: urgent nudge
- hook 21: human

**arguments against tiers:**
1. **complexity**: more tiers = more code paths
2. **diminished returns**: clone that ignores 5 nudges likely ignores 10
3. **extant safety net**: human escalation at 21 catches stuck clones

**answer via logic:** flat nudge is sufficient. no tiers needed.

**escalate to wisher?** no — answered via design logic.

### question #5: what triggers counter reset?

**vision says:** "on route.stone.set attempt: reset counter"

**but what counts as "attempt"?**

**i traced code.** when clone runs `--as passed`:
1. `stepRouteStoneSet` is called
2. guards run
3. if blocked → `setStoneGuardBlockerReport` is called
4. if passed → `setPassageReport` with status='passed'

**the question:** does counter reset on any invocation? or only on success?

**analysis:**
- if reset on any attempt: clone who tries and fails restarts at 0
- if reset only on success: clone who keeps fail gets perpetual nudge

**correct behavior:** reset on any attempt. the problem is clones who *never try*. clones who try are engaged.

**answer:** design decision. document: "counter resets on any `rhx route.stone.set --stone X --as passed` invocation, regardless of outcome."

### question #6: what if clone runs wrong stone command?

**situation:** route.drive shows stone A. clone runs `--as passed --stone B`.

**does A's counter reset?** no — command was for stone B, not A.

**is this a problem?** no — the counter is per-stone. stone A remains stuck. stone B gets progress.

**answer:** per-stone semantics handle this correctly.

### question #7: does counter persist across context compaction?

**situation:** clone context compacts. new summary. same session.

**does counter persist?** yes — filesystem state is external to clone memory.

**answer:** this is architectural. filesystem persists.

### question #8: what if human manually runs --as passed?

**situation:** human runs `rhx route.stone.set --stone X --as passed`.

**does counter reset?** yes — any invocation resets.

**is this right?** yes — the nudge is about inaction. human action demonstrates engagement.

**answer:** correct behavior. no special case.
