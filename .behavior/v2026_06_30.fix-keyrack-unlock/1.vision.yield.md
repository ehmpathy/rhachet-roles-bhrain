# vision: fix keyrack unlock

> reviewers always need keyrack unlocked. so unlock it ourselves, upfront —
> and if we can't, failfast with a clear hint instead of a fake malfunction.

## the outcome world

### before

a reviewer is invoked (peer review guard, or a human runs `rhx review`). the keyrack
daemon was never unlocked for `ehmpath/prep`. the brain credential fetch fails deep
inside `genContextBrain`, and the human sees something that *looks* like a malfunction:

```
🦉 patience, friend

🗿 route.review
   └─ ✋ blocked by constraints
```

or worse — the failure surfaces only after the guard has already spun up several peer
reviewers, wasting wall-clock time and tokens before anyone learns the real cause:
**the keyrack was simply never unlocked.**

90% of first-review attempts hit this. the human eventually digs out the fix, runs
`rhx keyrack unlock --owner ehmpath --env prep` by hand, then retries.

### after

the reviewer notices it needs `ehmpath/prep` creds and **unlocks the keyrack itself**
before any expensive work. the output stays quiet — **no bespoke ceremony**:

**on success — stay silent, or a single line.** either emit no unlock output at all, or one
line under the stonehead's header:

```
🗿 route.review
   ├─ unlocked ehmpath/prep
   └─ ... review proceeds
```

(the exact placement of that one line is a small ergonomic choice for the blueprint — a
line under the stonehead header, or silence. the point: no separate `🔓 keyrack unlock`
tree, no `✨ unlocked` flourish.)

**on failure — forward keyrack's own stdout, verbatim.** don't re-wrap or re-invent the
message. keyrack already emits a precise, actionable error; just surface it and exit with
its code:

```
🔓 keyrack unlock
   └─ ✋ <whatever keyrack itself printed — locked / absent / prikey hint / etc>
```

we do **not** author our own `✋ keyrack could not be unlocked …` block. keyrack owns that
vocabulary; we pass it through. (`rhx keyrack unlock` / `--json` already produces the
locked/absent/prikey guidance — see groundwork.)

### the "aha" moment

the value clicks the first time a reviewer "just works" with a locked keyrack — no
manual unlock, no retry, and no noisy output. and when it can't, the failure is keyrack's
*own* honest message, surfaced immediately — instead of a confusing malfunction surfaced
late after wasted work.

## user experience

### usecases

| goal | contract | timeline |
|------|----------|----------|
| run a review with a locked keyrack | `rhx review ...` | auto-unlocks, then proceeds |
| run reflect with a locked keyrack | `rhx reflect ...` | auto-unlocks (`env: test`), then proceeds |
| peer-review guard fires reviewers | `rhx route.stone.set --as arrived` | unlock happens once, upfront |
| diagnose a truly broken keyrack | read stderr at unlock step | immediate, before brain work |

### contract inputs & outputs

the unlock is driven by the **same `{ owner, env }` the reviewer already passes to
`genContextBrain`** — no new human-facing input. proposed internal contract:

```ts
// .what = ensure the keyrack daemon holds creds for this owner/env before brain work
// .why  = reviewers always need it; unlock upfront, failfast if impossible
// .note = likely implemented by shelling out to `rhx keyrack unlock --owner --env --json`
//         rather than the raw unlockKeyrackKeys() API (which needs a full ContextKeyrack)
await ensureKeyrackUnlocked({ owner: 'ehmpath', env: 'prep' });

const brain = await genContextBrain({
  choice: options.brain,
  creds: { keyrack: { owner: 'ehmpath', env: 'prep' } },
});
```

behavior:
- **already unlocked / creds in env** → no-op (idempotent, CI-friendly), no output
- **success** → proceed; emit at most one line (`unlocked ehmpath/prep`) or stay silent
- **unlock fails** → **forward keyrack's stdout verbatim and propagate its exit code.**
  keyrack already classifies (`exit 2` for locked/absent constraints, `exit 1` for infra
  malfunctions per `rule.require.exit-code-semantics`); we pass its message and code
  through rather than author our own. we only *ensure* the failure surfaces here (early),
  not later disguised as a brain malfunction.

### what it looks like to leverage

```bash
# locked keyrack, no manual unlock needed — quiet success
$ rhx review --rules 'briefs/**/*.md' --paths 'src/**/*.ts'
🗿 route.review
   ├─ unlocked ehmpath/prep
   └─ review complete

# unlock fails — keyrack's own message, forwarded verbatim
$ rhx review --rules 'briefs/**/*.md' --paths 'src/**/*.ts'
🔓 keyrack unlock
   └─ ✋ <keyrack's own locked/absent/prikey message>
# exit code = keyrack's own
```

## mental model

### how users would describe this

> "before a reviewer does anything, it makes sure the keyrack is open for the creds it
> knows it'll need. if it can't open it, it tells me right away — it doesn't pretend
> something broke."

### analogies

- **like a barista turning on the espresso machine before opening** — you know you'll
  need it, so you warm it up first rather than discovering it's cold mid-order.
- **like `ssh-add` before a command that needs the key** — load the credential into the
  agent up front so everything downstream just works.

### terms

| user term | system term |
|-----------|-------------|
| "unlock the keyrack" | send creds to the keyrack daemon for a session |
| "the combo" | `{ owner, env }` (e.g., `ehmpath/prep`) |
| "it just works" | proactive idempotent unlock before brain invocation |
| "real error" | failfast constraint (`exit 2`) at the unlock step |

## evaluation

### how well does it solve the goals?

| goal | coverage |
|------|----------|
| stop wasting time on locked-keyrack first-failures | **yes** — unlock proactively, upfront |
| stop disguising a locked keyrack as a malfunction | **yes** — failfast at unlock step with hint |
| reviewers always have the creds they need | **yes** — unlock keyed off the same `{ owner, env }` |

### pros

- removes the #1 false-failure for reviewers
- failure is honest, early, and actionable (`exit 2`, clear hint)
- no new human-facing input — reuses the `{ owner, env }` already in the contract
- idempotent — safe to call on every invocation

### cons

- adds an unlock step (small latency) on every review/reflect invocation
- couples the reviewer entrypoints to a keyrack-unlock concern
- duplication across `review.ts` and `reflect.ts` (shared helper mitigates)

### edgecases & pit-of-success

| edgecase | result |
|----------|--------|
| creds already in `process.env` (CI/test) | skip unlock entirely — no daemon needed *(env-skip behavior on the live `genContextBrain` path is unverified — see assumptions)* |
| keyrack already unlocked | cheap idempotent no-op — **re-unlock is instant** (confirmed by wisher) |
| ssh prikey absent / daemon down | failfast `exit 2` with unlock hint |
| both review and reflect | use `env: prep` (wisher: **both should use prep**) — reflect's current `env: test` should change to `prep` |
| interactive auth (passphrase/yubikey) | **not required for ehmpath** — keyrack handles auth (wisher-confirmed) |

## open questions & assumptions

### assumptions

1. **[answered by wisher]** unlock for `ehmpath` is non-interactive — **no passphrase
   needed; keyrack handles auth for us.** so auto-unlock is safe in CI and guard
   subprocesses (no blocking prompt). this clears the former top risk.
2. **[verified]** a programmatic unlock exists: `unlockKeyrackKeys({ owner, env }, ctx)`
   (`node_modules/rhachet/dist/domain.operations/keyrack/session/unlockKeyrackKeys.d.ts`),
   plus the CLI `rhx keyrack unlock --owner --env` (`invokeKeyrack.js:719-764`).
3. **[verified + wisher-decided]** reviewers pass `{ keyrack: { owner, env } }` —
   `review.ts:186` (`prep`), `reflect.ts:147` (currently `test`). **wisher: both should use
   `prep`** → reflect's `env: test` should change to `env: prep`, and the unlock targets
   `ehmpath/prep` for both entrypoints.
4. **[answered by wisher]** the CI env-skip holds — **keyrack itself uses `process.env`**.
   so when creds are already in env (CI), the keyrack unlock is a cheap no-op and creds
   resolve fine; auto-unlock-on-every-review will **not** break CI.
5. **[answered by wisher]** re-unlock when already unlocked is **instant** — confirmed by
   wisher. so unconditional auto-unlock on every invocation is fine; no status-check-first
   needed for cost reasons.
6. **[decided — forward keyrack's output]** on failure, **forward keyrack's own stdout and
   propagate its exit code** rather than author our own error block. keyrack already
   classifies per `rule.require.exit-code-semantics` (`exit 2` locked/absent constraints,
   `exit 1` infra malfunctions); pass-through preserves that. on success, stay silent or
   emit one line (`unlocked ehmpath/prep`).

## what is awkward?

- **mutating side effect on a read-path.** invoking a review now *changes* daemon state
  (unlocks creds). it's idempotent, instant (wisher-confirmed), and intended, but it means
  "run a review" quietly also means "unlock my keyrack." acceptable given reviewers always
  need it, but worth naming.
- **two entrypoints, one concern.** `review.ts` and `reflect.ts` both need the same
  unlock-before-brain step (now both `env: prep`) — a shared `ensureKeyrackUnlocked({ owner,
  env })` helper avoids drift, but the duplication of the *call site* remains until
  rhachet itself unlocks as part of `genContextBrain`.

**resolved (were awkward, now settled by wisher):**
- ~~interactive-auth uncertainty~~ → `ehmpath` needs no passphrase; keyrack handles auth.
- ~~"always prep isn't quite true"~~ → wisher says **both** review and reflect use `prep`;
  reflect's `env: test` should change to `prep`. so a hardcoded `ehmpath/prep` is now correct.
