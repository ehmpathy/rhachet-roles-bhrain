# vision: fix driver heap OOM

## the outcome world

### before
- guards with broad artifact patterns (`src/**/*`) crash with OOM
- THREE root causes contribute to memory exhaustion:
  1. `computeStoneReviewInputHash` reads ALL artifact files into memory to compute hash
  2. `stepReview.ts` loads ALL rules, refs, and targets via Promise.all simultaneously
  3. stdout logs every expanded artifact file path (700+ lines of output)
- 700+ files × avg file size = ~4GB heap exhaustion
- reviews never run — crashes before completion
- users must manually narrow artifact patterns or increase heap

### after
- guards work with any artifact pattern, any repo size
- `computeStoneReviewInputHash` uses git blob hashes (40 chars each)
- `stepReview` streams or batches file reads instead of Promise.all
- stdout shows glob patterns, not expanded file lists
- 700+ files = ~50KB of hash strings in memory
- reviews run successfully on first attempt
- users don't think about memory — it just works

### the "aha" moment
when a guard with `src/**/*` artifacts completes in <1s instead of OOM after 5 minutes.

## user experience

### usecases
1. **route driver** runs `rhx route.stone.set --as arrived` on execution stone
2. guard computes artifact hash to check for cached reviews
3. hash computation completes instantly (git blob hashes)
4. reviews run against diff-filtered files
5. judge evaluates, stone passes

### contract inputs/outputs

**current (broken):**
```
getAllStoneArtifacts() → string[] (file paths)
computeStoneReviewInputHash() → reads each file → OOM
stepReview() → Promise.all(files.map(readFile)) → OOM
stdout → logs every file path (700+ lines)
```

**proposed:**
```
getAllStoneArtifacts() → string[] (file paths)
computeStoneReviewInputHash() → git ls-tree → blob hashes → hash string
stepReview() → stream or batch file reads
stdout → logs glob pattern, not expanded paths
```

### timeline
| step | before | after |
|------|--------|-------|
| enumerate artifacts | 100ms | 100ms |
| compute hash | OOM @ 5min | <1s |
| run reviews | never reached | proceeds |

## mental model

### how users describe it
"the guard checks if my code changed since last review"

### analogies
- **git status** — git knows what changed without read of files
- **content-addressable storage** — same content = same hash
- **cache key** — fast to compute, identifies content

### terms
| user term | our term |
|-----------|----------|
| "artifact" | file matched by guard's artifact patterns |
| "hash" | sha256 of combined git blob hashes |
| "cache hit" | prior review exists for this hash |

## evaluation

### how well does it solve the goal?

| criteria | score | notes |
|----------|-------|-------|
| eliminates OOM | excellent | zero file reads |
| same semantics | excellent | content hash still accurate |
| performance | excellent | O(n) hashes vs O(n) file contents |
| compatibility | good | works for tracked files; needs handler for untracked |

### pros
- **zero file reads** — git already has the hashes
- **O(1) memory per file** — 40-char hash vs file content
- **same cache semantics** — same content = same hash
- **scales to any repo** — 10k files still just ~400KB of hashes

### cons
- **untracked files** — need `git hash-object` (still no read into our process)
- **git dependency** — requires git repo (already assumed by route system)
- **binary files** — git blob hash includes them (fine for our purposes)

### edge cases

| case | handler |
|------|---------|
| untracked file | `git hash-object <file>` computes hash without load |
| modified file | `git hash-object <file>` for current content |
| deleted file | absent from `git ls-tree`, skip or error |
| binary file | git blob hash works fine |
| symlink | git tracks symlink target, hash reflects that |

## open questions & assumptions

### assumptions (all verified)
1. **[answered]** all artifact files are in a git repo — true, routes require git by design
2. **[answered]** git blob hash is deterministic — true, SHA-1 of (type + size + content)
3. **[research]** `git ls-tree` is fast for 1000s of files — expected O(n), verify in implementation

### questions triaged

| question | status | resolution |
|----------|--------|------------|
| batch efficiency of ls-tree | [research] | verify with 100-1000 files in implementation |
| untracked files in hash | [answered] | yes, maintain current semantics via hash-object |
| modified file hash source | [answered] | use `git hash-object` not `git ls-tree HEAD` — critical for correct worktree semantics |

### critical discovery from self-review

`git ls-tree HEAD` returns the **committed** blob hash. for modified files, this is WRONG — it would hash the old version. implementation MUST use `git hash-object <file>` for any modified or untracked file to hash the current worktree content.

### external research needed
- none — git commands are well-documented and stable

## groundwork

### external research
none — no external APIs or services; git is the dependency

### internal research

**root cause 1: computeStoneReviewInputHash**
- `src/domain.operations/route/guard/computeStoneReviewInputHash.ts` lines 29-33
- reads all artifact files via `fs.readFile` in a loop
- concatenates content, then hashes

**root cause 2: stepReview Promise.all bulk read**
- `src/domain.operations/review/stepReview.ts` lines 412-439
- `readFileContent()` function called via Promise.all for all rules, refs, and targets
- loads all file contents simultaneously into memory
- fix: stream or batch file reads instead of Promise.all (keep `--mode hard` content behavior)

**root cause 3: stdout artifact expansion**
- stdout logs every expanded artifact file path
- 700+ files = 700+ lines of output
- should show glob pattern instead

**verified git provides blob hashes:**
```bash
$ git ls-tree HEAD src/index.ts
100644 blob 524c9da26ff98f18a1fdd3dba62488e6778c92d3  src/index.ts
```

**verified batch capability:**
```bash
$ git ls-tree -r HEAD -- src/index.ts src/domain.objects/
# returns blob hash for each file
```

### contracts to conform to
- `computeStoneReviewInputHash(input: { stone, route }): Promise<string>` — same signature
- hash must be deterministic for same artifact content

### required test coverage
- **stdout artifact display**: test that glob patterns appear in output, not expanded file lists
- **regression prevention**: snapshot tests to ensure stdout format never regresses

### vocab to reuse
- "artifact" — files matched by guard's artifact patterns
- "hash" — sha256 digest used for cache lookup

## what is awkward?

### feels off
- **untracked files** — the current code reads them, but should they even be artifacts? if they're not tracked, they won't be reviewed. this might reveal a design question: should artifacts only be tracked files?

### tradeoffs
- **git dependency** — acceptable; routes already require git
- **two code paths** — tracked (ls-tree) vs untracked (hash-object); could unify via hash-object for all

### mental model friction
- users might wonder "why does it need git?" — answer: git already computed the hashes, we reuse them
