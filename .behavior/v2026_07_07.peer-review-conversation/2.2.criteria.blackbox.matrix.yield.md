# 2.2 blackbox coverage matrix — peer-review conversation

distills `2.1.criteria.blackbox` into coverage matrices. one matrix per related usecase set. `ind:` = independent (input/condition), `dep:` = dependent (expected outcome).

---

## matrix A — the peer-contemplation gate

covers usecases 1, 2, 4, 5, 8. governs whether a stone may progress.

dimensions:
- **ind: verdict** — does the reviewer have findings? `has-findings` (≥1 blocker or nitpick) vs `clean` (0/0)
- **ind: taken-coverage** — how many current-iteration `.given` have a paired `.taken`? `all` / `some` / `none`
- **ind: taken-freshness** — is the paired `.taken` for the current `(hash, iter)`? `fresh` / `stale`

| ind: verdict | ind: taken-coverage | ind: taken-freshness | dep: progress | dep: reply-prompt shown? |
|--------------|---------------------|----------------------|---------------|--------------------------|
| has-findings | all                 | fresh                | allowed       | no                       |
| has-findings | some                | fresh                | blocked       | yes (names the absent reviewer) |
| has-findings | none                | —                    | blocked       | yes (names all reviewers) |
| has-findings | all                 | stale                | blocked       | yes (stale = treated as absent) |
| clean        | none                | —                    | allowed       | no (clean needs no `.taken`) |
| clean        | all                 | fresh                | allowed       | no (a `.taken` is harmless but not required) |

gaps flagged: none. every `verdict × coverage × freshness` combination that can occur maps to an outcome. `clean × some/stale` is not enumerated because a clean reviewer has no critique to pair against — the freshness/coverage axes collapse for it (documented, not a gap).

---

## matrix B — the absent-file crystal-clear message

covers usecases 3, 10. when a driver signals contemplation/promise before the file exists.

dimensions:
- **ind: review-kind** — `peer` (`.taken.by_self`) vs `self` (self-review articulation)
- **ind: level-varies** — is the target path's level guard-assigned and possibly shifted from a prior attempt? `yes` (self) / `no` (peer path is keyed, not leveled)

| ind: review-kind | ind: level-varies | dep: names exact write-path | dep: shows paired read-path | dep: states why | dep: states "guard assigns level, copy verbatim" |
|------------------|-------------------|-----------------------------|-----------------------------|-----------------|--------------------------------------------------|
| peer             | no                | yes (`.taken.by_self.$slug`) | yes (`.given.by_peer.$slug`) | yes | n/a (peer path is not leveled) |
| self             | yes               | yes (current-level path)     | n/a (self-review has no paired peer file) | yes | yes |

gaps flagged: none. the read-path column is n/a for self because a self-review contemplates the artifact, not a peer critique; the level column is n/a for peer because the peer path is keyed by `(iter, hash, idx)`, not by a guard-assigned level.

---

## matrix C — conversation visibility (opt-in)

covers usecases 6, 7. whether a reviewer sees the prior exchange.

dimensions:
- **ind: conversation-flag** — declared on the reviewer? `opt-in` / `omitted`
- **ind: prior-exchange** — do prior `.given` + `.taken` exist for the stone? `present` / `absent`
- **ind: fix-in-artifact** — is the driver's claimed fix actually present? `present` / `absent` (only meaningful when the reviewer can see the exchange)

| ind: conversation-flag | ind: prior-exchange | ind: fix-in-artifact | dep: reviewer receives context | dep: prior blocker verdict |
|------------------------|---------------------|----------------------|-------------------------------|----------------------------|
| opt-in                 | present             | present              | yes                           | dropped (fix confirmed)    |
| opt-in                 | present             | absent               | yes                           | re-raised (fix not found)  |
| opt-in                 | absent (first round)| —                    | yes (just the current `.given`, or empty) | fresh critique, no prior to weigh |
| omitted                | present             | present              | no                            | unchanged from today's blind behavior |
| omitted                | present             | absent               | no                            | unchanged from today's blind behavior |

gaps flagged: none. `fix-in-artifact` is only a live axis when `conversation-flag = opt-in` and `prior-exchange = present`; for the omitted rows the reviewer cannot weigh a fix, so the axis collapses (documented).

---

## matrix D — the reply-prompt render surfaces

covers usecases 2, 11. the halt message must be identical wherever it appears.

dimensions:
- **ind: surface** — where the driver meets the halt: `stophook` / `--as arrived` / `--as passed`

| ind: surface   | dep: reply-prompt shown? | dep: content identical to other surfaces? |
|----------------|--------------------------|-------------------------------------------|
| stophook       | yes                      | yes                                       |
| --as arrived   | yes                      | yes                                       |
| --as passed    | yes                      | yes                                       |

gaps flagged: none. single independent dimension with three values, all mapped. the `identical` column is the crux — it forces a single shared formatter, not three drifted copies.

---

## matrix E — the legible transcript (human read)

covers usecase 9.

dimensions:
- **ind: rounds** — how many peer-review rounds the stone went through? `one` / `many`

| ind: rounds | dep: each `.given` sits beside its paired `.taken` | dep: readable as dialogue |
|-------------|----------------------------------------------------|---------------------------|
| one         | yes                                                | yes                       |
| many        | yes (each round's pair matched by shared key)      | yes                       |

gaps flagged: none.

---

## decomposition signal

no matrix exceeds 3 independent dimensions. matrix A (the gate) is the widest at 3, and it stays legible because the third axis (freshness) collapses for the `clean` verdict. no matrix explodes; no usecase bundles unrelated concerns. the five matrices map cleanly to five separable behavioral boundaries:

- A: the gate (progress vs block)
- B: the absent-file guidance
- C: the opt-in visibility
- D: the render consistency
- E: the human legibility

this separation is a healthy sign — each boundary can be built, tested, and recomposed on its own.
