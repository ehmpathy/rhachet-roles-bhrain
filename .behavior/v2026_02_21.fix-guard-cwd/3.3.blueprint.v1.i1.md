# blueprint: fix $route variable in guard judge commands

## summary

remove the `cwd: input.route` override from `runOneStoneGuardJudge` so that judge commands execute from the repo root. this makes `$route` work intuitively — the path is relative to where the user invoked the command.

## the fix

### root cause

in `runOneStoneGuardJudge.ts`:
- line 55 sets `cwd: input.route`
- line 45 substitutes `$route` with `input.route`

when `input.route = '.behavior/v2026_02_21.my-feature'`:
1. command runs from `.behavior/v2026_02_21.my-feature`
2. `$route` = `.behavior/v2026_02_21.my-feature`
3. the judge looks for stones at the doubled path

### solution

remove `cwd: input.route` from the `execAsync` call. commands will run from `process.cwd()` (repo root) where `$route` paths resolve correctly.

## migration notes

### guards that used `--route .` workaround

these guards will break because `--route .` from repo root = repo root, not route directory.

**before (workaround)**:
```yaml
judges:
  - npx rhx route.stone.judge --mechanism approved? --stone $stone --route .
```

**after (correct)**:
```yaml
judges:
  - npx rhx route.stone.judge --mechanism approved? --stone $stone --route $route
```

### affected guards in this repo

all guards in this repo use `$route` already, so no migration needed. the fix makes them work correctly.
