# vision: route.stone.set --as rewound

## the outcome world

### before: stuck in validation limbo

a human delegates verification to an agent. the agent runs self-reviews, discovers it needs human input, and emits a handoff. the human completes their task.

now what?

the agent's reviews are stale — they assumed the old state. the judges locked in verdicts based on pre-handoff conditions. the passage system sees "already reviewed" and wants to reuse cached results.

the human has two bad options:
1. manually delete files in `.route/` (error-prone, violates encapsulation)
2. blow away the whole route and start from stone 1 (wasteful)

### after: clean slate with one command

```sh
rhx route.stone.set --stone 5.3.verification --as rewound
```

all cached reviews invalidated. all judges cleared. all promises reset. time enforcement markers removed.

the stone is back at its entrypoint — ready for fresh evaluation with the new reality.

### the "aha" moment

> "oh — i don't have to fight the system. rewinding respects the work i did while giving me a clean slate for what changed."

the value clicks when you realize: rewound doesn't discard your artifact. it discards the *validation state* that was based on the old artifact.

## user experience

### primary usecase: post-handoff restart

**goal**: human completed handoff action, wants agent to re-verify with fresh eyes

**timeline**:
1. agent runs stone, hits blocker requiring human action
2. agent emits handoff: "human, please do X"
3. human does X
4. human runs: `rhx route.stone.set --stone <stone> --as rewound`
5. agent runs stone again with fresh reviews/judges

**contract**:
```
input:
  --stone <pattern>   # fuzzy match stone name
  --as rewound        # the rewind action

output:
  - for specified stone AND all subsequent stones:
    - all *.guard.review.* files: deleted
    - all *.guard.judge.* files: deleted
    - all *.guard.promise.* files: deleted
    - all *.guard.selfreview.*.triggered.* files: deleted
  - passage.jsonl: appends {stone, status: 'rewound'} for each (no timestamp)
    - note: prior approvals are NOT deleted — they remain in the log
    - but they become INVALID: route.drive treats any approval that precedes
      a rewind (for same or earlier stone) as stale
  - stdout: confirmation tree showing stones rewound and what was cleared
```

### secondary usecase: fix-and-retry

**goal**: human realizes the artifact was wrong, fixes it, wants fresh validation

**timeline**:
1. stone passes with flawed artifact (reviews missed something)
2. human spots the flaw, fixes artifact content
3. human runs: `rhx route.stone.set --stone <stone> --as rewound`
4. stone re-runs with proper validation of fixed artifact

### tertiary usecase: development iteration

**goal**: developer iterating on a stone's guard rules, wants to test fresh

**timeline**:
1. developer tweaks review prompts in guard file
2. developer runs: `rhx route.stone.set --stone <stone> --as rewound`
3. stone re-evaluates with new review logic

## mental model

### how users describe it

> "rewind puts me back at the starting line of the stone. all the validation checkpoints are cleared, but my actual work (the artifact) stays."

### analogies

| analogy | mapping |
|---------|---------|
| git reset --soft | keeps changes (artifact), clears commits (reviews) |
| video game checkpoint reload | back to save point, progress after that wiped |
| exam retake | same material, fresh grading |
| terraform destroy + apply | not quite — rewound only clears validation, not artifact |

### terms mapping

| user says | system term |
|-----------|-------------|
| "start over" | rewound |
| "clear the reviews" | invalidate guard artifacts |
| "fresh slate" | reset validation state |
| "retry from scratch" | rewind + re-run stone |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| enable clean restart after handoff | excellent | direct solution |
| preserve artifact work | excellent | only clears validation |
| simple mental model | good | one command, clear semantics |
| safe by default | excellent | only clears derived state |

### pros

- **surgical**: only removes validation state, not the work itself
- **auditable**: passage.jsonl records the rewind event
- **idempotent**: rewinding an already-rewound stone is a no-op
- **composable**: works with current route.drive flow

### cons

- **no partial rewind**: clears all validation, can't preserve specific reviews
- **no undo**: once rewound, must re-run full validation

### edgecases & pit of success

| edgecase | behavior | pit of success |
|----------|----------|----------------|
| stone doesn't exist | error with helpful message | fail-fast, no silent failure |
| stone already rewound | no-op, still logs event | idempotent, safe to retry |
| stone never had validation | no-op | graceful degradation |
| mid-review rewind | clears partial state | consistent end state |
| multiple stones match pattern | error, ask for specificity | prevents accidental bulk rewind |

## open questions & assumptions

### assumptions

1. **rewound clears all validation types** — reviews, judges, promises, triggers
2. **rewound doesn't touch the artifact** — only derived validation state
3. ~~**rewound is human-only**~~ — agents CAN rewind (safer option)
4. **passage.jsonl gets a rewound entry** — for audit trail

### questions for wisher

1. ~~**scope: stone-level or route-level?**~~ **RESOLVED: CASCADE FROM STONE**
   - wisher confirmed: affects the specified `--stone` AND all subsequent stones
   - rewind 3.blueprint → also rewinds 4.roadmap, 5.execution, etc.
   - rationale: downstream stones were validated while upstream was valid

2. ~~**should agents be able to rewind?**~~ **RESOLVED: YES**
   - wisher confirmed: agents CAN rewind — it's a safer option
   - if a robot wants to redo a task, that's fine

3. ~~**should rewound clear approval status?**~~ **RESOLVED: INVALIDATED BY LOG ORDER**
   - wisher confirmed: approvals stay in passage.jsonl (append-only log)
   - but they become INVALID when a rewind entry for an earlier stone appears after them
   - invalidation-by-order: log sequence determines validity

4. ~~**should there be a --force for bulk rewind?**~~ **RESOLVED: MOOT**
   - cascade is the default behavior (rewind stone N → also rewinds N+1, N+2, etc.)
   - no separate flag needed

5. ~~**is the secondary usecase (fix-and-retry) valid?**~~ **RESOLVED: YES**
   - verified in `computeNextStones.ts`: route.drive SKIPS stones with passage markers
   - even if artifact hash changes (fresh reviews would run), the passage marker prevents re-evaluation
   - rewound clears passage status so stone can be re-evaluated

## what is awkward?

### ~~approval interaction~~ RESOLVED

~~if a stone was approved by a human, rewinding it clears that approval.~~

wisher confirmed: approvals are KEPT in passage.jsonl but INVALIDATED by a rewind to an earlier stone.

mechanism: passage.jsonl is append-only. when route.drive reads the log, it scans for `{status: 'rewound'}` entries. any approval that precedes a rewind (for the same stone or an earlier one) is treated as stale. the log order determines validity — no deletion required.

### naming tension

"rewound" implies going backward in time. but we're not restoring old state — we're *clearing* state to enable fresh forward progress.

alternatives considered:
- `--as reset` — too generic, conflicts with git mental model
- `--as invalidated` — accurate but wordy
- `--as cleared` — doesn't convey the "start over" intent
- `--as fresh` — adjective feels wrong for `--as` pattern

**decision**: "rewound" is best available. the metaphor is "rewind the tape to the start of this track" — you'll play forward again with fresh recording.

### partial rewind desire

sometimes you want to keep peer reviews but clear self-reviews (or vice versa). current design doesn't support this.

**resolution**: keep it simple for v1. if partial rewind proves necessary, add `--only reviews|judges|promises|triggers` later.
