# blackbox criteria: enbrief route creation

## usecase.1 = learn route creation basics

```
given(driver wants to create a thought route)
  when(driver reads the brief)
    then(brief explains what a stone is)
      sothat(driver understands stones are milestones on a route)
    then(brief explains what a guard is)
      sothat(driver understands guards are optional validation gates)
    then(brief explains the `.route/v$isodate.$slug/` convention)
      sothat(driver knows where to create route directories)
    then(brief explains `.$variant/` for special routes)
      sothat(driver knows they can use `.behavior/` or custom variants)
```

## usecase.2 = create a route directory

```
given(driver understands route creation)
  when(driver creates `.route/v$isodate.$slug/` directory)
    then(directory is created)
      sothat(route has a home)
  when(driver creates `0.wish.md`)
    then(wish document captures the goal)
      sothat(context is preserved for future reference)
  when(driver creates `N.name.stone` file)
    then(stone is recognized by route.drive)
      sothat(driver can navigate the route)
```

## usecase.3 = define stone content

```
given(driver wants to add a stone)
  when(driver creates `1.vision.stone`)
    then(stone file contains markdown instructions)
      sothat(the work to be done is clear)
  when(driver creates `2.research.stone`)
    then(stone file contains markdown instructions)
      sothat(the work to be done is clear)
  when(driver creates stone with numeric prefix)
    then(stones are ordered by prefix)
      sothat(route.drive navigates in order)
```

## usecase.4 = add validation guards

```
given(driver wants to require validation before passage)
  when(driver creates `N.name.guard` matched to `N.name.stone`)
    then(guard is recognized by route.stone.set)
      sothat(passage requires guard validation)
  when(guard contains `judges:` section)
    then(judges are invoked on passage attempt)
      sothat(automated validation runs)
  when(guard contains `reviews.self:` section)
    then(self-review prompts are presented)
      sothat(driver reflects before passage)
  when(guard requires `approved?` judge)
    then(human approval is required)
      sothat(human reviews the work before passage)
```

## usecase.5 = bind route to branch

```
given(driver has created a route)
  when(driver runs `rhx route.bind.set --route .route/v$isodate.$slug`)
    then(route is bound to current branch)
      sothat(route.drive knows which route to navigate)
  when(driver runs `rhx route.bind.get`)
    then(current bound route is displayed)
      sothat(driver knows which route is active)
  when(driver runs `rhx route.bind.del`)
    then(route bind is removed)
      sothat(driver can unbind from a route)
```

## usecase.6 = walk the route

```
given(driver has bound a route)
  when(driver invokes route.drive)
    then(next stone is displayed)
      sothat(driver knows what work to do)
  when(driver completes the stone's work)
    then(driver can mark stone as passed via `rhx route.stone.set --stone N.name --as passed`)
      sothat(progress is recorded)
  when(stone has a guard)
    then(guard is validated before passage)
      sothat(quality gates are enforced)
  when(all stones are passed)
    then(route is complete)
      sothat(driver knows the journey is done)
```

## usecase.7 = parallel stones (advanced)

```
given(driver wants multiple stones to run in parallel)
  when(driver creates stones with same numeric prefix: `3.1.a.stone`, `3.1.b.stone`)
    then(stones with same prefix can be worked in parallel)
      sothat(independent work streams proceed concurrently)
  when(one parallel stone is passed)
    then(other parallel stones remain available)
      sothat(each parallel track is independent)
  when(all parallel stones in a prefix group are passed)
    then(route.drive advances to next prefix)
      sothat(progression respects dependencies)
```

## usecase.8 = nested routes (advanced)

```
given(driver wants to reference another route from within a stone)
  when(stone instructions reference a sub-route)
    then(driver can bind to the sub-route)
      sothat(complex journeys can be decomposed)
  when(sub-route is completed)
    then(driver returns to parent route)
      sothat(progression continues)
```

## error cases

```
given(driver makes a mistake)
  when(driver creates guard without matched stone)
    then(guard is ignored or error is raised)
      sothat(orphan guards don't cause confusion)
  when(driver creates malformed guard yaml)
    then(parse error is surfaced)
      sothat(driver can fix the syntax)
  when(driver tries to pass stone without bound route)
    then(error indicates no route is bound)
      sothat(driver knows to bind first)
  when(driver tries to pass stone that doesn't exist)
    then(error indicates stone not found)
      sothat(driver knows to create the stone first)
```
