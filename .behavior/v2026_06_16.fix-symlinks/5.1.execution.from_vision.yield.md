## defect 2: blocker count inconsistency

### todos

- [x] investigate: find why r1-r6 all reference r6.md in second tree
- [x] fix: correct the review file reference logic

### findings

**root cause IDENTIFIED**: reviewers had duplicate slugs (all were `$rhx`), which caused Map key collisions.

**fix applied — slug standardization at parse time**:

1. added `standardizePeerReviewSlugs()` to `parseStoneGuard.ts`:
   - counts slug occurrences
   - adds `.N` suffix to duplicates (e.g., `$rhx.1`, `$rhx.2`)
   - leaves unique slugs unchanged

2. reverted to slug-based lookup in `formatGuardTree.ts` and `getAllRouteStoneGuardReviewPeerMeters.ts`:
   - slugs are now guaranteed unique at parse time
   - slug-based Maps work correctly without collisions

**why slug standardization > index-based lookup**:
- single source of truth — uniqueness enforced at source (parse time)
- all downstream code can rely on unique slugs
- no workarounds or special handle needed anywhere

- `src/domain.operations/route/guard/parseStoneGuard.ts` — added `standardizePeerReviewSlugs()` function
- `src/domain.operations/route/guard/parseStoneGuard.test.ts` — added case6 for slug standardization
