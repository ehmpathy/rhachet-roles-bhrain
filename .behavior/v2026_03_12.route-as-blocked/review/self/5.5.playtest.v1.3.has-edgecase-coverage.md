# self-review: has-edgecase-coverage

## question

are edge cases covered?

## findings

### what could go wrong?

| scenario | covered in playtest? |
|----------|---------------------|
| robot tries to block without articulation | yes - case1 shows nudge |
| robot creates articulation without trigger | yes - edge1 tests this |
| robot blocks then immediately unblocks | yes - case4 tests recovery |
| multiple block attempts on same stone | implicitly covered by flow |

### unusual but valid inputs

| input | coverage |
|-------|----------|
| stone name with special chars | not covered - standard name convention assumed |
| very long articulation content | not covered - not a likely failure mode |
| empty articulation file | not covered - system should accept any content |

### boundary tests

| boundary | coverage |
|----------|----------|
| first attempt (no trigger file) | case1 |
| second attempt (with trigger + articulation) | case2 |
| guard-initiated vs agent-initiated blocks | edge2 (delegated to acceptance tests) |
| blockers count = 5 (exact threshold) | delegated to acceptance tests |
| blockers count = 6 (above threshold) | delegated to acceptance tests |

### edge cases in playtest

1. **edge1**: articulation without trigger
   - tests that robot must see nudge first
   - ensures two-step flow enforced

2. **edge2**: guard-initiated blocks
   - documented as covered by acceptance tests
   - verifies distinction from agent-initiated blocks

## verdict

key edge cases are covered. unusual inputs (special chars, empty files) are not critical failure modes and are not tested.
