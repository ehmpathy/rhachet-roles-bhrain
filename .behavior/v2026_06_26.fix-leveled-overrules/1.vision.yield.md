# 1.vision — fix leveled overrules

> illustrate the world where `overruled` and `forced` respect leveled peer review.

---

## the core insight

today, `overruled` is **stone-wide and level-blind**. when a human overrules, the
`reviewed?` judge passes the *entire* stone — no matter how many review levels remain
unrun (route.ts:1139-1148).

this collides with the leveled model. leveled guards run cheap reviewers first (l1),
and only unlock expensive reviewers (l3) once every lower level reaches a **terminal**
verdict — `approved | exhausted | malfunction | constraint`
(isReviewPeerLevelTerminal.ts:22-28, runStoneGuardReviews.ts:507-538).

> .note = the terminal set was widened by main's #288 (fix-driver-overruled-even-error):
> a *broken* (malfunction) or *genuinely-constrained* (exit 2, blockers=0) reviewer is now
> terminal, so it auto-escalates without overrule. but a reviewer that ran fine and simply
> **rejects** (exit 2 WITH blockers) stays deliberately NON-terminal
> (computeReviewPeerVerdict.ts:51-52). the perma-blocker our wish targets is exactly this
> rejection case — the one slice #288 leaves unsolved.

so when a small-brained l1 reviewer parks a **perma-blocker** — a rejection it will never
retract — the level never turns terminal, l3 never unlocks, and the human's only escape
hatch (`overruled`) skips *all* review, even the l3 review they actually wanted.

the wish: make overrule **level-scoped**. overrule the level that is *stuck*, let its
blockers be forgiven, mark it terminal — and let the next level finally run.

## the outcome world

### day-in-the-life

a driver sits on stone `5.1.execution`. the guard has three review levels:

- **l1** — `mech-failhides` (small, cheap, fast)
- **l1** — `arch-hazards`
- **l3** — `behavior-intent-coverage` (large, expensive, slow)

the driver runs `--as passed`. l1 reviewers run. `mech-failhides` parks the same blocker
every round, even after the driver fixes all it can. it burns its budget on a false
positive. l3 has **never run** — it sits queued behind l1.

the human reads the l1 review, agrees it is a false positive, and runs:

```
rhx route.stone.set --stone 5.1.execution --as overruled
```

the system replies:

```
🦉 route.stone.set
   └─ ✓ overruled
      ├─ level 1, overruled
      └─ level 3, ready
```

the driver runs `--as passed` again. this time:

- l1 reviewers run as normal
- l1 still has blockers — but they are **overruled**, so l1 turns terminal at once
- l3 **finally unlocks and runs**
- l3 approves → stone passes

the human overruled *only the stuck level*. the expensive, valuable l3 review still ran.

### before / after

| | before | after |
|---|---|---|
| overrule l1 (l3 unrun) | whole stone passes; **l3 never runs** | l1 forgiven; **l3 runs**; l3 verdict still gates passage |
| force at l1 (l3 unrun) | approves all at once | overrules l1 only; **approval withheld** until terminal level |
| perma-blocker at l1 | stuck forever, or skip all review | overrule the one stuck level, keep the rest |
| single-level guard | overrule passes stone | overrule passes stone (**unchanged**) |

### the "aha" moment

> "overrule is not a bypass of *review* — it is a bypass of *one stubborn level*. the
> levels above still get their say."

the value clicks when the human sees they no longer face a lose-lose: no need to choose
between *stuck forever* and *skip the review I actually trust*.

## user experience

### usecases

| who | goal |
|---|---|
| human | unstick an l1 perma-blocker without loss of l3's review |
| human | push a stone through, but approve only once the final level has been seen |
| driver | after a human overrules, proceed and let higher levels run |

### contract — inputs

unchanged surface, new semantics:

```
rhx route.stone.set --stone <stone> --as overruled    # human only
rhx route.stone.set --stone <stone> --as forced        # human only
```

- **`overruled`** — overrules the **current active level** (the lowest non-terminal
  level). does not touch higher levels.
- **`forced`** — overrules the current active level, **and** grants approval **only if**
  the active level is the terminal (highest) level.

### contract — outputs

`overruled` names the level it scoped to, and the level that becomes ready as a
result (when a higher level unlocks):

```
🦉 route.stone.set
   └─ ✓ overruled
      ├─ level 1, overruled
      └─ level 3, ready
```

`forced` distinguishes the two cases:

```
# active level is NOT terminal — overrule only, no approval line at all
🦉 route.stone.set
   └─ ✓ forced
      └─ overruled = ✓ (level 1)

# active level IS terminal — overrule + approve
🦉 route.stone.set
   └─ ✓ forced
      ├─ overruled = ✓ (level 3)
      └─ approved  = ✓
```

> when approval is withheld (active level is not terminal), the output simply omits the
> approval line. there is no "withheld" placeholder — its absence is the signal.

### timeline — leveled overrule

```
t0  driver  --as passed         → l1 runs, perma-blocks; l3 queued
t1  human   --as overruled      → overrule stamped @ level 1
t2  driver  --as passed         → l1 runs, blockers forgiven, l1 terminal
                                  → l3 unlocks, runs, approves
                                  → reviewed? passes → stone passes
```

### timeline — leveled force (two stops)

```
t0  driver  --as passed         → l1 perma-blocks; l3 queued
t1  human   --as forced         → overrule @ l1; approval WITHHELD (l3 unseen)
t2  driver  --as passed         → l1 forgiven; l3 runs; l3 blocks too
t3  human   --as forced         → overrule @ l3; l3 is terminal level → APPROVE
t4  driver  --as passed         → all levels terminal, approved → stone passes
```

## mental model

### how a human would describe it to a friend

> "the reviewers come in tiers. cheap ones first, fancy ones last. sometimes a cheap one
> snags on something dumb and blocks the whole line. overrule lets me wave *that tier*
> through so the line advances — but the fancy reviewer at the top still gets to look.
> force is the same, except it only stamps final approval once the top tier has had its
> turn."

### analogies

- **airport security lanes** — overrule waves one stalled lane through; the gate check
  at the end still happens.
- **a relay race** — overrule hands the baton past a fallen runner; the anchor leg still
  runs.

### terms

| we say | they might say |
|---|---|
| active level | the tier that is current / stuck |
| terminal level | the last / highest tier |
| level terminal | that tier is done (approved, exhausted, or overruled) |
| overrule | wave this tier through |
| forced (withheld) | wave through but do not sign off yet |

## evaluation

### how well it solves the goals

| goal | solved? |
|---|---|
| overrule scoped to active level | ✓ overrule stamps + forgives only the active level |
| after overrule, l1 runs then turns terminal so l3 runs | ✓ level-unlock treats overruled level as terminal mid-pass |
| forced withholds approval until terminal level | ✓ approval gated on `activeLevel === terminalLevel` |

### pros

- **no lose-lose** — human keeps the review they trust while they unstick the one they do not.
- **backward compatible** — single-level guards behave exactly as today (active == terminal).
- **honest approval** — `forced` no longer signs off on design the top level never saw.
- **small surface** — same two verbs; semantics deepen, CLI shape unchanged.

### cons

- **multi-stop force** — a force on a 3-level stuck stone may need the human to force more
  than once (once per stuck level). this is intentional (each level deserves a look) but
  is more steps than today's one-shot force.
- **more state** — overrule markers now carry a level; passage.jsonl rows grow a field.

### edgecases & pit-of-success

| edgecase | behavior |
|---|---|
| single-level guard | active level == terminal level → overrule passes, force approves (today's behavior) |
| preemptive overrule (no reviews run yet) | active level = lowest = l1 → overrule scopes to l1 only |
| overrule l1, l3 then **approves cleanly** | stone passes; no second overrule needed |
| overrule l1, l3 then **blocks** | reviewed? still fails on l3 → human must overrule l3 (or driver fixes) — l3 is *not* auto-forgiven |
| rewind after overrule | rewind clears overrule markers (cross-stone invalidation already handled, getOneStoneGuardOverrule.ts:9-11) |
| `forced` at terminal level | overrule + approve in one shot |

the pit of success: a human **cannot** accidentally approve unseen levels. approval is
mechanically withheld until the top level is the active level.

## open questions & assumptions

### assumptions made

1. **"active level" = lowest non-terminal level.** the base terminal set (post-#288) is
   `approved | exhausted | malfunction | constraint`; this wish adds `overruled` to it for
   the active level. computed from peer meter statuses at the moment overrule/force runs.
2. **overrule marker carries its level.** `PassageReport` gains an optional `level` field
   (PassageReport.ts:14-45 currently has no level). an overrule with `level: N` forgives
   blockers and forces terminal *only* for reviewers at level N.
3. **overrule forgives within the same pass.** the level-unlock loop in
   runStoneGuardReviews.ts:507-538 must consult overruled levels so l3 unlocks in the
   *same* `--as passed` call that re-runs l1.
4. **terminal level = highest `level` among the guard's peer reviewers.**
5. **base terminal semantics from #288 are preserved.** malfunction and constraint stay
   auto-terminal (no overrule needed to escalate past a broken reviewer); a plain rejection
   (exit 2 with blockers) stays non-terminal. this wish only adds the *human-driven*
   `overruled` path on top — it does not narrow or revert #288's error-as-terminal rule.

### open questions — resolved

all three wisher questions were answered by vlad on 2026-06-28. recorded here as settled
contract decisions.

1. **[resolved] overrule is sticky — survives until rewind.** when l1 is overruled and the
   driver later edits the artifact (hash changes), the l1 overrule still holds. it is keyed
   to the level, not the hash, and only a rewind clears it. matches today's marker
   semantics (getOneStoneGuardOverrule.ts matches stone+status, hash-independent).
2. **[resolved] forced omits the approval line when approval is withheld.** when the active
   level is not the terminal level, `forced` overrules the active level and the output
   simply does not print an approval line. there is no `withheld` / `deferred` placeholder
   — the absence of the line is the signal.
3. **[resolved] a non-active level cannot be overruled.** an attempt to overrule a level
   that is not the current active level is rejected with guidance (e.g. "level 3 is not
   active yet"). overrule applies only to the active (lowest non-terminal) level; no
   preemptive or stacked overrules of future levels.

### intersection with main's #288 (must reconcile at blueprint/execution)

main's #288 (`fix-driver-overruled-even-error`) and this wish touch the same terminal/
overrule logic. no git conflict (it changed `src/`, we added `.behavior/` + briefs), but
two semantic seams must be honored so we extend #288 rather than regress it:

1. **preserve error-as-terminal escalation.** malfunction/constraint stay auto-terminal;
   a *broken* l1 must still let l3 run without any overrule. our level-scoped overrule is
   additive — it handles the *rejection* perma-blocker #288 left non-terminal.
2. **overrule scope diverges for multi-level guards.** #288's overrule is a stone-wide
   bypass (overrule → whole stone passes, `route.ts:1139-1148`). our wish narrows it: an
   overrule at a *non-terminal* level unlocks the next level instead of passing the stone;
   only an overrule at the *terminal* level passes the stone. the execution stone must
   update #288's multi-level overrule snapshots accordingly, while leaving the single-level
   and terminal-level cases behaving as #288 shipped.

## what is awkward

1. **multi-stop force feels heavier than one-shot force.** a human used to "force = done"
   now may force twice on a deeply-stuck stone. it is *correct* (each level earns a look),
   but it fights the muscle memory of force-as-escape-hatch. there is a second-order
   awkwardness too: when approval is withheld, the output *omits* the approval line rather
   than a printed note. the absence is the signal — quiet and clean, but a human who
   expects an explicit "withheld" mark may miss that approval did not happen. mitigation:
   the `overruled = ✓ (level N)` line makes clear only the level was waved, and the next
   `--as passed` re-halts on approval, so the gap surfaces fast.

2. **"active level" is implicit state.** the human runs `--as overruled` without a named
   level, and the system infers it. that is ergonomic, but it means the same command does
   different things across review progress. mitigation: always echo the level it scoped to
   (`overruled → level 1, overruled`) so there is no silent surprise (rule.forbid.surprises).

3. **overrule that forgives "any blockers" within a pass.** the wish states after
   overrule, l1 runs and *any* blockers are forgiven and the level turns terminal. this
   means an overruled level's review result is computed but its blocker count is
   deliberately ignored for unlock purposes. the tension: the review still *runs and costs
   budget*, yet its verdict cannot block. that is the point — but it warrants a comment at
   the cause so future readers do not "fix" it.

4. **forgiveness vs. the final judge.** an overruled level's blockers must be forgiven for
   *both* level-unlock (so l3 runs) *and* the final `reviewed?` total (so the stone can
   pass). a forgiveness in one place but not the other leaves l3 to run while the stone
   still cannot pass. the two forgiveness points must stay in sync — a single source of
   truth for "is this level overruled" is the safe design.
