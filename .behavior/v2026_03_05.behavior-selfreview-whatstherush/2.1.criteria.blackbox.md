# blackbox criteria: self-review rush detection

## usecase.1 = first challenge

```
given a clone runs `set --as promised` for a stone with self-reviews
  when the review is challenged (first time for this stone)
    then the clone sees the `🗿 patience, friend` prompt
      sothat they know to slow down and reflect
    then the clone sees `be here now` with identity reframe
      sothat they adopt reviewer role, not author role
    then the clone sees `trust the way` with implementation intentions
      sothat they have specific when-then cues for behavior
    then the clone sees `articulate into` with file path
      sothat they know where to write their articulation
    then the clone sees the `🌕 lets reflect` section with review guide
      sothat they have the full review instructions
```

## usecase.2 = rapid second attempt

```
given a clone was challenged on a self-review
  when the clone runs `set --as promised` again rapidly (before sufficient reflection)
    then the clone sees `🍂 what is the rush?` first
      sothat they feel the disappointment, not anger
    then the clone sees `...` on its own line
      sothat silence lets the message sink in
    then the clone sees zen reflections on haste
      sothat they understand the cost of rush
    then the clone sees the normal `🗿 patience, friend` prompt
      sothat the original review guidance follows
    then the clone sees the `🌕 lets reflect` section
      sothat they still have the review guide
```

## usecase.3 = rapid third+ attempt

```
given a clone has been confronted with "what is the rush?" already
  when the clone runs `set --as promised` again rapidly
    then the clone sees the same `🍂 what is the rush?` confrontation
      sothat the message remains consistent
    then the clone sees the normal `🗿 patience, friend` prompt
      sothat they are reminded to be patient
```

## usecase.4 = clone passes review

```
given a clone has been confronted with "what is the rush?"
  when the clone waits and passes the self-review
    then the rush detection state is cleared
      sothat the next challenge sequence starts fresh
    then the next stone's self-review uses normal flow
      sothat past impatience doesn't carry forward
```

## usecase.5 = first challenge after prior pass

```
given a clone previously passed a stone with rush confrontation
  when they encounter a new stone's first self-review challenge
    then they see the normal `🗿 patience, friend` prompt only
      sothat the rush confrontation only appears on repeated rapid attempts
```

## usecase.6 = articulation requirement

```
given a clone sees the patience prompt with articulation requirement
  when they write their articulation to {route}/review/self/{stone}.{slug}.md
  when they run `set --as promised --that {slug}`
    then the system validates the articulation file was created
      sothat they cannot bypass the reflection requirement
```

## edgecase.1 = multiple behaviors

```
given a clone works on behavior A and behavior B
  when they get challenged on behavior A's self-review
  when they get challenged on behavior B's self-review separately
    then rush detection is tracked per-behavior
      sothat impatience on one doesn't affect the other
```

## edgecase.2 = file deleted mid-session

```
given the rush files are accidentally deleted
  when the clone runs `set --as promised`
    then the system treats it as first challenge
      sothat there is graceful recovery
    then normal `🗿 patience, friend` appears
      sothat behavior degrades to safe default
```

---

> the experience must confront without punish, disappoint without anger 🦉🌙🍵
