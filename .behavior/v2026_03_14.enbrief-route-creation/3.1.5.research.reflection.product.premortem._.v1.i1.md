# premortem: enbrief route creation

## premortem: imagine we shipped and it failed

we shipped the brief, and it failed badly. why?

### failure scenario 1: brief drifts from reality

the brief documents commands and conventions that don't match actual behavior. users follow the brief, get errors, lose trust.

**trigger**: rhachet updates change the CLI syntax or route conventions; brief is not updated.

### failure scenario 2: brief is never found

users want to create routes but don't know the brief exists. it's `ref` mode (not auto-load), so it never enters their context.

**trigger**: no documentation mentions the brief; brains don't reference it; users stumble through trial-and-error anyway.

### failure scenario 3: brief is too abstract

the brief explains concepts but doesn't provide concrete examples. users read it, understand theory, but can't execute.

**trigger**: we prioritize completeness over practicality; examples are absent or broken.

---

## what obvious signs did we ignore?

- **no tests**: briefs are untested; we rely on human playtest which is infrequent
- **no version sync**: when conventions change, brief has no mechanism to stay in sync
- **ref mode risk**: `ref` briefs only load when requested; discoverability depends on other documentation

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| users will find the brief via brain context | if brains don't reference it, users won't find it | medium |
| CLI syntax is stable | if `rhx route.bind.set` changes, brief breaks | low (cli stable) |
| boot.yml pattern from ehmpathy works here | if driver role boot differs, code may fail | low (pattern verified) |
| ref mode is sufficient | if users need this frequently, say mode is better | medium |
| examples are optional | if users need examples to succeed, brief fails | medium |

---

## what is the nightmare scenario?

**nightmare**: the brief is published, users rely on it, then a major rhachet update changes route conventions. the brief is not updated. users follow the brief, create malformed routes, waste hours debug. they lose trust in documentation and rhachet.

- **regret**: publish documentation that causes more confusion than it solves
- **damage**: user trust erosion; support burden increases
- **recovery**: difficult — bad docs taint perception even after fix

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| brief drifts from reality | add version marker; link to CLI --help as source of truth | low |
| brief is never found | mention in driver readme.md; add to onboard docs | low |
| brief is too abstract | include minimal copy-paste example | low |
| no automated validation | include checksum or last-verified date | low |
| ref mode discoverability | consider future `say` mode for core concepts | medium (requires decision) |

---

## conclusions

primary risks are documentation drift and discoverability. mitigations are low-cost:
1. include concrete examples
2. link to CLI `--help` as authoritative reference
3. mention brief in readme.md and other onboard paths
4. version marker for future sync checks

this is a documentation behavior with low blast radius. failure modes are frustrate but recoverable.
