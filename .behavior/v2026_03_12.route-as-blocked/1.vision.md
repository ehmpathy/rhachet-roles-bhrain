# vision: route --as blocked

## extant infra (discovered via self-review)

the PassageReport domain object already supports `blocked` status:

```typescript
interface PassageReport {
  stone: string;
  status: 'passed' | 'approved' | 'blocked' | 'rewound';
  blocker?: string;  // what blocks passage
  reason?: string;   // why blocked/rewound
}
```

**scope**: wire extant types to a new CLI command тАФ no new types to build.

## user experience

### usecases

| actor | goal | contract |
|-------|------|----------|
| robot | signal "i need human help" | articulate into `$route/.route/blocker/$stone.md`, then `rhx route.stone.set --stone X --as blocked` |
| human | see what's blocked | read `$route/.route/blocker/$stone.md` or passage.jsonl |
| hook | detect blocked state | read passage.jsonl for `status: 'blocked'` |
| hook | suggest escalation | when blockers > 5, route.drive appends `--as blocked` as an option |

note: articulation file is required. the act of write forces reflection.

### route.drive with blockers > 5

when a robot is stuck (blockers > 5), route.drive suggests escalation:

```
ЁЯжЙ where were we?

ЁЯЧ┐ route.drive
   тФЬтФА where do we go?
   тФВ  тФЬтФА route = .behavior/v2026_03_12.route-as-blocked
   тФВ  тФФтФА stone = 3.blueprint
   тФВ
   тФЬтФА are we there yet? if so, run
   тФВ  тФФтФА rhx route.stone.set --stone 3.blueprint --as passed
   тФВ
   тФФтФА are you blocked? if so, run
      тФФтФА rhx route.stone.set --stone 3.blueprint --as blocked
```

this nudges robots to consider escalation when they're stuck in a loop.

### timeline

1. robot works on stone
2. robot hits genuine blocker
3. robot runs `--as blocked` (without articulation file)
4. system shows nudge: "failure is only the opportunity to begin again"
5. system creates `$route/.route/$stone.blocked.triggered` (proves robot saw nudge)
6. robot articulates into `$route/.route/blocker/$stone.md`
7. robot runs `--as blocked` again
8. system checks: `.triggered` exists AND `.blocked.md` exists
9. passage.jsonl records `{ stone: "3.blueprint", status: "blocked" }`
10. route.drive hook sees blocked state, no longer prompts
11. human reads articulation, provides guidance
12. robot continues with `--as passed` when complete

**key**: the articulation file forces reflection. same pattern as selfreview articulation.

- "it's like when you raise your hand in class because you're stuck"

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| robot can signal they're stuck | тЬЕ `--as blocked` after articulation |
| human knows where help is needed | тЬЕ `$route/.route/blocker/$stone.md` explains why |
| robot doesn't abuse escalation | тЬЕ `.triggered` file + required articulation file |
| route.drive hook knows state | тЬЕ can read blocked status |

### pros

- explicit > implicit: no need to guess if robot is stuck
- articulation file forces deep reflection, not quick string
- trigger file ensures nudge was seen
- works in non-interactive contexts (no [y/n] prompt)
- recoverable: blocked is a state, not an end

### cons

- two-step flow (trigger тЖТ confirm) adds friction

### edge cases & pit of success

| edge case | how we handle it |
|-----------|------------------|
| robot skips nudge, provides articulation directly | `.triggered` file check fails тАФ must see nudge first |
| robot marks blocked without effort | nudge reminds: "failure is only the opportunity to begin again" |
| robot writes shallow articulation | human can reject or ask for clarity |
| robot marks blocked repeatedly | passage.jsonl tracks all attempts, human can review |
| human doesn't notice blocked | route.drive hook surfaces blocked stones |
| robot unblocks themselves | they run `--as passed` when ready |

## open questions & assumptions

### assumptions

- robots will read the nudge before they provide reason
- humans will check passage.jsonl or route.drive output
- blocked is a temporary state, not permanent
- robot can self-unblock via `--as passed` (no approval needed)

### questions to validate

1. should route.drive actively notify human of blocked stones (email, slack)?
2. should blocked stones have a timeout or escalation ladder?

### resolved questions

- тЬЕ articulation file required (`$route/.route/blocker/$stone.md`)
- тЬЕ `.triggered` file proves robot saw nudge (same pattern as selfreview)
- тЬЕ `.triggered` files are gitignored (no clutter concern)
- тЬЕ .gitignore for .route/ should be stdouted on first write (extant behavior)
- тЬЕ challenges are opportunities: "failure is only the opportunity to begin again"

---

## what is awkward?

### what feels off

- two-step flow adds friction: first trigger, then confirm with articulation

### where design fights mental model

- robot might expect single command, not trigger тЖТ confirm

### uncomfortable tradeoffs

- required articulation adds ceremony, but ensures humans get context

## summary

`--as blocked` with articulation file gives robots an explicit escalation path:
1. attempt block тЖТ see nudge тЖТ `.triggered` file created
2. articulate into `$route/.route/blocker/$stone.md`
3. confirm block тЖТ passage.jsonl records blocked state
4. human reads articulation тЖТ provides guidance
5. robot continues with `--as passed`

**key design decisions**:
- articulation file required (forces deep reflection, not quick string)
- `.triggered` file proves robot saw nudge (same pattern as selfreview)
- no interactive prompts (works in all contexts)

**test coverage**:
- journey acceptance test to verify full flow: block attempt тЖТ nudge тЖТ articulation тЖТ passage recorded
