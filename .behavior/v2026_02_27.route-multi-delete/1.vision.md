# vision: multi-stone delete

## the outcome world

### before
deleting multiple stones requires running the command repeatedly:

```sh
➜ rhx route.stone.del --mode apply --stone 3.2
➜ rhx route.stone.del --mode apply --stone 3.1
➜ rhx route.stone.del --mode apply --stone 2.3
```

3 commands, 3 waits, 3 outputs to scan. friction compounds when pruning a route.

### after
one command, one operation:

```sh
➜ rhx route.stone.del --mode apply --stone 3.2 --stone 3.1 --stone 2.3
```

output aggregates all deletions:


### the "aha" moment
when you're pruning research stones to restart a phase, you type one command instead of hunting through shell history or retyping the same prefix 5 times.


## user experience

### usecases
1. **prune a phase** — delete all 3.1.research.* stones to restart research
2. **clean up false starts** — remove multiple draft stones that didn't pan out
3. **batch cleanup** — tidy a route by removing obsolete artifacts

### contract

```sh
rhx route.stone.del --stone <pattern> [--stone <pattern>]... [--mode plan|apply]
```

- `--stone` is repeatable; each pattern expands via glob
- `--mode plan` (default) shows what would be deleted
- `--mode apply` executes deletions

### timeline
1. user runs with `--mode plan` to preview
2. user confirms list looks right
3. user reruns with `--mode apply`
4. all matched stones deleted in single operation


## mental model

### how users describe it
> "you can pass multiple `--stone` flags to delete them all at once"

### analogies
- like `rm file1 file2 file3` instead of running `rm` three times
- like `git add` accepting multiple paths

### terms
users say "stones" — we say "stones". alignment is good.


## evaluation

### pros
- **reduced friction** — one command instead of N
- **atomic view** — see all deletions in one output
- **natural cli pattern** — repeatable flags are idiomatic

### cons
- **larger blast radius** — easier to delete more accidentally
  - mitigated by: `--mode plan` default, confirmation output

### edge cases

| case | behavior |
|------|----------|
| no stones match any pattern | exit 0, report "deleted = 0" |
| some patterns match, some don't | delete matches, skip patterns with no matches |
| same stone matched by multiple patterns | delete once, dedupe in output |
| `--stone` omitted | error: at least one --stone required |

### pit of success
- `--mode plan` is default — users see before they act
- output clearly lists each deleted stone
- patterns that match no files are silently skipped (no noise)
