# blackbox criteria matrix: peer-review-budgets

## matrix.1 = budget consumption

| ind: review result | ind: cache status | dep: budget change | dep: reviewer status |
|--------------------|-------------------|--------------------|-----------------------|
| success + blockers | not cached        | -1                 | active                |
| success + no blockers | not cached     | -1                 | approved (terminal)   |
| malfunction        | not cached        | 0                  | active                |
| n/a                | cached            | 0                  | approved (terminal)   |

**dimensions**: 2 independent × 2 dependent — manageable

---

## matrix.2 = exhaustion trigger and reset

| ind: budget left | ind: event | dep: status after | dep: budget after |
|------------------|------------|-------------------|-------------------|
| 1                | review succeeds | exhausted | 0 |
| 0 (exhausted)    | hash changes | exhausted | 0 |
| 0 (exhausted)    | --as rewound | active | full |
| 0 (exhausted)    | --add N | active | N |

**dimensions**: 2 independent × 2 dependent — manageable

---

## matrix.3 = level execution order

| ind: level 1 status | ind: level 2 status | dep: level 1 runs? | dep: level 2 runs? |
|---------------------|---------------------|--------------------|--------------------|
| queued              | queued              | yes                | no (waits)         |
| approved            | queued              | n/a (done)         | yes                |
| exhausted           | queued              | n/a (done)         | yes                |
| mixed (some approved, some exhausted) | queued | n/a (done) | yes |
| has blockers        | queued              | yes                | no (waits)         |

**dimensions**: 2 independent × 2 dependent — manageable

**note**: "mixed" means some level 1 reviewers approved, some exhausted — all terminal, so level 2 proceeds.

---

## matrix.4 = route halt condition

|------------------------------|------------------------------|-------------------|-------------------------|

| no                           | yes                          | no                | no (not yet)            |
| yes                          | no                           | no                | no (all approved)       |
| yes                          | yes                          | yes               | yes                     |

**dimensions**: 2 independent × 2 dependent — manageable

## matrix.7 = approval unification

## matrix.8 = budget scope across stones

| ind: stone | ind: rounds consumed on stone | dep: budget at stone start |
|------------|-------------------------------|----------------------------|
| stone.A    | 0                             | 3                          |
| stone.A    | 2                             | 3 → 1 (after 2 rounds)     |
| stone.B    | 0                             | 3 (fresh budget)           |
| stone.B    | 2                             | 3 → 1 (after 2 rounds)     |

**dimensions**: 2 independent × 1 dependent — manageable

**note**: budget is per stone per reviewer. each stone starts with fresh budget as declared in its guard. consumption on stone.A does NOT affect budget for stone.B.

---

## gap analysis

no gaps detected. all meaningful combinations are covered:

1. **budget consumption**: all success/fail × cache permutations covered
2. **exhaustion**: all trigger and reset paths covered
3. **level execution**: all terminal state combinations covered
4. **route halt**: the 2×2 matrix is complete
5. **extension scope**: bulk vs targeted covered
6. **defaults**: both fields covered
7. **approval**: all presence/absence combinations covered
8. **scope**: cross-stone behavior covered

---

## decomposition assessment

all matrices have ≤3 independent dimensions. no decomposition needed.

the design cleanly separates concerns:
- budget consumption (per review)
- exhaustion (per reviewer per route)
- level execution (per tier)
- route halt (aggregate)
- approval (unified)

each concern has a focused matrix. the behavioral boundaries are narrow and testable.
