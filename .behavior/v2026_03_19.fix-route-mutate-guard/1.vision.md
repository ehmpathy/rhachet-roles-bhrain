# 1.vision: fix route.mutate guard

## the outcome world

### before

a mechanic runs `rhx declapract.upgrade init` which creates a route at `.route/v2026_03_19.declapract.upgrade/`. they bind to it and start working. but when they try to write any artifact into their bound route — blocked. the guard sees `.route/` in the path and rejects it.

the guard is treating ALL `.route/**` paths as sacred metadata, but the route directory itself is where work happens.

```
❌ Write to .route/v2026_03_19.declapract.upgrade/artifact.md → BLOCKED
❌ Write to .behavior/xyz/.route/passage.jsonl → BLOCKED (correct)
```

both get blocked, but only the second should be.

### after

the guard distinguishes between:
1. **the route directory itself** — where artifacts live, where work happens
2. **the `.route/` subdirectory within a route** — where metadata lives (passage.jsonl, bind flags, etc.)

```
✅ Write to .route/v2026_03_19.declapract.upgrade/artifact.md → ALLOWED
❌ Write to .route/v2026_03_19.declapract.upgrade/.route/passage.jsonl → BLOCKED
❌ Write to .behavior/xyz/.route/passage.jsonl → BLOCKED
```

### the "aha" moment

> "oh! the guard protects the `.route/` subdirectory within a route, not routes that happen to be located AT `.route/`"

the mental model shifts from "all paths containing `.route/`" to "nested `.route/` directories that hold metadata".

## user experience

### use case 1: declapract.upgrade workflow

1. human runs `rhx declapract.upgrade init`
2. route created at `.route/v2026_03_19.declapract.upgrade/`
3. mechanic binds to the route via `rhx route.bind.set`
4. mechanic works on stones, writes artifacts directly to the route directory
5. artifacts like `1.upgrade.invoke.md` live at `.route/v2026_03_19.declapract.upgrade/1.upgrade.invoke.md`
6. metadata like passage.jsonl lives at `.route/v2026_03_19.declapract.upgrade/.route/passage.jsonl`
7. guard allows artifact writes, blocks metadata writes — pit of success

### use case 2: behavior routes (extant pattern)

1. route at `.behavior/v2026_03_19.fix-route-mutate-guard/`
2. artifacts at `.behavior/v2026_03_19.fix-route-mutate-guard/1.vision.md`
3. metadata at `.behavior/v2026_03_19.fix-route-mutate-guard/.route/passage.jsonl`
4. guard behavior unchanged — still works correctly

### use case 3: blocker explanations

**before**: blockers written to `$route/.route/blocker/*.md`
**after**: blockers written to `$route/blocker/*.md`

why? blockers are part of the route's output, not hidden metadata. they should be visible alongside artifacts.

### timeline

1. **bind** → route is associated with branch
2. **drive** → work through stones, write artifacts
3. **guard** → protected: stones, guards, `.route/**`; allowed: artifacts, `blocker/**`
4. **pass** → advance through route via proper flow

## mental model

### how users describe it

> "the guard protects route internals — the `.route/` folder where passage tracking and bind flags live. but the route directory itself is my workspace. i can write my artifacts there."

### analogy

think of a route like a project folder:
- **the folder itself** (`my-project/`) is where you put your work
- **the `.git/` subfolder** (`my-project/.git/`) is internal machinery — don't touch

similarly:
- **the route directory** (`.route/xyz/` or `.behavior/xyz/`) is where artifacts go
- **the `.route/` subfolder** within it (`xyz/.route/`) is internal machinery — protected

### terms

| user term | technical term |
|-----------|---------------|
| "my route" | bound route directory |
| "my artifacts" | files directly in route (not in `.route/` subfolder) |
| "route internals" | `.route/` subdirectory (passage.jsonl, bind flags) |
| "blockers" | `$route/blocker/*.md` — visible output, not hidden |

## evaluation

### does it solve the goals?

| goal | solved? |
|------|---------|
| allow writes to bound route directory | yes — no longer blocked |
| block writes to route metadata | yes — `.route/` subdirectory still protected |
| support declapract.upgrade routes | yes — routes at `.route/` work |
| move blockers to visible location | yes — `$route/blocker/` not hidden |

### pros

- **minimal change**: guard logic only needs path comparison refinement
- **backwards compatible**: extant `.behavior/` routes work identically
- **intuitive**: matches mental model of "workspace vs internals"
- **enables new patterns**: routes can live at `.route/` itself

### cons

- **slightly more complex guard logic**: must parse path more carefully
- **migration needed**: extant routes with blockers at `.route/blocker/` need migration

### edgecases and pit of success

| edgecase | handling |
|----------|----------|
| write to `$bound_route/artifact.md` | allowed — your workspace |
| write to `$bound_route/.route/passage.jsonl` | blocked — internals |
| write to `$other_route/artifact.md` | blocked — not your route |
| write to `$bound_route/blocker/defect1.md` | allowed — visible output |
| read from `$bound_route/2.criteria.stone` | blocked — stone protection unchanged |
| read from `$bound_route/2.criteria.guard` | blocked — guard protection unchanged |

## what is awkward?

### the `.route/` in `.route/`

when a route lives at `.route/v2026_03_19.declapract.upgrade/`, its metadata is at `.route/v2026_03_19.declapract.upgrade/.route/`. that's `.route/.../.route/`.

it's a bit recursive-looking. but it's consistent: the `.route/` subdirectory ALWAYS holds metadata, regardless of where the parent route lives.

### blocker location change

moving from `$route/.route/blocker/` to `$route/blocker/` is a breaking change for any code that reads blockers from the old location. need to audit consumers.

### guard complexity

the guard now needs to:
1. find bound route path
2. check if target path is WITHIN bound route but NOT in its `.route/` subdirectory → allow
3. check if target path is in `.route/` subdirectory → block
4. check if target is stone/guard → block

this is more nuanced than "grep for `.route/`". but it's the correct semantics.

## summary

the fix makes the guard smarter: it protects route internals (the `.route/` subdirectory) while allowing work in the route directory itself. this enables routes at `.route/` (like declapract.upgrade) while maintaining the safety guarantees.
