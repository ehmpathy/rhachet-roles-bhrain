  Current approach (OOM):
  for (const filePath of sortedFiles) {
    const content = await fs.readFile(filePath);  // loads 700+ files
    contents.push(content);
  }
  hash(contents.join('\n'));  // ~4GB in memory

  Better approach (zero file reads):
  // git ls-tree gives us hashes without reading file contents
  const result = exec(`git ls-tree -r HEAD ${paths.join(' ')}`);
  // parse blob hashes, combine them
  const blobHashes = result.lines.map(l => l.split(/\s+/)[2]);
  return hash(blobHashes.sort().join('\n'));  // ~50KB in memory

  Benefits:
  - Zero file reads — git already computed the hashes
  - O(n) strings, not O(n) file contents — just 40-char hashes
  - Same semantics — git blob hash is content-addressable

  For unstaged changes, git hash-object <file> computes the blob hash without us loading the file. Could even batch it: git
  hash-object --stdin-paths.
