# self-review: has-questioned-assumptions

## review: are there hidden technical assumptions?

### assumption: articulation path is `$route/.route/blocker/$stone.md`

**the assumption**: blocker articulation files go in a `blocker/` subdirectory.

**questioned**: why `blocker/` and not `blocked/`?

**why it holds**: `blocker` is a noun (the entity that blocks). `blocked` is an adjective (the state). nouns are better for directory names that contain entities. consistent with domain vocabulary.

### assumption: single triggered file pattern

**the assumption**: use single `.blocked.triggered` file, not the `.since`/`.uptil` pattern from selfreview.

**questioned**: why not use the same pattern as selfreview?

**why it holds**: selfreview uses time bounds for the 30-second patience enforcement. blocked has no time enforcement. simpler pattern fits simpler requirements. see has-questioned-deletables review for full analysis.

### assumption: three-state challenge decision

**the assumption**: `BlockedChallengeDecision = 'challenge:first' | 'challenge:absent' | 'allowed'`

**questioned**: why not boolean? why not more states?

**why it holds**:
- boolean can't distinguish "first attempt" from "has triggered but no articulation"
- three states is minimum needed: no triggered, triggered but no articulation, both present
- no need for fourth state (articulation without triggered is handled as challenge:first)

### assumption: blockers > 5 threshold

**the assumption**: suggest `--as blocked` when blockers exceed 5.

**questioned**: why 5? could this be wrong?

**why it holds**: 5 is a heuristic that indicates repeated attempts without progress. too low (e.g., 2) would trigger suggestion before robot has genuinely tried. too high (e.g., 20) would delay helpful nudge. 5 is consistent with extant hook behavior that considers blockers count.

**note**: could be made configurable in future. but add configuration now would be YAGNI.

### assumption: triggered file not cleaned up

**the assumption**: after successful block, `.blocked.triggered` file remains.

**questioned**: should it be cleaned up?

**why it holds**:
- triggered files are gitignored (no repo clutter)
- provides audit trail for debug
- cleanup would add complexity without clear benefit
- consistent with selfreview pattern (promise files persist)

### assumption: output uses optional booleans

**the assumption**: output is `{ challenged?: boolean; blocked?: boolean; ... }`

**questioned**: why not a union type like `{ result: 'challenged' | 'blocked' }`?

**why it holds**: optional booleans allow for clear conditionals:
- `if (result.challenged)` — nudge was shown
- `if (result.blocked)` — block succeeded

union type would work but optional booleans are idiomatic for this pattern and consistent with extant emit contracts.

## conclusion

all assumptions are justified. no hidden assumptions found that require change.
