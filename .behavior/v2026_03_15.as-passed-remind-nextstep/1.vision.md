
- try to run the same command again
- look for a different command

## user experience

### usecase: completing a stone

**goal:** mark stone as passed and continue driving the route

**timeline:**
1. driver reviews stone artifacts
2. satisfied, runs `rhx route.stone.set --stone X --as passed`
3. sees success message + next step reminder
4. runs `rhx route.drive`
5. receives next stone assignment

**contract inputs:** `--stone <name> --as passed`

**contract outputs:**
- confirmation of passage
- reminder: `run rhx route.drive to continue the route`

### usecase: completing final stone

**goal:** mark final stone as passed, route completes

**timeline:**
1. driver reviews final stone
2. runs `rhx route.stone.set --stone 5.execution --as passed`
3. sees success message
4. reminder says `run rhx route.drive to complete the route`
5. driver runs route.drive, sees route completion celebration

## evaluation

### how well does it solve the goal?

**excellent** — directly addresses the confusion by providing immediate, actionable guidance at the exact moment of uncertainty.

### pros

- zero learning curve — just read and follow
- momentum preserved — no pause to think
- consistent experience — same reminder every time
- simple implementation — just add output text

### cons

- slightly more verbose output
- experienced users may find it redundant (but harmless)

### edgecases

| edgecase | pit of success |
|----------|----------------|
| final stone passed | reminder adapts: "run route.drive to complete" |
| stone already passed | idempotent, shows same reminder |
| invalid stone name | error shown before reminder (fail fast) |
| `--as approved` | NO reminder — approval is not passage |
| `--as rewound` | NO reminder — rewind is not passage |
| `--as promised` | NO reminder — promise is not passage |

### scope constraint

the reminder appears ONLY on `--as passed`. other actions (`approved`, `rewound`, `promised`) do not emit the reminder because they do not constitute passage.

rationale:
- `approved` = human signs off on guard, but stone still needs `--as passed`
- `rewound` = stone reverted for re-review
- `promised` = self-review checkpoint, not passage

## open questions & assumptions

### assumptions

1. drivers always want to continue after passing a stone
2. `rhx route.drive` is the canonical next command
3. a single line reminder is sufficient (not a full help screen)

### questions (resolved)

1. should the reminder be optional/configurable?
   - **[answered] NO** — YAGNI. the wish doesn't ask for configuration. add later if needed.

2. should we distinguish "continue" vs "complete" for final stone?
   - **[answered] NO** — keep reminder consistent. route.drive handles completion message.

3. should the reminder include the route name for clarity?
   - **[answered] NO** — context is implicit. driver just passed a stone in that route.

### external research needed

none — all questions resolved via logic

## what is awkward?

### what feels off?

no major concerns. this is a simple, focused enhancement.

### tradeoffs

| tradeoff | assessment |
|----------|------------|
| verbosity vs clarity | clarity wins — one extra line is worth it |
| consistency vs context | keep it simple, same message every time |

### potential friction

- if route.drive command name changes, reminder becomes stale
  - mitigation: reminder text should reference the canonical command dynamically if possible, or be updated when command changes
