# handoff: integration/acceptance tests blocked by tooling bug

## blocker summary

**git.repo.test does not export keyrack keys to subprocess**

the `git.repo.test` skill calls `rhx keyrack unlock --owner ehmpath --env test` which unlocks keys in keyrack daemon memory, but it does NOT export them as environment variables to the npm subprocess. the jest env file checks `process.env[key]` which is empty.

## what i tried

### 1. ran git.repo.test with keyrack unlocked
```bash
rhx keyrack unlock --owner ehmpath --env test  # succeeded
rhx git.repo.test --what integration            # failed - keys not in process.env
```
keyrack shows "unlocked ehmpath/test" but tests still fail with "absent keys".

### 2. verified keyrack has the keys
```bash
rhx keyrack source --owner ehmpath --env test
# outputs:
# export XAI_API_KEY='xai-...'
# export OPENAI_API_KEY='sk-svcacct-...'
# export ANTHROPIC_API_KEY='sk-ant-...'
# export TAVILY_API_KEY='tvly-dev-...'
```
keys are set correctly in keyrack.

### 3. verified ~/.config/rhachet/apikeys.env exists
the file exists and contains all required keys. the `use.apikeys.sh` skill would work if sourced.

### 4. attempted inline env var invocation
```bash
OPENAI_API_KEY=$(rhx keyrack get ...) npm run test:integration
```
blocked by permissions - command substitution not allowed.

### 5. attempted source && npm chain
```bash
source .agent/repo=.this/role=any/skills/use.apikeys.sh && npm run test:integration
```
blocked by permissions - cannot chain commands with `&&`.

## why each approach failed

| approach | failure reason |
|----------|---------------|
| git.repo.test | skill bug: unlocks but doesn't export to subprocess |
| inline env vars | blocked by permissions (command substitution) |
| source && npm | blocked by permissions (command chaining) |

## what makes this insurmountable

1. **tooling bug in rhachet-roles-ehmpathy**: the `git.repo.test` skill needs to add `eval "$(rhx keyrack source --owner ehmpath --env test)"` before running `npm run test:*`

2. **permission constraints**: cannot run arbitrary shell commands that would work around the bug

3. **separate shell sessions**: each bash tool call is a separate shell, so sourcing in one doesn't affect the next

## foreman action needed

run the tests manually:

```bash
# option 1: use apikeys.sh (sources from ~/.config/rhachet/apikeys.env)
source .agent/repo=.this/role=any/skills/use.apikeys.sh && npm run test:integration && npm run test:acceptance

# option 2: use keyrack source directly
eval "$(rhx keyrack source --owner ehmpath --env test)" && npm run test:integration && npm run test:acceptance
```

OR fix the `git.repo.test` skill in `rhachet-roles-ehmpathy` to export keyrack keys before running tests.
