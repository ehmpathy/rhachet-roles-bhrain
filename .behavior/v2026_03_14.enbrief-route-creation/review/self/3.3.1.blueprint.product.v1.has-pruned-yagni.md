## deeper YAGNI reflection

### the onBoot hook question revisited — concrete analysis

re-read getDriverRole.ts to see what actually exists:

```typescript
hooks: {
  onBrain: {
    onBoot: [
      {
        command: './node_modules/.bin/rhx route.drive --mode hook',
        timeout: 'PT5S',
      },
    ],
    // ...
  },
}
```

**key finding**: driver's extant onBoot hook runs `route.drive`, NOT `rhachet roles boot`.

this means:
1. driver DOES have an onBoot array (so adding to it is valid)
2. but that hook does NOT trigger boot.yml load
3. we likely need BOTH:
   - `boot: { uri: __dirname + '/boot.yml' }` — declares WHAT to load
   - new onBoot entry for `rhachet roles boot --repo bhrain --role driver` — triggers WHEN to load

**the blueprint is correct**: it says "add boot and onBoot" — meaning add the boot property AND add a new onBoot entry. NOT replace the extant route.drive hook.

**revised analysis**: this is NOT a YAGNI violation. the onBoot command for `rhachet roles boot` is distinct from the extant `route.drive` hook. both are needed:
- `route.drive --mode hook` — checks bound route on brain start
- `rhachet roles boot` — loads boot.yml briefs on role init

**lesson learned**: "driver already has onBoot hooks" is misleading without checking WHAT those hooks do. different hooks serve different purposes.

### why this matters

adding an unnecessary onBoot command would:
1. duplicate boot triggers if extant hooks already call boot
2. add maintenance burden
3. create potential for double-boot bugs

not adding it when needed would:
1. leave boot.yml un-loaded
2. require a fix later

the safer YAGNI choice is: try minimal first (just `boot` property), verify in playtest.
