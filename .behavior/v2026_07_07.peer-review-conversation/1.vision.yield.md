# 1.vision — peer-review conversation

> make peer review a two-sided conversation: reviewers speak (`.given.by_peer`), drivers must answer (`.taken.by_self`), and reviewers can optionally hear the whole exchange (`--conversation`).

---

## the outcome world

### day-in-the-life

vlad enrolls a route. a stone gates on two peer reviewers: `architect` and `mechanic`.

the driver arrives at the stone. the guard runs the reviewers. each writes a `.given` file with blockers + nitpicks. today the driver reads them, maybe fixes some, and re-arrives — and the guard has **no idea whether the driver even read the feedback**. the reviewer, on its next round, sees the same artifact and often re-raises the same blocker because it can't tell what the driver did with its last critique. this is the "infinite churn".

in the outcome world, the guard now **halts the driver after peer review** the same way it halts after self-review — and it speaks in the same treestruct voice the driver already knows. the driver hits this exact stdout in **three** places: the `route.drive` stophook (so it can't quietly stop mid-conversation), and any `--as arrived` or `--as passed` attempt (so it can't progress the stone until every reviewer is answered):

```
🪨 run solid skill repo=bhrain/role=driver/skill=route.drive

🦉 the reviewers await your reply

🌕 lets respond
   │
   ├─ review.peer 1/2
   │  ├─ slug = architect
   │  ├─ verdict = 2 blockers, 1 nitpick
   │  ├─ contemplate from
   │  │  └─ .reviews/peer/1.execute._.review.i1.a1b2c3.r1._.given.by_peer.architect.md
   │  └─ articulate into
   │     └─ .reviews/peer/1.execute._.review.i1.a1b2c3.r1._.taken.by_self.architect.md
   │
   ├─ review.peer 2/2
   │  ├─ slug = mechanic
   │  ├─ verdict = 0 blockers, 3 nitpicks
   │  ├─ contemplate from
   │  │  └─ .reviews/peer/1.execute._.review.i1.a1b2c3.r2._.given.by_peer.mechanic.md
   │  └─ articulate into
   │     └─ .reviews/peer/1.execute._.review.i1.a1b2c3.r2._.taken.by_self.mechanic.md
   │
   └─ when you've contemplated each reviewer, run
      └─ rhx route.stone.set --stone 1.execute --as contemplated --that architect
```

the driver writes a `.taken` file that answers each critique — "fixed by X", or "holds because Y" — then signals `--as contemplated`. the articulation *is* the proof of contemplation: the guard cannot see whether the driver thought hard, but it can see that a `.taken` was written. only once **every** `.given` for the current iteration has a paired `.taken` can the stone progress.

on the next round, the `architect` reviewer is defined with `--conversation`. the guard expands the full conversation glob (every `.given` + `.taken` for this stone) and hands it to the reviewer as reference context. the reviewer now reads its own prior blocker **and the driver's response to it**. instead of blindly re-raising, it evaluates: "the driver claims fixed by X — is X actually present? yes → drop it. no → re-raise with 'X not found'." churn collapses.

### before / after

| dimension | before | after |
|-----------|--------|-------|
| driver reads feedback | hoped-for, unenforced | **required** — stone blocks until `.taken` extant |
| reviewer sees driver's response | never — blind to prior rounds | **optional** via `--conversation` |
| wasted budget | driver ignores feedback, burns rounds | eliminated — no progress without articulated response |
| reviewer churn | re-raises settled blockers | evaluates against the driver's stated fix |
| conversation legibility | scattered `.given` files only | full `.given`↔`.taken` pairs, one key |

### the "aha" moment

the value clicks the first time a reviewer **drops a blocker because the driver explained the fix** — a round that previously would have looped now converges. and the first time a human opens the `.reviews/peer/` dir and reads a stone's review as an actual back-and-forth dialogue instead of a pile of one-sided critiques.

## deliverables (handwave checklist)

the exact pieces this wish ships. handwave-level — the roadmap will decompose each.

- [ ] **`.taken.by_self` convention** — write driver responses to `.reviews/peer/$stone._.review.i$iter.$hash.r$idx._.taken.by_self.$slug.md`, mirroring the extant `.given.by_peer` key.
- [ ] **peer-response gate** — a stone with peer reviews cannot progress until **every** current-iteration `.given` has a paired `.taken` (clean 0/0 reviews exempt).
- [ ] **contemplation signal** — `rhx route.stone.set --as contemplated --that <slug>` records that a reviewer was contemplated (the `.taken` is the proof).
- [ ] **crystal-clear absent-file message** — if the driver signals `--as contemplated` before the `.taken` exists, the guard prints the exact path to write, the paired `.given` to read, and *why* — symmetric `contemplate from` / `articulate into`, never a vague failure.
- [ ] **"reviewers await your reply" stdout** — the treestruct halt message, rendered from a single shared formatter across **three surfaces**: the `route.drive` stophook, `--as arrived`, and `--as passed`.
- [ ] **`$conversation` guard substitution var** — the guard expands `$conversation` in a peer reviewer's `run` command to the conversation glob (`.given.by_peer.*` + `.taken.by_self.*` for the stone). opt-in per reviewer.
- [ ] **`review --conversation` native flag** — the built-in `review` skill accepts the glob and threads those files in as reference context.
- [ ] **one shared conversation expander** — a single `enumRouteGuardReviewConversationFiles` feeds both the guard var and the `review` flag, so the two paths can't drift (see R4).
- [ ] **clean-review skip** — a 0-blocker, 0-nitpick review requires no `.taken` (pending Q5).
- [ ] **self-review absent-file message parity** — apply the same crystal-clear, symmetric treatment to the *extant* self-review `challenge:absent` stdout. drivers routinely stumble here because the articulation path's level (`rN`) is assigned by the guard and **varies** between attempts (r1→r2→…, per `howto.run-self-reviews` "the level pitfall"). the message must print the exact current-level path verbatim and say plainly "copy this path — the guard sets the level, you do not" so a driver never computes it wrong.
- [ ] **exhaustive boundary test coverage** — the wisher's "most important part": pairing edgecases, stale-hash/iter, partial responses, opt-in on/off, clean reviews, the three render surfaces, and the absent-file messages (peer + self).

## user experience

### usecases & goals

| user | goal | how |
|------|------|-----|
| driver | prove i engaged with each critique | write `.taken` response per `.given`, then signal |
| driver | not waste budget re-triggering blind reviewers | conversation lets reviewer converge faster |
| route author | force accountability into the loop | just declare peer reviews as today — `.taken` gate is automatic |
| route author | let a reviewer see history | add `--conversation` to that reviewer's `run` (opt-in) |
| route author | keep a reviewer blind | omit `--conversation` — unchanged behavior |
| human | audit the exchange | read paired `.given`/`.taken` files under `.reviews/peer/` |

### contract inputs & outputs

**1. driver articulates a response (new gate, mirrors self-review):**

today, self-review gates via (`getSelfReviewArticulationPath.ts:11`, `setStoneAsPromised.ts`):
```bash
rhx route.stone.set --stone 1.execute --as promised --that <self-slug>
```

peer-review response follows the same shape — `--as contemplated` (settled by the wisher: the articulation is proof of contemplation, and it rhymes with the `contemplate from` pointer above):
```bash
# after fixing / deciding, driver writes the .taken file, then signals it contemplated reviewer <slug>
rhx route.stone.set --stone 1.execute --as contemplated --that architect
```
- **input**: `--that <peer-slug>` — which reviewer's `.given` this contemplates
- **precondition**: the `.taken` file at the mirrored path exists
- **output**: guard records the contemplation; when all `.given` of the current iteration are paired, peer review can progress
- **if the `.taken` is absent**: the guard must not fail vaguely. it prints the exact path to write and *why* — crystal clear, so the driver never guesses:

```
✋ contemplation absent for reviewer architect

   the guard looked for your response here, and did not find it:
   └─ .reviews/peer/1.execute._.review.i1.a1b2c3.r1._.taken.by_self.architect.md

   why: --as contemplated is a promise that you engaged each critique.
        the .taken file IS that engagement — absent it, there is no ground to stand on.

   what to do:
   ├─ contemplate from
   │  └─ .reviews/peer/1.execute._.review.i1.a1b2c3.r1._.given.by_peer.architect.md
   └─ articulate into
      └─ .reviews/peer/1.execute._.review.i1.a1b2c3.r1._.taken.by_self.architect.md
```

**2. file naming — mirror the given key exactly:**

given (today, `runStoneGuardReviews.ts:138`):
```
.reviews/peer/$stone._.review.i$iter.$hash.r$idx._.given.by_peer.$slug.md
```
taken (decided by wisher — same key, `.given.by_peer` → `.taken.by_self`):
```
.reviews/peer/$stone._.review.i$iter.$hash.r$idx._.taken.by_self.$slug.md
```
pairing is by `(stone, iter, hash, idx, slug)`. the `$slug` is retained so a response always names the reviewer it answers. `by_self` (the driver authored it) contrasts cleanly with `by_peer` (the reviewer authored it).

**3. `--conversation` on a peer reviewer (opt-in, guard-native):**

guard peer definition today (`RouteStoneGuard.ts:32-68`):
```yaml
reviews:
  peer:
    - slug: architect
      run: rhx review --rules '.agent/.../architect/**' --paths-with '$route/*.stone'
      budget: 3
```
with conversation:
```yaml
    - slug: architect
      run: rhx review --rules '...' --paths-with '$route/*.stone' --conversation $conversation
```
- the guard substitutes `$conversation` (alongside the extant substitution vars `$stone $route $hash $output`, plus `repoRoot`, `runStoneGuardReviews.ts:147-153`) with the expanded glob of the full conversation for this stone: every `.given.by_peer.*` and `.taken.by_self.*` file. (`rhx`/`rhachet` are not `$`-substituted; they are on `PATH` via `node_modules/.bin`, `runStoneGuardReviews.ts:157-160`.)
- `$conversation` is a **glob of filepaths**, so any skill — not just `review` — can consume it. the `review` skill additionally gains native `--conversation` support, threading those files in as reference context (analogous to the extant `--refs`, `review.ts:45`).

### timeline

```
arrive → guard runs peer reviews → .given files written
   → guard halts: "respond to each reviewer"
   → driver writes .taken per .given → --as contemplated --that <slug> (×N)
   → all paired? → stone progresses / next round
   → next round: reviewer with --conversation reads prior .given + .taken → converges
```

## mental model

### how a user describes it to a friend

> "peer review used to be a reviewer shouting into a void — it'd flag the same thing every round because it couldn't see what i did about it. now it's a real conversation: the reviewer writes its notes, i have to write back what i changed or why i disagree, and next round the reviewer actually reads my reply. way less looping."

### analogies

- **code review threads**: `.given` = a review comment; `.taken` = the "done / won't fix because…" reply; `--conversation` = the reviewer seeing the settled thread before re-reviewing.
- **email reply-with-quote**: the reviewer replies *in context* instead of starting a fresh cold email each time.
- **a conversation transcript**: `.given`/`.taken` pairs, one key, read top-to-bottom.

### terms: theirs vs ours

| user says | we say |
|-----------|--------|
| "the reviewer's notes" | `.given` (by_peer) |
| "my reply" | `.taken` (by_self) |
| "the whole thread" | conversation = glob of `.given` + `.taken` for a stone |
| "answering the review" | contemplate / articulate (`--as contemplated`) |
| "let the reviewer see history" | `--conversation $conversation` (opt-in) |

we deliberately reuse **given/taken** (already in the human-feedback template, `refs/template.[feedback].v1.[given].by_human.md`) and **articulate** (already the self-review verb) so the peer flow feels like a known idiom, not a new mechanism. the new word — **contemplate** — is the owl's own: `contemplate from` the critique, then signal `--as contemplated`. one word threads read → reflect → respond.

## evaluation

### how well it solves the goals

- **"eliminate driver wasting budget by ignoring feedback"** → directly solved: the stone **cannot progress** until a `.taken` exists for every `.given`. ignoring feedback is now structurally impossible, not merely discouraged.
- **"decrease infinite reviewer churn"** → solved for opted-in reviewers: `--conversation` gives the reviewer its own prior critique + the driver's response, so it can converge rather than re-raise.

### pros

- reuses the **extant self-review gate pattern** (promise/articulate) — low conceptual surface area.
- reuses **given/taken** vocabulary already present in the repo.
- `--conversation` is **opt-in** → zero forced behavior change for reviewers that work better blind.
- `$conversation` as a **glob** → any skill can consume it, not just `review`.
- the conversation becomes **human-auditable** as a legible dialogue.

### cons / costs

- adds a **mandatory step** to every peer-reviewed stone — more driver effort per round (this is the point, but it is friction).
- **more files** in `.reviews/peer/` (one `.taken` per `.given`).
- passing full conversation history **grows reviewer token cost** each round (bounded by opt-in).
- a driver could write a **low-effort `.taken`** ("done") to satisfy the gate — the gate proves engagement, not quality (mirrors the known limitation of self-review promises).

### edgecases & pit-of-success

| edgecase | keep-in-pit strategy |
|----------|----------------------|
| driver signals `--as contemplated` but wrote no `.taken` | gate checks the file **exists** (self-review's actual bar, `getSelfReviewChallengeDecision.ts:70-79`); if absent, it prints the exact path + paired `.given` + why (crystal clear, never vague) |
| `.given` re-runs → new `$hash`/`$iter` → stale `.taken` | pair on the **full key incl hash/iter**; a new iteration requires a fresh `.taken` |
| reviewer has 0 blockers + 0 nitpicks (clean) | no response required — no findings to articulate; don't force busywork |
| reviewer defined without `--conversation` | unchanged; `$conversation` simply not substituted for it |
| first round (no prior conversation) | `$conversation` expands to just the current `.given`(s) — or empty — reviewer handles gracefully |
| multiple reviewers, driver responds to some | gate blocks until **all** current-iteration `.given` are paired |

## open questions & assumptions

### assumptions made

1. the `.taken` authorship suffix is `by_self` (the driver) — **decided by the wisher**: `.given.by_peer` (reviewer) contrasts with `.taken.by_self` (driver). the full key still pairs on `(stone, iter, hash, idx, slug)`, so the `.given`/`.taken` swap plus the `by_peer`/`by_self` swap travel together.
2. the response gate reuses the self-review promise machinery (`setStoneAsPromised.ts`) rather than a wholly new mechanism.
3. clean reviews (0 blockers, 0 nitpicks) require no `.taken`.
4. `$conversation` expands to the current-stone conversation only (all iterations of this stone), not cross-stone history.

### questions (triaged)

each tagged `[answered]` (resolved now via logic/code), `[research]` (defer to research phase), or `[wisher]` (only the wisher can decide).

- **Q1 (name)** — `[wisher]` ✅ **ANSWERED by wisher this session**: `.taken.by_self` (contrast to `.given.by_peer`).
- **Q2 (verb)** — `[wisher]` ✅ **ANSWERED by wisher this session**: `--as contemplated --that <slug>`. the articulation (`.taken`) is the proof of contemplation, and the verb rhymes with the `contemplate from` read-pointer, so the whole loop shares one word. this is a new status (an eighth), accepted deliberately over an overload of `--as promised`. (statuses today: passed/approved/promised/blocked/rewound/arrived/overruled/forced, `route.ts:149-156`; `--that` is currently typed as "review.self slug to promise", `route.ts:314` — it will widen to cover peer slugs too.)
- **Q3 (scope of `$conversation`)** — `[wisher]`: current stone only, or the full route's review history? code permits either; the *choice* is a product decision (token cost vs cross-stone context). default proposed: current stone.
- **Q4 (content bar)** — `[answered]` via code: the extant self-review gate does **not** check content quality — it checks **file presence** (`getSelfReviewChallengeDecision.ts:70-79`), a **30s dwell**, and a **3-attempt plowthrough** (`:55-56, :98-125`). so "mirror self-review's exact check" = **presence + dwell + plowthrough**, no quality bar. a stricter content check on `.taken` would be net-new scope, not a mirror. (whether to add one is a follow-on `[wisher]` decision, but the baseline is now known.)
- **Q5 (clean reviews)** — `[wisher]`: confirm no response is required when a reviewer returns 0/0. logic favors "no response" (no findings to answer), but the wisher may want an explicit "acknowledged clean" record.

none — this is entirely internal to rhachet's route/guard/review machinery. no third-party APIs involved. no `[research]`-tagged questions remain.

## what is awkward?

- **naming (settled)**: `.given.by_peer` ↔ `.taken.by_self`. the driver authored the response, so `by_self` is authorship-correct and contrasts cleanly with the reviewer's `by_peer`. no lingering tension.
- **new status (settled)**: `--as contemplated` is a new eighth status. the wisher chose it over an overload of `--as promised` because the verb ties the loop together (`contemplate from` → `--as contemplated`). the cost is one more status in the enum; the benefit is a self-consistent vocabulary. `--that` widens from self-slugs to also cover peer-slugs.
- **`.reviews/peer/` is gitignored** (`findsertReviewPeerGitignore`, `runStoneGuardReviews.ts:119-121`) — yet the wish wants the conversation to be "easy to find." if `.taken` lives beside `.given` it's also gitignored, so the durable conversation record is local-only. is that acceptable, or should the conversation be committed? (relates to Q3.)
- **the gate proves engagement, not quality** — a driver can satisfy it with a hollow `.taken`. this is the same soft underbelly self-review promises already have; we inherit it rather than solve it.
- **token growth**: `--conversation` monotonically grows the reviewer prompt each round. opt-in bounds it, but a long-running stone with conversation enabled could get expensive — no truncation strategy is proposed yet.
- **`review` vs guard duplication**: the `review` skill must support `--conversation` natively **and** the guard must expand `$conversation` generically for arbitrary skills. two code paths must agree on what "the conversation glob" means — a single shared expander (one `enumRouteGuardReviewConversationFiles`) avoids drift.
