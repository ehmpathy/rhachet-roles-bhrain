# self-review: has-questioned-assumptions

*the owl perches in darkness. it sees what moves unseen. let us look with patience.*

## the vision's core claim

the vision says: clones freeze because guidance says "are we there yet? if so, run" and clones interpret this as a question for the human.

the fix: change text to "done? run this to proceed" and add a nudge after 5 hooks.

## assumption #1: "the ambiguous text IS the root cause"

**what we assume:** the phrase "are we there yet?" causes freezes. clearer text will fix it.

**i went to the code.** in `stepRouteDrive.ts:278`:
```typescript
lines.push(`   ├─ are we there yet? if so, run`);
```

this appears twice — once at line 278, once at line 297.

**i went to the example case.** the clone at `refs/example.case1.md` saw this exact text 21+ times. it never ran the command. it said "vision artifact ready. pending passage."

**what if the text is NOT the cause?**
- what if clones freeze because they're conditioned not to run commands without explicit human request?
- what if the problem is deeper: clones believe passage is a human action, not a clone action?
- what if no text change will fix a model-level belief?

**probing deeper:** the example clone didn't say "i don't understand what to do." it said "ready. pending." — it knew next step was passage, but believed *someone else* should do it.

**is text change enough?** text can clarify *what* to do. but if the clone believes it lacks *permission*, text won't help.

**counter-evidence:** the nudge adds "YOU run this command, don't wait" — this addresses permission explicitly. this is why the nudge matters more than the normal text change.

**verdict:** the normal text change alone might not be enough. the nudge is the real fix. assumption partially holds.

## assumption #2: "we can detect stuck state via hooks-without-attempt"

**what we assume:** we track hooks in `.route/drive.state.$stone.json`. after 5 hooks without `--as passed` attempt, we show nudge.

**i searched the code.** no such file exists. no `hooksWithoutAttempt` counter. this is new implementation.

**what could go wrong?**
- **when does counter reset?** the vision says "on route.stone.set attempt." but what counts as "attempt"?
  - does a *failed* attempt reset? (clone ran command, guard blocked)
  - does a *blocked* attempt reset? (clone ran command, approval required)
  - if yes: clone runs command, gets blocked, counter resets, must wait 5 more hooks for nudge. seems wrong.
- **what if clone is legitimately slow?** some stones require deep research. 5 hooks of real work isn't "stuck."
- **what about multiple stones?** if clone switches stones mid-work, does the counter persist?

**probing deeper:** the vision says "if counter >= 5: show nudge." but it doesn't say when counter resets to 0.

- if it resets on *any* attempt (even failed): clones who run command, fail guard, then hesitate start over at 0.
- if it resets only on *success*: stuck clones who keep trying but fail guard will never see nudge.
- if it never resets: counter grows forever, nudge appears permanently.

**issue found:** the vision doesn't specify reset conditions precisely.

**fix needed?** yes. clarify: "counter resets when stone passes or when clone runs `--as passed` (regardless of outcome)."

**verdict:** assumption needs refinement. reset conditions are ambiguous.

## assumption #3: "5 is the right number"

**what we assume:** 5 hooks without attempt = stuck.

**evidence:** none. the vision admits this.

**what if 5 is wrong?**
- for simple stones, 5 is a lot. clone could be stuck at hook 2.
- for complex stones, 5 is minimal. a thorough research stone might take 10+ hooks of real work.

**what does the wisher say?** the wish says "after 5 attempts of blockers without any attempt to run --as passed." the wisher chose 5.

**probing deeper:** is 5 arbitrary but stable? or will we tune it constantly?

**verdict:** 5 is wisher-specified, not junior-inferred. assumption is externally grounded. tune if needed.

## assumption #4: "nudge text will unstick clones"

**what we assume:** "YOU run this command, don't wait" is clear enough to change behavior.

**the nudge text:**
```
├─ 🐢 walk the way
│  ├─ your next step: mark stone as passed
│  ├─ YOU run this command, don't wait
│  ├─ run it to find out if you're done
│  └─ rhx route.stone.set --stone 1.vision --as passed
```

**what if clones still freeze?**
- the emoji 🐢 might be parsed as decoration, not signal.
- "walk the way" is metaphor. some clones parse literally: "walk where?"
- "YOU run this command" — what if clone reasons: "i am not a 'you', i am an assistant"?

**counter-evidence:** the nudge includes the literal command. even if clone ignores prose, it sees the executable.

**verdict:** nudge is robust. prose is flavor. command is explicit. assumption holds.

## assumption #5: "the 'are we there yet' phrase is shown to clones at all"

**what we assume:** clones see the route.drive hook output.

**i checked:** hook output appears in tool use results. clones see it.

**but what if context is compacted?** after compaction, clones see summary, not raw hook output.

**probing deeper:** does the summary include "are we there yet?" or just "route.drive showed guidance"?

**evidence:** compaction summaries mention route.drive but often elide details.

**implication:** if clones don't see the text after compaction, text changes won't help post-compaction clones.

**counter-evidence:** the nudge triggers after 5 hooks. that's 5 chances to see the text before compaction typically hits.

**verdict:** assumption mostly holds. early hooks reach clones. post-compaction is less certain.

## hidden assumptions found

### "clones understand 'as passed' semantics"

the vision assumes clones know that `--as passed` means "i claim done, system validates." but what if clones think `--as passed` means "i decree this is done, no check needed"?

**mitigation:** the nudge says "run it to find out if you're done." this teaches the validation model.

### "state file location is correct"

the vision says `.route/drive.state.$stone.json`. but current route artifacts use patterns like `.route/passage.jsonl`, `.route/*.guard.*.md`.

**consistency check needed:** is `drive.state.$stone.json` consistent with extant patterns? or should it be `.route/drive.$stone.state.json`?

### "the wisher wants both changes"

the wish mentions "after 5 attempts... show a more clear and assertive hook message." it doesn't explicitly mention the normal text change ("done? run this to proceed").

**did i infer?** yes. the vision added the normal text change as part of the fix. the wisher only asked for the nudge.

**is this wrong?** no — the normal text change is harmless improvement. the wisher's priority (nudge) is addressed. but i should note: normal text change was junior-added, not wisher-requested.
