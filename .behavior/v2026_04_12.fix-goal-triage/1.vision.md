# vision: actionable goal.triage.infer output

## the outcome world

### a day-in-the-life

a brain runs `rhx goal.triage.infer`. there are incomplete goals. the brain sees:

```
🦉 stillness. then clarity.

🔮 goal.triage.infer
   ├─ asks = 0
   ├─ uncovered = 0
   ├─ goals = 2
   │  ├─ complete = 0
   │  └─ incomplete = 2
   ├─ coverage = 0
   │
   ├─ incomplete goals
   │  ├─ git-repo-test-config-failfast [incomplete]
   │  │  ├─ absent: why.purpose, why.benefit, what.outcome, how.task, how.gate
   │  │  └─ to fix, run: `rhx goal.memory.set --slug git-repo-test-config-failfast --why.purpose "..."`
   │  │
   │  └─ git-repo-test-timeout-flag [incomplete]
   │     ├─ absent: why.purpose, why.benefit, what.outcome, how.task, how.gate
   │     └─ to fix, run: `rhx goal.memory.set --slug git-repo-test-timeout-flag --why.purpose "..."`
   │
   └─ triage required
      ├─ complete incomplete goals via `rhx goal.memory.set`
      └─ then re-run `rhx goal.triage.infer`
```

the brain knows exactly what command to run. no guessing, no reading help text, no searching.

note: `--scope` defaults to `route` when bound to a route, `repo` otherwise. typically brains work within routes, so no flag needed.

**before** — the output was descriptive but not actionable:

the brain doesn't know:

## changes

### 1. field flags already exist in goal.memory.set

goal.memory.set already supports field flags for partial updates:
```bash
rhx goal.memory.set --slug my-goal --why.purpose "enable fast test failures"
rhx goal.memory.set --slug my-goal --why.purpose "..." --why.benefit "..." --what.outcome "..."
```

supported flags: `--why.ask`, `--why.purpose`, `--why.benefit`, `--what.outcome`, `--how.task`, `--how.gate`

multiple flags can be used at once. no new feature needed — just emit these flags in triage output.

### 2. actionable guidance per goal in goal.triage.infer

each incomplete goal emits:
- the absent fields (extant behavior)
- the command to fill the first absent field (new, uses extant field flags)

why first field only?
- one step at a time = less cognitive load
- brain runs command, re-runs triage, sees next command
- avoids too many commands at once

### 3. actionable guidance per goal in goal.triage.next

**before** — generic hint at bottom:
```
└─ (2)
   ├─ slug = git-repo-test-timeout-flag
   ├─ why.ask = add --timeout flag for expected-fast test runs
   └─ status = enqueued → ✋ finish this first

     hint: `rhx goal.memory.get`
```

**after** — tip per goal with exact command:
```
└─ (2)
   ├─ slug = git-repo-test-timeout-flag
   ├─ why.ask = add --timeout flag for expected-fast test runs
   └─ status = enqueued → ✋ finish this first
      └─ tip: run `rhx goal.memory.get --slug git-repo-test-timeout-flag` to see the goal
```

same pattern as goal.triage.infer — show the exact command with slug filled in.

### 4. skill rename: `goal.infer.triage` → `goal.triage.infer`

**issue**: skill names should follow treestruct pattern with noun hierarchy first.
- `goal.triage.next` — correct (noun.noun.verb)
- `goal.infer.triage` — wrong (noun.verb.noun)

**fix**: rename to `goal.triage.infer` for symmetry:
- `goal.triage.infer` — infer triage state
- `goal.triage.next` — get next goal to work on

requires:
- rename shell entrypoint: `goal.infer.triage.sh` → `goal.triage.infer.sh`
- rename cli function: `goalInferTriage` → `goalTriageInfer`
- update hook command in `getAchieverRole.ts`
- update package.json exports if applicable

### 5. flag rename: `--mode` → `--when`

the `goal.triage.infer` skill changes:
- **before**: `--mode hook.onStop`
- **after**: `--when hook.onStop`

this aligns with `goal.triage.next` which already uses `--when hook.onStop`.

consistency: all hook-context flags use `--when`, not `--mode`.

### 6. bug fix: triage completeness must check status.choice

**bug discovered**: goals with `status.choice = 'incomplete'` but all fields filled are wrongly counted as "complete":

```
├─ complete goals
│  ├─ git-repo-test-config-failfast [incomplete]  ← WRONG
│  └─ git-repo-test-timeout-flag [incomplete]     ← WRONG
```

**root cause**: `getTriageState.ts` only checks field presence:
```ts
const goalsComplete = goals.filter(
  (g) => computeGoalCompleteness(g).complete === true,  // ignores status.choice!
);
```

**fix in `getTriageState.ts`** (not in `computeGoalCompleteness`):
```ts
// triage complete = not incomplete (already past triage stage)
const goalsComplete = goals.filter(
  (g) => g.status.choice !== 'incomplete',
);

// triage incomplete = status is incomplete (needs fields filled)
const goalsIncomplete = goals.filter(
  (g) => g.status.choice === 'incomplete',
);
```

**why not modify `computeGoalCompleteness`?**
- `computeGoalCompleteness` answers: "are all fields present?"
- this is needed to know when a goal CAN transition from `incomplete` → `enqueued`
- if we mix status into it, we lose that signal

**triage only cares about `incomplete` status:**
- `incomplete` = needs triage (fill fields, then transition to enqueued)
- `enqueued`, `inflight`, `blocked`, `fulfilled` = past triage stage, no action needed

**field completeness** only matters for goals with `status.choice === 'incomplete'` — to show which fields need fill before the goal can be enqueued.

### 7. update achiever role hook command

in `getAchieverRole.ts`, the onStop hook uses old name and `--mode`:
```ts
command: './node_modules/.bin/rhx goal.infer.triage --mode hook.onStop',
```

must change to:
```ts
command: './node_modules/.bin/rhx goal.triage.infer --when hook.onStop',
```

### 8. updated shell entrypoint header

```bash
# usage:
#   ./goal.triage.infer.sh                    # default scope (route if bound, repo otherwise)
#   ./goal.triage.infer.sh --scope repo       # explicit repo scope
#   ./goal.triage.infer.sh --when hook.onStop # hook mode
```

## user experience

### usecases & goals

| actor | goal | what they do |
|-------|------|--------------|
| brain | see incomplete goals | runs `rhx goal.triage.infer` |
| brain | complete a goal | copies command from output, fills in value, runs |
| brain | verify completion | re-runs triage, sees goal moved to "complete" |
| hook | halt on uncovered/incomplete | runs `--when hook.onStop`, exits 2 if issues |

### timeline: completing an incomplete goal

1. brain runs `rhx goal.triage.infer`
2. sees `git-repo-test-config-failfast` is incomplete, absent: `why.purpose`
3. sees command: `rhx goal.memory.set --slug git-repo-test-config-failfast --why.purpose "..."`
4. runs: `rhx goal.memory.set --slug git-repo-test-config-failfast --why.purpose "enable fast test failures"`
5. re-runs `rhx goal.triage.infer`
6. sees: absent: `why.benefit` with new command
7. repeats until goal is complete

### timeline: hook.onStop mode

1. hook runs `rhx goal.triage.infer --when hook.onStop`
2. if uncovered asks or incomplete goals: exits 2, shows same actionable output
3. if all clear: exits 0, silent

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| brain knows what command to run | yes | command shown per incomplete goal |
| no guesswork | yes | exact flags and slug in output |
| consistent flags | yes | `--when hook.onStop` matches other skills |
| backward compat | n/a | hard break — we control only consumer |

### pros

- zero ambiguity — brain sees exact command
- incremental — one field at a time, re-run to see next
- consistent — tree structure matches other skills
- copy-pasteable — command ready to execute

### cons

- slightly longer output per goal (worth it for clarity)
- `--mode` → `--when` is a break (but we control the only consumer)

### edgecases & pit of success

| edgecase | how we handle it |
|----------|------------------|
| goal has many absent fields | show command for first absent, brain re-runs after each |
| goal has no absent fields but wrong status | show status, no command (already complete) |
| all goals complete | show "all asks covered, all goals complete" (no commands) |
| hook mode | same actionable output to stderr, then exit 2 |

## what is awkward? (resolved)

### the `"..."` placeholder

`"..."` is vague. alternatives considered:
- `"<describe the purpose>"` — more instructive but longer
- `@stdin` — for longer values, but adds complexity

**decision**: `"..."` as minimal placeholder. universally understood as "fill this in". brain fills it inline.

### field-by-field vs all-at-once

re-run triage after each field feels tedious for goals with 5 absent fields. but all commands at once is too much.

**decision**: one field at a time. creates verification checkpoint. matches incremental workflow.

### breaking `--mode` → `--when`

extant hooks that use `--mode hook.onStop` will break. options:
1. hard break (update docs, update hooks)
2. soft break (alias `--mode` to `--when`, deprecation warning)
3. keep both forever

**decision**: hard break. we control the only consumer (`getAchieverRole.ts`). update hook in same PR. no external consumers to break.
