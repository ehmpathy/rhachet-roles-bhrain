# self-review: has-pruned-yagni

## review: what features are not needed right now?

### feature: configurable blockers threshold

**is it needed?** no

**why it can wait**: the `blockers > 5` threshold is hardcoded. could be configurable. but no user has requested this. default is reasonable. add configuration only when a real use case emerges.

**YAGNI**: configuration adds complexity without demonstrated need.

### feature: timeout or escalation ladder

**is it needed?** no

**why it can wait**: vision mentions "should blocked stones have a timeout or escalation ladder?" as an open question. no concrete use case exists. blocked state is already surfaced in route.drive. humans can act when they see it.

**YAGNI**: escalation automation is speculative.

### feature: notification to humans (email, slack)

**is it needed?** no

**why it can wait**: vision mentions this as an open question. route.drive already surfaces blocked stones. active notification can be built later if humans miss blocked stones.

**YAGNI**: notification infrastructure is significant scope for unvalidated need.

### feature: cleanup of triggered file after success

**is it needed?** no

**why it can wait**: triggered file is gitignored. no harm in its persistence. provides audit trail. cleanup is extra code for no user benefit.

**YAGNI**: cleanup adds complexity without user value.

### feature: validation of articulation file content

**is it needed?** no

**why it can wait**: blueprint requires articulation file to exist. could validate it contains required sections (what blocks me, what i tried, what i need). but this would add complexity. humans can reject shallow articulation via review.

**YAGNI**: content validation is scope creep. existence check is sufficient for MVP.

## conclusion

all YAGNI candidates identified. none are in the blueprint. blueprint contains only what is needed for MVP.
