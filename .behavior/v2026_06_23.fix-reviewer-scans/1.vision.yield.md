# vision: reviewer scans respect gitignore

## the outcome world

### before

reviewers crash when scanning repos that contain gitignored directories with restricted permissions:

```
EACCES: permission denied, scandir '.temp/genTempDir.symlink/2026-06-23T13-46-35.415Z.mvsafe-test.c583d639/sneaky-dir/snap-private-tmp'
```

the malfunction occurs because:
- `enumFilesFromGlob` uses `fast-glob` with `dot: true` (includes dotfiles)
- `fast-glob` does NOT automatically respect `.gitignore`
- only `node_modules` is explicitly excluded via `ignore` pattern
- `.temp` (and other gitignored dirs) are scanned, hitting permission errors

### after

reviewers respect gitignore when they enumerate **target files** — gitignored directories are never traversed.

- `.temp/`, `.env`, `dist/`, `coverage/` — invisible to target file enumeration
- permission errors become impossible (restricted files are excluded at glob level)
- reviews run faster (no wasted traversal of ignored directories)
- **rules and refs from `.agent/**/*` are always loaded** — never filtered by gitignore

### the "aha" moment

"oh — reviewers should only care about code that's in the repo. gitignored stuff is noise. but rules and refs must always load."

## user experience

### usecases

| usecase | glob pattern | result |
|---------|--------------|--------|
| review all ts files | `**/*.ts` | scans only git-tracked ts files |
| review src directory | `src/` | scans src, excludes any gitignored subdirs |
| review with .env present | `**/*` | `.env` excluded, no secrets scanned |

### contract inputs & outputs

> .note = the implemented contract evolved from the original sketch below.
>   two deliberate changes, both kept in sync with the code:
>   1. gitignore is decided by `git ls-files` (the source of truth), not by
>      globby's `gitignore` option — globby's internal walk lacked
>      `suppressErrors` and crashed on restricted dirs. globby still does the
>      glob walk; git decides what is ignored.
>   2. supplies do NOT apply gitignore at all. supplies are explicitly pointed
>      at via --rules / --refs, so they load whatever the glob matches — the
>      gitignored .agent/repo=* dirs and symlinked rule dirs alike. gitignore
>      is a subjects-only concern (subjects come from broad globs / diffs and
>      must skip build junk and restricted dirs). the glob walk uses
>      suppressErrors, so a restricted dir cannot crash a supplies scan either.
>      (an earlier `include` / `gitignore.except` allowlist was removed: a
>      symlinked non-.agent rules dir proved supplies must load whatever the
>      caller points at, so a gitignore filter on supplies was wrong.)

**enumFilesFromGlob** (low-level, updated):
- input: `{ glob, cwd?, gitignore? }`
- `gitignore: true` — exclude files git ignores (via `git ls-files`)
- `gitignore: false` (default) — return all glob matches (still excludes
  node_modules/.git, still suppressErrors)
- replace `fast-glob` with `globby` (for the walk; gitignore comes from git)

**enumFilesForReviewSupplies** (new wrapper for rules/refs):
- input: `{ glob, cwd? }`
- internally: `gitignore: false`
- loads whatever the glob matches, gitignored .agent/repo=* and symlinked
  rule dirs alike; never injects files the glob did not match

**enumFilesForReviewSubjects** (new wrapper for target files):
- input: `{ glob, cwd? }`
- internally: `gitignore: true`
- respects `.gitignore` fully

| function | use case | gitignore |
|----------|----------|-----------|
| `enumFilesForReviewSupplies` | rules, refs | `false` (load all glob matches) |
| `enumFilesForReviewSubjects` | target files | `true` (respect gitignore) |

### timeline

1. reviewer invoked via `rhx review` or guard
2. glob patterns passed to `enumFilesFromGlob`
3. files enumerated, gitignored paths excluded
4. review runs on relevant files only

## evaluation

### how well does it solve the goals

| goal | solved? |
|------|---------|
| no more EACCES crashes | yes — restricted paths never traversed |
| faster reviews | yes — no wasted traversal |
| intuitive behavior | yes — matches git semantics |

### pros

- eliminates entire class of permission errors
- faster scans (skips ignored dirs)
- matches user mental model (git-centric)
- no config needed — uses extant .gitignore

### cons

- new files must be `git add`ed before review (but this is correct behavior)
- adds globby dependency (replaces fast-glob, so net zero deps)

### edgecases & pit of success

| edgecase | behavior |
|----------|----------|
| nested .gitignore | respected (git semantics) |
| negated patterns (!pattern) | respected |
| untracked but not ignored | included (git add makes it reviewable) |
| symlink to ignored dir | symlink target checked, excluded if ignored |

### regression test

reproduce the exact defect scenario:

1. create test fixture at `src/domain.operations/review/.test/assets/example.gitignored/`
2. add `.gitignore` with `restricted/` entry
3. create `restricted/` dir with mode 000 (no read permission)
4. run `enumFilesForReviewSubjects({ glob: '**/*' })`

```ts
given('[case] gitignored directory with restricted permissions', () => {
  // fixture: example.gitignored/
  //   ├── .gitignore (contains: restricted/)
  //   ├── allowed.ts
  //   └── restricted/ (mode 000)
  //       └── secret.ts

  when('[t0] enumFilesForReviewSubjects scans **/*', () => {
    then('does not throw EACCES', async () => {
      // before fix: EACCES permission denied on restricted/
      // after fix: restricted/ excluded via .gitignore, no traversal
      const files = await enumFilesForReviewSubjects({
        glob: `${FIXTURE_PATH}/**/*`,
      });
      expect(files).toContain(`${FIXTURE_PATH}/allowed.ts`);
      expect(files.some(f => f.includes('restricted'))).toBe(false);
    });
  });
});
```

test fails before fix, passes after — defect cannot recur.

## what is awkward

### decision: globby replaces fast-glob

wisher decided: replace `fast-glob` with `globby`.

- `globby` has built-in `gitignore: true` option
- drop `fast-glob` entirely (not both deps)
- simple, well-maintained, minimal change

### scope complexity — solved

originally awkward: different glob calls need different gitignore behavior.

**solution**: two wrapper functions with clear names.
- `enumFilesForReviewSupplies` — rules/refs, always includes `.agent/**/*`
- `enumFilesForReviewSubjects` — target files, respects gitignore fully

callers pick the right function by name — no options to remember.
