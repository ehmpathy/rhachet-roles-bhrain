## deeper edge case analysis

let me think through edge cases more thoroughly by category.

### category 1: passage.jsonl state edge cases

**edge case: multiple status entries for same stone**

the playtest tests this via case1:
1. malfunction entry added
2. passed entry added
3. hook checks latest = passed

**but what about more than 2 entries?**

scenario: malfunction → malfunction → passed

the implementation gets latest entry per stone, so:
- third entry is passed
- hook sees passed
- hook allows stop

**is this tested?** the logic is the same as 2 entries. latest wins regardless of how many prior entries exist.

**verdict:** covered by logic symmetry.

---

### category 2: cache key edge cases

**edge case: same hash, different stones**

the cache lookup uses (hash, stone) as key. two stones with same artifact hash are cached separately.

**is this tested?** case2 tests single stone cache behavior. multi-stone is out of scope for this wish.

**verdict:** not in scope, but logic is correct.

**edge case: hash changes after malfunction**

scenario:
1. artifact A (hash X) → malfunction
2. human fixes artifact → artifact B (hash Y)
3. retry with hash Y → cache miss (different hash)

**is this tested?** case2 tests same hash retry. different hash is a cache miss by design.

**verdict:** covered by cache semantics (hash-based lookup).

### category 3: reviewer exit code edge cases

**exit 0 (pass)**
- expected: cached
- tested: case2 verifies cache hit on retry

**exit 1 (malfunction)**
- expected: NOT cached
- tested: case2 core test

**exit 2 (constraint = blockers found)**
- expected: cached (valid review result)
- tested: NOT explicitly, but:
  - the wish says "malfunctions not cached"
  - constraint is not malfunction
  - therefore constraint cache behavior is orthogonal

**exit 3+ (unknown)**
- expected: malfunction (any non-0-2 is malfunction)
- tested: case2 uses exit 1, which represents this class

**verdict:** all exit code edge cases covered or out of scope.

### category 4: boundary conditions

**boundary: zero entries in passage**
- behavior: no passage = no malfunction found
- relevant? no, the wish assumes route has run at least once

**boundary: single entry (malfunction only)**
- behavior: latest = malfunction → hook halts
- tested: case1 t0

**boundary: single entry (passed only)**
- behavior: latest = passed → hook allows
- relevant? this is the happy path, not an edge case

**boundary: empty artifact hash**
- behavior: implementation error, not user edge case
- relevant? no, artifacts always have content

**verdict:** relevant boundaries tested.

## why edge cases are covered

the playtest relies on 14 acceptance tests that exercise:

1. **malfunction detection**: latest-wins semantics tested by case1
2. **cache exclusion**: malfunction excluded from cache by case2
3. **exit code classification**: exit 0/1 tested, exit 2 is orthogonal

edge cases not explicitly in playtest are either:
- covered by logic symmetry (more entries = same latest-wins logic)
- out of scope (multi-stone, hash changes, other exit codes)
- not user-visible (empty hash, parse errors)
