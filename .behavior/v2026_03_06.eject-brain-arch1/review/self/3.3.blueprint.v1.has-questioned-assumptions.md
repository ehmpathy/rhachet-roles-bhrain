# self-review: questioned assumptions

## the question

for each technical decision in the blueprint, ask:
- what do we assume here without evidence?
- what if the opposite were true?
- is this architecture choice based on evidence or habit?
- what exceptions or counterexamples exist?
- could a simpler approach work?

---

## assumptions identified and questioned

### 1. "mv is the right operation"

**assumption:** we should use `mv` to relocate files.

**what if the opposite were true?** could use `cp` then delete. but that's 2 operations instead of 1.

**evidence:** the wish explicitly says "mv the files". mv is atomic.

**verdict:** ✓ assumption holds — mv is correct for ejection.

---

### 2. "the destination paths are correct"

**assumption:** files should land in these paths:
- `domain.objects/BrainArch1/` → same path in new repo
- `domain.operations/brain.replic.arch1/` → `domain.operations/arch1/`
- `architect briefs` → `.agent/repo=.this/role=architect/briefs/`

**what if the opposite were true?**
- could keep `brain.replic.arch1` name unchanged
- could put briefs elsewhere

**evidence:**
- rename to `arch1` simplifies the path (removes redundant `brain.replic` prefix since the whole repo is about brains)
- `.agent/repo=.this/role=architect/` is the rhachet convention for role-specific briefs

**verdict:** ✓ assumption holds — destination paths follow conventions.

---

### 3. "all 99 files must move together"

**assumption:** all files are interdependent and must move as a unit.

**what if the opposite were true?** some files might be independent and could stay or move separately.

**evidence checked:**
- grep for cross-dependencies confirmed tight coupling
- domain.objects are imported by domain.operations
- plugins import from core
- tests import from implementation

**verdict:** ✓ assumption holds — files are cohesive and should move together.

---

### 4. "the target repo exists and is ready"

**assumption:** `~/git/ehmpathy/rhachet-brains-bhrain` exists.

**what if the opposite were true?** the mv commands would fail.

**should we check?** the blueprint uses `$(realpath ...)` which will fail if path doesn't exist.

**improvement:** could add `mkdir -p "$INTO"` at the start of commands.

**verdict:** ⚠ assumption needs safeguard — add mkdir for target repo root.

---

### 5. "no files exist at destination"

**assumption:** destination paths are empty.

**what if the opposite were true?** mv would overwrite or fail based on options.

**evidence:** this is a new repo, so it should be empty. but the blueprint should be safe to re-run.

**improvement:** could add checks before mv, or use `mv -n` (no-clobber).

**verdict:** ⚠ assumption needs safeguard — consider idempotency.

---

### 6. "imports will break and that's expected"

**assumption:** after mv, imports like `@src/domain.objects/BrainArch1` will break and need update.

**what if the opposite were true?** if imports were external package refs, they wouldn't break.

**evidence:** currently imports are internal `@src/` paths. they will break. this is expected and documented in post-ejection tasks.

**verdict:** ✓ assumption holds — import breakage is expected and documented.

---

### 7. "test files should move with implementation"

**assumption:** `.test.ts` and `.integration.test.ts` files move with their implementations.

**what if the opposite were true?** tests could stay behind as integration tests for the dependency.

**evidence:** tests import internal implementation details. they must move with the code they test.

**verdict:** ✓ assumption holds — tests move with implementation.

---

### 8. "directory structure should mirror source repo"

**assumption:** we keep `src/domain.objects/`, `src/domain.operations/`, `src/domain.roles/` in the new repo.

**what if the opposite were true?** could flatten to `src/objects/`, `src/operations/`, `src/roles/`.

**evidence:**
- the `domain.` prefix is an ehmpathy convention for clarity
- other rhachet packages use this structure
- consistency aids navigation across packages

**verdict:** ✓ assumption holds — follow convention for consistency.

---

### 9. "brain role belongs in domain.roles/"

**assumption:** the brain role stays in `src/domain.roles/brain/` in the new repo.

**what if the opposite were true?** the brain role could be at root level since it's the primary role of this package.

**evidence:**
- other rhachet packages put roles in `domain.roles/`
- the role is just one component; the main value is in `domain.operations/arch1/`
- convention consistency > special-casing for "main" role

**verdict:** ✓ assumption holds — keep role in standard location.

---

### 10. "index.ts exports are not part of ejection"

**assumption:** no index.ts needs to move or be created as part of mv commands.

**what if the opposite were true?** roles-bhrain might have an index.ts that re-exports BrainArch1 types.

**evidence checked:**
```bash
# looked for index.ts files in ejected paths
# domain.objects/BrainArch1/ has no index.ts
# domain.operations/brain.replic.arch1/ has no root index.ts
```

**verdict:** ✓ assumption holds — no index.ts to move. new repo will need its own index.ts (documented in post-ejection tasks).

---

### 11. "git history doesn't need to be preserved"

**assumption:** using `mv` means git history for these files will not transfer to the new repo.

**what if the opposite were true?** could use `git filter-branch` or `git subtree split` to preserve history.

**evidence:**
- the wish says "mv the files" — explicit about mv, not history preservation
- new repo starts at 0.1.0 (fresh version sequence)
- history remains in roles-bhrain for reference
- git filter-branch is complex and error-prone

**alternative considered:** `git subtree split`
```bash
git subtree split -P src/domain.operations/brain.replic.arch1 -b arch1-history
```

**verdict:** ✓ assumption acceptable — mv is simpler, history stays in source repo for reference. if history is needed later, can use git subtree split retroactively.

---

### 12. "the ejection should happen all at once"

**assumption:** all 100 files move in a single operation.

**what if the opposite were true?** could eject incrementally (domain.objects first, then operations, etc.)

**evidence:**
- files are tightly coupled (operations import objects)
- incremental ejection would break the source repo between steps
- single operation is atomic — either all succeed or all fail

**verdict:** ✓ assumption holds — atomic ejection is safer.

---

### 13. "we only need to move typescript files"

**assumption:** only .ts files and .md briefs need to move.

**what if the opposite were true?** could be config files, test fixtures, or other assets.

**evidence checked:**
```bash
# looked for non-.ts files in ejected directories
# found: readme.md (in brain.replic.arch1/) — already in inventory
# found: .md briefs — already in inventory
# no other file types found
```

**verdict:** ✓ assumption holds — all file types are accounted for.

---

### 14. "genMockBrainArch1Context depends only on BrainArch1"

**assumption:** this file only imports from BrainArch1, so it moves.

**evidence checked:**
```typescript
import type { BrainArch1Context } from '@src/domain.objects/BrainArch1/BrainArch1Context';
import { genContextLogTrail } from './genContextLogTrail';
```

it also imports `genContextLogTrail`, which stays in roles-bhrain.

**what if we moved it?** the import `./genContextLogTrail` would break.

**improvement:** the new repo will need either:
- a copy of genContextLogTrail, OR
- a direct inline implementation, OR
- import from a shared package

**verdict:** ⚠ assumption needs update — must handle genContextLogTrail dependency.
