## error handler coverage

### rule.require.fail-fast

**question:** does blueprint specify error handlers for each operation?

**check:**

1. **getExitCodeClass** - pure classifier, no error cases. all inputs produce valid output.

2. **formatTreeBucket** - pure formatter. what if content is null/undefined?

   blueprint shows:
   ```
   ├─ [+] input: { label: string; content: string }
   ```

   content is typed as `string`, not `string | null`. what happens if caller passes undefined?

   **gap?** should blueprint specify validation?

   **analysis:** TypeScript type enforcement requires `string`. if caller passes undefined, it's a type error caught at compile time. no runtime validation needed.

   **verdict:** no gap

3. **runStoneGuardReviews** - executes shell commands.

   blueprint shows:
   ```
   ├─ [○] execa command with reject: false
   ├─ [+] capture exitCode as number
   ```

   `reject: false` means execa won't throw on non-zero exit. but what if execa itself fails (ENOENT)?

   **gap?** already identified in has-behavior-declaration-adherance review. edge case for future work.

4. **stepRouteDrive** - detects malfunction.

   blueprint shows:
   ```
   ├─ [+] on malfunction:
   │  ├─ emit escalation message
   │  └─ exit code 3 (immediate halt, not retry)
   ```

   this IS error handling. malfunction is an error state, we handle it by halt and escalate.

   **verdict:** error handler present
