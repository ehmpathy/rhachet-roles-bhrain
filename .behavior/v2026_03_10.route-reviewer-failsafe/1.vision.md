# vision: route reviewer failsafe

## the outcome world

### what does a day-in-the-life look like with this in place?

when a reviewer or judge fails during route.drive:
1. **the error is immediately visible** in both the CLI output and judge artifacts
2. **the failure reason is actionable** - you know WHY it failed, not just that it did
3. **the hook continues to fire** - even after failures, the next session/message triggers route.drive again
4. **recovery is clear** - the agent knows what to fix or what to escalate

### before/after contrast

| before | after |
|--------|-------|
| judge artifact: `exit code: 1` (no explanation) | judge artifact: full error message with stack trace |
| review artifact: just API key loading, no review | review artifact: clear failure reason if it crashed |
| hook stops firing after failure | hook continues to guide agent toward fix |
| human must dig through artifacts to debug | human sees problem immediately in CLI output |
| agent is stuck with no guidance | agent gets clear next-step instructions |

## user experience

### usecases fulfilled

1. **debugging failed reviews** - when a peer review command fails, the output clearly explains what went wrong
2. **understanding judge verdicts** - when a judge fails, the reason is parsed and displayed
3. **resuming after failure** - the hook keeps firing to guide the agent back on track
4. **escalating to humans** - when the agent can't fix the issue, the human gets clear context

### contract inputs & outputs

**reviewer/judge exit code contract:**

| exit code | meaning |
|-----------|---------|
| 0 | passed |
| 2 | ConstraintError - controlled failure (reviewer/judge worked, check failed) |
| other | MalfunctionError - reviewer/judge itself broke |

**artifact = forward stdout + stderr + exit code in tree buckets:**
```
🪶 judge N
   ├─ stdout
   │  ├─
   │  │
   │  │  [stdout from command]
   │  │
   │  └─
   ├─ stderr
   │  ├─
   │  │
   │  │  [stderr from command]
   │  │
   │  └─
   └─ passage blocked
      ├─ blocked by constraints | blocked by malfunction
      └─ exit code: [exit code]
```

- exit code 2 → "blocked by constraints" (reviewer/judge outputs `passed: false` and `reason:` in stdout)
- exit code != 2 → "blocked by malfunction" (capture whatever is there)

**malfunction behavior:**
- passage.jsonl records `{"stone": "...", "status": "malfunction"}`
- route.drive hook tells driver to halt
- route.drive hook tells user: "reviewer/judge malfunctioned, human must fix"
- driver cannot proceed until malfunction is fixed

**route.drive hook behavior** (improved):
- on constraint failure → continues to fire, guides agent to fix
- on malfunction → halts driver, escalates to human

### timeline

1. agent runs `route.stone.set --as passed`
2. guard runs reviews (peer review skill)
3. if review fails → artifact captures error details
4. guard runs judges
5. if judge fails → artifact captures reason
6. route.drive hook fires on next message
7. agent sees clear guidance: retry, fix, or escalate

## mental model

### how would users describe this to a friend?

"when the code review fails, it tells you why instead of just saying 'error'"

"the hook keeps bugging you until the stone passes or you fix the issue"

"you can actually debug failures without digging through cryptic artifacts"

### analogies

- **before**: a compiler that just says "error" with no line number
- **after**: a compiler that says "undefined variable 'foo' on line 42"

- **before**: GPS that loses signal and goes silent
- **after**: GPS that says "lost signal, last known position was X, try Y"

### terms

| user term | system term |
|-----------|-------------|
| "review failed" | peer review command exited non-zero |
| "judge failed" | judge mechanism returned `passed: false` |
| "stuck on stone" | route.drive block count > threshold |
| "hook stopped" | blocked by constraints / no bind / route complete |

## evaluation

### how well does it fix the goals?

1. **error clarity** - clear error messages in artifacts and CLI
2. **hook continuity** - hook fires until fix or escalation
3. **debuggability** - full context in artifacts for post-mortem
4. **recoverability** - agent knows how to retry or escalate

### pros

- silent failures become loud and observable
- artifacts contain full debugging context
- hook behavior is predictable and persistent
- agents can self-diagnose common issues

### cons

- more verbose output (tradeoff for clarity)
- artifacts may contain sensitive error details
- hook persistence may feel "nagging" if stuck

### edgecases and pit of success

| edgecase | pit of success design |
|----------|----------------------|
| review command doesn't exist | clear "command not found" in artifact |
| review times out | timeout captured in artifact with duration |
| review outputs to wrong path | fallback capture of stdout/stderr |
| judge can't parse review | explicit "parse failed" reason |
| hook fires in non-route repo | graceful "no route bound" message |
| constraint blocks skill | clear "blocked by X" with fix hint |

## open questions & assumptions

### assumptions we've made

1. **review skill should always produce output** - if it crashes before producing a review, that's a bug
2. **judge artifacts should be self-explanatory** - readers shouldn't need CLI context
3. **hooks should be persistent** - keep firing until fix or explicit stop
4. **errors are recoverable** - most failures can be fixed by the agent

### questions to validate

1. what caused the "blocked by constraints" message in the original failure?
2. should the hook have different behavior for "permission denied" vs "guard failed"?
3. how should we handle rate limits on review skills (API throttling)?
4. should stale artifacts be automatically cleaned up on retry?

## what is awkward?

### what feels off or forced?

1. **artifact is source of truth** - CLI output should just display the artifact, not have its own format
2. **hash-based caching** - complex logic for when to reuse vs re-run judges
3. **blocker types** - 'approval' vs other blockers have different hook behavior

### where does the design fight the user's mental model?

1. user expects "failed" to mean "something broke" but judge "failed" can mean "threshold exceeded"
2. user expects hook to "stop" but it persists (by design)
3. artifact format (passed/reason/stderr/stdout/command) is intuitive - no mental model conflict here

### tradeoffs that feel uncomfortable

1. **verbosity vs clarity** - long error messages are clearer but noisier
2. **persistence vs politeness** - hooks that keep firing can feel aggressive
3. **caching vs freshness** - cached judges are fast but may miss updates
