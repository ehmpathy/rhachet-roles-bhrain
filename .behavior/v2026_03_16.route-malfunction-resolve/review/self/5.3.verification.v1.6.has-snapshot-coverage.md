# self-review: has-snapshot-coverage (iteration 6)

## deeper reflection

the guide asks: "do snapshots capture input/output for caller visibility?"

let me question whether snapshots would actually help.

---

### what would snapshots capture?

**case1 [t0]:**
- stdout/stderr: "halted, guard malfunction..."
- exit code: 3

snapshot of this would show the malfunction message. but:
- the message is not a contract (implementation detail)
- exit code is the contract (3 = malfunction escalate)
- assertion `expect(combined.toLowerCase()).toContain('malfunction')` is sufficient

**case1 [t1]:**
- stdout/stderr: minimal (hook mode is silent on success)
- exit code: 0

snapshot of this would be empty or minimal. the contract is:
- exit code 0 = hook allows stop
- passage.jsonl has both entries

**case2 [t0]:**
- stdout: malfunction status in output
- exit code: non-zero

snapshot would show malfunction output. but the contract is:
- review artifact contains malfunction indicator
- exit code is non-zero

**case2 [t1]:**
- stdout: pass status
- exit code: 0

snapshot would show pass output. but the contract is:
- new artifact created (not cached)
- artifact contains pass indicator

---

### would snapshots help reviewers?

the repros artifact asked this question and concluded no:

> snapshots not required — assertions on exit codes and file contents are sufficient.

**why?**

1. the feature fixes internal state machine logic
2. user-visible output is not the contract fixed here
3. exit codes and artifact contents are the contracts
4. snapshots would capture incidental output, not the essence

---

### counter-argument considered

"but snapshots let reviewers see actual output without run code"

true, but for this feature:
- the output format was not changed
- the behavior (state transitions) was changed
- assertions on state are more precise than output snapshots

---

## conclusion

snapshots intentionally absent because:
1. feature fixes internal state logic, not output format
2. exit codes and artifact contents are the contracts
3. repros explicitly planned no snapshots
4. assertions are more precise than output snapshots for this use case
