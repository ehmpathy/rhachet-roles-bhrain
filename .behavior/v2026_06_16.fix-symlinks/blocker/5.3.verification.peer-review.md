## r4: ergo-snapshot-visual-blemishes (iteration 2)

### blocker claims

1. "Inconsistent cached status format" - reviews show 'approved, cached' while judges show '· cached'
2. "Visual noise with blank lines" - stdout/stderr blocks have blank `│  │` lines
3. "Inconsistent output formats" - guard tree vs artifact outputs have different structures
4. "Inconsistent indentation" - tree connectors after last-child use spaces instead of `│`

### why it does not apply

**all claimed blemishes are intentional design choices, not defects.**

1. **different cached formats**: reviews show status (approved/rejected) because that's their semantic - a review has a verdict. judges show `· cached` because they report results, not verdicts. the formats are contextually appropriate.

2. **blank lines in stdout/stderr**: the treestruct output format uses blank `│  │` lines as visual padding within sub.buckets. this is the documented pattern per `rule.require.treestruct-output.md`: "whitespace is required: blank `│` line below open and above close for visual space."

3. **different root structures**: `route.stone.set` outputs use the full structured tree format. artifact captures (stdout/stderr) use raw capture with tree wrap. these serve different purposes - one is formatted output, the other is raw capture.

4. **spaces after last-child**: after a `└─` connector, tree format correctly uses spaces instead of `│` because there is no peer continuation. this is standard tree format.

**these are all pre-extant patterns, not new defects introduced by this PR.**

git diff shows:
- formatGuardTree.test.ts.snap: added case16 for slug-based meters (clean output)
- other snapshot changes: reflect slug standardization changes, not new blemishes

the patterns flagged are foundational to how the treestruct output system works.
