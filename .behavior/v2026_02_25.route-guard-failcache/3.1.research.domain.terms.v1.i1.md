# research: domain terms

## summary

**no new domain objects required.**

the change is purely behavioral within the current `runStoneGuardJudges` operation. the cache lookup logic changes to "cache successes only" — no new entities, events, or literals needed.

---

## current domain objects (unchanged)

| object | type | purpose |
|--------|------|---------|
| RouteStoneGuardJudgeArtifact | entity | represents judge output with `passed: boolean` |
| RouteStoneGuard | literal | defines guard conditions |
| RouteStone | entity | represents a route milestone |

these objects already have all attributes needed for "cache successes only":
- `passed: boolean` — determines if result should be cached
- `hash: string` — determines cache key

## standard cache terminology (from industry)

### cache hit

> "A cache hit occurs when the requested data can be found in a cache."
> — [1] Wikipedia: Cache (computing)

### cache miss

> "A cache miss occurs when the requested data cannot be found in a cache."
> — [2] Redis Glossary

### selective cache (our approach)

not a standard term, but describes our pattern:

| result | cache behavior | term |
|--------|---------------|------|
| success (exit 0) | store, return on hit | **cache hit** |
| success (exit 0) | not stored yet | **cache miss** → execute → store |
| non-success | never stored | **cache skip** → always execute |

## parallel concepts from other domains

### culinary: mise en place vs fresh prep

- **mise en place** = pre-prepared ingredients ready to use (cached success)
- **fresh prep** = cut/prepare when needed (re-execute on miss)
- **spoiled ingredients** = discard, don't reuse (failed results)

**analogy:** only mise en place what stays fresh. spoiled ingredients are discarded, not stored.

### healthcare: test results

- **confirmed negative** = recorded, no retest needed (cached success)
- **inconclusive** = retest required (failed result, not cached)
- **positive** = triggers follow-up, not simply cached

**analogy:** inconclusive tests are repeated; only definitive results are recorded.

### construction: inspection stamps

- **passed inspection** = stamp goes in the log (cached success)
- **failed inspection** = must re-inspect after fix (not cached)
- **inspector absent** = reschedule, no record (transient error)

**analogy:** only passed inspections are recorded. failures require reinspection.

### bluecollar: quality control

- **approved parts** = move to inventory (cached success)
- **rejected parts** = rework queue (not cached, retry after fix)
- **machine malfunction** = no judgment made (transient error)

**analogy:** only approved parts enter inventory. rejected parts get another chance.

## proposed terminology

since no new domain objects are needed, the terminology change is in **comments and documentation**:

### before (current)
```ts
// cache all judges (passed or failed) - if judgeInputHash matches, reuse result
```

### after (proposed)
```ts
// cache only passed judges - failures retry fresh each attempt
```

## conclusion

the ubiquitous language already exists. the behavioral change ("cache successes only") uses standard cache terminology:
- **cache hit** = found prior passed result
- **cache miss** = no prior passed result, execute fresh
- **cache skip** = result was failure, never cached

no new terms to coin. implementation changes, terminology stays standard.
