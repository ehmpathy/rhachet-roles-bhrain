# research: external product claims

## summary

research on code review fatigue, tiered review patterns, budget/quota systems, and human-in-the-loop approval workflows. these patterns inform the peer-review-budgets design.

---

## claims

### code review fatigue and round limits

**[1] [FACT]** reviewer sessions capped at one hour prevent cognitive fatigue from degrading review quality. [practitioner, 2026]
> "reviewers should focus on limiting time spent to up to 1 hour per review session, after which it's easy for them to miss necessary quality checks"
- source: [Augment Code - Code Review Best Practices That Scale](https://www.augmentcode.com/guides/code-review-best-practices-that-scale)

**[2] [FACT]** defect detection peaks at 200 lines of code, then declines as reviewers fatigue. [practitioner, 2024]
> "Defect density peaks at 200 LOC, then declines as reviewers fatigue"
- source: [Super Productivity - Code Review Best Practices](https://super-productivity.com/blog/code-review-best-practices/)

**[3] [OPIN]** maximum turnaround on a review round should be one business day. [official, 2024]
> "the absolute maximum turnaround on a review round should be one business day"
- source: [Google Engineering Practices - Speed of Code Reviews](https://google.github.io/eng-practices/review/reviewer/speed.html)

**[4] [FACT]** 20-50 notes in a single review round overwhelms authors. [practitioner, 2021]
> "the danger zone generally begins in the range of 20-50 notes in a single round of review"
- source: [mtlynch.io - How to Do Code Reviews Like a Human](https://mtlynch.io/human-code-reviews-1/)

**[5] [KHUE]** if multiple rounds of reviews are common, consider root causes. [practitioner, 2026]
> "if multiple rounds of reviews are common, consider whether requirements are unclear, reviewers are being overly pedantic on minor issues, or authors are submitting under-cooked code"
- source: [Count.co - Code Review Cycle Time](https://count.co/metric/code-review-cycle-time)

### tiered review patterns

**[6] [SUMP]** a two-step review process distributes workload and ensures senior oversight. [practitioner, 2025]
> "a first review conducted by a peer developer, followed by code being passed to a lead or senior developer for final approval. Peer reviews distribute the workload, and senior reviews ensure that no single person is a bottleneck"
- source: [DEV Community - Scaling Code Reviews: A Two-Step Process](https://dev.to/martinrojas/scaling-code-reviews-a-two-step-process-for-large-engineering-teams-571o)

**[7] [OPIN]** senior reviewers should focus on architecture while peers focus on functionality. [practitioner, 2026]
> "Senior reviews add a final layer of polish, catching higher-level issues like redundancy or misaligned patterns. Peer reviews focus on functionality and edge cases"
- source: [Full Scale - Code Review Practices](https://fullscale.io/blog/code-review-practices-team-productivity/)

**[8] [FACT]** L5+ senior developers review APIs, class definitions, and crucial behavior changes. [official, 2024]
> "senior L5+ developers review APIs, class and interface definitions, or any other crucial behavior changes in the system"
- source: [sizovs.net - Code Review Best Practices](https://sizovs.net/code-review/)

### automated before human review

**[9] [FACT]** automation catches up to 65% of vulnerabilities early, reduces fix costs. [practitioner, 2026]
> "Code review automation accelerates development by 10–20%, catches up to 65% of vulnerabilities early, and reduces fix costs by identifying issues during coding instead of testing"
- source: [Sonar - What is Automated Code Review](https://www.sonarsource.com/resources/library/what-is-automated-code-review/)

**[10] [SUMP]** linting before human review eliminates style debates entirely. [practitioner, 2026]
> "The foundation of any automated review system is deterministic tooling that enforces consistency and catches syntax-level issues before they reach human reviewers. This layer eliminates style debates entirely"
- source: [Qodo - Best Automated Code Review Tools](https://www.qodo.ai/blog/best-automated-code-review-tools-2026/)

**[11] [FACT]** hybrid model is most effective: automation for breadth, humans for depth. [practitioner, 2026]
> "Human review remains essential; the most effective model is hybrid: automation for breadth (syntax, patterns, vulnerabilities) and humans for depth (architecture, business logic, contextual decisions)"
- source: [DEV Community - Automated Code Review Guide](https://dev.to/cpave3/automated-code-review-benefits-tools-implementation-2026-guide-5dgd)

### budget and quota systems

**[12] [FACT]** short-term rate limits protect servers; long-term quotas manage cost and monetization. [official, 2024]
> "Short term rate limits are focused on protecting servers and infrastructure from being overwhelmed within a short period of time, whereas long term quotas are focused on managing the cost and monetization of your API's resources"
- source: [Moesif - Best Practices for API Rate Limits](https://www.moesif.com/blog/technical/rate-limiting/Best-Practices-for-API-Rate-Limits-and-Quotas-With-Moesif-to-Avoid-Angry-Customers/)

**[13] [OPIN]** transparency is essential when designing rate limits as part of API contract. [practitioner, 2024]
> "Transparency is essential when designing rate limits as part of your API contract, including documenting your limits, how they're calculated, when they reset"
- source: [ByteByteGo - Design A Rate Limiter](https://bytebytego.com/courses/system-design-interview/design-a-rate-limiter)

**[14] [FACT]** token bucket approach has burst tolerance by design. [practitioner, 2025]
> "The token bucket approach has the key feature of burst tolerance by design, where accumulated tokens absorb legitimate traffic spikes without triggering rate-limit responses"
- source: [Redis - API Throttling Algorithms](https://redis.io/blog/api-throttling-algorithms-patterns/)

### LLM cost management

**[15] [FACT]** output and reasoning tokens cost 2-6x more than input tokens. [practitioner, 2026]
> "Output and reasoning tokens cost two to six times more than input tokens, and most teams don't realize how quickly they add up"
- source: [CodeAnt - Why Output & Reasoning Tokens Inflate LLM Costs](https://www.codeant.ai/blogs/input-vs-output-vs-reasoning-tokens-cost)

**[16] [OPIN]** a capable model that catches issues on first pass costs less than a cheaper model that misses problems. [practitioner, 2026]
> "For AI code review, a capable model that catches issues on the first pass costs less than a cheaper model that misses problems and requires multiple iterations"
- source: [MindStudio - AI Agent Token Budget Management](https://www.mindstudio.ai/blog/ai-agent-token-budget-management-claude-code)

**[17] [FACT]** intelligent model routing can cut LLM spend by 60-90%. [practitioner, 2026]
> "Intelligent routing layers that match each request to the least expensive model capable of delivering an acceptable answer can cut LLM spend by 60–90%"
- source: [Pluralsight - How to Cut LLM Costs](https://www.pluralsight.com/resources/blog/ai-and-data/how-cut-llm-costs-with-metering)

### developer productivity impact

**[18] [FACT]** teams lose 5.8 hours per developer per week to poor code review workflows. [practitioner, 2025]
> "Teams lose an average of 5.8 hours per developer per week to poor code review workflows"
- source: [Hatica - Painful Code Reviews Killing Developer Productivity](https://www.hatica.io/blog/painful-code-reviews-killing-developer-productivity/)

**[19] [FACT]** each context switch costs 15-25 minutes of productive time. [practitioner, 2024]
> "Each context switch costs 15-25 minutes of productive time according to research on developer productivity"
- source: [Graphite - How AI Code Review Reduces Review Cycles](https://graphite.com/blog/how-ai-code-review-reduces-review-cycles)

**[20] [FACT]** PRs under 400 lines see 30-50% reduction in time-to-first-review. [practitioner, 2026]
> "Teams that consistently keep PRs under 400 lines typically see a 30-50% reduction in time-to-first-review and a 40-60% reduction in total review cycle time"
- source: [Count.co - Code Review Cycle Time](https://count.co/metric/code-review-cycle-time)

### human-in-the-loop and escalation

**[21] [FACT]** automated escalation rules trigger when approvals are delayed beyond set duration. [practitioner, 2026]
> "Time-based escalation occurs when a request remains unapproved for a set duration (e.g., 24 hours) and is automatically escalated to the next level"
- source: [Cflow - Automated Escalation Rules](https://www.cflowapps.com/how-automated-escalation-rules-reduce-approval-bottlenecks/)

**[22] [SUMP]** human-in-the-loop is an architectural pattern for structured checkpoints. [official, 2025]
> "Human-in-the-Loop (HITL) is a design principle for AI systems that defines when and how humans review, approve, or override automated decisions"
- source: [SAP Community - Human-in-the-Loop SAP Agents](https://community.sap.com/t5/artificial-intelligence-blogs-posts/human-in-the-loop-sap-agents-approval-escalation-and-audit-series-2-part-5/ba-p/14372994)

**[23] [OPIN]** human override must be available for exception handler. [practitioner, 2026]
> "Human override and exception handling should be available, for example allowing approvers to pause escalation when they temporarily manage higher-priority work"
- source: [SBN Software - Automated Escalations in Permit Workflows](https://sbnsoftware.com/blog/how-to-configure-automated-escalations-in-permit-workflows/)

## convergence signals

**3+ sources agree**: tiered review (cheap/automated first, expensive/senior after) is effective
- sources [6], [7], [10], [11] all support run simpler checks before expensive reviews

**3+ sources agree**: review fatigue is real and requires limits
- sources [1], [2], [4] all document cognitive limits on review capacity

**3+ sources agree**: cost optimization via model/resource route is significant
- sources [15], [16], [17] all document 50-90% cost saves via intelligent route

---

## conflict detection

no significant conflicts detected. sources generally agree on:
- fatigue-based limits are necessary
- tiered review is effective
- hybrid automation+human is optimal
- budget/quota transparency is important

---

## anti-patterns

**[4], [5]** warn against: overwhelm authors with too many notes per round
> "the danger zone generally begins in the range of 20-50 notes"

**[10], [11]** warn against: use automation to replace human judgment
> "Human review remains essential... for depth (architecture, business logic, contextual decisions)"

**[18], [19]** warn against: slow review cycles that cause context switch
> "Each context switch costs 15-25 minutes of productive time"
