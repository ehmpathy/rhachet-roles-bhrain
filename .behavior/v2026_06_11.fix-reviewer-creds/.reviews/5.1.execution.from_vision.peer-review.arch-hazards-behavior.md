# blocker.3: Time assumptions via setInterval and wall-clock in withSpinner

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/reflect/stepReflect.ts
- src/domain.operations/review/stepReview.ts

withSpinner (duplicated in stepReflect and stepReview) captures Date.now() at start and runs setInterval(..., 100) to render elapsed time with  overwrites. This assumes stable wall-clock time (no skew, no NTP adjustments), predictable interval firing, and no blocking during the async operation. Clock changes or high load can produce negative/stuck elapsed times or un-cleared intervals, leading to intermittent display corruption.

**snippet**:
```ts
const withSpinner = async <T>(input: {
  message: string;
  operation: () => Promise<T>;
}): Promise<T> => {
  const frames = ['⠋', '⠙', '⠹', '⠸', '⠼', '⠴', '⠦', '⠧', '⠇', '⠏'];
  const startTime = Date.now();
  let i = 0;

  const render = (frame: string) => {
    const elapsed = Math.floor((Date.now() - startTime) / 1000);
    process.stdout.write(`\r   └─ elapsed: ${elapsed}s ${frame}  `);
  };

  render(frames[0]!);
  const interval = setInterval(() => {
    i = (i + 1) % frames.length;
    render(frames[i]!);
  }, 100);
```

The final atoms/repls arrays (and thus the ContextBrain) depend on the runtime return order of those provider functions and module initialization. Same input.brain can produce different effective brain lists or choice resolution across runs/environments.
