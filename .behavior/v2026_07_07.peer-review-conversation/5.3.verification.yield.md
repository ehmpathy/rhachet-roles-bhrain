### peer-review re-run trigger (the conversation convergence mechanism, per human direction)

the wish's core payoff — a reviewer converges after the driver responds — needs the reviewer to actually **re-run** once a `.taken` exists. the naive approach (fold the conversation into `computeStoneReviewInputHash`) was tried and **rejected**: a `.taken` is written to mirror its `.given`'s filename, which embeds the review hash, so a fold of taken content into the hash moves the hash off the given the taken answers — the contemplation gate (`getRouteGuardReviewPeerContemplationStatus`) could then never pair given↔taken. proven by `setStoneAsContemplated.test.ts [case2]`, which computes the hash before the taken and would orphan the given. so the hash stays **artifacts-only** (stable given↔taken pairs, no circularity).

instead the re-run is triggered in the **cache decision** (`runStoneGuardReviews.ts`, the "cached with blockers" branch): when a `.taken.by_self` newer than the cached `.given` exists, a reviewer that **opted into `--conversation`** falls through to re-run (with the conversation visible) rather than reuse the cached rejection. once it re-runs it overwrites the given (newer mtime), so the trigger clears and the loop is bounded by budget. a blind reviewer (no `--conversation`) always reuses — no wasted round.

- coverage: `runStoneGuardReviews.integration.test.ts` `[case24]` (conversation reviewer re-runs → rounds 2) + `[case25]` (blind reviewer holds cache → rounds 1) + `[case26]` (per-slug isolation: a `.taken` for conv-b re-runs only conv-b, conv-a holds cache) + `[case27]` (stale `.taken` older than the given → no re-run), each with a per-slug rounds+verdict snapshot; end-to-end via the `conversation-visibility` acceptance suite (drop vs re-raise).

### dogfood: the convergence proved live on this stone (per human direction, "all the way through the l3 reviewers too")

the feature was dogfooded on this very stone. at hash `i003` two L1 reviewers rejected: `repo-rules` (a real gap — the `blockedOn` journey test lacked per-step snapshots) and `mech-test-intent` (the goal-order snapshot, holds-as-intended). the driver:
1. **fixed** the repo-rules gap (added a sanitized snapshot to all 11 `[tn]` steps of `driver.route.drive.blockedOn.acceptance.test.ts`) and wrote a `.taken.by_self.repo-rules` that cites the fix + proof;
2. **articulated** a `.taken.by_self.mech-test-intent` that answers the order-swap challenge — verified against `setGoal.ts` (creation order = `update-readme-env` before `notify-slack-done`) and the test's preserved intent;
3. signalled `--as contemplated --that repo-rules` and `--that mech-test-intent`;
4. re-attempted `--as passed`.

both reviewers had exhausted their 3-round budget, so `route.guard.budget --for review --add 1` extended it (rounds 3 → budget 4). on the next `--as passed`, the cache-decision re-run trigger fired for **exactly** the two reviewers with blockers + a fresh `.taken` + `--conversation` (r1, r7) at a fresh iteration `i004`, while the 6 approved reviewers reused their `i003` cache. the outcome is the wish's core payoff, observed live:
- **`mech-test-intent`**: read the `.taken`, accepted the articulated rationale → **dropped the blocker → approved** (0 blockers, 0 nitpicks). the "aha moment": a reviewer converges because the driver explained it, no code change needed.
- **`repo-rules`**: read the `.taken`, saw the now-present snapshots → **dropped the blocker** (0 blockers), raised **1 nitpick** (`computeReviewerVerdict` duplicated the authoritative `computeReviewPeerVerdict`, a `rule.require.judge-derived-counts` single-source-of-truth breach).

### two extant test flakes deflaked (dirty-worktree perf, per human direction)

both were extant tests that timed out on a busy local worktree because `getGitBlobHashes` spawned one `git hash-object` per dirty file over the whole repo `src/**/*.ts`:
- `computeStoneReviewInputHash.test.ts [case2]` — rewritten to a hermetic `$route`-scoped temp fixture (a two-file artifact set) rather than the real repo src, so the hash depends only on the fixture. intent kept (hash derived from guard artifacts).
- `getGitBlobHashes` — the worktree-hash path now batches all files through one `git hash-object --stdin-paths` exec instead of one-per-file. behavior-safe (identical hashes; all 7 unit cases pass), and it kills the `[case6]` OOM-proof perf assertion timeout. the whole unit suite dropped 49s → ~28s as a side effect.

### L3 count-contract robustness — via the feature's own conversation channel
