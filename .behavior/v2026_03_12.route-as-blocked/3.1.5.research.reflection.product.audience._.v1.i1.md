# audience: route --as blocked

## who is the audience?

### primary: robots (AI agents)

robots that drive through behavior routes. they encounter obstacles outside their capability — design decisions that require human judgment, access to resources they lack, or domain expertise they do not have.

**context**: mid-route execution, stuck on a stone.
**constraint**: cannot proceed without human input.

### secondary: humans (route supervisors)

humans who supervise robot work. they return to terminals to check progress and need to know where robot needs help.

**context**: asynchronous supervision, may not monitor continuously.
**constraint**: time-limited, need clear signal where to focus.

### tertiary: hooks (automation systems)

automation that reads passage.jsonl to detect state changes. needs to know when a stone is blocked so it can trigger notifications or workflows.

**context**: event-driven, reads passage.jsonl.
**constraint**: needs machine-readable status, not prose.

---

## why do they care?

### robots

- **pain**: stuck with no clear way to signal it
- **goal**: explicit escalation path that is not shameful
- **frustration**: wasted time on guesses or hallucinations

### humans

- **pain**: return to find wasted time or broken work
- **goal**: know exactly where to step in
- **frustration**: unclear status — "did it give up? is it stuck? did it break?"

### hooks

- **pain**: no event to react to when robot is stuck
- **goal**: detect blocked state for automation
- **frustration**: silent failures that go unnoticed

---

## how much do they care?

| audience  | priority | urgency | stakes |
|-----------|----------|---------|--------|
| robots    | high     | high    | can proceed vs stuck forever |
| humans    | high     | medium  | wasted time vs focused help |
| hooks     | medium   | low     | notification vs manual check |

---

## what do they care about?

### robots

- **success criteria**: clear command to signal block, visible confirmation it worked
- **tradeoffs**: accept two-step flow (trigger → confirm) for safety
- **delight**: articulation forces reflection, often unblocks themselves

### humans

- **success criteria**: blocked state visible in route.drive, reason accessible
- **tradeoffs**: accept articulation file over inline reason
- **delight**: robot articulates clearly: what blocks, what tried, what needs

### hooks

- **success criteria**: `status: 'blocked'` in passage.jsonl
- **tradeoffs**: accept no additional metadata beyond reason path
- **delight**: consistent format with other statuses

---

## how will they experience it?

### robot journey

1. work on stone
2. hit genuine obstacle
3. run `rhx route.stone.set --stone X --as blocked`
4. see nudge: "failure is only the opportunity to begin again"
5. see articulation guidance
6. articulate into `$route/.route/blocker/$stone.md`
7. run `--as blocked` again
8. see confirmation: "passage = blocked"

**friction points**: two-step flow adds friction. mitigated by nudge content that often unblocks them.

### human journey

1. return to terminal
2. run `rhx route.drive`
3. see blocked stone with path to articulation
4. read articulation file
5. provide guidance
6. robot continues

**friction points**: must read separate file for full context. mitigated by clear path in stdout.

### hook journey

1. read passage.jsonl
2. detect `status: 'blocked'`
3. trigger notification or workflow

**friction points**: none. same format as other statuses.
