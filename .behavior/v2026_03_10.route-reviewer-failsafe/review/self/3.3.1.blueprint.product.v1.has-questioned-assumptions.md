# self-review: has-questioned-assumptions

## technical assumptions questioned

### assumption 1: exit code 2 = ConstraintError

**what we assume:** exit code 2 signals controlled failure (reviewer/judge worked, check failed).

**evidence:** extant code in runStoneGuardJudges uses exit 2 for blocked judges:
```ts
'bash -c \'echo "passed: false"; exit 2\''
```

**what if opposite?** some tools use exit 2 for different purposes.

**verdict:** this is a design contract from the vision, not an unvalidated assumption. the exit code values (0/2/other) are explicitly defined in the criteria.

---

### assumption 2: malfunction should halt immediately

**what we assume:** on malfunction, route.drive should exit 3 (no retry).

**evidence:** the wish says "the hook never fired again" - but this was a BUG, not desired behavior for constraints. for malfunctions, immediate halt IS desired.

**what if opposite?** retry malfunctions like constraints?

**counterexample:** a malfunction means the reviewer/judge process itself broke (syntax error, absent dependency). retry won't fix it. human must intervene.

**verdict:** assumption holds. immediate halt on malfunction is correct.

---

### assumption 3: tree bucket format won't break parsers

**what we assume:** artifacts are for human consumption, not parsed by code.

**evidence:** judgeReviewed looks for review FILES by hash, not artifact CONTENT.

**what if opposite?** if any code parses artifact content, format change would break it.

**investigation:** reviewed judgeReviewed code - it checks file existence, not content format.

**verdict:** assumption holds. format change is safe.

---

### assumption 4: CLI displays guard results, not artifacts

**what we assume:** formatRouteStoneEmit receives results from runStoneGuardJudges/Reviews.

**evidence:** from code review, formatRouteStoneEmit takes guard results as input parameter.

**what if opposite?** if CLI read artifacts directly, we'd only need artifact changes.

**verdict:** assumption verified. CLI displays results directly, so both artifact and result changes needed.

---

### assumption 5: passage.jsonl is needed for malfunction persistence

**what we assume:** route.drive reads passage.jsonl to detect malfunction across sessions.

**evidence:** route.drive hook fires on each message. must persist state between fires.

**what if opposite?** detect malfunction from guard result in same session?

**counterexample:** if session ends mid-guard (timeout, crash), malfunction state would be lost without persistence.

**verdict:** assumption holds. passage.jsonl persistence is required.

---

### assumption 6: formatTreeBucket should be shared, not inline

**what we assume:** 6 call sites justify a shared function.

**evidence:** used for stdout AND stderr in 3 files (formatRouteStoneEmit, runStoneGuardReviews, runStoneGuardJudges).

**what if opposite?** inline ~10 lines in each call site?

**counterexample:** 60 lines of duplicated logic vs 10 lines in shared function.

**verdict:** assumption holds. shared function is simpler.

---

### assumption 7: classifyExitCode should be a function, not a lookup

**what we assume:** a 3-line function is the simplest form.

**what if opposite?** use a lookup table:
```ts
const CLASS: Record<number, ExitCodeClass> = { 0: 'passed', 2: 'constraint' };
return CLASS[code] ?? 'malfunction';
```

**evaluation:** similar complexity. the if/else is more readable for 3 cases. lookup tables shine with more cases.

**verdict:** either approach works. current function is fine.

---

### assumption 8: signals (exit > 128) should be malfunction

**what we assume:** any exit code != 0 and != 2 is malfunction.

**evidence:** signals like SIGKILL (137) or SIGINT (130) result in exit codes > 128.

**what if opposite?** treat signals differently from errors?

**counterexample:** if a reviewer is killed by OOM, exit code 137 is correct to treat as malfunction - the reviewer broke.

**verdict:** assumption holds. signals are a type of malfunction.

---

### assumption 9: artifacts will be written before crash

**what we assume:** the artifact is written before the process exits.

**evidence:** artifacts are written synchronously in runStoneGuardJudges/Reviews.

**what if opposite?** if the process crashes mid-write, artifact might be incomplete.

**verdict:** this is an edge case beyond scope. we're improving visibility, not crash recovery. assumption is acceptable.

---

### assumption 10: guard commands can be any executable

**what we assume:** exit code capture works regardless of command type (bash, node, python).

**evidence:** execa captures exit code from any child process.

**what if opposite?** if some commands behave differently?

**verdict:** assumption holds. exit codes are OS-level, not language-specific.
