# research: OSS levers — achiever role

## context

the achiever role requires minimal external dependencies. the architecture uses:
- file-based persistence (YAML for goals, JSONL for logs)
- content hash for ask identification
- shell skills for CLI interface
- hooks for event-driven behavior

this research evaluates OSS tools for each capability.

---

## category 1: YAML parse/serialize

### [1] js-yaml

**source:** github.com/nodeca/js-yaml

**type:** [official] — npm package

**stats:** 35M+ weekly downloads, actively maintained

**quote:** "YAML 1.2 parser and serializer for JavaScript, supports custom types and schema."

**pros:**
- industry standard for Node.js YAML
- supports custom types
- well-documented
- no native dependencies

**cons:**
- large API surface for simple use cases

**recommendation:** use js-yaml — it's the de facto standard.

---

### [2] yaml (eemeli/yaml)

**source:** github.com/eemeli/yaml

**type:** [official] — npm package

**stats:** 15M+ weekly downloads

**quote:** "YAML 1.2 parser and stringifier with support for comments and custom types."

**pros:**
- preserves comments (useful for human-edited files)
- modern API

**cons:**
- less adoption than js-yaml

**recommendation:** consider if comment preservation is needed.

---

## category 2: JSONL append

### [3] Node.js fs.appendFile

**source:** nodejs.org/api/fs.html

**type:** [official] — Node.js built-in

**quote:** "asynchronously append data to a file, create the file if it does not exist."

**pros:**
- built-in, no dependency
- atomic append (with flags)
- well-tested

**cons:**
- low-level API

**recommendation:** use fs.appendFile — no external dependency needed.

---

### [4] append-json-file (npm)

**source:** npmjs.com/package/append-json-file

**type:** [practitioner] — npm package

**quote:** "append a JSON object to a file as a new line."

**pros:**
- simple API
- handles newline

**cons:**
- unmaintained (last update years ago)
- unnecessary abstraction

**recommendation:** avoid — use fs.appendFile directly.

## category 3: content hash

### [5] Node.js crypto

**source:** nodejs.org/api/crypto.html

**type:** [official] — Node.js built-in

**quote:** "crypto module provides cryptographic functionality that includes hash functions."

**example:**
```js
const hash = crypto.createHash('sha256').update(content).digest('hex').slice(0, 8);
```

**pros:**
- built-in, no dependency
- supports multiple algorithms
- fast (native implementation)

**cons:**
- verbose API

**recommendation:** use crypto — no external dependency needed.

---

### [6] hash-fns (ehmpathy)

**source:** npmjs.com/package/hash-fns

**type:** [practitioner] — npm package

**quote:** "simple hash functions for common use cases."

**pros:**
- simple API: `toHashSha256Sync(content)`
- consistent with other ehmpathy packages

**cons:**
- external dependency

**recommendation:** consider if repo already uses ehmpathy packages.

## convergence signals

### file-based persistence is sufficient

**sources [3, 7, 21] agree:** file-based persistence (fs/promises, YAML, JSONL) is sufficient for goal storage. no database needed.

### built-ins over dependencies

**sources [3, 5, 7, 17] agree:** Node.js built-ins (fs, crypto, Date) cover most needs. minimize external dependencies.

### zod for validation

**sources [13, 14] agree:** schema validation is valuable. zod is preferred for TypeScript projects.

---

## anti-patterns from sources

### anti-pattern 1: vector databases for structured data

**achiever stores structured goals, not embeddings.** vector databases (pinecone, chroma) are overkill. file-based YAML is sufficient.

### anti-pattern 2: ORM for file-based storage

**achiever uses files, not a database.** ORMs (prisma, typeorm) don't apply.

### anti-pattern 3: over-abstraction

**sources [4] warn:** unnecessary abstraction layers (e.g., append-json-file) add dependencies without value.

## final recommendations

| capability | recommendation | rationale |
|------------|----------------|-----------|
| YAML parse | js-yaml | industry standard, well-maintained |
| JSONL append | fs.appendFile | built-in, no dependency |
| content hash | crypto | built-in, fast |
| file operations | fs/promises | built-in, async |
| glob patterns | fast-glob | fast, supports ignore |
| shell skills | bash | universal, simple |
| schema validation | zod | TypeScript-first, great errors |
| CLI args | minimist | minimal, focused |
| timestamps | Date.toISOString() | built-in, ISO format |
| domain objects | domain-objects | if repo uses DDD patterns |
