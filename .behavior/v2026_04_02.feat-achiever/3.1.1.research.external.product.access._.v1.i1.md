# research: external product access

## summary

the achiever role requires **no external remote access**. all persistence is local filesystem-based:
- `.goals/` directories for goal storage
- YAML files for goal content
- JSONL files for append-only logs (asks inventory, coverage)
- flag files for status markers
- git for branch detection

this is an intentional design choice aligned with industry best practices for AI agent memory systems.

---

## lesson.1: file-based persistence is industry standard for ai agent memory

multiple sources confirm that file-based storage is the preferred approach for AI agent goal and memory persistence.

> "File-based memory ensures persistence where files decouple memory from process lifecycle with no data loss when a process dies, and memory files are transparent and editable in any text editor to see exactly what the agent knows." [1]

> "Because memory is plain text, it lives in Git repositories, allowing you to commit the agent's 'knowledge,' revert changes if the agent goes off-rails, and branch the memory." [1]

> "A 4-layer file-based memory architecture provides persistent, scalable, and accessible memory for AI agents, developed after experimenting with in-memory caches to database-backed solutions." [2]

**convergence signal:** sources [1], [2], [3] all recommend file-based persistence over database solutions for AI agent memory.

## lesson.2: layered storage architecture separates concerns

best practice is to separate storage into distinct layers by purpose and update frequency.

> "A persistent agent needs three separate storage strategies: Working State (current task, active context, runtime flags, overwritten each session), Event Memory (append-only storage of what happened and what was learned), and Identity/Config (core parameters, behavioral policies, long-term goals that are rarely updated)." [3]

**application to achiever:**
- `asks.inventory.jsonl` = event memory (append-only)
- `asks.coverage.jsonl` = event memory (append-only)
- `*.goal.yaml` = identity/config (goals, rarely updated)
- `*.status=*.flag` = runtime state (frequently updated)

## lesson.3: autogpt removed external vector db in favor of local files

notably, AutoGPT evolved away from external database dependencies.

> "The AutoGPT team found that typical agent runs didn't actually generate enough distinct facts to require an expensive vector index, and by late 2023 they removed external vector DB support entirely, defaulting to a simple local file for memory storage." [4]

> "BabyAGI leveraged Pinecone for long-term memory, meaning it would embed and store results so that when the task list looped back around, it could recall prior outcomes relevant to new tasks." [5]

**conflict noted:** BabyAGI uses vector DB [5] while AutoGPT removed it [4]. the difference: BabyAGI optimizes for semantic retrieval across many tasks; AutoGPT found simple file storage sufficient for typical workloads.

**application to achiever:** goal tracking does not require semantic search — simple enumeration and hash-based lookup suffices. local files are appropriate.

## lesson.4: jsonl is ideal for append-only event logs

JSONL format is the industry standard for append-only logs and state management.

> "Append-Friendly: New records can be easily appended to the end of a JSONL file. This is crucial for log files, as you just append a new line to the end of the file with no need to modify extant content." [6]

> "Memory Efficiency: JSONL is designed for streaming. You can process data as it arrives, making it perfect for real-time analytics, log processing, and API streaming. JSONL allows processing line by line, using only the memory needed for one record." [6]

> "Data Safety: An append-only log reduces the risk of data corruption if the process crashes during a write (no half-written JSON files replacing valid ones)." [7]

> "JSONL's line-based format makes error handling straightforward. Best practices include: Parse line-by-line - Wrap each JSON.parse() in try-catch, Log errors - Record line number and raw content for debugging, Skip or fail - Choose whether to continue processing or halt on error." [6]

**convergence signal:** sources [6], [7], [8] all recommend JSONL for append-only event logs.

## lesson.5: yaml is appropriate for structured configuration and data

YAML is the standard for human-readable structured data files.

> "Use consistent indentation with spaces (usually two or four spaces throughout your file). More specifically, use spaces only — never tabs, as YAML parsers treat tab characters as errors, and two spaces is the community standard." [9]

> "Use descriptive key names to make your configuration smooth and easy to understand by other people." [9]

> "Quote values that look like booleans, numbers, or nulls but aren't, and use only true/false for booleans to avoid YAML 1.1 / 1.2 differences." [10]

> "Use | for content where newlines matter, > for flowing prose." [10]

**application to achiever:** `*.goal.yaml` files use YAML for human-readable goal content with multi-line fields (ask, task, gate).

## lesson.6: filesystem vs database tradeoffs favor filesystem for this usecase

the achiever role's requirements align with filesystem strengths.

> "Performance can be better than database storage, and accessing files in a filesystem is quite simple and lightweight. Filesystems have great sequential read/write throughput and are well suited to append-only logs and single-user scratchpads." [11]

> "Filesystems are ubiquitous, simple, and portable, with very low overhead to mounting a local filesystem." [12]

> "If your application has simple access patterns and storage requirements, a filesystem can work just fine." [11]

**anti-patterns noted:**

> "There are no ACID (Atomicity, Consistency, Isolation, Durability) operations in file systems, which means there is no guarantee." [13]

> "File system does not offer concurrency, whereas the DBMS provides a concurrency facility." [13]

**application to achiever:** single-agent, single-session access pattern. no concurrent writes. ACID not required — append-only logs are naturally crash-safe.

## lesson.7: content-addressable storage via hashes provides integrity

git's content-addressable model validates the hash-based ask identification design.

> "Git is a content-addressable filesystem. At the core of Git is a simple key-value data store where you can insert any kind of content into a Git repository, for which Git will hand you back a unique key you can use later to retrieve that content." [14]

> "If two pieces of content are identical, they receive the same identifier, and any change, no matter how minor, produces a new hash." [15]

> "Because objects are identified and stored based on a hash of their contents, you will never have duplicate copies of the same objects. Integrity checking is easy, as bit flips are easily detected when the hash of corrupted content does not match its name." [15]

**application to achiever:** asks are identified by content hash — deterministic, no counter needed, natural deduplication.

## lesson.8: flag files are a proven unix pattern for state markers

flag files are an established pattern for lightweight state indication.

> "Flag files can be used to create control mechanisms for state management, such as suspending warnings and alarms during system maintenance." [16]

> "Flag files must follow a naming convention, typically formatted as ignore_ProcessName, where ProcessName is the monitored process name." [16]

> "File system 'flags' alter state at startup and are used to persist changes between restarts if necessary." [17]

**application to achiever:** `*.status=$choice.flag` files encode status in filename — visible without file parse, glob-friendly.

## lesson.10: event sourcing validates append-only log pattern

event sourcing provides theoretical foundation for the append-only design.

> "Instead of storing only the current state of data in a relational database, event sourcing stores the full series of actions taken on an object in an append-only store, which acts as the system of record that can be used to materialize the domain objects." [20]

> "Write throughput improves, especially for the presentation layer, because append-only writes avoid the row-level lock contention that update-in-place systems create." [20]

> "Every change is recorded, enabling full traceability." [21]

> "In an event-sourced system you can replay the history into a new database and restore everything up to the point of the last event, whereas in a CRUD system a corrupted database might mean irreversible loss of state." [20]

**application to achiever:** `asks.inventory.jsonl` and `asks.coverage.jsonl` are append-only event logs — full audit trail, no data loss on crash.

## lesson.11: atomic write-rename pattern ensures safe persistence

the write-rename pattern prevents data corruption on crash.

> "The Create-Write-Rename pattern is the gold standard for safe file persistence in application development." [22]

> "An atomic write means that the process of updating a file is seen as a single, indivisible operation by other programs or threads. If the write process is interrupted, the original file either remains completely unchanged, or the new file is fully and correctly in place." [22]

> "Write all the data to a new temporary file in the same directory, close the temporary file (ensuring all data is flushed to disk), then rename/move the temporary file over the final destination file." [23]

**application to achiever:** goal file updates should use write-temp-rename pattern for crash safety.

## lesson.12: git branch-aware storage is established pattern

storing configuration per-branch is a recognized git pattern.

> "Git supports `includeIf` conditions with `onbranch` patterns - if the name of the currently checked out branch matches the pattern, the include condition is met." [24]

> "A highly recommended pattern involves separating baseline settings from machine-specific overrides." [25]

**application to achiever:** `.goals/$branch/` provides branch-specific goal storage. goals on main are forbidden to prevent accidental commits.

## conclusion

the achiever role's local filesystem persistence design is well-supported by industry practice:

| design choice | validation |
|---------------|------------|
| file-based persistence | AutoGPT removed vector DB; industry recommends files for agent memory [1,2,3,4] |
| JSONL for append-only logs | standard format, crash-safe, memory-efficient [6,7,8] |
| YAML for goal content | human-readable, git-friendly [9,10] |
| hash-based ask identification | git content-addressable model validates pattern [14,15] |
| flag files for status | proven unix pattern [16,17] |
| branch-specific storage | established git configuration pattern [24,25] |
| append-only event logs | event sourcing pattern validates audit trail [20,21] |

**no external remote access required.** the design is self-contained, portable, and aligned with best practices.
