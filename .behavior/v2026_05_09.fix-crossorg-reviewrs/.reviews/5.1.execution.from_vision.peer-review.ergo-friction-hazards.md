# blocker.3: enumFilesFromGlob lacks acceptance tests for error paths

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/domain.operations/review/enumFilesFromGlob.integration.test.ts

The function enumFilesFromGlob is tested for positive cases and edge cases like empty results, but there are no acceptance tests covering error scenarios such as invalid glob patterns, non-existent directories, or permission issues. Friction hazards must be covered with acceptance tests; untested error paths risk shipping broken user experiences. For example, if the glob is malformed, fast-glob may throw an error that is not clearly communicated or handled.

**snippet**:
```ts

  given('[case1] a positive glob pattern', () => {
    when('[t0] glob matches multiple files', () => {
      then('returns all matched files sorted', async () => {

          glob: `${ASSETS_PATH}/**/*.ts`,
        });
        expect(files).toEqual([
          `${ASSETS_PATH}/bar.ts`,
          `${ASSETS_PATH}/foo.ts`,
          `${ASSETS_PATH}/nested/qux.ts`,
        ]);
      });
    });
```
