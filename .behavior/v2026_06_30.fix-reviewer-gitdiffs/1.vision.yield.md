# 1.vision — fix reviewer gitdiffs

## the wish, distilled

reviewers currently receive **full file content** for every file touched by a diff, but
**no visibility into what actually changed in this pr vs priors**. they see the whole file,
flat, with no signal about which lines are new work and which are prior context.

the fix: keep giving reviewers full file content (they need the context), but **also** hand them
the **exact diff** — so they can see what changed, focus their attention there, and avoid
churning code this pr never touched.

> "fix forward, but dont churn whats not touched"

context budget is not a concern (1mm window) — we can afford both the diff and the full content.

---

## the outcome world

### day-in-the-life

a reviewer (the brain, invoked via `review` / a guard) is asked to review a pr.

**before:** the reviewer gets a `<target>` section with each file's full content in a fenced block.
it reads the whole file. it has no idea which 4 lines out of 400 were touched by this pr. so it
either (a) reviews everything and recommends fixes to long-standing code the author never touched,
creating churn and noise, or (b) guesses at what changed and reviews blind.

**after:** the reviewer gets, for each changed file, **both**:
1. the **diff hunks** — exactly what this pr added/removed vs the merge-base
2. the **full file content** — for surrounding context

now it knows: "these are the lines that changed, this is the file they live in." it focuses its
blockers and nitpicks on the changed lines, uses the rest of the file purely as context, and stops
recommending rewrites of untouched code.

### before/after contrast

| dimension | before | after |
|-----------|--------|-------|
| what changed | invisible | explicit diff hunks |
| full context | yes | yes (unchanged) |
| reviewer focus | whole file | changed lines, file as context |
| churn risk | high — flags untouched code | low — primed to fix-forward |
| signal/noise | low | high |

### the "aha" moment

the value clicks when a reviewer **stops flagging a prior pattern** in an untouched part of a
file — because the diff made it obvious that block wasn't part of this pr — and instead writes a
tight, relevant blocker on the three lines that actually changed.

## user experience

### usecases & goals

| actor | goal |
|-------|------|
| guard-driven review (route stones) | review only this pr's changes, with full context, low churn |
| ad-hoc `review --diffs since-main` | "what did i change since main? review it, in context" |
| pr-gate review | focus feedback on the delta, not the whole touched file |

the primary goal: **prime the reviewer to focus on the delta** while still trusting it with full
file context.

### contract inputs & outputs

inputs are **unchanged** — this is a presentation/enrichment change, not a new flag:

- `--diffs since-main | since-commit | since-staged` — already selects changed files
- `--paths <glob>` — already selects files by path
- `--focus pull | push` — already chooses paths-only vs content-injected

what changes is what the reviewer **sees** inside `<target>`: when a target file has an associated
diff (it came from `--diffs`, or it's a changed file), the prompt now carries that file's diff
hunks **in addition to** its full content.

### what it looks like to leverage

```bash
# guard / ad-hoc: review changes since main, in full context
rhx review \
  --rules '.agent/**/rule.*.md' \
  --diffs since-main \
  --focus push \
  --output .review/out.md
```

the reviewer's `<target>` section now reads (sketch):

```
## target (REVIEW THIS SECTION ONLY)

<target>
### src/domain.operations/review/compileReviewPrompt.ts

#### diff (what changed in this pr — focus here)
```diff
@@ -49,6 +49,12 @@
   // push focus: inject contents
   return input.targets
-    .map((target) => `### ${target.path}\n\n\`\`\`\n${target.content}\n\`\`\``)
+    .map((target) => renderTargetWithDiff(target))
     .join('\n\n---\n\n');
```

#### full content (for context — do not churn unchanged lines)
```
<...entire file...>
```
</target>
```

### timeline

1. caller invokes `review` with `--diffs since-main` (as today)
2. system enumerates changed files **and now also captures each file's diff hunks**
3. system reads full content (as today)
4. prompt compiler renders, per changed file: diff hunks **then** full content
5. reviewer reads diff → orients → reads content for context → emits delta-focused findings

## mental model

### how a user describes it to a friend

> "code review used to dump the whole file on the reviewer with no clue what i actually changed.
> now it shows the diff *and* the file — so the reviewer knows what to look at and stops nitpicking
> stuff i never touched."

### analogies / metaphors

- **github pr view**: you see the diff by default, and can "expand" to full file context. we give the
  reviewer both at once.
- **track changes in a document**: the redlines show edits; the surrounding paragraphs give context.
- **a surgeon's chart**: the diff is "here's the incision site"; the full file is "here's the whole
  patient" — operate on the site, mind the whole.

### terms — theirs vs ours

| user says | we say |
|-----------|--------|
| "the diff" / "what changed" | diff hunks (`git diff`) |
| "the whole file" | full content / target content |
| "don't nitpick old code" | fix-forward; don't churn untouched lines |
| "focus on my changes" | prime reviewer on the delta |

## evaluation

### how well it solves the goals

- **directly** solves the core wish: reviewer now sees the delta explicitly.
- low risk: inputs unchanged, only the `<target>` rendering is enriched.
- composes with the current `--focus`, `--diffs`, `--goal` knobs.

### pros

- delta is explicit → focused, relevant feedback
- full context retained → no loss of reviewing power
- no new flags → zero learning curve, no migration
- big context window makes "both" affordable

### cons / costs

- **token cost rises**: every changed file now carries diff + full content. acceptable per the wish
  (1mm window), but token estimates / breakdown reports must account for the diff bytes.
- **diff capture adds git calls**: need `git diff <range>` *with* patch output (not just `--name-only`).
- **push-only injection**: the diff (and content) are injected **only** in `--focus push`. in
  `--focus pull` the brain has tool use and fetches on its own — it reads files **and** can run
  `git diff` itself — so we inject neither; we just tell it the diff range/base so it knows what
  "this pr" means. this keeps pull clean (the brain pulls all it needs) and avoids a push/pull hybrid.

### edgecases & pit-of-success

| edgecase | pit-of-success treatment |
|----------|------------------------|
| target came from `--paths` only, never changed | no diff present → render content only (no empty diff section) |
| brand-new untracked file | **still render it in the diff**, explicitly marked as a **new file** (e.g. git's `new file mode` / `--- /dev/null` header, every line a `+` add). wisher decision (2026-06-30): double down — make crystal clear *what is the diff* vs *what is the context*, even when the whole file is new. do not silently omit the diff. — see A5 below |
| deleted file | supply a **one-line marker only**, not the removal diff or content. wisher decision (2026-07-01): render as ` --- \n - deleted $path/to/file \n --- `. the reviewer only needs to *know* the file is gone, not re-read every deleted line (dead code, wasted tokens). |
| file changed but diff range empty (e.g. since-staged with no files staged) | render content only; no false empty diff |
| **`since-staged` snapshot divergence (A1)** | content is read from the **worktree** (`stepReview.ts:464`) but `since-staged` diffs index-vs-HEAD (`enumFilesFromDiffs.ts:87`) — disk content can include **unstaged** edits the staged diff omits. pair the diff with the same tree the content reflects, **or** label the gap ("diff = staged only; content = worktree"). since-main/since-commit use `git diff <base>` which already compares base to the worktree, so they align — no action. |
| binary file changed | git diff says "Binary files differ" → pass that through, don't inject garbage |
| huge diff | still cheaper than full file; render as-is |
| `--focus pull` | inject **neither diff nor content** — the brain pulls on its own: it reads files and runs `git diff <base>` itself. we pass the range/base so it knows what "this pr" means. injection (diff + content) is push-only. |
| **line-number correlation (A8)** | diff hunk headers carry `@@ -X,Y +A,B @@` line numbers, but injected content is fenced **without** line numbers (`compileReviewPrompt.ts:53`). the output demands `locations: ["path:42"]`. the reviewer must map hunk line-numbers onto unlabeled content. pit-of-success: the diff's own `+A` line numbers are the authoritative anchor for changed lines — prompt copy should point the reviewer at hunk headers for `locations`, so it need not count content lines by hand. |
| **snippet source ambiguity (A9)** | output requires a `snippet.code` copied from the file (`compileReviewPrompt.ts:143`). with a diff present, the reviewer might copy a `+`/`-`-prefixed diff line instead of the clean source. prompt copy must say: quote `snippet.code` from the **clean content**, never the diff's `+`/`-` lines. |
| **removed-line flagging (A10) — defect-class** | a diff's `-` lines are code this pr **deletes**. the current prompt (`compileReviewPrompt.ts:75-81`) has no `+`/`-` semantics, so the reviewer could flag a `-` line as a violation — noise about code that no longer exists, a regression this feature itself would introduce. prompt copy must state: `-` lines are removed (never flag them; they are gone), `+` lines and surrounding context are what exists and may be flagged. |

the core rule: **a diff section appears only when a real diff is present for that target**, so the
reviewer is never confused by an empty or false "what changed" block. and: **the rendered diff
base must equal the file-selection base** (IR2) so a file is never shown a diff computed against a
different reference than the one that selected it.

## open questions & assumptions

### assumptions made

1. the wish targets the `review` operation (`src/domain.operations/review/*`), specifically the
   `<target>` rendering in `compileReviewPrompt.ts`. — **high confidence**, verified the gap lives there.
2. the diff to show is "this pr vs priors" = the same range already used to *select* files
   (`since-main` via merge-base, `since-commit`, `since-staged`). we reuse that range to *render* the
   diff, and the **render base must equal the selection base** (IR2). — **high confidence**.
3. "both diff and content" injection is **push-only**; `--focus pull` injects neither — the brain
   pulls files and runs `git diff` itself (we pass the range/base). — **wisher-confirmed (2026-06-30)**.
4. token-budget reports (`tokens.expected.*`, the 60%/75% thresholds) should count the diff bytes too.
   "budget not a concern" (R4) is a **permission** tied to a large-context brain, not a license to skip
   accounting — small-context brains must still failfast cleanly. — **high confidence**.
5. "reviewers" = **all** brain reviews. verified `compileReviewPrompt` has a single caller (`stepReview`,
   exposed via `src/contract/cli/review.ts`), so one change point reaches ad-hoc **and** guard/peer
   reviews (IR1). — **high confidence; execution must still confirm guard peer-review invokes `rhx review`,
   not a bespoke prompt.**
6. diff and content describe the **same snapshot**. **false for `since-staged`** (A1) — disk content can
   exceed the staged diff. execution must pair them coherently or label the gap. — **must-handle edgecase.**
7. "no churn" (R3) means *do not churn untouched code*, **not** *ignore breakage the change itself
   causes* — prompt copy should still permit flags on untouched code the diff breaks. — **nuance to encode.**

### open questions — triaged

- **Q1 [answered]** — for `--focus pull`, inject **neither** diff nor content. wisher decision
  (2026-06-30): pull already pulls on its own — the brain reads files **and** runs `git diff` itself,
  so injection would create a push/pull hybrid. injection (diff + content) is **push-only**; in pull
  we pass the range/base so the brain knows what "this pr" means and pulls the diff itself.
- **Q2 [answered]** — a `--paths`-selected, unchanged file renders **content-only, no diff section**.
  answered by the core rule "a diff section appears only when a real diff is present"; absence of a
  diff block is unambiguous, so no "no changes" note is needed.
- **Q3 [answered]** — **diff-first, then content.** answered by the wish: "primed to focus on [the
  diff] in particular" → lead with what the reviewer should focus on.
- **Q4 [answered]** — diff base = the **selection base** (merge-base for since-main,
  `enumFilesFromDiffs.ts:80`). answered by code + IR2 (render-base ≡ selection-base); we never show
  commits that landed on main after the branch point.
- **Q5 [answered]** — R3 strictness: the reviewer **may flag untouched code that *this change
  breaks*** — "flag-if-broken" is in-scope. wisher decision (2026-06-30): "flag if broken too." the
  line stays fix-forward for unrelated prior code, but breakage the change itself causes is fair to
  flag. prompt copy must encode: focus on the diff; do not churn unrelated untouched code; **do**
  flag untouched code the change breaks.
- **IR1 [answered]** — guard/peer review **does** invoke `rhx review` (the single path), confirmed by
  the wisher (2026-06-30) and consistent with the single-caller grep (`compileReviewPrompt` ←
  `stepReview` ← `cli/review.ts`). so the one change point reaches ad-hoc **and** guard/peer reviews.

### what to research externally

none — no external dependencies. the only "external" tool is `git diff`, already used in-repo. no
internal-research items remain — IR1 is confirmed.

## groundwork

### external research

none — no external dependencies. the feature is entirely internal; the sole external tool is local
`git`, already invoked throughout `enumFilesFromDiffs.ts`.

### internal research (verified)

the gap and the contracts to conform to:

- **diff hunks are discarded today**: `enumFilesFromDiffs.ts:76-101` builds `git diff ... --name-only`
  and returns only paths. the patch text is never captured.
  - since-main uses `git diff $(git merge-base <main> HEAD) --name-only` (`enumFilesFromDiffs.ts:80`)
    — i.e. changes since branch point, not changes on main. our rendered diff must use the **same**
    base to stay consistent.
  - untracked files are added separately (`enumFilesFromDiffs.ts:118-119`) and won't appear in a
    standard `git diff` — new-file treatment needed.
  - **deleted files are currently dropped**: the enumeration filters via `fs.stat(...).isFile()`
    (`enumFilesFromDiffs.ts:122-135`); a deleted path throws ENOENT → returns null → excluded. so to
    fulfill the wisher's "show deleted files," the enumeration must change to **keep** deleted paths
    (e.g. derive them from `git diff --diff-filter=D` / the name-status, not from a worktree `stat`).
    a deleted path needs neither diff nor content — just its path, rendered as a one-line
    ` - deleted $path ` marker (wisher, 2026-07-01).

- **targets contract = `{ path, content }`**: `compileReviewPrompt.ts:11-25` — to carry a diff we must
  extend the target shape (e.g. add an optional `diff` field) or pass diffs alongside.

- **target rendering, push vs pull**: `compileReviewPrompt.ts:43-56`
  - pull (`:44-48`): lists paths, instructs brain to open files
  - push (`:51-55`): injects ` ### path \n ```\n content\n``` `
  - this is the exact spot the diff section gets added.

- **prompt focus instructions**: `compileReviewPrompt.ts:75-81` and `:101-105` — the `<target>`
  section is declared "REVIEW THIS SECTION ONLY". new copy should tell the reviewer to **focus on the
  diff** and use full content as **context only, don't churn unchanged lines** — encoding the wish's
  fix-forward intent.

- **content read**: `stepReview.ts:480-485` reads `targetContents`. diff capture slots in near here
  (or in `enumFilesFromDiffs`) so both flow into `compileReviewPrompt`.

- **token accounting**: `stepReview.ts:493-519` builds `genTokenBreakdownReport` over rule+target
  contents, and `compileReviewPrompt.ts:148-169` enforces 60%/75% thresholds. diff bytes must be
  included so estimates stay honest.

- **vocab to reuse**: owl vibes in stdout (`🦉`, treestruct `├─ └─`), `since-main|since-commit|since-staged`
  range names, `focus: pull|push`, `goal: exhaustive|representative`, `target/rules/refs` sections.

### stdout / format patterns to match

- error/warn blocks use `🦉 woah there` + treestruct hints (`stepReview.ts:263-274`, `:404-424`).
- the diff fence should use ` ```diff ` so it renders as a diff in markdown review artifacts.

## verification — boundary-condition coverage (required)

every boundary condition above must be locked by a **snapshot-backed** test, scoped to the right
grain. the snapshots show exactly **how each file change is supplied** to the reviewer — what lands
in the `<target>` section — so a human can spotcheck the rendered shape in the PR.

### the three file-change kinds — how each is supplied (wisher-required snapshots, 2026-06-30)

| kind | diff supplied | content supplied | snapshot must show |
|------|---------------|------------------|--------------------|
| **new file** | yes — `new file` header, every line a `+` add | yes — full new content | the `new file` marker + the all-`+` diff, then the full content |
| **edited file** | yes — hunks (`+`/`-`/context) | yes — full current content | diff-first (hunks), then the full current content |
| **deleted file** | **no** — one-line marker only | **none** (file gone) | just ` --- \n - deleted $path \n --- ` — no removal diff, no content |

each of these three is a distinct snapshot so the reviewer's supplied shape is crystal clear and any
drift in *how* a change is presented surfaces in the diff of the snapshot.

### scope by grain

| grain | what | test kind |
|-------|------|-----------|
| `compileReviewPrompt` (pure render of `<target>`) | the supplied shape per kind/case | **unit + snapshot** — cheapest, most legible; the snapshot *is* the contract a human eyeballs |
| diff capture (git boundary) | the right hunks per range | **integration** against a real fixture repo |
| `stepReview` (end-to-end) | the emitted `input.prompt.md` artifact for a multi-file scenario | **integration/acceptance** + snapshot |

### boundary cases → snapshot asserts (at `compileReviewPrompt` unit grain)

| case | snapshot asserts |
|------|------------------|
| push: new / edited / deleted file | the three rows above |
| push: unchanged path-only file (no diff) | content only, no diff section |
| push: binary file changed | `Binary files differ`, no garbage |
| push: `since-staged` divergence (A1) | the gap label present |
| pull: any | neither diff nor content; paths + range/base instruction |
| A8 line-anchor | prompt copy points `locations` at hunk headers |
| A9 snippet source | prompt copy says quote from clean content |

### diff capture → integration (fixture repo)

- since-main (merge-base), since-commit, since-staged each capture the right hunks
- new/untracked → new-file diff; deleted → deletion diff; worktree-vs-staged divergence reproduced

## what is awkward?

1. **target shape stretch**: a "target" has been `{ path, content }`. adding `diff` means a target is
   now content **plus** a derived view of that content. it's coherent, but the name "target" now
   carries two representations of the same file. acceptable, but worth a clear field name (`diff`).

2. **diff/content duplication**: the changed lines appear twice (once in the diff, once in content).
   intentional per the wish, but it *feels* redundant and inflates tokens. the 1mm budget justifies it.

3. **pull-mode split brain** — *resolved (Q1)*: pull injects neither diff nor content; the brain
   pulls both itself. push injects both. clean split, no hybrid. the earlier awkwardness is gone.

4. **paths-without-diffs asymmetry**: when targets mix diff-selected and path-selected files, some
   targets show a diff and some don't. correct behavior, but the `<target>` section becomes visually
   uneven. acceptable — the per-file "diff present only when real" rule keeps it honest.

5. **untracked/new files** — *settled (wisher, 2026-06-30)*: a brand-new file's diff ≈ its content,
   so it duplicates. but the wisher chose to **keep the diff and mark it explicitly as a new file**
   (git's `new file` / `--- /dev/null` header). the duplication is accepted on purpose: it makes
   crystal clear *what is the diff* vs *what is the context*, even for wholly new files. no silent
   omission.
