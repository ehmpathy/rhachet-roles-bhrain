# vision: fix driver overruled even error

## primary deliverable: test coverage

**the most important part of this behavior is test coverage.**

the behavior may already function (we suspect it does for tier escalation). but without snapshotted tests, we have no proof and no protection against regression.

| priority | deliverable |
|----------|-------------|
| **P0** | snapshot tests that prove broken reviewers don't block escalation |
| **P0** | snapshot tests that prove overrule bypasses malfunction/constraint |
| P1 | fix overrule to bypass malfunction (if not already correct) |
| P1 | fix malfunction as terminal (if not already correct) |

the tests come first. they will reveal whether the fixes are even needed.

## the outcome world

### before

a driver is stuck at a guarded stone. the human approves the work and runs `--as overruled` to bypass reviewer thresholds. but one reviewer has a malfunction (e.g., missing rule files, network error, bad config). the stone remains blocked:

```
🦉 the way speaks for itself

🗿 route.stone.set
   ├─ stone = 5.3.verification
   ├─ ✗ blocked by reviewer malfunction
   │
   ├─ r8: mech-test-scope-purity (l2)
   │   ├─ malfunction
   │   └─ BadRequestError: no rule files found at ...
```

the human's overrule intent is ignored. the driver is permanently blocked until the reviewer infrastructure is fixed.

additionally, a malfunctioning l1 reviewer blocks l2/l3 from running — the system waits for l1 to reach a "terminal" state, but malfunction is not considered terminal.

### after

`--as overruled` overcomes all reviewer states, including malfunctions and constraint errors:

```
🦉 the way speaks for itself

🗿 route.stone.set
   ├─ stone = 5.3.verification
   ├─ ✓ passed
   │   └─ reason: human overruled (bypasses malfunction)
```

and malfunctioned/constrained reviewers are treated as terminal for tier escalation — l2/l3 can run even when l1 has errors:

```
   ├─ r1: repo-rules (l1, 3/3)
   │   ├─ malfunction (terminal)  ← now allows escalation
   │
   ├─ r5: arch-criteria (l2, 2/2)
   │   ├─ running...              ← no longer blocked
```

### the "aha" moment

"my reviewer broke, but the human already blessed the work. why should infra failures override human judgment?"

the human's `--as overruled` should be the final word.

## user experience

### usecases

| usecase | before | after |
|---------|--------|-------|
| reviewer config error (missing rules) | permanently blocked | overrule bypasses |
| reviewer network failure | permanently blocked | overrule bypasses |
| l1 reviewer crashes | l2/l3 never run | l2/l3 run (malfunction = terminal) |
| reviewer rate limited | blocked until retry | overrule bypasses |

### contract inputs & outputs

**extant contract (unchanged)**:
```sh
rhx route.stone.set --stone X --as overruled
```

**behavior change**:
- currently: overrule only bypasses review threshold verdicts
- after: overrule bypasses all reviewer states (malfunction, constraint, verdict)

**tier escalation change**:
- currently: `terminal = approved | exhausted`
- after: `terminal = approved | exhausted | malfunction | constraint`

### timeline

1. driver completes work on guarded stone
2. driver runs `--as arrived`
3. reviews run; some pass, one malfunctions
4. human reviews the work directly, decides it's good
5. human runs `--as overruled`
6. driver runs `--as passed` → **succeeds** (overrule respected)

## mental model

### how users describe it

"overrule means overrule — even if the reviewer broke"

"broken reviewers shouldn't block the road; they should get out of the way"

### analogies

- **broken traffic light**: a cop can wave you through even when the light is broken. the cop's authority (human overrule) supersedes the malfunctioning light (broken reviewer).

- **fire alarm malfunction**: if the alarm is broken but you smell smoke, you evacuate. the malfunction doesn't override your judgment.

### terms

| user term | system term |
|-----------|-------------|
| "reviewer broke" | malfunction (exit ≠ 0, ≠ 2) |
| "reviewer complained" | constraint (exit 2, blockers=0) |
| "reviewer rejected" | constraint (exit 2, blockers>0) |
| "overrule" | human bypasses all reviewer states |
| "terminal" | reviewer has reached final state, escalation allowed |

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| overrule bypasses malfunction | check overrule BEFORE malfunction handling |
| overrule bypasses constraint | check overrule BEFORE constraint handling |
| malfunction allows escalation | add `malfunction` to terminal states |
| constraint allows escalation | add `constraint` to terminal states (TBD) |
| test coverage | snapshot tests for each scenario |

### pros

- human judgment is final (as expected)
- reviewer infra issues don't block drivers
- tier escalation works even with broken reviewers
- no change to CLI contract

### cons

- a malfunction might hide a real problem (mitigation: human must explicitly overrule)
- test coverage requires crafting malfunction scenarios

### edge cases

| edge case | behavior |
|-----------|----------|
| all reviewers malfunction + no overrule | stone blocked (malfunction surfaced) |
| all reviewers malfunction + overrule | stone passes (human judgment final) |
| l1 malfunction + l2 not run yet | l2 runs (malfunction = terminal) |
| l1 constraint + l2 not run yet | l2 runs (constraint = terminal) |
| overrule then reviewer fixes itself | overrule still applies (idempotent) |

## open questions & assumptions

### assumptions

1. `malfunction` should be treated as terminal for tier escalation (same as `exhausted`)
2. `constraint` with blockers=0 should also be terminal (genuine constraint, not rejection)
3. overrule check should happen early in `setStoneAsPassed()`, before malfunction/constraint handling
4. no CLI contract changes needed

### answered via logic

1. should `constraint` (exit 2, blockers=0) also allow tier escalation?
   - **yes** — it's a terminal state (reviewer can't proceed without external action)

2. should overrule bypass only the specific malfunctioned reviewer, or all?
   - **all** — overrule is a blanket "i reviewed this myself"; per-reviewer granularity is excessive complexity

## what is awkward?

1. **overrule happens after reviews run** — conceptually, if the human overrules, why run reviews at all? but this is by design: reviews still provide signal, overrule just bypasses blocking.

2. **malfunction as terminal feels permissive** — a broken reviewer might be hiding real issues. mitigation: the malfunction is still surfaced in output; human must consciously overrule.

3. **two concerns conflated** — the wish asks for (a) overrule bypasses malfunction AND (b) malfunction allows escalation. these are related but distinct fixes. both are needed for the described scenario.
