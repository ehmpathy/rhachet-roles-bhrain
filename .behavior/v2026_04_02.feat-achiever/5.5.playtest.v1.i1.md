## edgey paths (via acceptance tests)

the acceptance tests cover these edge cases:

| edge case | test file | expected behavior |
|-----------|-----------|-------------------|
| main branch forbidden | `achiever.goal.lifecycle` | error message |
| empty goals list | `achiever.goal.lifecycle` case3 | returns `(none)` |
| status transitions | `achiever.goal.lifecycle` case1 | enqueued -> inflight -> fulfilled |
| partial goals | `achiever.goal.lifecycle` case1 | status = incomplete for partial |
| scope auto-detect: bound | `achiever.goal.lifecycle` (manual) | detects `--scope route` |
| triage halts on incomplete | manual.4, manual.6 | exit 2 when incomplete, exit 0 when complete |

**verification:** the acceptance test snapshots show the exact expected output for each case.

---

## design notes

**simplified design:** `incomplete` is a workflow status, not separate metadata.
- partial goals automatically get `status=incomplete`
- complete goals get `status=enqueued` by default
- no `.meta.yaml` files — status flag encodes completeness
- lifecycle: `incomplete -> enqueued -> inflight -> fulfilled` (with `blocked` branch)
