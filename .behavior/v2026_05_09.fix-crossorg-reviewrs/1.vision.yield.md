# vision: fix cross-org keyrack calls

## the outcome world

### day-in-the-life

a developer in the `ahbode` org installs `rhachet-roles-bhrain` and runs a review. it just works — keyrack finds their XAI_API_KEY from their `ahbode` keyrack manifest without them thinking about orgs.

### before/after contrast

| before | after |
|--------|-------|
| bhrain calls `keyrack get --org ehmpathy` | bhrain calls `keyrack get` (no --org) |
| works only in ehmpathy repos | works in any org's repos |
| user sees "org mismatch" errors | keyrack infers org from manifest |

### aha moment

"wait, i didn't have to configure anything about orgs — it just read my keyrack.yml and worked"

## user experience

### usecases

1. **cross-org developer**: installs bhrain in their repo (any org), runs review
2. **ehmpathy developer**: same as before, no change in workflow
3. **ci pipeline**: sets XAI_API_KEY env var, bhrain uses it (extant path)

### contract

```
input: (none — keyrack infers from cwd's manifest)
output: XAI_API_KEY credential for xai brain
```

### what it looks like

```bash
# in an ahbode repo
$ rhx review --rules 'rules/*.md' --paths 'src/*.ts' --output review.md
# keyrack get called without --org
# manifest in cwd has org: ahbode
# keyrack looks up ahbode.prep.XAI_API_KEY
# review runs
```

### timelines

1. install bhrain package
2. ensure keyrack.yml has correct org
3. unlock keyrack: `rhx keyrack unlock --owner ehmpath --env prep`
4. run review — just works

## evaluation

**goal**: bhrain works in repos outside ehmpathy org

**solution**: remove hardcoded `--org ehmpathy` from keyrack get call

**result**: 100% — keyrack infers org from manifest, works in any org

### pros

- zero config for users
- extant ehmpathy repos unchanged
- follows keyrack's intended design (infer from manifest)
- one-line fix

### cons

- none identified

### internal research

**verified the defect location**:
- file: `src/domain.operations/credentials/getXaiCredsFromKeyrack.ts`
- line 34: `--org ehmpathy` is hardcoded in the keyrack get command
- this is the only place `--org` appears in src/

**verified the fix**:
- remove `--org ehmpathy` from the command
- keyrack CLI will read org from cwd's keyrack.yml (verified via `rhx keyrack get --help` where default is `@this`)
- empirical test: `rhx keyrack get --owner ehmpath --key XAI_API_KEY --env prep --json` without `--org` returned `org: ehmpathy` from manifest
- extant behavior preserved for env var path (line 17-28)

**verified no other credential getters**:
- only one file in `src/domain.operations/credentials/`
