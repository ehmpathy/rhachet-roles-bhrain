## deeper reflection

### which edge case would catch a real bug?

the most likely real bug is:
- **empty output malfunction** - reviewer crashes before write

this is covered by "empty stdout/stderr" edge case. if the artifact is empty, the foreman will notice.

### which edge case is most important?

the **malfunction halt** (happy path 5) is the most critical edge case. the original defect was:
- hook never fired again after malfunction

if this test fails, the defect is not fixed.

---

## what would fail if a behavior was broken?

| broken behavior | which playtest fails |
|-----------------|---------------------|
| exit code not captured | happy path 3: no `exit code: 1` visible |
| stdout not in tree bucket | happy path 1: no `├─ stdout` visible |
| stderr not captured | happy path 3: no `stderr error` visible |
| malfunction not detected | happy path 5: no exit code 3 |
| wrong status label | happy path 2: `blocked by malfunction` instead of `blocked by constraints` |

each playtest isolates a specific behavior. if one breaks, exactly one test fails.

---

## what if the foreman skips a test?

the playtest is ordered by criticality:
1. passed flow (baseline)
2. constraint flow (common case)
3. malfunction flow (critical case)
4. judge classification (completeness)
5. malfunction halt (critical safety)

if foreman runs only tests 1, 3, 5, they cover:
- normal operation
- malfunction capture
- malfunction escalation

this is the minimum viable verification.

## what edge case did I almost miss?

- **what if two malfunctions occur in sequence?**
