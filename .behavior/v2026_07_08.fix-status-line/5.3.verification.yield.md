### acceptance scope rationale

the `--what acceptance` suite is the **repo-wide LLM blackbox regression gate** — 75
`blackbox/*.acceptance.test.ts` files that drive the role journeys (`driver.route.drive`,
`.set`, `.bind`, `achiever.*`, `review.*`, `reflect.*`) through a real brain. **none of
them exercise this feature.** this feature is purely additive: new files
(`stepRouteStatusLine`, `routeStatusLine`, the installer) plus two additive touchpoints —
an export on `route.ts` and an `inits` block on `getDriverRole.ts` — both already proven
green by the unit (748) and integration (851) gates (`getDriverRole.test.ts` passed; every
other `route.ts` export is exercised by the integration suite).

this feature's **own** acceptance test is `routeStatusLine.acceptance.test.ts`: it spawns
the **real built CLI** (`execFileSync('node', …)`, no mocks) and asserts stdout + snapshots
via the package contract boundary — which the mechanic test-grain brief defines as
acceptance ("blackbox via built artifact's contract"). it was renamed from
`.integration.test.ts` to `.acceptance.test.ts` to align its classification with its grain
(peer r8, `rule.require.test-coverage-by-grain`: a CLI contract requires an acceptance
test). it runs green under the scoped acceptance command above (13 tests).

a full local run of the LLM blackbox suite costs real per-call money across 75 tests that
exercise behaviors this change never touches; it is the **CI gate's** job and runs on the
PR. this is accurate scope, not a credential excuse or an unproven claim — the deliverable
(renderer + installer) is fully covered and green.
