# vision: expand $route in artifact globs

## the outcome world

### before

a behaver writes a guard with artifact patterns:
```yaml
artifacts:
  - "$route/5.1.execution.phase0_to_phaseN.v1.i1.md"
```

bhrain says "artifact not found" even when the file is at `.behavior/xyz/5.1.execution...`. two bugs compound:
1. `$route` is NOT expanded — bhrain looks for literal `$route/` directory
2. `cwd: input.route` is used — glob runs from inside route dir, breaking path resolution

### after

the same guard "just works". bhrain:
1. expands `$route` to the actual route path (e.g., `.behavior/v2026_03_24.xyz/`)
2. runs glob from repo root (no `cwd` override)

artifacts are found. guards pass. `$route` behaves consistently everywhere — reviews, judges, AND artifact globs.

### the "aha" moment

a behaver writes `$route/some-artifact.md` in their guard, runs the route, and it works on the first try. no debugging, no confusion.

## user experience

### usecase

behavers define guards with artifact patterns that reference the route directory. they expect `$route` to work because it already works in reviews and judges.

### contract

**input**: guard yaml with `artifacts: ["$route/pattern.md"]`
**output**: bhrain finds files matching `.behavior/xyz/pattern.md` (actual route path)

### timeline

1. behaver writes guard with `$route` in artifact pattern
2. behaver runs route
3. bhrain expands `$route` → actual route path (e.g., `.behavior/v2026_03_24.xyz/`)
4. glob runs from repo root, matches files at expanded path
5. guard proceeds with found artifacts

## mental model

### how users describe it

"$route is a variable that points to the route directory. use it in artifact patterns when your artifacts live in the route folder."

### analogy

`$route` is like `$HOME` in shell scripts — a variable that expands to a known path.

### terms

| user says | we say |
|-----------|--------|
| "route folder" | route directory |
| "the $route variable" | route path variable |
| "artifact pattern" | artifact glob |

## evaluation

### does it solve the goal?

yes. three changes:
1. expand `$route` to `input.route` (the actual route path)
2. remove `cwd: input.route` — glob runs from repo root
3. prefix default pattern with route path (when no guard artifacts specified)

### pros

- consistent with how `$route` works in reviews/judges
- no new concepts for users
- fixes the broken behavior

### cons

- none significant

### edge cases

| case | behavior |
|------|----------|
| no guard artifacts specified | default pattern `${input.route}/${stone.name}*.md` |
| no `$route` in custom pattern | pattern used as-is, relative to repo root |
| `$route` appears multiple times | all instances expanded |
| pattern has no matches | returns empty array |

## open questions & assumptions

### assumptions (verified)

- [verified] `input.route` is the correct expansion value — matches `vars.route` in reviews/judges
- [verified] `getAllStoneArtifacts` is the only place that needs this fix — grep confirmed in self-review r2

### questions

- [answered] should we use `.route` or `input.route`? → use `input.route` (the actual route path like `.behavior/xyz/`)
- [research] should artifact patterns also support `$stone`, `$hash`, `$output` like judges do? → defer to research phase; for now, only `$route` is needed to unblock bhuild

### hard requirements

- **never use `cwd: input.route`** — globs must run from repo root with expanded paths

## what is awkward?

### the wish was misleading

the wish suggested expanding `$route` to `.route`, but `.route` is not where routes live. routes are at `.behavior/xyz/...`. this was caught during self-review.
