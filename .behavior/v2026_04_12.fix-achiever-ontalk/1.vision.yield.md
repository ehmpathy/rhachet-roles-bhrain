# vision: fix achiever onTalk hook

this is a fix behavior — the vision was already specified in `v2026_04_02.feat-achiever/1.vision.md`. this document clarifies what fulfillment looks like.

---

## the outcome world

### day-in-the-life

a human sends a message to claude code. behind the scenes:

1. the `UserPromptSubmit` hook fires
2. `goal.triage.infer --when hook.onTalk` runs
3. the message is hashed and appended to `asks.inventory.jsonl`
4. a short reminder appears in the system output
5. the brain continues without halting

the human never notices — but every ask is now tracked. when the session ends, `onStop` can verify all asks are covered by goals.

### before/after contrast

| before (broken) | after (fixed) |
|-----------------|---------------|
| hook fires but does no useful work | hook accumulates ask to inventory |
| asks lost between messages | asks persist in `asks.inventory.jsonl` |
| `onStop` can't verify coverage | `onStop` can verify all asks covered |
| full triage output on every message | short reminder only |

### the "aha" moment

the human runs `rhx goal.triage.infer` and sees:

```
🔮 goal.triage.infer --scope repo
   ├─ asks = 5
   ├─ uncovered = 2
   ...
```

wait — *5 asks*? the human only remembers 3. they check `.goals/$branch/asks.inventory.jsonl`:

```jsonl
{"hash":"a1b2c3","content":"fix the flaky test","receivedAt":"2026-04-12"}
{"hash":"d4e5f6","content":"update the readme","receivedAt":"2026-04-12"}
{"hash":"g7h8i9","content":"oh and also the changelog","receivedAt":"2026-04-12"}
{"hash":"j0k1l2","content":"actually skip the changelog","receivedAt":"2026-04-12"}
{"hash":"m3n4o5","content":"ping me on slack when done","receivedAt":"2026-04-12"}
```

every ask was captured. even the ones the human forgot they said. that's the aha.

## user experience

### usecases

1. **ask accumulation** — every peer message saved with hash
2. **coverage verification** — `onStop` can check all asks have goals
3. **audit trail** — human can inspect what was asked in order

### contracts

**hook behavior** (not user-invoked, runs automatically):

```
trigger: UserPromptSubmit hook fires
input: user message via stdin
output: reminder to stderr, exit 0
side effect: append ask to asks.inventory.jsonl
```

**reminder output** (per original vision):

```
🦉 to forget an ask is to break a promise. remember.

🔮 goal.triage.infer --from peer --when hook.onTalk
   ├─ from = peer:human
   ├─ ask
   │  ├─
   │  │
   │  │  {the user's message}
   │  │
   │  └─
   │
   └─ consider: does this impact your goals?
      ├─ if yes, triage before you proceed
      └─ run `rhx goal.triage.infer`
```

### timelines

1. **immediate** — message arrives, hook fires, ask saved
2. **during session** — asks accumulate in inventory
3. **on stop** — `onStop` hook verifies coverage
4. **later** — human can audit `asks.inventory.jsonl`

## evaluation

### how well does it solve the goals?

**primary goal: accumulate asks** — solved. every message saved with hash.

**secondary goal: enable coverage verification** — solved. `onStop` can check inventory against coverage.

**tertiary goal: non-disruptive** — solved. short reminder, exit 0, no halt.

### pros

- simple append-only persistence
- deterministic hash for deduplication
- minimal output (reminder only)
- preserves message order

### cons

- no deduplication at write time (same message hashed twice = two entries)
- no cleanup of old asks (inventory grows forever)

### edge cases & pit of success

| edge case | pit of success |
|-----------|----------------|
| empty message | skip (no meaningful ask to track) |
| huge message | show full message in reminder (compaction could lose context) |
| rapid messages | each gets own entry, order preserved |
| binary/emoji | hash handles any utf-8 |

## open questions & assumptions

### assumptions

1. **stdin contains the message** — claude code pipes user prompt to UserPromptSubmit hooks
2. **order matters** — we preserve insertion order in jsonl
3. **dedup not needed at write** — coverage map handles dedup semantics

### questions (answered)

1. **[answered]** does claude code actually pipe stdin to UserPromptSubmit hooks?
   - yes. claude code passes JSON to stdin with a `prompt` field that holds the user message.

2. **[answered]** is the stdin the raw user message or preprocessed?
   - preprocessed JSON object. extract prompt via `jq -r '.prompt'`

### external research needed

none — this is implementation of already-specified behavior

## what is awkward?

### stdin assumption

we assume claude code pipes the user message to stdin. if it doesn't, the hook will save empty strings. need to verify this assumption.

**mitigation**: check claude code docs or test empirically.

### reminder verbosity

the original vision specifies a fairly verbose reminder (10+ lines). this fires on every message. might feel noisy.

**mitigation**: acceptable. the brain sees it, the human doesn't. full message required since compaction could lose conversation context.

### no dedup at write

same message sent twice = two inventory entries. this is intentional (preserves audit trail) but might confuse users.

**mitigation**: coverage map handles semantics. inventory is raw log.
