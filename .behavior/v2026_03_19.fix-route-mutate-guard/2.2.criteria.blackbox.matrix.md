# 2.2.criteria.blackbox.matrix

## matrix 1: guard decision by path type and privilege

dimensions:
- **ind: route bound** ‚Äî whether a route is bound to the current branch
- **ind: privilege** ‚Äî whether privilege is granted via `rhx route.mutate grant allow`
- **ind: path type** ‚Äî what kind of path is accessed
- **dep: verdict** ‚Äî allowed or blocked

| ind: route bound | ind: privilege | ind: path type | dep: verdict |
|------------------|----------------|----------------|--------------|
| no | ‚Äî | artifact | allowed |
| no | ‚Äî | .route/* subdir | allowed |
| no | ‚Äî | *.stone | allowed |
| no | ‚Äî | *.guard | allowed |
| yes | no | artifact | **allowed** |
| yes | no | .route/* subdir | blocked |
| yes | no | *.stone | blocked |
| yes | no | *.guard | blocked |
| yes | yes | artifact | allowed |
| yes | yes | .route/* subdir | allowed |
| yes | yes | *.stone | allowed |
| yes | yes | *.guard | allowed |

**note**: bold **allowed** on row 5 is the NEW behavior. previously, artifact writes to routes at `.route/` were blocked.

## matrix 2: guard decision by route location

dimensions:
- **ind: route location** ‚Äî `.route/` or `.behavior/`
- **ind: path type** ‚Äî artifact or .route/* subdir
- **dep: verdict** ‚Äî allowed or blocked

| ind: route location | ind: path type | dep: verdict |
|---------------------|----------------|--------------|
| `.route/xyz/` | artifact | allowed |
| `.route/xyz/` | .route/* subdir | blocked |
| `.behavior/xyz/` | artifact | allowed |
| `.behavior/xyz/` | .route/* subdir | blocked |

**symmetry check**: both route locations behave identically. ‚úì

## matrix 3: blocker location

dimensions:
- **ind: route location** ‚Äî `.route/` or `.behavior/`
- **dep: blocker path** ‚Äî where blockers are written

| ind: route location | dep: blocker path |
|---------------------|-------------------|
| `.route/xyz/` | `.route/xyz/blocker/*.md` |
| `.behavior/xyz/` | `.behavior/xyz/blocker/*.md` |

**NOT** at `.route/xyz/.route/blocker/*.md`. blockers are visible artifacts, not hidden metadata.

---

## gaps analysis

### gap 1: cross-route writes

**absent combination**: route bound at A, write to route B

**why absent**: explicitly out of scope. the guard only checks paths within the bound route.

### gap 2: nested routes

**absent combination**: route bound at `.behavior/outer/inner/`

**why absent**: explicitly out of scope. nested routes are not supported.

---

## decomposition signal

the matrix has 3 independent dimensions in matrix 1. this is manageable.

if we added more dimensions (e.g., read vs write, bash vs file tool), the matrix would explode. but:
- read vs write: same behavior (blocked = blocked for both)
- bash vs file tool: same behavior (guard checks both)

no decomposition needed. the behavioral boundaries are clear.
