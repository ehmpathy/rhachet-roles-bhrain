# research: audience — achiever role

## who is the audience?

### primary: robot brains (Claude Code, AI agents)

the robot brain itself is the primary user. achiever enhances the brain's ability to:
- detect goals from peer input
- persist goals across context compression
- track goal status through completion
- avoid forgotten asks

**context:** robot brains operate in sessions that can span hours. context windows compress. without external persistence, goals drift and blur.

**constraints:**
- must operate via skill invocation (shell scripts)
- must persist to filesystem (survives context loss)
- must integrate with extant hook system (onTalk, onStop)

### secondary: humans who work with robot brains

humans who delegate tasks to robot brains benefit when:
- their multi-part requests are all tracked
- they don't have to repeat themselves
- they can inspect `.goals/` to see what's queued

**context:** humans send messages like "fix the test and update the readme and notify me when done" — three asks in one message. today, the brain often forgets the later asks.

**constraints:**
- must not burden the human with extra ceremony
- must work transparently (human doesn't need to invoke skills manually)
- must provide visibility via filesystem (`.goals/` is inspectable)

### tertiary: future role developers

developers who build new roles may reference achiever patterns:
- goal persistence patterns
- hook-based triage flow
- nested domain object schemas

**context:** achiever establishes patterns for goal-oriented roles.

## why do they care?

### robot brains

| pain | how achiever helps |
|------|-------------------|
| goals blur together | discrete goal files with explicit schema |
| context loss forgets goals | filesystem persistence survives compression |
| no structure for "what to do" | forced articulation of why/what/how |
| can't verify completion | gates provide verification criteria |

### humans

| pain | how achiever helps |
|------|-------------------|
| "did it forget?" anxiety | goals are persisted and inspectable |
| re-ask the same request | goal survives across sessions |
| unclear what's queued | `.goals/` shows all open work |
| trust gap | transparency builds confidence |

### future role developers

| pain | how achiever helps |
|------|-------------------|
| no goal-track pattern to reference | achiever establishes the pattern |
| unclear how to persist structured data | Goal schema + YAML persistence demonstrates |
| hook integration unclear | onTalk/onStop handlers demonstrate |

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| robot brains | high | immediate | without achiever, goals are lost |
| humans | high | continual | trust in robot reliability |
| future role developers | medium | deferred | pattern reference when needed |

### robot brains — high priority, immediate urgency

every session without goal persistence risks forgotten asks. the brain wants to be reliable but lacks the infrastructure.

### humans — high priority, continual urgency

humans experience the pain every time they re-ask a request. they've learned to work around it (shorter tasks, one-at-a-time), but want to delegate more complex work.

### future role developers — medium priority, deferred urgency

not urgent until they need to build a similar feature. but when they do, achiever patterns will accelerate their work.

## what do they care about?

### robot brains

| criteria | why it matters |
|----------|----------------|
| schema forces foresight | prevents shallow "just do it" goals |
| persistence survives context | goals don't disappear mid-work |
| status track is explicit | brain knows what's inflight vs enqueued |
| gates are required | brain must articulate verification |

**tradeoffs they'd accept:**
- verbose schema if it prevents ambiguity
- triage ceremony if it guarantees no ask is lost

**tradeoffs they'd reject:**
- complexity that slows every interaction
- opaque persistence they can't inspect

### humans

| criteria | why it matters |
|----------|----------------|
| transparency | can inspect `.goals/` anytime |
| no ceremony required | just talk naturally, goals are tracked |
| reliability | goals survive across sessions |
| evidence of completion | status.reason shows what was verified |

**tradeoffs they'd accept:**
- brain pauses briefly to triage (worth it for reliability)
- `.goals/` files in repo (visibility is a feature)

**tradeoffs they'd reject:**
- explicit goal invocation per request
- noisy output on every message

## how will they experience it?

### robot brain journey

```
t0: session starts
    → achiever role loaded via boot.yml
    → hooks registered (onTalk, onStop)

t1: human sends message with multiple asks
    → hook.onTalk fires
    → ask appended to asks.inventory.jsonl
    → brain continues work (not halted)

t2: brain works on first understood goal
    → may or may not run goal.infer.triage inline
    → partial work occurs

t3: session ends
    → hook.onStop fires
    → brain halted until triage complete
    → brain runs goal.infer.triage
    → uncovered asks surfaced
    → brain creates goals via goal.memory.set
    → coverage recorded
    → session may end

t4: next session
    → goals persist in .goals/
    → brain runs goal.memory.get
    → resumes work on extant goals
```

### human journey

```
t0: human opens Claude Code
    → achiever is transparent (no visible difference)

t1: human sends complex request
    → "fix the test and update the readme"
    → brain acknowledges

t2: human optionally checks .goals/
    → sees both goals tracked
    → feels confident

t3: later, human returns
    → asks "what's queued?"
    → brain reads goal.memory.get
    → reports extant goals

t4: brain completes goals
    → marks fulfilled with evidence
    → human can verify completion
```

### friction points

| point | mitigation |
|-------|------------|
| triage ceremony at session end | necessary evil — guarantees no ask lost |
| verbose goal schema | forces clarity — prevents shallow goals |
| `.goals/` file clutter | status flags make active vs fulfilled clear |

### smooth experience markers

| marker | achieved by |
|--------|-------------|
| human never re-asks | asks.inventory captures all input |
| brain knows what's queued | goal.memory.get provides full picture |
| completion is evident | status=fulfilled with reason |
| no surprise mid-task pivots | hooks remind brain to consider impact |
