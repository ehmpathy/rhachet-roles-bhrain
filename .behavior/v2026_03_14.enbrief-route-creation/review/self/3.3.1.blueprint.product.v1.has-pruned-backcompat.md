## deeper reflection: what if boot.yml format changes?

### scenario

boot.yml uses `always.briefs.ref` format. what if rhachet changes this format in the future?

### analysis

this is NOT a backcompat concern for THIS blueprint. this is a forward-compat concern for FUTURE rhachet versions. the blueprint creates boot.yml per CURRENT rhachet spec. if rhachet spec changes, boot.yml would need update — but that's expected maintenance, not a backcompat issue.

### why this matters

backcompat = "does this change break extant consumers?"
forward-compat = "will future changes break this new code?"

forward-compat is NOT the blueprint's concern. the blueprint is correct for today's rhachet. future rhachet changes would require future updates — that's normal maintenance.

**lesson**: don't conflate backcompat (break extant) with forward-compat (future-proof). focus on what the wish asked: don't break extant.

## deeper reflection: onBoot hook order

### scenario

the blueprint adds `rhachet roles boot` before the extant `route.drive --mode hook` entry. what if the order is wrong?

### analysis

I stated "briefs should load before route.drive" but did not verify this assumption. let me think about it more carefully:

1. `rhachet roles boot` loads boot.yml, which loads briefs into context
2. `route.drive` shows the next stone and its instructions

**does route.drive need the brief?**: NO. route.drive shows the stone's content (from the .stone file), not from briefs. briefs are reference documentation for the human/brain.

**so order doesn't matter here**: route.drive doesn't consume the howto brief. they're independent operations.

**correction**: my earlier rationale was flawed. the order doesn't matter because these hooks serve unrelated purposes. I assumed order mattered without evidence.

**lesson**: verify assumptions about dependencies. just because two things run in sequence doesn't mean they depend on each other.
