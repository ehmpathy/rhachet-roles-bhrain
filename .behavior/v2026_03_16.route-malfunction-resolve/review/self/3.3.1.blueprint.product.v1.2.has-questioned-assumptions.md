### re-examine each assumption

#### assumption 1: passage.jsonl is append-only

**previous verdict:** valid assumption — based on evidence.

**deeper question:** what if someone manually edits passage.jsonl?

**analysis:** manual edits would break log semantics. but this is explicitly documented behavior. the route system depends on append-only semantics. if someone manually edits, that's user error outside our scope.

**holds?** ✅ yes — append-only is a design constraint, not an assumption.

#### assumption 2: last entry per stone = current state

**previous verdict:** valid assumption — explicitly stated by wisher.

**deeper question:** could there be race conditions where entries are written out of order?

**analysis:** passage.jsonl writes are sequential within a route execution. the route driver is single-threaded. race conditions would only occur if multiple processes write to the same passage.jsonl simultaneously, which is unsupported usage.

**holds?** ✅ yes — sequential execution guarantees order.

#### assumption 3: exitClass === 'passed' means success

**previous verdict:** valid assumption — follows extant pattern.

**deeper question:** could exitClass have other values that mean success?

**analysis:** exitClass is derived from exit codes via `getOneExitClass()`. values are: 'passed' (exit 0), 'constraint' (exit 2), 'malfunction' (all else). only 'passed' means success by definition.

**holds?** ✅ yes — exit code 0 is the only success.

#### assumption 4: malfunctions should not be cached

**previous verdict:** valid assumption — explicitly stated by wisher.

**deeper question:** could there be scenarios where a cached malfunction is useful?

**analysis:** a malfunction means the reviewer is broken, not that the artifact has issues. a cache would prevent the fixed reviewer from a fresh run. there's no scenario where this is desirable.

**holds?** ✅ yes — cached malfunctions would always be wrong.

#### assumption 5: one review per hash is sufficient

**previous verdict:** valid assumption — follows extant pattern.

**deeper question:** could the same artifact hash need different reviews?

**analysis:** if content is identical (same hash), the review should be identical. the reviewer code is assumed stable. if reviewer logic changes, the review artifact contains the reviewer identity, so stale reviews would be detected.

**holds?** ✅ yes — content identity implies review identity.
