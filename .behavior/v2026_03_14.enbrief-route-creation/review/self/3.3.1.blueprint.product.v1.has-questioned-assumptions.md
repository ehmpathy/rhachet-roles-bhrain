### assumption 6: onBoot hook not needed (just `boot` property)

**what if opposite?**: what if driver needs explicit boot command like mechanic?

**evidence check**: mechanic role has:
```typescript
onBoot: [
  { command: './node_modules/.bin/rhachet roles boot --repo ehmpathy --role mechanic', timeout: 'PT60S' }
]
```

but driver already has onBoot hooks for passage check and notify. the `boot` property enables boot.yml load. the onBoot command executes `rhachet roles boot` which loads boot.yml.

**deeper analysis**: if `boot` property alone is sufficient, we don't need onBoot command. if onBoot command is required to trigger boot.yml load, we need it.

**resolution**: the `boot` property defines WHAT to load. the onBoot hook triggers WHEN to load. driver may need the onBoot command to trigger boot.yml load.

**correction**: added onBoot command to blueprint scope. this was incorrectly simplified in deletables review.

## deeper reflection on architecture choice

**why boot.yml and not inline briefs configuration?**

the boot.yml pattern separates "what gets loaded" from "how the role is defined". this follows separation of concerns. getDriverRole.ts defines the role structure; boot.yml defines what content loads when.

**could a simpler approach work?**

alternative: inline the ref briefs directly in getDriverRole.ts. but this would:
- mix configuration with code
- require code changes to add/remove briefs
- lose the declarative boot.yml pattern proven in architect role

the boot.yml pattern is not over-engineered — it matches extant patterns and enables non-code changes to boot configuration.
