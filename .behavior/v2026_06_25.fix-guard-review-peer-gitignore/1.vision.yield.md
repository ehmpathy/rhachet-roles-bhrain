# 1.vision — fix guard review peer gitignore

## the wish

> we want to findsert a gitignore within the `$route/.review/peer/*` dir
> so that they stop spamming git
> whenever the guard runs reviews

---

## the outcome world

### day-in-the-life

a driver navigates a thought route. each guarded stone runs peer reviews. every
review writes a fresh artifact into `$route/.reviews/peer/`:

```
.reviews/peer/1.vision._.review.i1.a3f9.r1._.given.by_peer.architect.md
.reviews/peer/1.vision._.review.i1.a3f9.r2._.given.by_peer.mechanic.md
.reviews/peer/1.vision._.review.i2.b7c2.r1._.given.by_peer.architect.md
...
```

these artifacts pile up fast — one per reviewer, per iteration, per hash. with
this wish fulfilled, the driver runs `git status` and sees a **clean tree**. the
peer-review noise is gone. only intentional artifacts (passage records, bind
flags) remain tracked.

### before / after contrast

| | before | after |
|---|---|---|
| `git status` after a guard run | dozens of untracked `by_peer.*.md` files | clean — peer artifacts ignored |
| reviewing a real diff | scroll past review spam to find signal | signal only |
| accidental commit of review noise | easy (`git add .` catches it all) | impossible (gitignored) |
| the `.gitignore` itself | absent | tracked, self-documenting |

### the "aha" moment

the value clicks the first time a driver runs `git status` mid-route and sees
only their actual work. the guard did its job — ran reviews, wrote artifacts the
driver can still read on disk — but git never noticed. the artifacts are
**present for humans, invisible to version control**.

## user experience

### usecases & goals

- **driver**: navigate a route without git noise drowning out real changes.
- **reviewer (peer)**: still write review artifacts to disk; their flow is unchanged.
- **human**: review a PR diff that contains only meaningful changes.

### contract inputs & outputs

this is an internal mechanism — no new CLI surface. the contract is a domain
operation that mirrors the two siblings already in the codebase:

```ts
// src/domain.operations/route/gitignore/findsertReviewPeerGitignore.ts
findsertReviewPeerGitignore(input: {
  route: string;
}): Promise<{ path: string; action: 'created' | 'unchanged' }>
```

- **input**: `{ route }` — the route directory (e.g. `.behavior/v2026.../`)
- **output**: `{ path, action }` — the gitignore path + whether it was written or already correct

gitignore content (mirrors `findsertSelfReviewGitignore`):

```
# ignore all peer-review files
*
!.gitignore
```

### how it's leveraged

the guard already creates the `.reviews/peer/` dir before writing artifacts.
the findsert slots in right there:

```ts
// runOneStoneGuardReview (runStoneGuardReviews.ts:109-110)
const reviewsDir = path.join(input.route, '.reviews', 'peer');
await fs.mkdir(reviewsDir, { recursive: true });
await findsertReviewPeerGitignore({ route: input.route }); // ← new line
```

### timeline

1. guard runs a peer review
2. `.reviews/peer/` dir is ensured
3. `.gitignore` is findserted (created once, unchanged thereafter)
4. review artifact is written → already ignored by git
5. driver sees a clean `git status`

## mental model

### how a user describes it to a friend

> "the review bot writes a bunch of scratch files while it grades my work — this
> just tells git to ignore them, like a `node_modules` for reviews."

### analogies

- **`node_modules/.gitignore`**: machine-generated, present on disk, invisible to git.
- **a scratchpad**: you keep your notes, but they don't go in the final report.

### terms: theirs vs ours

| they say | we say |
|---|---|
| "review spam" / "noise" | peer-review artifacts |
| "the review folder" | `$route/.reviews/peer/` |
| "tell git to ignore" | findsert `.gitignore` |
| "make it once" | findsert (idempotent) |

## evaluation

### how well it solves the goal

directly. the wish is "stop spamming git" — a tracked `.gitignore` with `*` +
`!.gitignore` makes every artifact invisible while keeping the gitignore itself
tracked and self-documenting. the pattern is **proven** — two siblings already
ship in this exact codebase.

### pros

- tiny, surgical change — one new operation + one call site
- mirrors `findsertSelfReviewGitignore` exactly → low risk, easy review
- idempotent: created once, `unchanged` thereafter (no write churn)
- the `.gitignore` is itself tracked (`!.gitignore`) → self-documenting in the repo

### cons

- another findsert call in the guard hot path (negligible — a stat + maybe one write)
- if a human *wants* to commit a specific review artifact, they must `git add -f`
  (acceptable — matches self-review behavior)

### edgecases & pit-of-success

| edgecase | handling |
|---|---|
| `.gitignore` already correct | `action: 'unchanged'`, no write |
| `.gitignore` present but wrong content | overwritten to correct content |
| `.gitignore` present as a directory (EISDIR) | follows sibling behavior — must be covered by tests |
| dir not yet created | findsert does `mkdir -p` first |
| concurrent guard runs | findsert is idempotent; worst case a redundant identical write |

## open questions & assumptions

### assumptions

1. **path is `.reviews/peer/` (plural), not `.review/peer/`** — the wish says
   `.review/peer/*` but the actual code writes to `$route/.reviews/peer/`
   (`runStoneGuardReviews.ts:109`). **confirmed by the wisher: plural
   `.reviews/peer/`, no rename.** the singular in the wish was a typo.
2. the gitignore content should mirror `findsertSelfReviewGitignore`
   (`*` + `!.gitignore`), to ignore all artifacts too.
3. the call site belongs in `runOneStoneGuardReview`, right after the `mkdir`.
4. **error-handler must mirror `findsertSelfReviewGitignore`, NOT
   `findsertRouteGitignore`.** the two precedents diverge: selfReview uses
   `.catch((error) => { if ENOENT return null; throw error })` (rethrows real
   errors), while route uses `.catch(() => null)` (swallows *all* errors — a
   `failhide` per `rule.forbid.failhide`). we mirror the safe selfReview variant
   so a permission/EISDIR error surfaces loudly instead of a silent skip of the
   gitignore. (`findsertSelfReviewGitignore.ts:23-28` vs
   `findsertRouteGitignore.ts:25-27`.)

### open questions (triaged)

- **[answered] confirm the path**: `.reviews/peer/` (plural), or a rename to
  `.review/peer/`? **answered by the wisher: plural `.reviews/peer/`, no rename.**
  the code already uses plural everywhere (`runStoneGuardReviews.ts:109`,
  `enumRouteGuardReviewPeerFiles.ts:18`), so execution builds to the plural path
  with zero rename scope.
- **[answered] leaf vs parent dir**: should we findsert at the parent `.reviews/`
  or the `peer/` leaf? answered now via precedent: `findsertSelfReviewGitignore`
  targets the leaf (`review/self/`), and the `peer/` dir holds only peer artifacts,
  so the **leaf** is correct. no wisher input needed.
- **[answered] which error-handler to mirror**: answered now via code —
  mirror `findsertSelfReviewGitignore` (rethrow non-ENOENT), not the
  `findsertRouteGitignore` failhide (see assumption #4 above).
- **[answered] ignore-all vs allowlist**: answered now via logic — the peer dir
  holds only machine-generated artifacts, so `*` + `!.gitignore` (ignore-all) is
  correct; future must-track files can add a one-line `!exception`.

## what is awkward?

- **the wish's path typo** (`.review` vs `.reviews`) is the one real snag. it
  forces a "did you mean?" check before we proceed — easy to settle, but worth
  flagging so we don't build to the wrong path.
- **three near-identical findsert operations** would now exist
  (`route`, `selfReview`, and the new `reviewPeer`). this is mild WET duplication.
  per `rule.prefer.wet-over-dry`, three instances is exactly the threshold where
  abstraction *could* be considered — but the content strings and target subdirs
  differ, and the extant two were left un-abstracted, so mirroring is the
  lower-risk move. worth a note, not a blocker.
- no other part fights the mental model — this is a textbook "ignore generated
  files" pattern.
