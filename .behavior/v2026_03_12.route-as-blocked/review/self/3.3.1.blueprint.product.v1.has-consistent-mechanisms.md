# self-review: has-consistent-mechanisms

## review: are mechanisms consistent with extant patterns?

### challenge decision pattern

**extant pattern**: `getSelfReviewChallengeDecision` returns `'challenge:time' | 'challenge:first' | 'challenge:absent' | 'allowed'`

**blocked pattern**: `getBlockedChallengeDecision` returns `'challenge:first' | 'challenge:absent' | 'allowed'`

**consistency**: follows same pattern. omits `challenge:time` because blocked has no time enforcement. this is intentional simplification, not inconsistency.

### triggered file pattern

**extant pattern**: selfreview uses `.since` and `.uptil` files for time bounds

**blocked pattern**: uses single `.blocked.triggered` file

**consistency**: simpler pattern for simpler requirements. blocked has no time enforcement. single file sufficient to prove nudge was seen. not inconsistent — appropriately adapted.

### subdirectory pattern

**extant pattern**: `promise/` subdirectory for selfreview operations

**blocked pattern**: `blocked/` subdirectory for blocked operations

**consistency**: follows same organizational pattern. domain operations grouped by feature.

### stdout emit pattern

**extant pattern**: `formatRouteStoneEmit` with tree structure, owl header

**blocked pattern**: extends same format with blocked-specific nudge content

**consistency**: reuses extant format infrastructure. adds new content variant, not new format system.

### passage report pattern

**extant pattern**: `setPassageReport({ status: 'passed' | 'approved' | ... })`

**blocked pattern**: `setPassageReport({ status: 'blocked' })`

**consistency**: reuses extant operation. adds new status value. no mechanism change.

## conclusion

all mechanisms follow extant patterns. blocked-specific simplifications are justified by simpler requirements, not arbitrary divergence.
