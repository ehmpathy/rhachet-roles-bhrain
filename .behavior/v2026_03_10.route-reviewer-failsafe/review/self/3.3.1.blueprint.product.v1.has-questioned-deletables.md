# self-review: has-questioned-deletables

## components questioned

### classifyExitCode.ts - KEEP

**can this be removed?** no.

**if deleted and had to add back, would we?** yes. the exit code contract (0=passed, 2=constraint, other=malfunction) is the core semantic of the feature. without a dedicated function, this logic would be duplicated in `runStoneGuardReviews`, `runStoneGuardJudges`, and `stepRouteDrive`. single source of truth is worth it.

**simplest version?** a 3-line function. already minimal:
```ts
if (code === 0) return 'passed';
if (code === 2) return 'constraint';
return 'malfunction';
```

---

### formatTreeBucket.ts - KEEP

**can this be removed?** no.

**if deleted and had to add back, would we?** yes. the vision explicitly requires tree bucket format for stdout/stderr. reused for stdout AND stderr across BOTH reviews AND judges. 4 call sites minimum.

**simplest version?** could inline the logic in each call site, but that's 4x duplication of ~10 lines. a shared function is simpler.

---

### blocker: 'malfunction' - DELETE

**can this be removed?** **YES.**

**why delete?** the `blocker` field distinguishes block types when `status='blocked'`:
- status='blocked', blocker='review.peer' → blocked by peer review
- status='blocked', blocker='judge' → blocked by judge

but when `status='malfunction'`, the status is self-explanatory. `blocker='malfunction'` is redundant.

**action:** remove `blocker: 'malfunction'` from blueprint. keep only `status: 'malfunction'`.

---

### route.malfunction fixture - DELETE

**can this be removed?** **YES.**

**why delete?** extant tests create inline guards with echo commands:
```ts
'bash -c \'echo "passed: false"; exit 2\''
```

we can do the same for malfunction:
```ts
'bash -c \'echo "stdout"; exit 1\''
```

no separate fixture directory needed. inline test commands are sufficient and simpler.

**action:** remove `route.malfunction/` from filediff tree.

---

### formatRouteStoneEmit: tree buckets for ALL guards - SIMPLIFY

**can this be simplified?** **YES.**

**observation:** the vision shows tree bucket output for FAILED guards (exit != 0). for passed guards (exit 0), the current summary is sufficient:
```
└─ finished 10.4s ✓
```

**simplification:** only emit tree buckets for guards that fail. passed guards keep the current compact format.

**action:** update blueprint to clarify: tree buckets only for exit != 0.

---

### stepRouteDrive malfunction detection - KEEP

**can this be removed?** no.

**if deleted and had to add back, would we?** yes. without malfunction detection, route.drive treats malfunction like constraint failure and retries 21 times. immediate escalation on malfunction is the core feature.

**simplest version?** early return with exit 3 when status='malfunction' detected in passage.jsonl.

---

## changes to blueprint

1. **remove blocker: 'malfunction'** - redundant, status is self-explanatory
2. **remove route.malfunction fixture** - inline test commands are sufficient
3. **clarify tree bucket scope** - only for failed guards (exit != 0)

---

## summary

- 2 components deleted: `blocker: 'malfunction'`, `route.malfunction/`
- 1 component simplified: tree buckets only for failed guards
- 3 components kept as-is: classifyExitCode, formatTreeBucket, stepRouteDrive
- blueprint updated to reflect deletions and simplifications

---

## why the remaining components hold

### classifyExitCode holds because:
- it embodies the exit code contract (0/2/other)
- single source of truth prevents drift between callers
- if contract changes, one place to update

### formatTreeBucket holds because:
- used in 3 files: formatRouteStoneEmit, runStoneGuardReviews, runStoneGuardJudges
- each file uses it twice: stdout and stderr
- 6 call sites justify a shared function

### stepRouteDrive malfunction detection holds because:
- without it, malfunction would retry 21 times before escalation
- immediate halt on malfunction is the core requirement from the wish

### PassageReport.status: 'malfunction' holds because:
- persistence is required for route.drive to detect malfunction across sessions
- without it, the state would be lost on restart

### formatRouteStoneEmit tree buckets hold because:
- the vision explicitly requires visible stdout/stderr for failed guards
- this is the observable fix for "the error was not clear"

### runStoneGuardReviews/Judges changes hold because:
- they write artifacts that persist for later debugging
- they return results that CLI displays immediately
- both need exit code classification for malfunction detection
