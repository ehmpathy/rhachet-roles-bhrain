# 1.vision — fix review peer measurement

> extract the reviewer's verdict reliably: try the deterministic parse first, then fall
> back to a cheap brain when a raw reviewer brain won't conform to the numeric contract.

---

## the outcome world

### day-in-the-life

a human enrolls raw brain clis as peer reviewers, not the built-in review skill:

```sh
rhx enroll claude --roles reviewer,mechanic,architect
```

each reviewer brain reads the huge repo and emits a review. the guard needs one thing
from that review: **how many blockers, how many nitpicks**. today it parses that with a
regex (`getReviewCountsFromContent`). when a brain conforms — `0 blockers`, `2 nitpicks` —
the regex reads it and the stone proceeds.

but raw brains drift. one ends its review with:

```
looks solid overall — no major blockers, just a couple of style nits to clean up.
```

no numeric count. today the guard promotes this to `💥 malfunction` and blocks the stone.
the human is stuck: the review was *fine*, the reviewer just phrased its verdict in prose.

**after this wish:** when the regex finds no number, the guard hands that short review
text to a cheap, fast brain (default `fireworks/deepseek/v4-flash`) with a strict schema
and asks: "how many blockers, how many nitpicks?" the brain reads the small text (not the
huge repo) and returns `{ blockers: 0, nitpicks: 2 }`. the stone proceeds. the human never
sees a spurious malfunction.

### before / after

| | before | after |
|---|--------|-------|
| conforming reviewer | regex reads counts → proceed | regex reads counts → proceed (unchanged) |
| non-conforming reviewer | `💥 malfunction`, stone blocks | cheap brain extracts counts → proceed |
| truly-absent verdict | `💥 malfunction` | cheap brain can't find a verdict → `💥 malfunction` (still fails loud) |
| cost | 0 (regex only) | ~0 (regex) + one tiny brain call only on the fallback path |

### the "aha" moment

the value clicks when you realize **the expensive work already happened**. the reviewer
brain already read the whole repo and wrote its findings. all we're missing is a machine-
readable count. paying a full brain to re-review would be absurd — but paying a *tiny*
brain to read a few hundred bytes of already-written review and pull out two integers is
nearly free. we extract the contract we need instead of forcing the reviewer to conform.

## user experience

### usecases & goals

- **primary**: a human runs peer reviews with enrolled raw brain clis and wants the stone
  to pass/fail on the *substance* of the review, not on whether the reviewer happened to
  phrase its count in the exact numeric form.
- **secondary (unchanged)**: a human runs the built-in `review` skill, which already
  conforms — the deterministic path handles them with zero added cost.

### contract inputs & outputs

the change lives inside the peer-review measurement, so the *human-facing* cli contract is
unchanged — same `rhx route.stone.set --as arrived` flow. what changes is the internal
domain.operation that measures a single review.

today (deterministic only), the shape flattens absence into a fake zero:

```ts
// getReviewCountsFromContent.ts (today)
getReviewCountsFromContent(input: { content: string })
  : { detected: boolean; blockers: number; nitpicks: number }
// note the hazard: when detected=false it still returns blockers: 0, nitpicks: 0
// — a fiction a forgetful caller reads as "clean". see getReviewCountsFromContent.ts:56-57.
```

proposed — a **discriminated union** return shape, so absence carries no count fields at all.
the compiler forces every caller to narrow on `detected` before it can touch a number; the
`?? 0` fiction becomes unrepresentable, and `rule.forbid.failhide` is enforced by the type
system, not by convention.

```ts
// the leaf shape — a tactic does not know it sits in a waterfall, so it carries no `tactic`
type ReviewCounts =
  | { detected: false }
  | { detected: true; blockers: number; nitpicks: number };

// the orchestrator shape — stamps WHICH tactic yielded the verdict, only on the true branch
type ReviewCountsResolved =
  | { detected: false }
  | { detected: true; blockers: number; nitpicks: number; tactic: 'deterministic' | 'probabilistic' };
```

- **deterministic tactic** = today's `getReviewCountsFromContent`, re-shaped to `ReviewCounts`
  (pure, no brain). same detection logic, honest absence:

```ts
// getReviewCountsFromContent.ts (transformer — pure)
getReviewCountsFromContent(input: { content: string }): ReviewCounts
```

- **probabilistic tactic** = new leaf that calls the cheap brain, same `ReviewCounts` shape:

```ts
// getReviewCountsViaBrain.ts (communicator — calls brain)
getReviewCountsViaBrain(
  input: { content: string },
  context: ContextBrain<BrainChoice>,
): Promise<ReviewCounts>
```

- **root orchestrator** = waterfalls the two tactics, stamps `tactic`:

```ts
// getReviewCounts.ts (root namespace — orchestrates the tactics)
getReviewCounts(
  input: { content: string },
  context: ContextBrain<BrainChoice>,   // brain injected for the fallback tactic
): Promise<ReviewCountsResolved>
```

internally the probabilistic leaf leans on the proven pattern from `stepReview.ts:757`.
note `role: { briefs: [] }` stays empty — the extractor only tallies the reviewer's own
labels, so the reviewer brief set adds cost without value (and would bias it toward the
strict numeric contract the fallback exists to relax). the tiny gloss lives in the prompt:

```ts
const extracted = await context.brain.brain.choice.ask({
  role: { briefs: [] },   // empty by design — see note above + stepReview.ts:761 precedent
  prompt: `here is a code review's output. count the blockers and nitpicks it reports.
           trust the review's own labels. if an issue is unlabeled: blocker = must-fix,
           nitpick = optional. if the review reports a clean pass, detected=true with 0/0.
           if you cannot find a verdict at all, detected=false.\n\n---\n${input.content}`,
  schema: {
    output: z.discriminatedUnion('detected', [
      z.object({ detected: z.literal(false) }),
      z.object({
        detected: z.literal(true),
        blockers: z.number().int().min(0),
        nitpicks: z.number().int().min(0),
      }),
    ]),
  },
});
return extracted.output;
```

### what leverage looks like

the orchestrator reads as narrative:

```ts
// deterministic first — free, exact
const viaRegex = getReviewCountsFromContent({ content: input.content });
if (viaRegex.detected) return { ...viaRegex, tactic: 'deterministic' };

// probabilistic fallback — cheap brain reads the small review text
const viaBrain = await getReviewCountsViaBrain({ content: input.content }, context);
if (viaBrain.detected) return { ...viaBrain, tactic: 'probabilistic' };

// neither tactic found a verdict — the false branch carries no counts to fake
return { detected: false };
```

the call site in `runOneStoneGuardReview.ts:221` swaps its direct regex call for the
orchestrator. today it blind-destructures `const { blockers, nitpicks } = counts` (`:213`) —
under the union that will no longer compile, which is the point: the caller must narrow on
`detected` first. the `exitCode === 0 && !detected → malfunction` promotion stays, but now it
only fires after *both* tactics fail, so it truly means "no verdict exists," not "the verdict
wasn't a number." the artifact may still record `0/0` for the malfunction case — there the
`exitClass: 'malfunction'` carries the "don't trust these" signal, so it stays unambiguous.

### what the human sees (stdout)

the guard already renders each reviewer as a 4-line block via `formatGuardReviewerTree.ts`.
today the status line carries only `verdict + duration`:

```
🗿 guard: reviewed?
   ├─ r1: peer/mechanic (l1, 1/∞)
   │  ├─ approved 12.4s
   │  ├─ 0 blockers ✓
   │  ├─ 2 nitpicks 🟠
   │  └─ review: .reviews/peer/...r1.mechanic.md
   └─ r2: peer/architect (l1, 1/∞)
      ├─ approved 9.1s
      ├─ 0 blockers ✓
      ├─ 0 nitpicks ✓
      └─ review: .reviews/peer/...r1.architect.md
```

**[wisher confirmed]** when a count came from the fallback, a dedicated branch appears directly
below the nitpick row — grouped with the counts it produced — and it names the brain that
tallied them: `tallied by reviewer@$bhrain`. the deterministic path shows no such branch (the silent
common case), so the branch's *presence* is the whole signal:

```
🗿 guard: reviewed?
   ├─ r1: peer/mechanic (l1, 1/∞)
   │  ├─ approved 12.4s
   │  ├─ 0 blockers ✓                        ← deterministic: reviewer emitted `2 nitpicks`, no tally branch
   │  ├─ 2 nitpicks 🟠
   │  └─ review: .reviews/peer/...r1.mechanic.md
   └─ r2: peer/architect (l1, 1/∞)
      ├─ approved 9.1s
      ├─ 0 blockers ✓
      ├─ 1 nitpick 🟠
      ├─ tallied by reviewer@fireworks/deepseek/v4-flash  ← probabilistic: reviewer wrote prose, sub-brain tallied it
      └─ review: .reviews/peer/...r1.architect.md
```

mechanically: the `finished` state in `ReviewerTreeState` (`formatGuardReviewerTree.ts:24-31`)
gains a `tactic: 'deterministic' | 'probabilistic'` field plus the brain slug; the detail-line
build (`:102-131`) inserts one extra branch — `tallied by reviewer@${brain}` — right after the
`nitpicks` line (before the `review:` path), only when `tactic === 'probabilistic'`. a snapshot
test locks both renders. the `reviewer@` prefix names the role that ran the tally (a reviewer
sub-brain), so the slug reads as `role@brain` — consistent with how brains are addressed.

### timeline

1. reviewer brain runs (seconds–minutes), emits review to stdout.
2. deterministic parse (instant). if it finds counts → done, no tally branch.
3. only on miss: one cheap brain call on a few hundred bytes (sub-second, fractions of a cent).
4. guard classifies the review with the extracted counts; a `tallied by reviewer@$bhrain` branch appears.

## mental model

### how a user describes it to a friend

> "the reviewer already did the hard part — it read the whole repo and wrote up what's
> wrong. i just need the tally. if it wrote the tally as a number, great. if it wrote it in
> english, i have a little cheap robot read the summary and tell me the number."

### analogies

- **a translator at the finish line**: the marathoner (big reviewer brain) already ran the
  race. we just need someone to read their handwritten time and write it on the scoreboard.
- **OCR for verdicts**: the regex is exact-match text search; the brain is OCR that reads
  messy handwriting when the clean print isn't there.
- **waterfall / fallback chain**: like `git config` local → global → system, we try the
  cheapest source of truth first and fall through only on miss.

### terms — theirs vs ours

| user says | we say |
|-----------|--------|
| "read the count" | `getReviewCounts` (orchestrator) |
| "the exact-match way" | deterministic tactic (`getReviewCountsFromContent`) |
| "have the cheap robot read it" | probabilistic tactic (`getReviewCountsViaBrain`) |
| "it couldn't tell" | `detected: false` → `malfunction` |
| "cheap robot" | `context.brain.choice.ask` on `fireworks/deepseek/v4-flash` |

## evaluation

### how well it solves the goals

- ✅ **non-conforming reviewers no longer spuriously block** — the core pain in the wish.
- ✅ **cheap** — brain call only on the fallback path, on tiny input, on a small model.
- ✅ **waterfall as specified** — deterministic tactic first, probabilistic fallback, root
  namespace orchestrates the tactics (exactly the wish's structure).
- ✅ **failloud preserved** — a review with genuinely no verdict still becomes a malfunction.

### pros

- reuses the proven `context.brain.choice.ask({ schema })` pattern already in `stepReview.ts`.
- clean grain separation: pure transformer (regex) + communicator (brain) + orchestrator.
- the `tactic` field on the true branch makes the path observable/testable and snapshot-friendly.
  **[wisher confirmed]** it also surfaces in the guard's treestruct — a `tallied by reviewer@$bhrain`
  branch below `approved` — so a human always sees when a count came from the fallback, and which
  brain tallied it.
- deterministic path stays hot and free for the common (conforming) case.
- **discriminated-union return** makes the failhide unrepresentable: the `{ detected: false }`
  branch carries no count fields, so a caller *cannot* read a fake `0/0`. the compiler enforces
  `rule.forbid.failhide` at the extraction boundary.

### cons / tradeoffs

- introduces a brain dependency into the guard's measurement path (previously pure). the guard
  holds no brain context today (`ContextCliEmit.ts:21`), so a brain context threads in-process
  through 5 hops (route cli → … → runOneStoneGuardReview). a wider blast radius than a leaf
  change, but each hop is a one-line context-type widen — accepted per the wisher's decision.
- probabilistic tactic is non-deterministic — tests must use `when.repeatably` per
  `rule.require.repeatable-for-llm-tests`.
- a mis-extracting brain could turn a real malfunction into a false "approved" (or vice versa).
  the union hardens the *structure* (no fake 0/0), but a *confident wrong* `detected: true` is
  still possible — mitigated by the "trust the labels" prompt, the small model's narrow task,
  and adversarial boundary tests.

### edgecases & pit-of-success

| edgecase | contract keeps user safe by |
|----------|-----------------------------|
| reviewer emits `3 blockers` in prose then `0 blockers` in summary | deterministic tactic already takes the LAST match; fallback never runs |
| reviewer emits only prose "no blockers, some nits" | fallback brain reads it → `{ detected: true, blockers: 0, nitpicks: N }` |
| reviewer emits truly empty / garbage | both tactics return `{ detected: false }` → `malfunction` (fail loud); no count fields to fake |
| brain itself errors / times out | fallback surfaces as malfunction, never silent-passes |
| review is genuinely clean | `{ detected: true, blockers: 0, nitpicks: 0 }` → approved, correct — a *declared* 0/0, distinct from the union's countless `{ detected: false }` |
| conforming reviewer (built-in skill) | deterministic tactic wins, zero brain cost |

## open questions & assumptions

### assumptions made

1. the `.choice.ask({ schema: { output } })` API is real (verified, `stepReview.ts:757`). the
   guard does **not** hold a brain context today — **confirmed via code**: `ContextCliEmit`
   carries only `cliEmit` callbacks (`ContextCliEmit.ts:21-33`), and `runOneStoneGuardReview`
   takes **no context at all** (`runStoneGuardReviews.ts:103-112`); reviewers run as exec
   subprocesses. **[wisher confirmed] direction: thread a brain context in-process** (not a
   subprocess). the path, hop by hop, each hop widens `ContextCliEmit` →
   `ContextCliEmit & ContextBrain<BrainChoice>`:

   ```
   contract/cli/route.ts:858   build brain via genContextBrain (same maker as review.ts:186)
     └─ stepRouteStoneSet.ts:48
        └─ setStoneAsPassed.ts:55  (calls runStoneGuardReviews at :309)
           └─ runStoneGuardReviews.ts:286
              └─ runOneStoneGuardReview.ts:103  (adds a context arg; today it has none)
                 └─ getReviewCounts(input, context)  ← the orchestrator, where the brain is used
   ```
2. **[wisher confirmed]** fallback brain is `fireworks/deepseek/v4-flash`, fixed and not
   configurable (the wish names it; it is the `DEFAULT_TEST_BRAIN` in `genTestBrainContext.ts:22`).
3. **[wisher confirmed]** the fallback runs **only** when `exitCode === 0` (a review that ran but
   phrased its verdict oddly). a non-zero exit is a real failure and is never "rescued" by a brain.
4. **[answered — decided with wisher]** the return shape is a **discriminated union**
   (`{ detected: false } | { detected: true; blockers; nitpicks }` at the leaf,
   `+ tactic` at the orchestrator), not a flat `{ detected, blockers, nitpicks }`. this removes
   the `?? 0` fiction at `getReviewCountsFromContent.ts:56-57` and makes the failhide a compile
   error. it does ripple to the call site (`runStoneGuardReviews.ts:213` blind-destructure must
   narrow first) — an accepted, welcome cost. `role: { briefs: [] }` stays empty for the
   extractor (it only tallies the reviewer's own labels).

### questions for the wisher — triaged

- **[answered — wisher confirmed] brain-context shape**: the guard has no brain context today
  (`ContextCliEmit.ts:21`, `runStoneGuardReviews.ts:103`). **decision: thread a brain context
  in-process** down the call chain (route cli → stepRouteStoneSet → setStoneAsPassed →
  runStoneGuardReviews → runOneStoneGuardReview → getReviewCounts), *not* a subprocess. brain
  built at the cli entrypoint via `genContextBrain` (the maker already used by `review.ts:186`).
- **[answered — wisher confirmed] non-zero exit rescue**: **no.** the fallback runs *only* when
  `exitCode === 0`. a non-zero exit is a real failure and is never "rescued" by the brain.
- **[answered — wisher confirmed] prod default model**: `fireworks/deepseek/v4-flash`, fixed.
  **not configurable** per route/guard — one default, keep it simple.
- **[answered — wisher confirmed] observability**: **yes, surface it.** the guard's treestruct
  shows a `tallied by reviewer@$bhrain` branch below the nitpick row when a count came from the

  the deterministic path shows no such branch (the silent, common case).
- **[research] adversarial-prose accuracy**: how reliably does a cheap model count / honor a
  `detected:false` on mentioned-then-cleared or empty reviews? answer in the blueprint's
  boundary tests (`rule.require.repeatable-for-llm-tests`), not via a question to the wisher.

## groundwork

### external research

none — no external dependencies. the fallback uses the extant in-repo brain abstraction
(`context.brain.choice.ask`) and already-installed brain packages
(`rhachet-brains-fireworksai` for `deepseek/v4-flash`). verified the schema-based `.ask`
call is present and exercised in prod code at `src/domain.operations/review/stepReview.ts:757`.

### internal research

verified extant behavior the vision must conform to:

- **deterministic parser**: `src/domain.operations/route/guard/getReviewCountsFromContent.ts:39`
  — `getReviewCountsFromContent({ content }) → { detected, blockers, nitpicks }`. takes the
  LAST numeric match per dimension; `detected` true only when both dimensions carry a number.
- **the waterfall insertion point**:
  `src/domain.operations/route/guard/runStoneGuardReviews.ts:221` — where
  `exitCode === 0 && !counts.detected` currently promotes straight to `💥 malfunction`. the
  probabilistic tactic slots in *before* this promotion.
- **brain schema-ask pattern**: `src/domain.operations/review/stepReview.ts:757` —
  `context.brain.brain.choice.ask({ role, prompt, schema: { output: zodSchema } })` returns
  `{ output, metrics }`. this is the exact pattern the probabilistic tactic reuses.
- **contract to honor**: `.agent/repo=bhrain/role=reviewer/briefs/contract.reviewer-output.md`
  — numeric-only contract, `N blockers` / `N nitpicks`, else malfunction. the probabilistic
  tactic *relaxes the input* the human's reviewer must produce, but the guard's *internal*
  invariant (a numeric count must be present to proceed) is preserved.
- **output types**: `src/domain.objects/Driver/RouteStoneGuardReviewArtifact.ts:11` —
  `RouteStoneGuardReviewPeerArtifact` carries `{ blockers, nitpicks, exitCode, exitClass }`.
- **verdict logic**: `src/domain.operations/route/guard/reviewPeerMeter/computeReviewPeerVerdict.ts:24`
  — already waterfalls verdicts (malfunction → constraint → queued → approved → exhausted →
  rejected). our change feeds it trustworthy counts; its logic is untouched.
- **grain/name conventions**: repo uses `get*` for retrieval and threads `(input, context)`.
  the orchestrator `getReviewCounts` + leaf tactics `getReviewCountsFromContent` (transformer)
  and `getReviewCountsViaBrain` (communicator) fit `rule.require.get-set-gen-verbs` and the
  transformer/communicator/orchestrator grain model.
- **llm test rule**: `rule.require.repeatable-for-llm-tests` — the probabilistic tactic's
  tests must use `when.repeatably(REPEATABLE_CONFIG)`.

### what is awkward?

- **a pure guard path gains a brain dependency.** the measurement was deterministic and
  cheap; threading a brain context into it is the main friction. it may require widening the
  context type the guard receives. worth confirming the plumbing before committing to a shape.
- **naming collision risk**: `getReviewCounts` (orchestrator) vs `getReviewCountsFromContent`
  (deterministic leaf) vs `getReviewCountsViaBrain` (probabilistic leaf) — three similar
  names. it reads well as a tactic family but a reviewer could find them easy to confuse.
- **non-determinism at a trust boundary.** the whole point of the guard is a trustworthy
  verdict. delegating the count to a probabilistic brain — even a cheap one on tiny input —
  sits uncomfortably against `philosophy.verification-strictness`. the mitigation (only rescue
  `exitCode===0`, strict schema, `detected` flag, boundary tests) must be airtight.
- **"couldn't find" is doing double duty**: deterministic-miss and brain-says-no-verdict both
  land at `detected:false`. that's correct, but the human should be able to tell *why* it
  malfunctioned (odd phrasing the brain also couldn't parse vs. genuinely empty). the `tactic`
  field helps but the stderr message may need to distinguish them.
