# blackbox criteria: fix driver heap OOM

## usecase.1 = guard with broad artifact pattern completes without OOM

given a guard with artifact pattern `src/**/*` that matches 700+ files
  when the driver runs `rhx route.stone.set --as arrived`
    then the guard completes successfully
      sothat users can use broad globs without memory configuration
    then the guard completes in under 10 seconds
      sothat performance is acceptable for interactive use
    then heap usage stays under 512MB
      sothat default node heap limits are sufficient

given a guard with artifact pattern that matches 1000+ files
  when the driver runs `rhx route.stone.set --as arrived`
    then the guard completes without OOM
      sothat any repo size is supported

## usecase.2 = review with many target files completes without OOM

given a review invocation with `--paths 'src/**/*'` that matches 700+ files
  when the review runs
    then the review completes successfully
      sothat users can review broad scopes
    then heap usage stays under 512MB
      sothat default node heap limits are sufficient

given a review with 100+ rule files
  when the review runs
    then the review completes without OOM
      sothat large rule sets are supported

## usecase.3 = stdout shows glob patterns, not expanded file lists

given a guard with artifact pattern `src/**/*`
  when the driver runs `rhx route.stone.set --as arrived`
    then stdout shows `src/**/*` (the glob pattern)
      sothat output is scannable
    then stdout does NOT show 700 individual file paths
      sothat output stays concise

given a review with `--paths 'src/**/*.ts'`
  when the review runs
    then stdout shows `src/**/*.ts` (the glob pattern)
      sothat users see their input, not the expansion

## usecase.4 = hash computation is deterministic

given a guard with artifacts that have not changed
  when the driver runs twice
    then the computed hash is identical both times
      sothat cache lookups work correctly

given a file that is modified but not committed
  when the driver computes the artifact hash
    then the hash reflects the current worktree content
      sothat modified files invalidate the cache
    then the hash does NOT reflect the committed content
      sothat stale cache hits are prevented

given a file that is untracked (new, not staged)
  when the driver computes the artifact hash
    then the hash includes the untracked file content
      sothat new files invalidate the cache

## usecase.5 = regression prevention via test coverage

given the stdout artifact display behavior
  when the test suite runs
    then snapshot tests verify glob patterns appear in output
      sothat regressions are caught in CI
    then snapshot tests verify expanded paths do NOT appear
      sothat output bloat is prevented
