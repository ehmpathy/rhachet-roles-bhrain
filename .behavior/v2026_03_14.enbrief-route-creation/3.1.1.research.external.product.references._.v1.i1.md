# research: references for enbrief route creation

## summary

this document compiles authoritative references that ground our knowledge for route creation documentation. references are organized by domain: workflow systems, documentation patterns, and implementation standards.

---

## foundational references

### ref.1 = the checklist manifesto

**credibility**: [book] — seminal work on checklist effectiveness

**citation**: Gawande, Atul. *The Checklist Manifesto: How to Get Things Right*. Metropolitan Books, 2009.

**url**: https://atulgawande.com/book/the-checklist-manifesto/

**key concepts**:
- checklists reduce errors in complex work
- two types: PAUSE AND CHECK vs READ AND DO
- works across domains: aviation, medicine, software

**relevance**: thought routes are structured checklists with validation gates.

---

### ref.2 = progressive disclosure (nielsen norman group)

**credibility**: [official] — usability research authority

**url**: https://www.nngroup.com/articles/progressive-disclosure/

**key concepts**:
- show essential info first, details on demand
- improves learnability, efficiency, error rate
- defer advanced features to secondary screens

**relevance**: brief should use progressive disclosure — basics first, advanced patterns (parallel stones, nested routes) later.

---

### ref.3 = command line interface guidelines

**credibility**: [official] — community standard

**url**: https://clig.dev/

**key concepts**:
- prioritize clarity over fast type
- provide help at the touchpoint
- error messages should be descriptive + actionable

**relevance**: CLI commands (`rhx route.bind.set`, etc.) should follow these guidelines.

---

### ref.4 = convention over configuration

**credibility**: [reference] — established pattern

**url**: https://en.wikipedia.org/wiki/Convention_over_configuration

**key concepts**:
- decrease developer decisions without lose flexibility
- developer only specifies unconventional aspects
- examples: Rails, ASP.NET MVC, Laravel

**relevance**: route conventions (`.route/v$isodate.$slug/`, `N.name.stone`) are CoC patterns.

---

### ref.5 = github actions workflow conventions

**credibility**: [official] — platform documentation

**url**: https://docs.github.com/en/actions/about-workflows

**key concepts**:
- workflows in `.github/workflows/` directory
- yaml-based workflow definitions
- standardized location enables toolchain integration

**relevance**: similar pattern — routes in `.route/v$date.$slug/`, stones as workflow steps.

---

### ref.6 = quality gates in CI/CD

**credibility**: [practitioner] — industry consensus

**urls**:
- https://testrigor.com/blog/software-quality-gates/
- https://www.infoq.com/articles/pipeline-quality-gates/
- https://www.perforce.com/blog/sca/what-quality-gates

**key concepts**:
- automated checkpoints before phase progression
- can include tests, coverage, security scans, approvals
- improve quality AND velocity (not tradeoffs)

**relevance**: guards are quality gates for thought work.

---

### ref.7 = git branch name conventions

**credibility**: [official] — platform guidance

**urls**:
- https://learn.microsoft.com/en-us/azure/devops/repos/git/git-branch-guidance
- https://graphite.dev/guides/git-branch-name-conventions

**key concepts**:
- lowercase + hyphens
- include ticket numbers for traceability
- consistent patterns enable automation

**relevance**: branch names pair with route binds (`route.bind.set`).
