# 1.vision — driver status line

## .what (one breath)

a driver walks a route, stone by stone. today, an observer must run `rhx route.drive`
(or read the scrollback) to learn which stone the driver is on. this wish paves a
glance: the current stone name, rendered live in the claude code status line, so
anyone who watches the session sees the step at a glance — no command, no scroll.

rendered as:

```
🗿 1.vision
```

**on `:maui head:`** — this is the **moai stone-head** 🗿, the driver's own icon. the
driver navigates *stones*, and a moai is a stone head — so the prefix marks "the stone
the driver's head is at." (the owl 🦉 is the repo's human-chat mascot; the *driver
artifact* is the stone, hence the moai.) render the moai glyph `🗿 <stone>` as the
clean default; the literal label `:maui head:` is the origin label.

---

## the outcome world

### day-in-the-life

a human opens a claude code session bound to a route. at the very bottom of the
terminal, always in view, sits one quiet line:

```
🗿 1.vision
```

the driver does its work, self-reviews, passes the stone. moments later the line
updates itself:

```
🗿 5.1.execution.from_vision
```

the human never typed a thing. they glanced down, saw the step, and looked away.
when the route finishes (or no route is bound), the line simply isn't there — no
noise, no stale value.

### before / after

| | before | after |
|---|--------|-------|
| "which stone are we on?" | run `rhx route.drive`, read the tree, find `stone =` | glance at the status line |
| observing a long autonomous drive | scroll back through output | one line, always pinned |
| onboarding a watcher | "run this command to check progress" | "look at the bottom of the screen" |
| stone changes | invisible until you re-ask | line updates on its own |

### the "aha"

the value clicks the first time the driver passes a stone and the observer sees
the status line tick forward **without asking**. progress becomes ambient. the
route stops being a thing you query and becomes a thing you feel.

## user experience

### who & what goal

- **the observer / co-pilot human** — wants ambient awareness of where the
  autonomous driver is, without interrupting or querying.
- **the driver's operator** — wants to confirm at a glance that the driver is on
  the expected stone before walking away or context-switching.

### the contract (two layers)

this wish is two cooperating contracts, per the ehmpathy pattern (an init that writes
settings.json + a command the harness runs):

**layer 1 — the installer (init hook)**
a **dedicated driver init** that **merges a `statusLine` block into
`.claude/settings.json`** (idempotently), so the harness knows to render our line.

**the pattern = the ehmpathy permissions installer.** the model is *not* the
print-only `sessionstart.notify-permissions` — it is the real settings-writer,
`init.claude.permissions.sh` + `init.claude.permissions.jsonc` (verified first-hand in
`rhachet-roles-ehmpathy/dist/domain.roles/mechanic/inits/`). that init ships a `.jsonc`
config and uses **bash + jq** to merge it into `.claude/settings.json`, preserve all
other keys, back up on change, and no-op when already applied. our init copies that
mechanism exactly, but for the `statusLine` key instead of `.permissions`.

#### how the init writes settings.json (mirrors `init.claude.permissions.sh`)

the bash does, in order (line refs to the ehmpathy installer we mirror):

1. **locate** the target: `PROJECT_ROOT="$PWD"`;
   `SETTINGS_FILE="$PROJECT_ROOT/.claude/settings.json"` (installer runs from repo
   root at boot). — permissions.sh:32-33
2. **load** the shipped config beside the init file: resolve the init's own dir
   (`$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)`), read `init.claude.status-line.jsonc`
   (a `.jsonc` that holds the `statusLine` block), strip `//` comments via
   `grep -v '^\s*//' | sed 's|[[:space:]]//.*||'` (preserves `://`), then `jq -c '.'`.
   — permissions.sh:36-49
3. **ensure** the file: `mkdir -p "$(dirname "$SETTINGS_FILE")"`; if absent,
   `echo "{}" > "$SETTINGS_FILE"`. — permissions.sh:51-57
4. **merge** with jq into a tmp file — the one line that differs from permissions.sh:
   ```bash
   jq --argjson sl "$STATUS_LINE_CONFIG" '.statusLine = $sl.statusLine' \
     "$SETTINGS_FILE" > "$SETTINGS_FILE.tmp"
   ```
   `.statusLine = $sl.statusLine` sets our key and **leaves every other key
   untouched** (hooks, permissions, …), exactly as the permissions jq preserves
   non-`.permissions` keys. — permissions.sh:63-75
5. **idempotency check**: compare before/after semantically —
   `jq -e --slurpfile before … --slurpfile after … -n '$before[0].statusLine == $after[0].statusLine'`;
   if equal, `rm` the tmp and exit 0 with "already configured". — permissions.sh:77-86
6. **backup + atomic swap** on change: `cp "$SETTINGS_FILE" "$BACKUP_FILE"` (ISO-dated),
   then `mv "$SETTINGS_FILE.tmp" "$SETTINGS_FILE"`. — permissions.sh:88-92
7. `set -euo pipefail` + an `ERR` trap that prints the failing line — fail-fast, fail-loud.

#### the shipped `.jsonc` (what gets merged)

```jsonc
// init.claude.status-line.jsonc — shipped beside the init, merged verbatim
{
  "statusLine": {
    "type": "command",
    "command": "./node_modules/.bin/rhx route.status.line --syntax plaintext",
    "padding": 0
  }
}
```

#### registration (the one code change beyond the init files)

`getDriverRole.ts` currently has **no `inits` block** (confirmed:
`src/domain.roles/driver/getDriverRole.ts` defines `skills`, `briefs`, `hooks` but no
`inits`). the mechanic role registers its installer via
`inits: { dirs, exec: [{ cmd: __dirname + '/inits/init.claude.sh' }] }`. so the build
adds an analogous `inits` block to the driver role that points at
`__dirname + '/inits/init.claude.status-line.sh'`. boot then runs it, same as the
mechanic's permissions init runs under `rhachet roles boot --role mechanic`.

- **home:** `src/domain.roles/driver/inits/` (a reusable driver-role capability — every
  repo that boots the driver inherits it, per the wisher).
- **target:** the committed `.claude/settings.json` (shared with the team).
- **input:** none (reads `$PWD` + the shipped `.jsonc`).
- **output:** settings.json gains the `statusLine` block; stdout a short owl note (or
  "already configured" on no-op).

**layer 2 — the renderer (a driver skill)**
the command the harness invokes (repeatedly). its job is to **emit the current stone
name**, prefixed.

- input: claude code status payload on **stdin** (JSON with `cwd`,
  `workspace.current_dir`, `worktree.path`, `session_id`, …)
- output (stdout): `🗿 <stone.name>` (the moai stone-head prefix) — or **an empty line**
  when no route is bound / no next stone
- exit 0 on the expected paths (bound → the line; unbound/complete → empty line). a
  **genuine fault is not swallowed**: it is logged to stderr and the process exits
  non-zero, so the harness blanks the line (fail-open *via the harness*, not a catch-all
  exit 0). this satisfies `rule.forbid.failhide` — see the amendment note below

> **amendment (build-time reconciliation).** an earlier draft of this vision said
> "renderer errors: catch all → exit 0 always." that directly violates the repo's
> `rule.forbid.failhide` (a mega-blocker: never swallow a real error). the peer reviewer
> `mech-failhides` rejected the catch-all-exit-0 pattern outright. the reconciled contract
> is **fail-loud, not fail-hidden**: the renderer catches only to log the fault to stderr
> and then rethrows, so the fault is observable *and* the harness still blanks the line on
> the non-zero exit. the "never break the session" goal is preserved (the harness blanks on
> any non-zero exit); only the *mechanism* changed from "swallow" to "surface + rethrow".

### what it looks like to leverage

nobody "invokes" this by hand in the happy path — that's the point. the observer just
watches. the only human touch is the one-time nudge to register the installer (or it
auto-registers on the next `roles boot` / session start, same as the other hooks).

to debug, a human can run the renderer directly from the repo root:

```bash
# amended (build-time): the shipped renderer reads no stdin, takes no --syntax flag,
# and is invoked via `node -e` (not `rhx`, whose banner would pollute the one line).
# it uses cwd for the bound-route lookup, so run it from the repo root.
node -e "import('rhachet-roles-bhrain/cli/route').then(m => m.routeStatusLine())"
# -> 🗿 1.vision
```

### timeline

```
session start
  └─ installer init runs → settings.json has statusLine block   (once, idempotent)

... driver works ...
  ├─ harness fires statusLine (after each assistant msg, debounced 300ms)
  │    └─ renderer reads bound route + next stone → "🗿 1.vision"
  ├─ driver passes 1.vision
  └─ harness fires statusLine again
       └─ renderer → "🗿 5.1.execution.from_vision"

route unbound / complete
  └─ renderer emits an empty line → line disappears
```

## evaluation

### how well it solves the goal

directly. the goal is "observers glance at the current step." a pinned status line is
the canonical claude code surface for exactly this — always visible, auto-refreshed,
zero interaction. it fits the harness's intended use of `statusLine`.

### pros

- ambient, zero-friction awareness — no command, no scroll
- self-updating on every assistant turn (harness-driven)
- mirrors an established repo pattern (ehmpathy init writes settings.json)
- reuses extant stone-lookup logic (`getRouteBindByBranch`, `getAllStones`,
  `computeNextStones`) — no new domain truth invented
- degrades to silence when unbound (no stale/misleading value)

### cons / costs

- the renderer runs often (debounced 300ms, on every turn). stone lookup globs
  the route dir + reads artifacts. must stay **fast**, or the harness cancels it and
  the line flickers/blanks.
- writing into settings.json is a shared, human-owned file — the installer must be
  careful and idempotent (never clobber a human's own `statusLine`).
- one more hook in an already-long SessionStart list.

### edge cases & pit-of-success

| edge case | contract keeps you safe by |
|-----------|---------------------------|
| no route bound | renderer emits an empty line → line just absent |
| route bound, all stones passed | emit an empty line (or a done marker) → no confusion |
| multiple routes bound to branch | `getRouteBindByBranch` throws; the fault propagates, exits non-zero, harness blanks the line (fault surfaced on stderr, not swallowed) |
| cwd is a worktree / subdir | the harness runs the renderer with cwd = project root, so the lookup scans from cwd directly (no stdin payload, no chdir) |
| human already set a `statusLine` | installer skips-and-warns: never overwrites a human's own `statusLine` (vision Q2) |
| stone lookup slow | keep the read cheap; the lookup globs the route dir once per refresh — fast enough, no cache in v1 |
| renderer errors | **fail-loud, not fail-hidden**: log the fault to stderr and rethrow → non-zero exit → harness blanks the line. satisfies `rule.forbid.failhide` while the session stays safe |

## proposed v1 scope (a firm recommendation, not just options)

after the harder requirements pass, i take a position rather than leave everything
open:

1. **build the renderer first, and treat it as the whole of v1's core value.**
   `route.status.line --syntax plaintext` is the piece that delivers the wish. it is small,
   testable in isolation, and useful even if a human installs the config by hand. do
   not let it wait on the installer.

2. **install via a dedicated driver init (wisher's call).** a new init under
   `src/domain.roles/driver/inits/` writes the `statusLine` key into the committed
   `.claude/settings.json`, per the `achiever/inits/init.claude.sh` backup→write
   pattern. it is a reusable driver-role capability — every repo that boots the driver
   inherits it. registered under `SessionStart.hooks`, authored `repo=bhrain/role=driver`.

3. **prefix = the moai stone-head `🗿` (resolved by the wisher).** `:maui head:` is the
   moai — the driver's own stone-head icon. render `🗿 <stone>`. the literal
   `:maui head:` text is the alternative, but the glyph is the clean default.

this trims v1 to: **one renderer skill + one line added to the driver's boot write**,
with the moai prefix settled. everything heavier (bespoke init, cache, refreshInterval,
done-marker) is deferred until asked.

## open questions & assumptions

### assumptions made

1. **the prefix is the moai stone-head `🗿`** (the driver's icon), resolved by the
   wisher — `:maui head:` names the moai. render `🗿 <stone>`; the literal text is the
   fallback label.
2. **two-layer design is wanted** (installer writes settings.json + renderer skill),
   matching "just like ehmpathy runs updates on init." the alternative — a plain
   SessionStart print — can't stay pinned, so it doesn't satisfy "glance anytime."
3. **stone name = `RouteStone.name`** (e.g. `1.vision`, `5.1.execution.from_vision`),
   not the path, not the guard — exactly as the wish says "just the stone name."
4. the renderer emits **an empty line** when unbound (line absent) rather than a
   placeholder.
5. **the renderer's cwd/branch matches bind detection.** `getRouteBindByBranch` scans
   from `process.cwd()` and reads the current git branch. i assume the statusLine
   subprocess runs at the repo root on the bound branch. if the harness spawns it from
   a different cwd, the renderer must derive its anchor from the payload's
   `workspace.current_dir` / `worktree.path` instead of blind `process.cwd()`. this is
   a real build constraint, not a given.

   > **amendment (build-time): the cwd-anchor risk is accepted, not mitigated.** the
   > repo-rules blockers `rule.forbid.cwd-outside-gitroot` + `rule.require.explicit-stdin-flags`
   > removed the stdin-payload/chdir mechanism entirely, leaving `process.cwd()` as the
   > sole anchor. the claude code harness documents that it runs the statusLine command
   > with **cwd = project root**, so under the documented contract the anchor is correct.
   > the residual risk: *if* a future harness ever spawns from a subdirectory, the bind
   > glob (`**/.route/.bind.*.flag`, at-or-below cwd) would find no flag and the renderer
   > would emit an empty line — indistinguishable from "unbound". this tradeoff is
   > **accepted for v1** (the documented contract holds today; the failure mode is a blank
   > line, never a crash). if the harness contract changes, the fix is a payload-anchored
   > lookup guarded to stay within gitroot — deferred until there is evidence it is needed.
6. **"current stone" is singular.** `@next-one` returns the first incomplete stone.
   but this route system supports **parallel** stones (same numeric prefix, e.g.
   `3.1.a` / `3.1.b`). the wish says "which stone" (singular), so i assume the first
   stone is acceptable to show — but a parallel group is a real counterexample where
   one name under-tells the truth.
7. **`settings.json` means the shared, committed file.** `.claude/settings.json` is
   git-tracked here, so an auto-write is shared with every teammate on the repo. i
   assume that is intended (a team capability), not a personal `settings.local.json`
   preference. the wish did say "settings.json" — but the shared-vs-personal
   consequence was inferred, not stated.

### external research (groundwork)

- **claude code `statusLine` contract — verified** against official docs
  (https://code.claude.com/docs/en/statusline.md):
  - shape: `settings.json → statusLine: { type: "command", command, padding?, refreshInterval? }` ✔
  - invocation: command receives a **JSON payload on stdin** with `cwd`,
    `workspace.current_dir`/`project_dir`, `session_id`, `worktree.*`, etc. ✔
  - output: stdout lines rendered; ANSI color supported; **non-zero exit or no output
    ⇒ blank line** ✔
  - refresh: event-driven (after each assistant message, on compact, on mode change),
    debounced 300ms, in-flight run cancelled if superseded; optional `refreshInterval`
    for timer-based ✔
  - trust: same trust gate as hooks; disabled by `disableAllHooks: true`
  - **constraint that shapes design:** it runs often and blocks the line while running
    → the renderer must be fast and never error-out.

## what is awkward?

1. **`:maui head:` — resolved, no longer awkward.** an earlier draft misread this as a
   mascot conflict. it is not: `:maui head:` is the **moai stone-head** 🗿, the driver's
   own icon. the owl is the repo's human-chat mascot, but the *driver artifact* is the
   stone, so a stone-head prefix is exactly right. the only micro-choice left is glyph
   (`🗿`) vs literal label (`:maui head:`) — cosmetic, default to the glyph.
2. **a status line that runs stone-lookup every ~turn.** globbing the route dir on
   every refresh is a little wasteful for "print one short string." tolerable, but it
   nudges toward a small cache keyed on `session_id` + route mtime. worth a thought,
   not worth over-engineering in v1.
3. **writing into a human-owned settings.json.** the installer edits a file the human
   also edits by hand. idempotency + "don't clobber" discipline matters more here than
   the feature itself; getting it wrong is more annoying than the feature is helpful.
4. **two artifacts for one small line.** an installer *and* a renderer feels heavy for
   "show a word." but it's the honest shape of the claude code contract — anything
   lighter can't stay pinned. the weight is inherent, not accidental.
