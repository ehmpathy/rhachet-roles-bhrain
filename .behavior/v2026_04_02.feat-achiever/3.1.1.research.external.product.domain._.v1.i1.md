# research: domain — achiever role

## domain objects

### entities

#### Goal (primary entity)

**convergence:** 6+ sources agree on goal as the core entity.

| source | term | structure |
|--------|------|-----------|
| OKR (atlassian [1]) | Objective | qualitative target with key results |
| asana [8] | Goal | workspace/team/individual level objectives |
| cognitive science [17] | Goal | "internal representations of desired states" |
| locke & latham [19] | Goal | directive function for attention and effort |
| TOTE model [20] | Desired State | reference point for test-operate loop |

**quote [17]:** "goals are defined as internal representations of desired states." — APA PsycNet

**quote [19]:** "goals affect performance through four mechanisms, with the first directional: goals direct attention and effort toward goal-relevant activities and away from goal-irrelevant activities." — locke & latham

**achiever map:** the `Goal` entity captures the desired state with `what.outcome`, the directive function with `how.task`, and verification via `how.gate`.

---

#### Task (work item entity)

**convergence:** all task management systems use this entity.

| source | term | key attributes |
|--------|------|----------------|
| GTD [3] | Next Action | "physical, visible action to move forward" |
| todoist [6] | Task | id, content, project_id, due, priority, labels |
| linear [5] | Issue | team, assignee, project, cycle, status |
| jira [7] | jiraissue | type, summary, assignee, due_date, status |
| asana [8] | Task | "basic unit of action with assignee, name, notes" |

**quote [3]:** "the GTD method rests on the idea that one moves all items of interest out of the mind by external record and then breaks them into actionable work items with known time limits." — wikipedia GTD

**quote [4]:** "a next action should be the next physical, visible action to move the project closer to its goal." — todoist GTD guide

**achiever map:** the `how.task` field within Goal captures the next action. goals are not tasks — goals are outcomes; tasks are actions toward outcomes.

---

#### Ask (input entity)

**convergence:** GTD and agent architectures both capture input separately.

| source | term | purpose |
|--------|------|---------|
| GTD [3] | Inbox | "collection point for unprocessed items" |
| babyagi [10] | Task Queue | "prioritized list of tasks" |
| langchain [12] | Message History | "tracks the conversation" |

**quote [3]:** "inbox — collection point for unprocessed items." — GTD

**achiever map:** `asks.inventory.jsonl` is the inbox. each peer input is an Ask that must be processed into Goals.

#### Coverage (audit entity)

**convergence:** event source and append-only patterns.

| source | term | purpose |
|--------|------|---------|
| event source [claim 12] | Event Log | "append-only logs preserve causal sequence" |
| langchain [13] | Checkpointer | "save the state after every node execution" |

**achiever map:** `asks.coverage.jsonl` maps each Ask hash to a Goal slug — audit trail for triage.

## relationships

### containment hierarchy

```
Scope (route | repo)
└── .goals/
    ├── asks.inventory.jsonl (append-only)
    ├── asks.coverage.jsonl (append-only)
    └── $offset.$slug.goal.yaml + .status=$choice.flag
```

**convergence:** all systems use containment hierarchy.

| source | hierarchy |
|--------|-----------|
| clickup [23] | Workspace > Spaces > Folders > Lists > Tasks > Subtasks |
| asana [8] | Workspace > Team > Project > Section > Task > Subtask |
| notion [24] | Workspace > Database > Data Source > Page > Block |
| workflowy [25] | single infinite nested list |

**quote [23]:** "the Workspace contains your entire organization and all of your work, with one Workspace recommended per organization." — clickup

**achiever note:** two-level scope: route (local to behavior) or repo (branch-scoped).

---

### coverage relationship

```
Ask (hash) ──covers──> Goal (slug)
```

| source | pattern |
|--------|---------|
| event source | event -> aggregate |
| langchain [13] | "checkpointers persist state after every node execution" |

**achiever map:** `asks.coverage.jsonl` records which hash is covered by which slug.

---

### dependency relationship

```
Goal (blocked) ──when.goal──> Goal (blocker)
```

| source | pattern |
|--------|---------|
| jira [7] | issuelink with issuelinktype |
| linear [5] | blocked-by relationships |
| cognitive science [18] | "sub-goal must be instrumental for the primary goal" |

**quote [18]:** "the hierarchical goals model envisions sub-goals as instruments that contribute to the primary goal. a sub-goal must be instrumental for the primary goal: fulfilment of the sub-goal facilitates fulfilment of the primary goal." — wiley

**achiever map:** `when.goal` captures blocked-by relationship. `when.event` captures external blockers.

---

### status lifecycle

```
enqueued ──start──> inflight ──complete──> fulfilled
    │                   │
    └───blocked<────────┘
         │
         └──unblock──> enqueued
```

**convergence:** status lifecycle is universal.

| source | statuses |

**achiever note:** four states: blocked, enqueued, inflight, fulfilled. matches wish requirements.

## composition for wish

### triage flow composition

```
peer input (hook.onTalk)
  └── append to asks.inventory.jsonl [Ask entity]

session end (hook.onStop)
  └── goal.infer.triage
      ├── read asks.inventory.jsonl [getAll asks]
      ├── read asks.coverage.jsonl [getAll coverage]
      ├── read .goals/*.goal.yaml [getAll goals]
      └── return uncovered asks + extant goals

brain triage
  └── for each uncovered ask:
      └── goal.memory.set --covers $hash [setCreate or setUpdate]
          ├── write .goals/$offset.$slug.goal.yaml
          ├── write .goals/$offset.$slug.status=$choice.flag
          └── append to asks.coverage.jsonl
```

**quote [15]:** "if the goal is complex, agentic AI models will break it down into smaller, more manageable sub-goals via task decomposition. this enables the system to focus on complex tasks in a hierarchical manner." — analytics vidhya

**quote [20]:** "an organism tests the initial state of the world against its desired state; it operates (enacts actions) if there's a discrepancy. the process is then repeated until the actual and desired states match, after which the system exits." — TOTE model

### persistence composition

| artifact | format | purpose |
|----------|--------|---------|
| asks.inventory.jsonl | JSONL | append-only inbox |
| asks.coverage.jsonl | JSONL | append-only audit |

**achiever note:** file-based persistence aligns with langchain's namespace + key pattern.

## anti-patterns from research

### anti-pattern 1: vector database for simple goals

**sources [10, 12]:** babyagi uses pinecone, langchain mentions vector stores.

**quote [10]:** "BabyAGI leverages Pinecone for long-term memory. it embeds and stores results so that when the task list loops back around, it can recall prior outcomes relevant to new tasks."

**achiever decision:** goals are small, structured, enumerable. vector search is overkill. file-based persistence is sufficient and simpler.

---

### anti-pattern 2: overly complex hierarchy

**sources [23, 8]:** clickup has 6 levels, asana has 5 levels.

**achiever decision:** two-level scope (route | repo) is sufficient. avoid hierarchy explosion.

---

### anti-pattern 3: stateless triage

**sources [10, 11]:** early agents lost context between loops.

**quote [11]:** "both early agents maintain persistent lists of tasks, store their thought processes and intermediate results as episodic memory for plan and learn cycles."

**achiever decision:** triage runs in the direct brain with full context, not a nested call.
