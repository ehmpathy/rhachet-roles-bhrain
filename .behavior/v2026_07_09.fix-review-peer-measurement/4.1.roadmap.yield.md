# 4.1 roadmap ‚Äî fix review peer measurement

> the ordered, checklist-style execution plan for the blueprint
> (`3.3.1.blueprint.product.yield.md`). each phase declares its dependencies, the briefs to
> read first, the behavioral acceptance criteria it satisfies, and the exact verification that
> proves it done. phases are ordered so each builds only on already-green predecessors.

---

## how to drive this roadmap

- **one phase at a time, top to bottom.** a phase may start only when every phase it depends on
  is checked done AND green.
- **every phase ends green.** the verification command must pass before the box is checked.
  a red verification means the phase is not done ‚Äî fix at cause, never skip.
- **read the phase's briefs before you write its code.** they encode the rules the peer
  reviewers will hold you to.
- **tests are co-equal with code.** a code phase and its test phase ship together; a phase is
  not done until its tests are green.

### briefs every phase should hold

- `3.1.3.research.internal.product.code.prod._.yield.md` ‚Äî the extant prod seams this fix
  threads through (parser shape, malfunction seam, 5-hop context chain, brain maker, schema-ask).
- `define.domain-operation-grains` ‚Äî transformer / communicator / orchestrator, so each new op
  lands in the right grain with the right test scope.
