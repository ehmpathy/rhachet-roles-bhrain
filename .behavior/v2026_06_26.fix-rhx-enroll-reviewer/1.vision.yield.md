# 1.vision — fix rhx enroll reviewer timeout

## tl;dr

the `.behavior/*` l3 guards that run `rhx enroll claude` time out **only in this repo**. the
leading, well-evidenced hypothesis: this repo (and only this repo) self-depends via
`"rhachet-roles-bhrain": "link:."`.

that self-link makes `node_modules/rhachet-roles-bhrain` a symlink to the repo root, which
contains `node_modules/rhachet-roles-bhrain` again → an **infinite symlink cycle**:

```
node_modules/rhachet-roles-bhrain → ..(repo root)
  → node_modules/rhachet-roles-bhrain → ..(repo root)
    → node_modules/rhachet-roles-bhrain → ... ∞
```

git refuses to walk this (`fatal: ... beyond a symbolic link`), but rhachet's own recursive
directory walkers **follow symlinks via `statSync` with no cycle guard**, so a walk that ever
enters node_modules would recurse forever → the guard subprocess hits its timeout.

**status of the causal chain**: the cycle is *verified to exist* and the cycle-blind walkers are
*verified to ship* — but the precise walk that loops during the l3 enroll subprocess is *not yet
reproduced* (see [research] Q4/Q5). only l3 reproduces because only l3 spawns a fresh `claude`
(see [answered] Q9). diagnosis must reproduce the hang before any fix is commit.

every other repo consumes `rhachet-roles-bhrain` as a **flat, published package** (just
`dist/` + `rhachet.repo.yml`, no nested `node_modules`) → no cycle → guards run fast.

---

## the outcome world

### day-in-the-life

a driver works a `.behavior/*` route. they reach a `5.x` stone whose guard runs level-3 peer
reviews — each an `rhx enroll claude --model sonnet --roles ... -p '...'` subprocess. the
reviews spin up, think, and return blockers/nitpicks in the same handful of minutes they take
in every other repo. the route flows.

### before / after

| | before | after |
|---|---|---|
| l3 guard (`rhx enroll claude`) in bhrain repo | hangs until timeout, every time | completes normally |
| same guard in other repos | works fine | still works fine |
| root difference | `link:.` self-cycle | cycle removed / walkers cycle-safe |
| developer experience | "guards are broken here, skip them" | guards trustworthy everywhere |

### the "aha"

the bug is not in `enroll`, not in claude, not in the reviewer prompts. it is one line in
`package.json` (`"rhachet-roles-bhrain": "link:."`) interacting with a cycle-blind directory
walk. remove the cycle (or make the walk cycle-safe) and the symptom vanishes.

## user experience

- **usecase**: drive any `.behavior/*` route to completion in the bhrain repo, including the
  expensive l3 review gates, without guards stalling.
- **contract leveraged**: `rhx enroll claude --model <m> --roles <r,...> -p '<prompt>'` as
  invoked by `5.x.*.guard` peer reviews (`level: 3`).
- **timeline today**: invoke guard → subprocess spawns `claude` → claude/rhachet walks the
  filesystem → walk enters the self-cycle → recurses until the guard's timeout kills it → guard
  reports failure/stall.
- **timeline fixed**: invoke guard → subprocess spawns `claude` → filesystem walk terminates →
  review runs → blockers/nitpicks returned.

## mental model

- to a friend: *"the repo had itself listed as one of its own dependencies, so a folder pointed
  back into itself forever. anything that tried to walk the folder tree got stuck in an endless
  loop. we cut the loop."*
- analogy: two mirrors facing each other — infinite reflections. `link:.` is the second mirror.
- terms: users say *"the reviewer guards hang"*; we say *"recursive symlink cycle + cycle-blind
  `statSync` traversal."*

## evaluation

### how well it solves the goal

directly. the goal is "find the cause and fix it." cause is identified and verified; the fix
targets the cause (the cycle), not the symptom (the timeout).

### pros

- tiny blast radius if fixed at the self-link (one `package.json` line).
- restores parity with every other repo.
- optional upstream hardening protects all future repos from the same footgun.

### cons / tradeoffs

- the `link:.` self-link is presumably **intentional** — it lets the repo dogfood its own
  unreleased roles (the boot hooks read `--repo bhrain` from the installed
  `rhachet-roles-bhrain` package). naively deleting it may break local role lookup. the fix
  must preserve dogfooding while removing the cycle.
- the most robust fix (cycle-guard the walkers) lives **upstream in the `rhachet` package**, not
  in this repo — so a full "solve at cause" fix may require a coordinated rhachet release.

### edgecases / pit of success

- a walker that uses `lstatSync` + skips symlinked dirs (as `setDirectoryReadonlyExecutable`
  already does) is inherently cycle-safe — that pattern is the pit of success to replicate.
- excluding `node_modules` from any in-repo recursive scan also closes the hole.

## open questions & assumptions

### assumptions

1. the timeout is caused by an unbounded recursive filesystem walk that follows the self-cycle.
   **verified**: the cycle is real; rhachet ships cycle-blind walkers; git explicitly stops at
   the cycle while naive `statSync` walks would not.
2. the `link:.` self-link is intentional (dogfooding local roles), not an accident. **unverified
   — must confirm with wisher.**

### triaged open questions

- **[answered] why does the self-link exist?** to dogfood the repo's own freshly-added
  roles/skills locally. evidence: `git blame` puts `package.json:127` in commit `10299b06`
  (`feat(route): add artifact gate enforcement via route.bounce`, 2026-03-12), which in the same
  commit added a new driver skill (`route.bounce.sh`) + a PreToolUse hook in `.claude/settings.json`
  — i.e. the repo needed to consume its own live roles without a publish. any fix must preserve
  local dogfood.
- **[answered] is the self-link committed / intentional / current?** yes. clean worktree
  (`git diff`/`git status` empty), committed 2026-03-12. the fix targets a tracked source line.
- **[wisher] preferred fix locus**: patch this repo (remove/replace the self-link, e.g. via
  `yalc` local consumption already a devDependency) vs. upstream `rhachet` (cycle-guard the
  walkers) vs. both?
- **[wisher] is a change to the upstream `rhachet` package in scope**, or only changes within
  this repo?
- **[research] how do we prove the fix worked?** define a timed reproduction probe: spawn the
  enroll subprocess with a tiny prompt and assert it returns under N seconds (hangs before fix,
  completes after). doubles as the diagnosis repro.
- **[research] what else breaks if the self-link is removed?** enumerate every consumer of the
  `rhachet-roles-bhrain` package name (boot hooks' `--repo bhrain` lookup, `package.json` exports
  map) and confirm the replacement (e.g. `yalc` or a `dist`-only link) still satisfies them.
- **[answered] why do only l3 guards time out, not l1?** in `5.3.verification.guard`, l1
  reviewers run `$rhx review` **in-process** (lines 232–265) and reuse the already-booted parent;
  only l3 runs `$rhx enroll claude -p` (lines 269–282), which spawns a **fresh `claude`**
  subprocess that re-runs process startup + the filesystem scan that meets the cycle. this also
  resolves the parent/child paradox: the parent boots once (clean), l1 reuses it, only l3's
  fresh-spawn re-triggers the vulnerable scan.

### [research] must confirm in diagnosis (not yet pinned)

- the **exact** walk that loops in the enroll path. candidates, in order of suspicion:
  1. claude-code's / rhachet's startup file scan in the freshly-spawned `claude -p` subprocess.
  2. a rhachet `roles link` / `countFilesInDirectory` pass (`countFilesInDirectory` in
     `node_modules/rhachet/dist/.../symlinkResourceDirectories.js` is cycle-blind and explicitly
     skips the loop-guarded sibling when the source is within the repo — i.e. exactly the
     `link:.` case, lines 94–99).
  3. `getAllFilesFromDir` (`node_modules/rhachet/dist/infra/filesystem/getAllFilesFromDir.js`,
     "follows symlinks", no cycle guard) — though the `.agent/*` symlinks observed point to clean
     `dist/` paths, so boot alone did not reproduce in the parent session.
- **paradox to settle in diagnosis**: this parent session booted fine, so the rhachet boot
  hooks (which read clean `.agent → dist` paths) are likely *not* the looping walk. the diagnose
  phase should reproduce the enroll subprocess in isolation and capture where it hangs.

## what is awkward?

- the fix lever most available (this repo's `package.json`) is **not** the most correct fix
  locus (upstream `rhachet` walkers). solving only here leaves the footgun loaded for the next
  repo that self-links; solving only upstream requires a coordinated release.
- the self-link is a deliberate convenience (dogfooding) that silently arms a filesystem cycle —
  the design fights the user's mental model ("a dependency on myself should be harmless").
- the rhachet codebase already *knows* about the hazard (one walker guards against it) yet two
  sibling walkers do not — an inconsistency that makes the bug feel like an oversight rather than
  a deep design problem.
