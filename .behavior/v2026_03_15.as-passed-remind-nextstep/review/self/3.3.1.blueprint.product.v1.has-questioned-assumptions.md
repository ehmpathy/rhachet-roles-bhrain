# self-review: 3.3.1.blueprint.product.v1.has-questioned-assumptions

## assumptions reviewed

### 1. assumption: reminder should be part of formatRouteStoneEmit

**evidence:** prod research [citation 4, 5] shows this file handles all route.stone.set output format.

**if the opposite were true?**
could add reminder in `setStoneAsPassed.ts` instead. but that would couple passage logic with output format, which violates separation of concerns.

**verdict:** assumption holds — formatRouteStoneEmit is the correct location

---

### 2. assumption: single constant for reminder text

**evidence:** vision shows identical reminder text for both guarded and unguarded stones.

**what exceptions exist?**
none per vision — reminder is the same regardless of guard status.

**verdict:** assumption holds — DRY principle applies

---

### 3. assumption: check passage === 'allowed' to show reminder

**evidence:** vision explicitly states reminder appears ONLY when passage is allowed, not blocked.

**what if opposite?**
if we showed reminder on blocked, driver would be confused — they can't proceed until they fix blockers.

**verdict:** assumption holds — criteria explicitly requires this condition

---

### 4. assumption: no need to modify formatGuardTree

**evidence:** formatRouteStoneEmit already joins the guard tree output with other lines. add reminder in the caller is simpler than modify formatGuardTree.

**could a simpler approach work?**
yes, this IS the simpler approach — single change location vs two files.

**verdict:** assumption holds — minimizes touch points

---

### 5. assumption: unit tests are necessary (not just acceptance)

**evidence:** repo pattern shows formatGuardTree.test.ts has 475 lines of unit tests.

**what if acceptance only?**
acceptance tests are slower (~5-10s each) and harder to debug. unit tests provide fast feedback.

**verdict:** assumption holds — but could be reconsidered for faster delivery
