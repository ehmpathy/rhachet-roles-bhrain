## assumption 5: "looks like right now that's not true" means a bug exists

**what do we assume without evidence?**
that the wisher observed a real bug, not a misread of the system.

**what evidence supports this?**
- code review confirms the bug: `.find()` finds ANY malfunction, not latest per stone
- no test exists for malfunction → pass → unblock flow
- the behavior the wisher describes (stuck after fix) matches the code

but: the wisher's expectation ("if the last status is passed, the malfunction was fixed") is the natural, intuitive behavior. the current behavior is a surprise. that makes it a bug in the UX sense.

**verdict: assumption valid ✅** — code confirms the bug exists

## assumption 7: all malfunctions should be unblocked by pass

**what if there's a case where malfunction should persist?**

consider: what if stone A malfunctions, then stone B passes?
- current code: finds A's malfunction → halted
- proposed code: checks latest status of EACH stone → A still malfunctioned

wait — does the proposed fix only check the STONE that malfunctioned, or ALL stones?

**re-read vision:**
"route.drive --mode hook checks LATEST status for each stone"

this means: for EACH stone, get latest status. if ANY stone's latest is 'malfunction', halt.

**is this correct?**
yes — a stone that malfunctioned and was never resolved should still halt.
the fix is: latest status per stone, not "any malfunction ever recorded".

**verdict: assumption valid ✅** — per-stone latest status is correct

## assumption 8: exit code 1 always means malfunction

**what evidence supports this?**
- `getExitCodeClass.ts` defines: 0 = passed, 2 = constraint, other = malfunction
- exit code 1 falls under "other" → malfunction

**what if a reviewer legitimately exits 1 for a different reason?**
- if it's a genuine failure (like "file not found"), that's still a malfunction
- the convention is: 0 = success, 2 = constraint, all else = problem

**verdict: assumption valid ✅** — exit code convention is established
