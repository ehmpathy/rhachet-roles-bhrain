# vision: peer-review-budgets

## the outcome world

### day-in-the-life

**before**: a developer drives through a route. one peer reviewer finds an issue. they fix it. the reviewer finds another. fix. another. fix. the loop continues — 8 rounds later, they've burned through API tokens and time. worse: the expensive opus reviewer spent 3 rounds on lint issues that a cheap haiku reviewer would have caught.

**after**: each peer reviewer has a budget AND a level. cheap reviewers run first (level 1), expensive reviewers run after (level 2+). the developer sees all reviewers in the guard's reviews section:
```
🗿 route.stone.set
   ├─ stone = 3.blueprint
   ├─ passage = blocked
   └─ guard
      ├─ artifacts
      │   └─ ...
      └─ reviews
          ├─ r1: cheapo-linter (l1, 0/10, queued)
          │   └─ awaits arrival
          ├─ r2: cheap2-format (l1, 0/10, queued)
          │   └─ awaits arrival
          └─ r3: primo-review (l2, 0/3, awaits)
              └─ awaits l1
```

after level 1 clears and level 2 runs with blockers:
```
🗿 route.stone.set
   ├─ stone = 3.blueprint
   ├─ passage = blocked
   └─ guard
      ├─ artifacts
      │   └─ ...
      └─ reviews
          ├─ r1: cheapo-linter (l1, 2/10, approved)
          │   ├─ cached ✓
          │   └─ review: .route/3.blueprint.guard.review.i2.abc123.r1.md
          ├─ r2: cheap2-format (l1, 1/10, approved)
          │   ├─ cached ✓
          │   └─ review: .route/3.blueprint.guard.review.i1.def456.r2.md
          └─ r3: primo-review (l2, 3/3, exhausted)
              ├─ finished 5.0s ✓
              └─ review: .route/3.blueprint.guard.review.i3.ghi789.r3.md
```

and the route halts with:
```
🦉 two roads, one step...

🗿 route.drive
   └─ ✋ blocked: reviewer exhausted

one or more reviewers exhausted their budget:
   └─ primo-review (0/3 rounds)

options:
   ├─ increase budget
   │  └─ rhx route.guard.budget --for review --add 2
   └─ approve as-is
      └─ rhx route.stone.set --stone 3.blueprint --as approved
```

### the "aha" moment

the value clicks when:
1. you realize runaway review loops cost you hours and tokens
2. you see exactly which reviewer is stuck and how many rounds it took
3. you have a clear decision point: invest more rounds or approve manually
4. the budget forces intentionality — each round is a conscious choice
5. cheap reviewers caught the lint issue before expensive opus burned 3 rounds on it

## user experience

### usecases

| goal | contract |
|------|----------|
| limit review rounds | declare `budget: N` in guard |
| prioritize cheap before expensive | declare `level: 1` (cheap) vs `level: 2` (expensive) |
| monitor round consumption | observe via `rhx route.drive` output |
| extend budget mid-route | `rhx route.guard.budget --for review --add N` |
| approve despite exhaustion | `rhx route.stone.set --stone $stone --as approved` |

### contract inputs

**guard file declaration** (builds on inflight PR for peer review structure):
```yaml
reviews:
  peer:
    # level 1: cheap reviewers run first
    - slug: cheapo-linter
      budget: 10
      level: 1  # NEW: lower level = runs first
      run: rhx review --brain haiku --rules $route/rules/lint.md --paths $artifacts

    - slug: cheap2-format
      budget: 10
      level: 1
      run: rhx review --brain haiku --rules $route/rules/format.md --paths $artifacts

    # level 2: expensive reviewer runs after level 1 clears
    - slug: primo-review
      budget: 3
      level: 2
      run: rhx review --brain opus --rules $route/rules/*.md --paths $artifacts
```

**tier execution order**:
- level 1 (cheapo, cheap2) runs first — catches basic issues cheaply
- level 2 (primo) runs after — only if level 1 clears
- rationale: don't waste expensive brain tokens on code that still has lint issues

**budget extension command**:
```sh
rhx route.guard.budget --for review --add 2
# or with peer slug filter:
rhx route.guard.budget --for review --peer primo-review --add 2
```

### contract outputs

**review artifact** — unchanged pattern:
```
$stone.guard.review.i$iteration.$hash.r$index.md
```

**exhaustion state** — tracked in passage.jsonl:
```jsonl
{"stone":"3.blueprint","status":"blocked","blocker":"review.peer","reason":"r1 exhausted (3/3 rounds)"}
```

**stdout on exhaustion**:
```
🦉 patience, friend...

🗿 review --peer primo-review
   ├─ budget: 0/3
   └─ status: exhausted

reviewer has exhausted its budget.
rest of reviewers will continue.
```

### timeline

```
t0: driver begins stone
t1: level 1 reviews run (round 1)
    - cheapo-linter: blockers found, round 1/10 consumed
    - cheap2-format: approved (done)
    - primo-review: awaits level 1
t2: driver fixes, level 1 reviews run (round 2)
    - cheapo-linter: approved (done)
    - level 1 now complete → level 2 unlocked
t3: level 2 reviews run (round 1)
    - primo-review: blockers found, round 1/3 consumed
t4: driver fixes, level 2 reviews run (round 2)
    - primo-review: blockers found, round 2/3 consumed
t5: driver fixes, level 2 reviews run (round 3)
    - primo-review: blockers found, round 3/3 consumed → EXHAUSTED
    - all reviewers now terminal
t6: route halts (any exhausted), human prompted for decision
t7a: human extends budget → reviews continue
t7b: human approves as-is → route proceeds
```

## mental model

### how users describe it

> "each reviewer gets a round budget. once it's spent, i decide whether to keep iteration or approve what we have."

> "it's like a code reviewer gets 3 passes — after that, we discuss instead of iterate."

> "cheap linters run first to catch the obvious stuff. only after they pass does the expensive architecture reviewer look at it."

### analogies

| analogy | mapped concept |
|---------|---------|
| **sports timeouts** | each team (reviewer) gets N timeouts (rounds). once spent, the game continues without them |
| **interview rounds** | candidate gets N interview rounds. if still uncertain, committee decides |
| **revision limits** | journal paper gets N revision rounds. after that, accept/reject decision |
| **medical triage** | nurse (cheap) screens first; doctor (expensive) sees only patients who pass triage |
| **code review escalation** | junior reviewer catches lint; senior reviewer focuses on architecture |

### terms

| user term | system term | concept |
|-----------|-------------|---------|
| "rounds" | `budget` | max review invocations |
| "exhausted" | terminal state | reviewer hit budget limit |
| "extend budget" | `--add N` | increase rounds |
| "approve anyway" | `--as approved` | human override |
| "tier" / "level" | `level` | execution order (lower = first) |
| "cheap" / "smol brain" | level 1 | low-cost reviewers |
| "expensive" / "big brain" | level 2+ | high-cost reviewers |

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| prevent runaway loops | yes | hard cap on rounds per reviewer |
| give humans control | yes | explicit budget declaration + extension mechanism |
| maintain review quality | yes | exhaustion prompts human review, not silent skip |
| preserve cache semantics | yes | budget consumption tracked per-hash, approved reviews stay cached |
| optimize cost | yes | cheap reviewers run first, expensive only after cheap clears |

### pros

1. **predictable cost** — know max rounds before start
2. **intentional iteration** — each round is a conscious choice
3. **human in the loop** — exhaustion triggers decision, not silent failure
4. **per-reviewer granularity** — different budgets for different reviewers
5. **composable with extant approval flow** — exhaustion triggers same approval mechanism
6. **cost-efficient tier order** — cheap reviewers catch basic issues before expensive reviewers run

### cons

1. **added complexity** — one more field to understand
2. **budget choice anxiety** — "what's the right number?"
3. **potential to game** — could set high budget to avoid decisions

### edgecases & pit of success

| edgecase | how handled |
|----------|----------|
| budget not specified | default to unlimited (backwards compat) |
| level not specified | default to 1 (lowest tier, runs first) |
| all reviewers exhausted | same as any exhausted: prompt human |
| budget extended after exhaustion | reviewer un-exhausts, continues from current state |
| hash changes after exhaustion | stay exhausted; only `--as rewound` resets |
| stone rewound (`--as rewound`) | exhaustion resets to full budget |
| cached review from prior run | doesn't consume budget (already approved) |
| malfunction per review | doesn't consume budget (not a successful review) |
| level N has blockers | level N+1 waits; driver must clear level N first |
| level N reviewer exhausted | level N+1 proceeds; exhaustion is terminal (like approved) |
| all level N terminal | level N+1 unlocked; reviewers at that level run |
| any reviewer exhausted after all terminal | route halts, requires human approval |

## open questions & assumptions

### assumptions

1. **budget applies per stone per reviewer** — budget declared in guard, exhaustion tracked per stone. each stone gets fresh budget.
2. **inflight PR for peer review structure will land** — slug and run fields exist
3. **exhaustion is terminal until rewound** — no automatic retry, even on hash change. only `--as rewound` resets exhaustion.
4. **approval is singular** — approval for exhaustion also covers any awaited approval judge. wish explicit: "its still just one approval flag."

### decisions (answered via logic or wisher)

1. **budget scope**: per stone per reviewer. each stone starts with fresh budget.

2. **hash change after exhaustion**: stay exhausted. hash change does NOT reset exhaustion — only `--as rewound` resets it.

3. **default budget**: unlimited (backwards compat). if budget not specified, behave as before — no limit.

4. **extension granularity**: per-reviewer via `--peer` filter. wish example shows `rhx route.guard.budget --for review --peer architecture-review --add 2`.

5. **partial approval**: one approval covers all. wish explicit: "once they approve for either, they approve for both. its still just one approval flag."

6. **level progression when one reviewer exhausted**: exhaustion does NOT block level progression. exhausted reviewers are terminal (like approved), so level N+1 proceeds. exhaustion just prevents infinite loops — it doesn't gate the next tier. after all reviewers terminal, if any exhausted → route halts and requires human approval.

## what is awkward?

### exhaustion vs approval interplay

the wish says:
> "if any of the reviewers were exhausted, the route must halt and ask the human to approve by hand"

but also:
> "once they approve for either, they approve for both. its still just one approval flag."

this creates:
- exhaustion triggers an **implicit** `approved?` judge
- this implicit judge shares state with any **explicit** `approved?` judge
- user sees "approve" prompt but it's for a different reason than the explicit judge

**feels forced**: the user might not realize they approve the explicit judge too. or vice versa.

**possible resolution**: make the approval prompt explicit about what they approve:
```
you approve:
   ├─ exhausted reviewers (primo-review)
   └─ awaited approval judge (if any)
```

### budget as round count vs token count

the wish says `budget: $rounds` — round count. but users might expect token/cost budget.

**tradeoff**: round count is simpler and deterministic. token count would require parse of review output for cost metrics.

**comfortable with**: round count is the right abstraction for this use case.

### what happens on hash change? (resolved)

if exhausted on hash A, and the artifact changes to hash B:
- stay exhausted — hash change does NOT reset exhaustion
- only `--as rewound` resets exhaustion to full budget

**resolved**: exhaustion is per route, persists across hash changes. prevents infinite loops even when artifact evolves.

### level vs tier naming

the wish uses both "tiers" and "level". the field is `level: $number`. users might say "tier 1" or "level 1" interchangeably.

**comfortable with**: `level` as the system term, accept "tier" as user vernacular.

### exhaustion and level progression (resolved)

if level 1 has 2 reviewers, and one exhausts while the other approves:
- exhaustion = terminal (like approved), level 2 proceeds
- exhaustion just prevents infinite loops, doesn't gate next tier
- after all reviewers terminal, if any exhausted → route halts, requires human approval

**resolved**: exhaustion allows level progression. human approval is the single decision point at the end.
