# self-review: has-questioned-deletables

## review: can components be removed?

### setBlockedTriggeredReport + getBlockedTriggeredReport

**can this be removed?** no

**why it holds**: the triggered file proves robot saw the nudge. without it, robot could skip the nudge entirely by just create articulation file first. this two-step flow is explicitly required by the vision.

### getBlockedChallengeDecision

**can this be removed?** considered consolidation with selfreview challenge decision

**why it holds**: blocked flow is simpler — no time enforcement, no plowthrough (attempt count). consolidate would add complexity. keep separate for clarity.

### setStoneAsBlocked

**can this be removed?** no

**why it holds**: this is the core operation. must exist.

### formatRouteStoneEmit changes

**can this be removed?** considered reuse of extant nudge format

**why it holds**: blocked nudge has different content (failure quote, articulation guidance). needs its own format. but shares structure with selfreview nudge — that's intentional consistency.

### blocked/ subdirectory

**can this be removed?** considered inline in stones/ directory

**why it holds**: follows pattern of `promise/` subdirectory for selfreview operations. consistent with extant structure.

## simplification applied

1. **no time enforcement**: unlike selfreview, blocked does not require 30 second wait. simpler flow.
2. **no plowthrough**: blocked does not allow attempt-count bypass. must have articulation.
3. **single triggered file**: unlike selfreview which has `.since` and `.uptil`, blocked uses single `.triggered` file.

## conclusion

all components are necessary. simplifications already applied. no further deletions possible without break requirements.
