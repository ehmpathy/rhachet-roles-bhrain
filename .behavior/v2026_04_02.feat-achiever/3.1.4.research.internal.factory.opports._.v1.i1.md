# research: factory opportunities — achiever role

## the constraint

**the bottleneck: acceptance test cycle time**

acceptance tests require:
1. `genTempDirForRhachet` — clone fixtures + setup symlinks (~500ms)
2. `npx rhachet roles link` — register role in temp dir (~1-2s)
3. shell invocation of skill (~1-2s)

total: 3-5s per acceptance test case.

**if we could fix one item:** reduce `rhachet roles link` overhead.

most of that time is spent on symlink path resolution. for achiever tests, we don't need the full rhachet ceremony — we just need the skill entrypoints accessible.

**but:** this improvement benefits all future roles, not just achiever. it's a systems-level optimization worth deferred to a dedicated improvement cycle.

**for this behavior:** the 3-5s cycle time is acceptable. we can work around it by:
- front-load logic verification in unit tests (~100ms)
- use integration tests for persistence (~1s)
- use acceptance tests sparingly for end-to-end confidence

## waste

### what handoffs slow you down?

none identified. achiever is single-developer scope.

### what setup is repeated that could be cached?

| setup | frequency | cacheable? |
|-------|-----------|------------|
| genTempDir | per test case | no — needs isolation |
| rhachet roles link | per acceptance test | could be cached per test file |
| npm run build | per change | yes — already incremental via tsc |

**opportunity:** acceptance tests could share a single rhachet-linked temp dir if tests are read-only on role files. but test isolation is more valuable than speed here.

### what partial work sits idle?

none. achiever is a new role — no legacy to maintain.

### what manual process is error-prone?

**fixture creation:** manual goal YAML file construction is error-prone:
- easy to miss required fields
- status.choice must be valid enum
- nested structure is verbose

**mitigation:** create fixture factory functions:
```typescript
const createGoalFixture = (overrides?: Partial<Goal>) => ({
  slug: 'test-goal',
  why: { ask: 'test', purpose: 'test', benefit: 'test' },
  what: { outcome: 'test' },
  how: { task: 'test', gate: 'test' },
  status: { choice: 'enqueued', reason: 'test' },
  source: 'peer:human',
  createdAt: '2026-04-02',
  updatedAt: '2026-04-02',
  ...overrides,
});
```

## small batches

### can you verify incrementally?

yes — the layered test strategy enables this:

| layer | verifies | feedback |
|-------|----------|----------|
| unit | Goal schema, validation | <100ms |
| integration | persistence, file I/O | ~1s |
| acceptance | skill invocation, hooks | 3-5s |

### what's the smallest verifiable unit?

1. **Goal domain object** — can be tested in isolation with zero I/O
2. **content hash function** — pure function, instant verification
3. **JSONL parse/serialize** — pure function, instant verification

### can you get feedback before the whole is done?

yes. each component (Goal, Ask, Coverage, goal.memory.set, goal.memory.get, goal.infer.triage) can be built and tested independently.

recommended build order:
1. Goal domain object + unit tests
2. Ask/Coverage domain objects + unit tests
3. goal.memory.set + integration tests
4. goal.memory.get + integration tests
5. goal.infer.triage + acceptance tests
6. hook handlers + integration tests

## density

### what % of verification time is test execution vs setup?

| test type | setup | execution | ratio |
|-----------|-------|-----------|-------|
| unit | ~0ms | ~50ms | 100% execution |
| integration | ~500ms (genTempDir) | ~500ms | 50% execution |
| acceptance | ~2-3s (genTempDirForRhachet + link) | ~1-2s | ~40% execution |

**observation:** acceptance test setup dominates. but this is acceptable because:
1. acceptance tests are few (3-5 cases per skill)
2. integration tests cover most behavior
3. unit tests provide rapid feedback

### where is time wasted that could be reclaimed?

| waste | impact | actionable? |
|-------|--------|-------------|
| rhachet roles link | ~1-2s per acceptance test | yes, but systemic change |
| symlink setup | ~500ms per test | no, needed for isolation |
| shell overhead | ~500ms per invocation | no, part of acceptance test definition |

**decision:** accept current overhead. the real opportunity is to minimize acceptance test count by increased unit/integration coverage.

## conclusion

**the opportunity:** fixture factory functions

the highest-leverage quick win is factory functions for test fixtures:
- `createGoalFixture(overrides)`
- `createAskFixture(overrides)`
- `createCoverageFixture(overrides)`

these:
1. eliminate manual YAML construction errors
2. provide sensible defaults
3. enable quick customization for edge cases
4. can be reused across unit/integration/acceptance tests

**deferred opportunities:**
- faster rhachet roles link (systemic, benefits all roles)
- hook test pattern documentation (benefits driver + achiever)

**no blockers** — the current factory setup is sufficient for achiever development.
