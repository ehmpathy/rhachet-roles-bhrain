# research: reflection — audience

## who is the audience?

### primary: drivers (agents who execute routes)

the main beneficiaries are **agents (mechanical or human) who drive through routes with guarded stones**. these are:

- **mechanical agents** (claude, etc.) — execute route workflows autonomously
- **developers** — drive routes manually in feature development
- **ci pipelines** — drive routes as part of automated workflows

they interact with peer reviews directly and feel the pain of runaway review loops.

### secondary: route authors (architects who define guards)

**route architects who configure guard files** with reviewer declarations:

- **behavior route designers** — define wish → vision → blueprint flows
- **ci/cd engineers** — set up automated review gates
- **team leads** — establish review standards and cost controls

they configure the budget and level values. they decide which reviewers are cheap vs expensive.

### tertiary: budget holders (stakeholders who care about costs)

**managers and finance who track API token spend**:

- **technical managers** — need visibility into review costs
- **finance/ops** — track LLM token consumption
- **platform teams** — monitor overall system costs

they care indirectly — they see the bill, not the loops.

---

## why do they care?

### primary (drivers)

| pain | relief |
|------|--------|
| runaway review loops | hard cap stops infinite iteration |
| wasted time on endless fixes | budget forces intentional rounds |
| no visibility into round count | clear status shows budget consumption |
| no control over when to stop | explicit decision point at exhaustion |
| expensive reviewers wasted on lint | cheap reviewers run first via levels |

### secondary (route authors)

| pain | relief |
|------|--------|
| unpredictable review costs | declared budget = predictable max |
| no way to prioritize reviewers | level order gives explicit control |
| reviewers configured once, run forever | budget + level = fine-grained control |

### tertiary (budget holders)

| pain | relief |
|------|--------|
| unpredictable API costs | budget caps provide upper bound |
| expensive models on trivial issues | level order uses cheap first |

---

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| primary (drivers) | high | high | blocks their work daily |
| secondary (route authors) | medium | medium | affects team productivity |
| tertiary (budget holders) | low | low | affects cost reports monthly |

## what do they care about?

### primary (drivers)

**criteria for success:**
- clear indication of budget consumption
- obvious decision point when exhausted
- easy to extend budget or approve
- no hidden costs or surprises

**tradeoffs accepted:**
- accept to configure budget per reviewer
- accept to pause and decide at exhaustion

**tradeoffs rejected:**
- automatic approval on exhaustion (loses human oversight)
- silent skip of review (breaks quality gate)

**delight vs disappointment:**
- delight: upon view of cheap reviewer catch lint before expensive reviewer runs
- disappointment: confusion about exhaustion vs approval prompts

### secondary (route authors)

**criteria for success:**
- simple declaration syntax (`budget: N`, `level: N`)
- sensible defaults (unlimited budget, level 1)
- per-reviewer granularity
- backwards compatible with extant guards

**tradeoffs accepted:**
- added complexity in guard declaration
- need to think about budget values

**tradeoffs rejected:**
- changes that break extant guard files
- mandatory budget/level (should default)

**delight vs disappointment:**
- delight: one field each, clear semantics
- disappointment: complex interaction rules

### tertiary (budget holders)

**criteria for success:**
- visible budget caps in guard definitions
- predictable maximum costs

**tradeoffs accepted:**
- need to set budgets thoughtfully

**tradeoffs rejected:**
- no observability into what was spent

## how will they experience it?

### primary (drivers) — touchpoints

1. **start**: run `rhx route.drive`, see reviewer status with budget/level
2. **iteration**: fix code, rerun, see budget decrement
3. **exhaustion**: see clear "reviewer exhausted" message with options
4. **decision**: choose extend budget OR approve as-is
5. **continue**: route proceeds after decision

**friction points:**
- first-time learn of what "exhausted" means
- which command to run (extend vs approve)
- confusion if both explicit approval judge AND exhaustion-triggered approval

**smooth experience:**
- tree output shows budget/level/status at a glance
- exhaustion message shows exact commands to run
- one approval covers both exhaustion and explicit judge

### secondary (route authors) — touchpoints

1. **configure**: add `budget:` and `level:` to guard peer review declarations
2. **test**: run route, observe behavior
3. **adjust**: tweak values based on experience

**friction points:**
- choice of right budget value (no prior art)
- level interaction with exhaustion

**smooth experience:**
- defaults work (unlimited budget, level 1)
- syntax mirrors extant slug/run pattern

### tertiary (budget holders) — touchpoints

1. **review**: see guard files with budget declarations
2. **monitor**: track API costs (outside this feature scope)

**friction points:**
- no direct interaction with feature

**smooth experience:**
- budget values visible in guard files
- can estimate max cost from declarations
