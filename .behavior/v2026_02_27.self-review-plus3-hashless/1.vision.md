# vision: self-review hashless after $hashbar attempts

## the outcome world

### day-in-the-life with this in place

a clone works on a stone artifact. they complete their work and attempt to pass:

```
🗿 route.stone.set
   ├─ stone = 1.vision
   └─ passage = blocked (self-review required)

🌕 lets reflect
   │
   ├─ review.self 1/2
   │  ├─ slug = all-done
   │  └─ ...
   │
   └─ promise its done? if so, run
      └─ rhx route.stone.set --stone 1.vision --as promised --that all-done
```

the clone does the self-review, finds gaps, fixes them. artifact hash changes. they attempt to promise again — same "lets reflect" output (clone must re-review).

the clone re-reviews, finds more gaps, fixes again. hash changes again. same "lets reflect" output.

the clone reviews one more time, makes final polish. third change. this time:

```
🗿 route.stone.set
   └─ stone = 1.vision
   └─ passage = progressed (self-review 1/2 promised)
```

the clone doesn't know why it finally worked — maybe they got it right, maybe the system decided they'd iterated enough. the mechanism is invisible. what matters: the `all-done` promise is now firm. even if later self-reviews (like `tests-pass`) cause more artifact changes, the `all-done` promise remains valid.

### before/after contrast

| before | after |
|--------|-------|
| clone reviews → finds gap → fixes → new hash → must re-review → finds gap → fixes → new hash → infinite loop | clone reviews and iterates → eventually system accepts → later changes don't invalidate it |
| self-review.2 can invalidate self-review.1's promise | promises are hashless → firm checkpoints |
| frustration: "i already reviewed this 5 times!" | relief: "it finally accepted my promise" |

### the "aha" moment

the value clicks when a clone has done solid review passes on the `design-complete` self-review, then the `tests-pass` self-review triggers another artifact fix. without hashless promises, the clone would need to re-promise `design-complete` even though they already thoroughly reviewed it. with hashless promises, the checkpoint holds firm.

## user experience

### usecases and goals

1. **thorough self-review with iteration** — clone catches gaps, fixes them, without infinite loops
2. **firm checkpoints** — once a self-review is promised, it stays promised
3. **sequential self-reviews don't invalidate each other** — `tests-pass` fixes don't reset `design-complete`
4. **configurable rigor** — route authors can tune `hashbar` per self-review

### guard config with hashbar

**current syntax:**
```yaml
reviews:
  self:
    - slug: all-done
      say: |
        did you complete all that was requested?

    - slug: tests-pass
      say: |
        do all tests pass?
```

**new syntax with hashbar:**
```yaml
reviews:
  self:
    - slug: all-done
      hashbar: 3           # optional, default: 3
      say: |
        did you complete all that was requested?

    - slug: tests-pass
      hashbar: 5           # more iterations allowed for this one
      say: |
        do all tests pass?
```

`hashbar` = how many hash changes we consider before the promise becomes hashless.
- `hashbar: 3` (default) → after $hashbar challenge attempts, promise is firm
- `hashbar: 1` → strict: one hash change means re-review forever
- `hashbar: 0` → always hashless (promise never invalidates)

### contract inputs & outputs

**triggered watermark files (no change needed):**
```
.route/$stone.guard.selfreview.$slug.$hash.triggered  (one per hash)
```
→ glob `$stone.guard.selfreview.$slug.*.triggered` and count = attempt count

**promise artifacts:**
```
.route/$stone.guard.promise.$slug.$hash.md  (current: hash-bound)
.route/$stone.guard.promise.$slug.md        (new: hashless after $hashbar reached)
```

### timeline (internal mechanics, invisible to clone)

```
t0: clone attempts --as passed
    → self-review triggered for "all-done"
    → creates: $stone.guard.selfreview.all-done.$hash1.triggered
    → glob count = 1

t1: clone does review, fixes artifact, attempts --as promised
    → hash changed → challenge
    → creates: $stone.guard.selfreview.all-done.$hash2.triggered
    → glob count = 2

t2: clone re-reviews, fixes again, attempts --as promised
    → hash changed → challenge
    → creates: $stone.guard.selfreview.all-done.$hash3.triggered
    → glob count = 3

t3: clone re-reviews, makes final touch, attempts --as promised
    → hash changed → glob count (4) > $hashbar (3)
    → check mtime of newest triggered file > 90s? yes → promise accepted
    → creates: $stone.guard.promise.all-done.md (hashless)
    → promise is now firm

t4: later self-review "tests-pass" causes more artifact changes
    → "all-done" promise still valid (hashless)
```

the clone only sees "blocked (self-review required)" or "progressed" — never:
- attempt count
- $hashbar value
- "invalidated" status (no distinction between first attempt and hash-changed)

## mental model

### how users describe this to a friend

> "the self-review catches you if you try to rush. you have to actually review and iterate. once you've done that, your checkpoint is locked in and later changes don't force you to re-review stuff you already checked."

### analogies that fit

- **progressive trust**: like 2FA that eventually remembers your device
- **save points in games**: once you reach a checkpoint, later failures don't send you back before it
- **captcha that gets easier**: after you prove you're not a bot a few times, it stops the challenge

### terms users vs we use

| user says | we say |
|-----------|--------|
| "it finally accepted" | $hashbar reached, hashless promise created |
| "kept challenged me" | hash mismatch, attempt counter incremented |
| "checkpoint" | promise artifact |
| "it didn't forget my earlier review" | promise stability (hashless) |

## evaluation

### how well does it solve the goals?

| goal | how well? |
|------|-----------|
| eliminate infinite loops | ✅ $hashbar caps the iteration |
| preserve firm checkpoints | ✅ hashless promises don't invalidate |
| maintain review quality | ✅ multiple challenges ensure thorough review |
| invisible mechanism | ✅ clone doesn't know the hashbar value |
| configurable per-review | ✅ route authors set hashbar in guard |

### pros

1. **bounds iteration** — system prevents runaway loops
2. **firm checkpoints** — prior self-reviews stay valid even with later changes
3. **maintains rigor** — challenges force actual review work
4. **invisible** — clone can't game it by attempt count
5. **configurable** — route authors tune hashbar per self-review
6. **backwards compatible** — default hashbar: 3 preserves reasonable behavior

### cons

1. **edge case complexity** — need to handle watermark vs promise differently

### edgecases and pit of success

| edgecase | how contract keeps users safe |
|----------|-------------------------------|
| clone tries to skip review by quick changes | mtime of newest triggered file must be > 90s even after $hashbar reached |
| self-review.1 promised, then self-review.2 causes changes | self-review.1's hashless promise stays valid |
| clone manually deletes triggered files to reset count | each hash creates new file, deletion of old ones doesn't help much |
| prior hash-bound promises found alongside new hashless ones | hashless promise takes precedence if present |
| hashbar: 0 | always hashless — promise never invalidates (strict trust) |
| hashbar: 1 | strict — one hash change means re-review forever (no hashless mode) |

### what's awkward

1. **name choice**: "hashbar" — is this clear enough? alternatives: `hashbound`, `hashbar`, `maxhash`
2. **debug**: if an issue occurs, harder to diagnose without visible attempt count
3. **guard migration**: routes with no hashbar get default 3 — backwards compatible

## summary

the vision is:
- **triggered watermarks** already exist per-hash; glob count = attempt count (no new state)
- **after glob count > $hashbar** (default: 3), check if newest triggered file mtime > 90s
- **if both conditions met**, the promise becomes hashless
- **promises are hashless** once earned, so they don't invalidate when later self-reviews cause artifact changes
- **the mechanism is invisible** — clone just sees "blocked" or "progressed", never "invalidated"
- **hashbar is configurable** per self-review in the guard file

this prevents runaway loops while the rigor that self-reviews provide remains, without clones able to game the system. route authors can tune the balance between strictness and practicality.
