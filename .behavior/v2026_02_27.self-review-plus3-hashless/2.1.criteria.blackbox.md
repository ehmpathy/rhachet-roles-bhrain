# blackbox criteria: self-review hashless after $hashbar attempts

## usecase.1 = self-review iteration within $hashbar

```
given a stone with reviews.self defined (hashbar: 3)
  when clone attempts --as passed (first time)
    then passage = blocked (self-review required)
    then shows "lets reflect" with first self-review slug
    then shows promise command
      sothat clone knows how to proceed

  when clone does self-review, fixes artifact, attempts --as promised
    then passage = blocked (self-review required)
    then shows same "lets reflect" output
      sothat clone must re-review (no "invalidated" status shown)

  when clone re-reviews, fixes again, attempts --as promised (attempt 3)
    then passage = blocked (self-review required)
      sothat clone continues to iterate (still within $hashbar)
```

## usecase.2 = self-review accepts after $hashbar reached

```
given a stone with reviews.self defined (hashbar: 3)
  given 3 triggered files already exist for slug "all-done" (different hashes)
  given newest triggered file mtime > 90 seconds ago
  when clone attempts --as promised --that all-done
    then passage = progressed (self-review 1/N promised)
    then hashless promise artifact created (.route/$stone.guard.promise.all-done.md)
      sothat promise is firm and won't invalidate on future hash changes
```

## usecase.3 = firm checkpoint survives later changes

```
given a stone with multiple reviews.self (all-done, tests-pass)
  given "all-done" already has hashless promise (earned after $hashbar)
  when clone works on "tests-pass" self-review
  when clone fixes artifact (hash changes)
  when clone attempts --as passed
    then "all-done" promise still valid
    then only "tests-pass" blocks passage
      sothat prior checkpoints are firm
```

## usecase.4 = configurable hashbar per self-review

```
given a guard file with:
  reviews:
    self:
      - slug: all-done
        hashbar: 2
      - slug: tests-pass
        hashbar: 5

  when clone iterates on "all-done" 2 times
    then promise accepted after 2 hash changes
      sothat strict reviews can have lower threshold

  when clone iterates on "tests-pass" 5 times
    then promise accepted after 5 hash changes
      sothat lenient reviews can have higher threshold
```

## usecase.5 = default hashbar when not specified

```
given a guard file with no hashbar specified:
  reviews:
    self:
      - slug: all-done
        say: did you complete all?

  when clone iterates 3 times (hash changes 3x)
    then promise accepted after 3rd attempt
      sothat default hashbar: 3 applies
```

## usecase.6 = time enforcement still applies

```
given a stone with reviews.self (hashbar: 3)
  given 4 triggered files exist (> $hashbar)
  given newest triggered file mtime < 90 seconds ago
  when clone attempts --as promised
    then passage = blocked
    then shows "patience, friend" message
      sothat clone can't bypass review by quick changes
```

## usecase.7 = no "invalidated" status shown

```
given a stone with reviews.self
  when clone attempts --as promised (first time)
    then shows "blocked (self-review required)"

  when clone fixes artifact, attempts --as promised again
    then shows "blocked (self-review required)"
    then does NOT show "invalidated" or any distinction
      sothat mechanism is invisible to clone
```

## usecase.8 = hashless promise takes precedence

```
given a stone with reviews.self
  given both hash-bound promise ($slug.$hash.md) and hashless promise ($slug.md) exist
  when system checks if slug is promised
    then hashless promise takes precedence
    then promise is considered valid
      sothat legacy hash-bound files don't interfere
```
