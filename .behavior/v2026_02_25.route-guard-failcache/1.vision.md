# vision: route guard failcache

## the outcome world

### before

vlad runs `rhx route.stone.set --stone 1.vision --as passed`. a transient infrastructure glitch causes the judge to fail (file absent, network timeout, npm cache corruption). the failure gets cached.

the infra issue clears itself minutes later. vlad tries again. same cached failure. tries again. cached. the route is stuck in purgatory — the judge never re-runs because "we already know it failed."

vlad's only escape: manually find and delete the cache file, or wait for cache expiry. friction. frustration. wasted time debugging phantom failures.

### after

vlad runs `rhx route.stone.set --stone 1.vision --as passed`. same transient glitch. judge fails with exit code 1.

**the failure is not cached.**

vlad (or the system) tries again 30 seconds later. infra recovered. judge runs fresh. passes. route proceeds.

the key insight: **failures from unexpected errors are transient by nature — they should always be retried.** only deliberate "blocked by constraints" verdicts (exit code 2) represent stable state worth caching.

### the "aha" moment

> "oh, the cache only remembers successes and explicit blocks. crashes and glitches just... disappear on retry. the system heals itself."

## user experience

### usecases

1. **route driving with flaky infra** — agent drives a route, judge hits a transient npm/network issue, route continues on next attempt without manual intervention

2. **iterative stone work** — human works on a stone, judge fails due to absent approval, human grants approval, next run picks it up fresh

3. **ci/cd resilience** — automated pipeline hits a cache corruption, retries succeed without human debugging

### contract inputs & outputs

**judge exit codes:**
| exit code | meaning | cached? |
|-----------|---------|---------|
| 0 | passed | ✅ yes |
| 2 | blocked by constraints | ✅ yes |
| 1 | unexpected error | ❌ no |
| other | unexpected error | ❌ no |

**cache key:** `${stone}.${mechanism}.${artifactHash}`

**cache behavior:**
- on success (exit 0): cache result, skip future runs for same artifact
- on explicit block (exit 2): cache result, skip future runs for same artifact
- on error (exit 1, other): do not cache, always retry

### timeline

```
t0: vlad runs route.stone.set
t1: judge executes, hits infra glitch, exits 1
t2: system notes "not cached" (error state)
t3: vlad retries (or system auto-retries)
t4: judge executes fresh, infra recovered, exits 0
t5: system caches success
t6: future runs skip judge (cached pass)
```

## mental model

### how users describe it

> "the route system only remembers things that matter — passes and explicit blocks. everything else gets a fresh chance."

> "it's like a teacher who forgets you tripped on the way to the exam but remembers your actual grade."

### analogies

- **http caching**: you cache 200s and maybe 404s, but never 500s — those are transient server issues
- **circuit breaker pattern**: failures trigger retries; only sustained patterns change behavior
- **git merge conflicts**: git doesn't cache "merge failed due to disk full" — it just fails and you retry

### terms

| user says | system term |
|-----------|-------------|
| "it crashed" | exit code 1 (unexpected error) |
| "it's blocked" | exit code 2 (blocked by constraints) |
| "it passed" | exit code 0 (success) |
| "try again" | cache miss → fresh execution |
| "already checked" | cache hit → skip execution |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| transient failures don't stick | ✅ yes — exit 1 never cached |
| explicit blocks still cached | ✅ yes — exit 2 cached |
| successes still cached | ✅ yes — exit 0 cached |
| no manual cache clearing needed | ✅ yes — errors auto-retry |

### pros

- **self-healing**: transient errors clear on retry without human intervention
- **simple mental model**: "cache the verdict, not the crash"
- **backwards compatible**: extant cache files still work; new behavior only affects future writes
- **fail-fast friendly**: unexpected errors surface immediately, don't hide behind stale cache

### cons

- **slightly more judge executions**: errors retry each time (acceptable cost for correctness)
- **potential for retry storms**: if judge has persistent bug, retries every time (mitigated by: fix the bug)

### edgecases & pit of success

| edgecase | handling |
|----------|----------|
| judge exits 1 repeatedly | not cached, retries each time — surfaces the real bug |
| judge exits 2 then artifact changes | new artifact hash → cache miss → fresh judge |
| judge exits 0 then code changes | same artifact hash → cache hit (correct: artifact unchanged) |
| cache file corrupted | treat as cache miss → fresh judge |
| mixed exit codes across retries | last stable state (0 or 2) gets cached |

### what's awkward?

1. **exit code semantics must be documented** — judges need to know: "exit 2 for explicit blocks, exit 1 for errors"
2. **extant cached failures** — old cache files with exit 1 results will persist until cleared or expired (one-time migration concern)

---

## summary

> cache successes and explicit blocks. never cache crashes. let the system heal.
