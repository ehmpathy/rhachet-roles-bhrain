# evaluation: peer-review-budgets

## summary (as implemented)

implemented per-reviewer budget limits and level-based execution order for peer reviews.

**key deliverables implemented**:
1. upgraded `RouteStoneGuardReviewPeer` from `string` to structured object with `slug`, `run`, `budget`, `level`
2. added `RouteStoneGuardReviewPeerMeter` domain object to track rounds consumed per reviewer
3. execute reviewers by level (higher levels first, level N-1 waits for level N terminal)
4. halt route when any reviewer exhausted, prompt human for budget extension or approval
5. added `route.guard.budget` skill for budget extension

**terminal verdicts**: `approved | exhausted`

## divergence analysis

### divergences found

| section | blueprint declared | actual implemented | divergence type |
|---------|-------------------|-------------------|-----------------|
| getStoneGuardBlockerReport.ts | [~] updated | unchanged | structural (location) |
| route.guard.budget.ts | [+] separate file | in route.ts | structural (location) |
| PassageReport.blocker | `review.peer.concerned` | `review.peer` | name (backwards compat) |

### divergence resolution

| divergence | resolution | rationale |
|------------|------------|-----------|
| getStoneGuardBlockerReport.ts not modified | acceptable | exhaustion handle logic implemented in setStoneAsPassed.ts via computeExhaustionState — more appropriate location as exhaustion is detected at stone passage, not blocker report generation |
| route.guard.budget.ts not separate file | acceptable | CLI entrypoint (routeGuardBudget) implemented in src/contract/cli/route.ts alongside other route CLI functions — maintains consistency with extant CLI organization pattern |
| PassageReport.blocker kept as `review.peer` | acceptable | blueprint wanted rename to `review.peer.concerned` to distinguish from `review.peer.exhausted`, but `review.peer` maintains backwards compatibility with extant routes that may reference this blocker type; the new `review.peer.exhausted` value was added as specified |

**verdict**: all three divergences are acceptable. two are structural (where functionality lives), one is a name decision for backwards compatibility. the functionality is fully implemented. no repair needed.
