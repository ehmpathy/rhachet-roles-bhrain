## deeper questions: could there be a simpler way?

### alternative: ci-level test split

**what if we didn't eject at all?**

instead, configure ci to skip brain tests when brain files unchanged:
```yaml
# .github/workflows/test-brain.yml
on:
  push:
    paths:
      - 'src/domain.operations/brain.replic.arch1/**'
      - 'src/domain.objects/BrainArch1/**'
      - 'src/access/sdk*/**'
```

**pros:**
- zero code moves
- zero import changes
- immediate cost savings

**cons:**
- same repo, still coupled releases
- can't reuse brain independently
- version confusion when arch1 diverges from roles

**verdict:** this achieves cost savings but not clean separation. if the only goal is cost reduction, this might suffice. but the wish explicitly says "eject into its own repo" — so full separation is intended.

---

### alternative: monorepo with packages

**what if we used a monorepo instead of two repos?**

```
rhachet-bhrain/
├── packages/
│   ├── brains-bhrain/
│   └── roles-bhrain/
└── pnpm-workspace.yaml
```

**pros:**
- single git history
- shared tooling
- easier coordinated releases

**cons:**
- still one ci pipeline (unless nx/turborepo caching)
- more complexity than two simple repos
- not the standard ehmpathy pattern

**verdict:** adds complexity without clear benefit over two repos. the wish chose two repos — trust the decision.

---

### is npm package separation necessary?

**the wish says:** "eject ... into its own repo, rhachet-brains-bhrain"

**evidence:** explicit. the wisher chose full separation.

**what if we didn't:** cost savings via ci config, but no clean module boundary.

**verdict:** npm separation is the chosen approach. proceed.
