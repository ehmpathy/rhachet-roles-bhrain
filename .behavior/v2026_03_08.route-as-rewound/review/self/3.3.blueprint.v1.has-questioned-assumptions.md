# self-review: has-questioned-assumptions

## reflection

I paused. took a breath. read the blueprint again, line by line, as if I had not written it.

what would a skeptical senior engineer question? what assumptions did I make that I didn't even notice?

I found 5 technical assumptions worth questioning. for each, I asked "what if the opposite were true?"

---

## assumption 1: cascade scope is correct

**the assumption**: rewind stone N → also rewinds N+1, N+2, etc.

**what if the opposite were true?** what if we only rewound the single stone?

**evidence**: the wisher explicitly confirmed cascade behavior. downstream stones were validated while upstream was valid — if upstream is rewound, downstream validation is stale. this is not habit; it's the correct design.

**verdict**: assumption is valid, backed by wisher confirmation.

---

## assumption 2: deletion is the right approach for guard artifacts

**the assumption**: delete `*.guard.review.*`, `*.guard.judge.*`, etc.

**what if we preserved them?** we could mark them as "stale" instead.

**counterargument**: the hashbar already handles cache invalidation. if the artifact hash changes, fresh reviews run. deletion is simpler — no "stale" flag infrastructure needed.

**verdict**: assumption is valid. deletion is simpler and correct.

---

## assumption 3: passage.jsonl is append-only

**the assumption**: we only append to passage.jsonl, never rewrite.

**what if we could rewrite?** we'd need locks, risk data loss, more complex operations.

**evidence**: the extant codebase uses `setPassageReport` which appends. this is deliberate — append-only logs are simpler and safer.

**verdict**: assumption is valid, matches extant design.

---

## assumption 4: approval invalidation by log order works

**the assumption**: scan passage.jsonl, find approval, check for rewind after it.

**what if log order is unreliable?** could concurrent writes scramble order?

**evidence**: passage.jsonl uses jsonlines format with single-line appends. node's fs.appendFile is atomic for small writes. for large-scale concurrency, we'd need locks, but the current design assumes single-agent operation.

**risk**: if two agents write simultaneously, order could be non-deterministic.

**mitigation**: the scenario is rare (human approves, another agent rewinds simultaneously). for v1, the risk is acceptable.

**verdict**: assumption is valid for v1. if concurrency proves problematic, add flock.

---

## assumption 5: we can reuse extant patterns

**the assumption**: `setPassageReport`, `findOneStoneByPattern`, `getAllStones` work as-is.

**what if they don't?** what if setPassageReport can't handle `status: 'rewound'`?

**evidence**: I checked PassageReport domain object — it currently only allows `'passed' | 'approved' | 'blocked'`. the blueprint correctly identifies `[~] PassageReport.ts` as an update.

**action needed**: none, already accounted for in filediffs. verified the type update is listed.

**verdict**: assumption is valid.

---
