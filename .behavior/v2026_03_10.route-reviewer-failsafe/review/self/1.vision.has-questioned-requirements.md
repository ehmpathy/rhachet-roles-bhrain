# self-review: has-questioned-requirements

## requirement 1: judge artifacts should contain clear error messages

- **who said this?** the wish describes empty judge artifacts as a problem
- **evidence?** the artifacts show only `exit code: 1` with no explanation
- **what if we didn't?** humans would need to re-run commands manually to debug
- **scope check:** correctly scoped - we need the artifact, not CLI output, because artifacts persist for post-mortem
- **simpler way?** could just log to stderr, but artifacts are already the contract

**verdict:** requirement is valid

## requirement 2: hook should keep to fire after failures

- **who said this?** the wish says "hook never triggered again"
- **evidence?** the output shows "blocked by constraints" after the first failure
- **what if we didn't?** agent would be stuck with no guidance
- **scope check:** need to investigate whether "blocked by constraints" is a permission issue or a bug
- **simpler way?** no simpler alternative

**verdict:** requirement is valid, but root cause may be different than assumed

## requirement 3: review skill should always produce output

- **who said this?** inferred from the empty review artifact
- **evidence?** review artifact shows only API key load messages
- **what if we didn't?** judges would fail silently when reviews crash
- **scope check:** this is a defensive requirement - fail-fast is better than fail-silent

**verdict:** requirement is valid

## summary

all three requirements are valid and properly scoped. the key insight is that requirement 2 (hook continuity) may have a different root cause than initially assumed - "blocked by constraints" suggests a permission/constraint issue rather than a hook bug.
