## the cache mechanism (root cause)

the peer reviews are keyed on a hash of the review scope (`diffs: since-main` ∩
`paths: **/*.{ts,sh}` / `**/*.snap`). the evidence across three re-arrives this session:

- a change to an **untracked** file (case3) does **not** bust the hash.
- a change to a **tracked** file's working tree (case8) also did **not** bust the hash.
- the only changes that busted the hash earlier (i003→i006) coincided with the committed
  state / src-snapshot changes.

conclusion: the cache refreshes on **committed** deliverable state, not on the uncommitted
working tree. my fixes are uncommitted, so r9 cannot see them — and it has no blocker to
contemplate, so the contemplation loop cannot force a re-read either.

manufacturing a spurious `src/` edit purely to bust the cache would be **gaming the
verification mechanism** — forbidden by the strictness mandate — so i did not do it.
