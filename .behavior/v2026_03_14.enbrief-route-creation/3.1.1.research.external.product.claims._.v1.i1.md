# research: claims for enbrief route creation

## summary

this research gathers claims from external sources relevant to route creation documentation. the concepts of thought routes map to established patterns: checklists, quality gates, progressive disclosure, and convention-based systems.

---

## claims

### claim.1 = checklists reduce errors in complex work

**[FACT]** well-designed checklists reduce errors and improve outcomes across domains.

> "no matter how expert you may be, well-designed check lists can improve outcomes" [1]

> "each hospital had a reduction in complications. the average reduction was more than a third. and we saw a significant drop in deaths as well." [1]

**convergence**: sources [1], [2] agree that checklists work in aviation, medicine, and software.

**sources**:
- [1] [The Checklist Manifesto | Atul Gawande](https://atulgawande.com/book/the-checklist-manifesto/) — 2009 [book]
- [2] [Software Development Plan Checklist | Manifestly](https://www.manifest.ly/use-cases/software-development/software-development-plan-checklist) — 2025 [practitioner]

---

### claim.2 = milestones are checkpoints for progress

**[FACT]** milestones provide decision points and progress visibility.

> "milestones such as 'Phase 1 Complete' or 'Beta Version Approved' act as important checkpoints that provide opportunities to celebrate progress, assess project health, and catch potential delays before they impact the final delivery date." [3]

> "a milestone doesn't necessarily equate to a release—a version of a project could be composed of multiple milestones depending on planning and available resources." [4]

**sources**:
- [3] [The Modern Developer's Workflow | Medium](https://medium.com/@averageguymedianow/the-modern-developers-workflow-best-practices-for-peak-productivity-7655be24947f) — Oct 2025 [blog]
- [4] [CakeDC Git Workflow](https://www.cakedc.com/git-workflow) — 2024 [practitioner]

---

### claim.3 = quality gates prevent defects from propagation

**[FACT]** quality gates are automated checkpoints that validate code meets criteria before progression.

> "a software quality gate is an automated checkpoint in the SDLC that evaluates if code meets a predefined set of criteria before it can proceed to the next phase of the development pipeline." [5]

> "software quality gates serve as decision points that can automatically block code with issues/defects, ensuring a higher-quality product and preventing problems from progressing to later phases." [5]

> "making good use of quality gates not only can improve the quality of your software, but it can also improve your delivery speed." [6]

**convergence**: sources [5], [6], [7] agree quality gates improve both quality and velocity.

**sources**:
- [5] [Software Quality Gates | testRigor](https://testrigor.com/blog/software-quality-gates/) — 2025 [practitioner]
- [6] [Pipeline Quality Gates | InfoQ](https://www.infoq.com/articles/pipeline-quality-gates/) — 2024 [practitioner]
- [7] [Quality Gates | Perforce](https://www.perforce.com/blog/sca/what-quality-gates) — 2024 [official]

---

### claim.4 = quality gates should be incremental, not all at once

**[OPIN]** start small and add gates incrementally.

> "start small – introduce gates incrementally instead of all at once, for example start with a quick smoke test integrated to dev deployment pipeline." [8]

> "limit them to main phases because the more gates you add, the more complex the flow becomes, which could lead to costly delays." [5]

**sources**:
- [8] [CI/CD Quality Gates | DevOps Institute](https://www.devopstraininginstitute.com/blog/10-cicd-quality-gates-for-production-level-reliability) — 2025 [tutorial]

---

### claim.5 = progressive disclosure reduces cognitive load

**[FACT]** progressive disclosure shows essential information first, details on demand.

> "progressive disclosure is an interaction design pattern used to make applications easier to learn and less error-prone. it defers advanced or rarely used features to a secondary screen." [9]

> "progressive disclosure improves 3 of usability's 5 components: learnability, efficiency of use, and error rate." [9]

> "progressive disclosure involves layer information so you don't present everything to users at once, make some information available only at secondary or tertiary levels of navigation." [10]

**convergence**: sources [9], [10], [11] agree on the cognitive benefits.

**sources**:
- [9] [Progressive Disclosure | Nielsen Norman Group](https://www.nngroup.com/articles/progressive-disclosure/) — 2023 [official]
- [10] [Progressive Disclosure | I'd Rather Be Write](https://idratherbewriting.com/ucd-progressive-disclosure/) — 2024 [blog]
- [11] [Progressive Disclosure | Wikipedia](https://en.wikipedia.org/wiki/Progressive_disclosure) — 2024 [reference]

### claim.6 = convention over configuration reduces decision fatigue

**[FACT]** conventions decrease the number of decisions developers must make.

> "convention over configuration is a software design paradigm used by software frameworks that attempts to decrease the number of decisions that a developer is required to make without necessarily lose flexibility." [12]

> "CoC frees developers from make mundane decisions, such as whether to name a database column id, postId, posts_id or pid." [13]

> "the Convention over Configuration pattern allows new developers to learn a system quickly. once developers understand the convention, they can quickly start develop without worry about write configurations." [13]

**convergence**: sources [12], [13], [14] all cite Rails, ASP.NET MVC as examples.

**sources**:
- [12] [Convention over configuration | Wikipedia](https://en.wikipedia.org/wiki/Convention_over_configuration) — 2024 [reference]
- [13] [Convention Over Configuration | DevX](https://www.devx.com/terms/convention-over-configuration/) — 2024 [practitioner]
- [14] [Patterns in Practice | Microsoft Learn](https://learn.microsoft.com/en-us/archive/msdn-magazine/2009/february/patterns-in-practice-convention-over-configuration) — 2009 [official]

### claim.7 = CLI help should be at the touchpoint

**[OPIN]** help should be accessible where users need it.

> "for those moments when users can't figure out how to proceed, it's important to provide easily accessible help, directly at the touchpoint." [15]

> "add a --help command to your CLI provides users with an essential piece of documentation. it lets new users discover all the commands and options available to them." [16]

**sources**:
- [15] [CLI Design Guidelines | Thoughtworks](https://www.thoughtworks.com/insights/blog/engineering-effectiveness/elevate-developer-experiences-cli-design-guidelines) — 2024 [practitioner]
- [16] [Command Line Interface Guidelines](https://clig.dev/) — 2024 [official]

---

### claim.8 = documentation must match the current version

**[SUMP]** version drift causes broken commands and lost trust.

> "ensure that the CLI documentation matches the latest version of the command line interface. version mismatches lead to broken commands, frustrated developers, and lost trust." [17]

**sources**:
- [17] [CLI Docs Checklist | Infrasity](https://www.infrasity.com/blog/cli-docs-checklist) — 2025 [blog]

---

### claim.9 = branch names should follow conventions

**[FACT]** consistent branch name aids collaboration and tools.

> "use a consistent name convention for your feature branches to identify the work done in the branch, and you can also include other information such as who created the branch." [18]

> "document the conventions in your project's contribution guidelines or README file so that new contributors can easily follow them." [18]

> "stick to lowercase for branch names and use hyphens to separate words." [19]

**convergence**: sources [18], [19], [20] agree on lowercase + hyphens.

**sources**:
- [18] [Git Branch Name Convention | Alibaba Cloud](https://www.alibabacloud.com/blog/git-branching-naming-convention---best-practices_597423) — 2024 [practitioner]
- [19] [Name conventions for Git Branches | Medium](https://medium.com/@abhay.pixolo/naming-conventions-for-git-branches-a-cheatsheet-8549feca2534) — 2024 [blog]
- [20] [Git branch guidance | Microsoft Learn](https://learn.microsoft.com/en-us/azure/devops/repos/git/git-branching-guidance?view=azure-devops) — 2024 [official]

### claim.10 = self-document names reduce need for comments

**[FACT]** descriptive names explain purpose without external documentation.

> "self-document (or self-describe) source code and user interfaces follow conventions that enable use of the system without prior specific knowledge." [21]

> "clear and descriptive names make the code easier to understand. when names accurately describe their purpose, other developers can quickly grasp the functionality without need extensive comments." [22]

**sources**:
- [21] [Self-document code | Wikipedia](https://en.wikipedia.org/wiki/Self-documenting_code) — 2024 [reference]
- [22] [10 Tips For Self-Document Code | Nile Bits](https://www.nilebits.com/blog/2024/05/self-documenting-code/) — May 2024 [blog]

### claim.11 = centralized documentation reduces cognitive load

**[FACT]** scattered documentation creates friction; centralization helps.

> "search for scattered documentation can be time-consume and mentally tax, but centralized documentation ensures that developers can find what they need quickly and easily." [23]

> "insightful documentation significantly alleviates cognitive load by externalize critical context and information." [24]

**sources**:
- [23] [Build an IDP with Backstage | Axelerant](https://www.axelerant.com/blog/reduce-developer-cognitive-load-with-backstage-idp) — 2024 [practitioner]
- [24] [Reduce Cognitive Load | Bytex](https://bytex.net/blog/reducing-cognitive-load-the-developers-guide-to-efficient-coding/) — 2024 [blog]

### claim.12 = docs-as-code prevents documentation drift

**[OPIN]** documentation should change with the code.

> "the software developer process breaks when technical documentation describes a different reality than the repo, but docs-as-code fixes this because the knowledge changes with the code, in the same review cycle." [25]

**sources**:
- [25] [Ultimate Developer Onboard Checklist | Port.io](https://www.port.io/blog/developer-onboarding-checklist) — 2024 [practitioner]
