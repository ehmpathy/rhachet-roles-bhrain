# vision: route malfunction resolve

## the outcome world

### day-in-the-life with this in place

a mechanic works through a route. the reviewer script has a bug and exits with code 1 (malfunction). the route correctly halts and escalates to human.

the human investigates, fixes the reviewer script, and tells the mechanic to try again. the mechanic runs `rhx route.stone.set --stone 3.review-malfunction --as passed` and this time the reviewer passes. passage.jsonl now shows:

```jsonl
{"stone": "3.review-malfunction", "status": "malfunction"}
{"stone": "3.review-malfunction", "status": "passed"}
```

the mechanic continues to work. when the session stop hook fires, `route.drive --mode hook` sees the LATEST status for that stone is `passed`, so the malfunction is considered resolved. the hook does NOT halt — the mechanic can proceed normally.

### before/after contrast

**before:**
- malfunction recorded in passage.jsonl
- even after stone passes, old malfunction entry still exists
- `route.drive --mode hook` finds ANY entry with status === 'malfunction'
- hook stays halted forever, even though malfunction was resolved

**after:**
- malfunction recorded in passage.jsonl
- stone passes, new entry added with status === 'passed'
- `route.drive --mode hook` checks LATEST status for each stone
- latest status is 'passed' → malfunction resolved → hook unblocked

### the "aha" moment

"oh, passage.jsonl is a log not a map — the last entry wins!"

## user experience

### usecases fulfilled

1. **malfunction recovery** — human fixes the issue, mechanic retries, route proceeds
2. **no stuck state** — mechanic doesn't need human to manually edit passage.jsonl
3. **cache integrity** — broken reviewers don't pollute the cache for subsequent attempts

### contract inputs & outputs

**scenario 1: malfunction → fix → pass**
```
input: reviewer malfunctions (exit 1)
output: passage = malfunction, hook halted

input: human fixes reviewer, mechanic retries
output: passage = passed, hook unblocked
```

**scenario 2: reviewer malfunction not cached**
```
input: reviewer exits 1, then fixed, then re-run with same artifact hash
output: fresh review runs (not cached broken result)
```

### timeline

1. t0: artifact created
2. t1: reviewer runs → exits 1 → malfunction recorded
3. t2: human fixes reviewer
4. t3: mechanic retries stone.set --as passed
5. t4: reviewer runs fresh (not cached!) → exits 0 → passed recorded
6. t5: session ends, hook fires
7. t6: route.drive sees latest status = passed → hook allows stop

## mental model

### how users would describe this

"if the reviewer breaks, fix it and try again. once it passes, you're good to go."

### analogies

- like a flight checklist: if a check fails, you fix it and re-check. once it passes, you don't stay grounded because it failed before.
- like a test suite: a failed test doesn't permanently mark the build as broken. once you fix and re-run, the build is green.

### terms alignment

| user says | system terms |
|-----------|--------------|
| "it broke" | malfunction (exit 1) |
| "fix it and try again" | resolve malfunction via stone.set --as passed |
| "we're good now" | latest status = passed |

## evaluation

### how well it solves the goals

| goal | solution quality |
|------|-----------------|
| malfunction doesn't block forever | ✅ latest status wins |
| broken reviewers don't cache | ✅ only cache successful reviews |
| clear mental model | ✅ last entry wins (log semantics) |

### pros

- no manual passage.jsonl edits needed
- idempotent: a repeated pass doesn't break anything
- consistent with judges (already only cache passed)
- log semantics are familiar and intuitive

### cons

- passage.jsonl grows with each status change (acceptable)
- requires knowledge that order matters in jsonl (documented)

### edgecases and pit of success

| edgecase | how it's handled |
|----------|------------------|
| multiple malfunctions, then pass | last entry is 'passed' → resolved |
| pass, then malfunction | last entry is 'malfunction' → halted |
| same hash, different reviewers | each reviewer tracked by index |

## open questions & assumptions

### assumptions — verified

1. **[verified]** passage.jsonl is append-only (never edited, only appended)
   - evidence: code uses `fs.appendFile` for passage writes
2. **[verified]** last entry for a stone represents current state
   - evidence: wisher explicitly said "if the last status is passed"
   - evidence: `getOnePassageReport` returns latest entry per stone
3. **[verified]** malfunction cache is undesired for reviews (like judges)
   - evidence: wisher said "only successes get cached"
   - evidence: judges already filter by `&& passed`

### questions — triaged

1. **[answered]** should we clean up old malfunction entries, or keep the full log?
   - answer: keep full log for auditability
   - rationale: log semantics enable replay/debug and audit trails
2. **[answered]** should there be a command to manually clear malfunction state?
   - answer: no, a stone pass is the natural resolution
   - rationale: separate command would be redundant and could cause inconsistent state

### external research needed

none — this is internal domain logic

## what is awkward?

### what feels off

- the current code uses `.find()` which gets first match, not last match
- reviews cache ALL prior artifacts by hash, malfunctions included
- this asymmetry with judges (which only cache passed) is a surprise

### where design fights mental model

the current behavior (any malfunction = permanent halt) contradicts the natural expectation that "fix and retry" should work.

### uncomfortable tradeoffs

none — the fix aligns with user mental model and judge precedent.

## summary

the current codebase has two bugs:

1. **malfunction check uses `.find()` instead of latest status** — line 50-51 in stepRouteDrive.ts finds ANY malfunction, not the latest status
2. **reviews cache all artifacts with malfunctions** — line 133-135 in runStoneGuardReviews.ts reuses ALL prior reviews without filter by exitClass

the fix:
1. change malfunction detection to check latest status per stone
2. change review cache to skip malfunctions (like judges do)

the tests prove both behaviors work correctly.
