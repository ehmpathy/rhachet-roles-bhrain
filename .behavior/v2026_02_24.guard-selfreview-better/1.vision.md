# vision: guard-selfreview-better

## the outcome world

**before**: a clone reaches a self-review checkpoint, sees the guide, and immediately runs `rhx route.stone.set --as promised --that all-done`. they treat self-review as a tollbooth to rush through, not a quality gate to embrace.

### what's the "aha" moment where the value clicks?

the "aha" moment comes when a clone, challenged by the time enforcement, actually goes back and does the review properly — and catches a real issue they would have missed.

the second "aha" moment comes when a human notices that clone outputs are consistently higher quality because the self-review friction made them actually review instead of just rubber-stamp.

## user experience

### what usecases do folks fulfill? what goals?

| actor | goal | interaction |
|-------|------|-------------|
| clone | complete stone milestone | encounters self-review as natural sanctuary, not obstacle |
| clone | promise self-review is done | must wait minimum time before promise is accepted |
| human | get higher quality outputs | clones actually review, catch issues before peer review runs |
| human | observe review behavior | timestamps in promise artifacts show review durations |

### what contract inputs & outputs do they leverage?

**trigger flow** (new: tracks when review was first shown)

```
route.stone.set --stone 1 --as passed
```

returns self-review prompt **and** records `triggeredAt` timestamp in route metadata:
- `.route/1.stone.guard.selfreview.all-done.$hash.triggered.md`
- contains timestamp of when clone first saw this review

**promise flow** (new: validates time elapsed)

```
route.stone.set --stone 1 --as promised --that all-done
```

checks:
1. was this review triggered? (yes, because we got here)
2. how long ago was it triggered?
3. if < 30 seconds, block with challenge
4. clone must wait and try again after 30 seconds pass

### what would it look like to leverage them?

**happy path: clone takes their time**

```
$ rhx route.stone.set --stone 1 --as passed

🌕 lets reflect
   ... (full warm message with before/after instructions)

[clone reads artifacts, reviews carefully for 2 minutes]

$ rhx route.stone.set --stone 1 --as promised --that all-done

✓ promise recorded
   ├─ review.self = all-done
   ├─ duration = 2m 14s
   └─ status = thorough 🦉
```

**challenged path: clone rushes**

```
$ rhx route.stone.set --stone 1 --as passed

🌕 lets reflect
   ... (full warm message)

$ rhx route.stone.set --stone 1 --as promised --that all-done

🦉 patience, friend
   ... (challenge message)

[clone reflects, decides to actually review]

[2 minutes later]

$ rhx route.stone.set --stone 1 --as promised --that all-done

✓ promise recorded
   ├─ review.self = all-done
   ├─ duration = 2m 31s (after challenge)
   └─ status = thorough 🦉
```

### what timelines do they go through?

1. **t=0**: clone attempts to pass stone, sees self-review trigger
2. **t=0**: system records `triggeredAt` timestamp
3. **t=?**: clone runs promise command
4. **t=?**: system checks elapsed time since trigger
   - if time >= 30 seconds: accept promise
   - if time < 30 seconds: challenge, must wait
5. **t=?**: promise recorded with duration metadata

## mental model

### how would users describe this to a friend?

> "when you hit a self-review checkpoint, it actually makes you slow down and breathe. it's like the system knows we're tempted to just click through, so it frames it as 'hey, this is your dedicated review time, enjoy it'. and if you try to speed through, it won't let you — you have to actually wait and reflect before you can promise."

### what analogies or metaphors fit?

1. **meditation timer** — like a meditation app that won't let you mark "done" until at least X minutes have passed. the time must pass; there is no shortcut.

2. **cool-down period** — like email apps that make you wait before you send an angry email. the pond must settle before you can proceed.

3. **tea time** — you cannot rush tea. it steeps for the time it needs. so too with reflection.

### what terms would they use vs what terms would we use?

| user term | our term | notes |
|-----------|----------|-------|
| "review timeout" | minimum review duration | they see it as a timer |
| "rush alert" | time enforcement challenge | they see the challenge |
| "review timer" | triggered timestamp track | we track, they experience |
| "zen mode" | warm frame / calm intro | the positive frame |

## evaluation

### how well does it solve the goals?

| goal | solution | score |
|------|----------|--------|
| make clones proactively do thorough reviews | warm frame reframes self-review as gift, not chore | ★★★★☆ |
| provide generic review instructions | before/after sections with universal review wisdom | ★★★★★ |
| enforce minimum review time | timestamp track + challenge system | ★★★★★ |

### what are the pros?

1. **psychological frame** — self-review framed as gift/sanctuary changes clone behavior more than enforcement alone
2. **time-based enforcement** — catches obvious rubber-stamp (12 second reviews)
3. **no escape hatch** — the time must pass; there is no shortcut
4. **observability** — duration metadata enables analysis of review behavior over time
5. **generic instructions** — before/after sections provide universal value regardless of specific review guide

### what are the cons?

1. **token overhead** — longer prompts mean more tokens per self-review
2. **state management** — triggered timestamp records adds complexity
3. **fixed threshold** — 30 seconds may feel long for simple reviews
4. **clock manipulation** — a determined clone could just wait, not actually review

### what edgecases exist and how do our contracts keep users in a pit of success?

| edge case | how we handle it |
|-----------|------------------|
| clone reviews before command runs | time starts at trigger, not at artifact creation; review-then-trigger is still valid |
| extremely long reviews (hours) | no upper limit; long reviews are fine |
| multiple self-reviews in sequence | each review has its own triggered timestamp |
| hash changes mid-review | triggered timestamps are per-hash; new hash = new timer |
| clone restarts session | triggered timestamps persist in .route/ files; timer continues |
| clone runs promise without trigger | error: must trigger first (current behavior) |

### what's awkward?

1. **trigger track state** — we need to persist "when was this review shown" which adds a new artifact type
2. **UX verbosity** — the warm frame adds significant output; some may find it too chatty

## proposed contract changes

### new artifact type

```
.route/$stone.guard.selfreview.$slug.$hash.triggered
```

content (mtime used for time enforcement):
```
slug: $slug
hash: $hash
```

### modified promise command

```bash
rhx route.stone.set --stone $stone --as promised --that $slug
```

time enforcement: if < 30 seconds, command is blocked until 30 seconds pass.

### modified promise artifact

no timestamp fields in promise content (simplified):
```markdown
# promise: $slug

- stone: $stone
- hash: $hash
```

## summary

this vision transforms self-review from a tollbooth to rush through into a **dedicated sanctuary** with:

1. **warm frame** that reframes the moment as a gift
2. **generic instructions** before/after the specific guide
3. **time enforcement** that blocks until the pond has settled

the result: clones actually do self-reviews, catch more issues themselves, and produce higher quality outputs before peer reviews ever run.

🦉 wisdom comes to those who pause and reflect.
