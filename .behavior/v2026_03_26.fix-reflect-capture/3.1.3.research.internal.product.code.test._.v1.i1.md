## pattern 3: plan mode no side effects [REUSE]

**location**: `src/domain.operations/reflect/savepoint/setSavepoint.integration.test.ts:52-65`

**current pattern**:
```typescript
when('[t0] mode is plan', () => {
  then('does not write files', async () => {
    const savepoint = setSavepoint({ scope: scene.scope, mode: 'plan' });
    // paths are computed but files not written
    expect(fs.existsSync(savepoint.patches.stagedPath)).toBe(false);
    expect(fs.existsSync(savepoint.patches.unstagedPath)).toBe(false);
  });
});
```

**relation**: this test FAILS with the new implementation in plan mode.

**problem**: the new approach writes files first (shell redirect), then reads metadata. in plan mode, we should NOT write files.

**solution**: in plan mode, fall back to execSync with maxBuffer for small diffs, or compute hash differently. OR accept that plan mode writes to a temp location then cleans up.

## gap: plan mode with shell redirect

**problem identified**: the vision says "shell redirect writes directly to file" but plan mode should NOT write files.

**options**:

1. **plan mode uses old approach with maxBuffer**: for plan mode only, use execSync with `maxBuffer: 50 * 1024 * 1024`. plan mode is for preview, large diffs are rare in preview.

2. **plan mode writes to temp, then deletes**: write to temp dir, compute metadata, delete temp files, return savepoint. wasteful but consistent.

3. **plan mode skips hash entirely**: return placeholder hash like `'plan000'` for plan mode. hash only matters for deduplication, which only matters in apply mode.

**recommendation**: option 1 (plan mode uses old approach). plan mode is for preview, users rarely preview 50MB diffs. if they do, they'll hit the 50MB limit and get a clear error.
