# vision: fix reviewer creds

## the outcome world

### before

a mechanic runs `rhx review --brain fireworks/fastcode/qwen3-coder`:

```
🦉 patience, friend

🗿 route.review
   └─ ✋ blocked by constraints
```

no message. no hint. exit code 2. the mechanic knows a constraint exists but has no idea what. they must read source code or ask a human.

### after

the same command surfaces what's wrong:

```
🦉 patience, friend

🗿 route.review
   └─ ✋ blocked by constraints

✋ FIREWORKS_API_KEY is required
   ├─ brain: fireworks/fastcode/qwen3-coder
   └─ run: rhx keyrack unlock --owner ehmpath --env prep
```

clear. actionable. no need to dig.

### the "aha" moment

fireworks becomes the default brain. credentials work seamlessly — keyrack fetch with clear error hints. no silent failures, no guesswork. mechanics don't need to specify `--brain` for standard reviews.

## mental model

### how users would describe this

> "when i pick a brain, the CLI figures out what credentials it needs and fetches them from keyrack. if creds are absent, it tells me exactly what to do."

### analogies

- **like docker login** — authenticate once, use everywhere
- **like ssh-agent** — keys loaded once, available to all commands

### terms

| user term | system term |
|-----------|-------------|
| "brain" | brain choice slug (e.g., `fireworks/fastcode/qwen3-coder`) |
| "creds" | API key (e.g., `FIREWORKS_API_KEY`) |
| "keyrack" | credential store at `~/.rhachet/keyrack/` |
| "unlock" | fetch creds from keyrack into process.env |

## evaluation

### how well does it solve the goals?

| goal | coverage |
|------|----------|
| fireworks as default | **yes** — replaces xai entirely |
| error messages surface | **yes** — ConstraintError caught and printed |
| better default brain | **yes** — `fireworks/deepseek/v4-flash` is better, cheaper, faster |
| generic for future suppliers | **partial** — handoff created, not fully generic yet |

### pros

- immediate fix for fireworks
- default brain is now better, cheaper, and faster
- error messages no longer swallowed
- pattern exists for more suppliers
- keyrack.yml already declares per-supplier credentials

### cons

- still requires per-supplier conditional in bhrain repo
- not fully generic until rhachet has credential discovery

### edgecases & pit-of-success

| edgecase | result |
|----------|----------|
| creds already in env | skip keyrack, use env (CI-friendly) |
| keyrack locked | clear message + unlock command |
| keyrack key absent | clear message + set command |
| unknown brain supplier | falls through, ConstraintError surfaces from brain module |

## groundwork

### external research

none — no external dependencies. fireworks api key pattern is same as xai.

### internal research

verified extant patterns:

**credential fetch pattern** — `getXaiCredsFromKeyrack.ts:14-113`:
- checks `process.env.XAI_API_KEY` first (CI/test environments)
- calls keyrack CLI via `execSync`
- handles grant statuses: `granted`, `locked`, `absent`, `blocked`
- sets `process.env.XAI_API_KEY` for rhachet compatibility
- returns supplier object for future compatibility

**brain selection conditional** — `review.ts:183-187`:
```ts
const isXaiBrain = options.brain.startsWith('xai/');
if (isXaiBrain) {
  await getXaiCredsFromKeyrack();
}
// no equivalent for fireworks
```

**error handler gap** — `review.ts:228-233`:
```ts
} catch (error) {
  if (error instanceof BadRequestError) {
    process.exit(2);
  }
  throw error;  // ConstraintError rethrown, message lost
}
```

**keyrack.yml declarations** — both suppliers declare their keys:
- `rhachet-brains-xai/.agent/keyrack.yml`: `XAI_API_KEY` under `env.test`
- `rhachet-brains-fireworksai/.agent/keyrack.yml`: `FIREWORKS_API_KEY` under `env.test`

**vocab to reuse**:
- pattern: `options.brain.startsWith('fireworks/')`
- note: xai support dropped, no `isXaiBrain` needed

**architectural shift** — move from `getXyzCredsFromKeyrack` to context-based creds:
- OLD: `await getXaiCredsFromKeyrack()` — bhrain fetches creds, sets `process.env`
- NEW: pass `{ creds: { keyrack: { owner, env } } }` via context — brains repo fetches creds internally

**established pattern** — `rhachet-brains-fireworksai` two-level credential strategy:
- keyrack shorthand: `{ keyrack: { owner, env } }` — brains repo auto-discovers key
- explicit getter: `() => Promise<{ FIREWORKS_API_KEY: string }>` — custom source
- env fallback: `process.env.FIREWORKS_API_KEY` preserved for backwards compat
- bhrain usage: always `owner: ehmpath, env: prep` for reviews

**stdout patterns to match** — from `getXaiCredsFromKeyrack.ts:60-93`:
```
🦉 patience, friend

✋ keyrack is locked
   ├─ owner: ehmpath
   └─ run: rhx keyrack unlock --owner ehmpath --prikey ~/.ssh/ehmpath --env prep
```

## what is awkward?

### per-supplier conditionals (future)

for now, only fireworks is supported (xai dropped). if we add more suppliers later, we'd need:
```ts
if (isFireworksBrain) context = { ...context, creds: { keyrack: { owner: 'ehmpath', env: 'prep' } } };
if (isAnotherBrain) context = { ...context, creds: { keyrack: { owner: 'ehmpath', env: 'prep' } } };
// ... grows linearly
```

this is simpler than the old `getXyzCredsFromKeyrack` pattern (no external function, just context config), but still fights the open-closed principle. the wish acknowledges this by a handoff request to rhachet for generic credential discovery.

### reflect.ts duplication

`reflect.ts` has the same brain selection conditional. both review.ts and reflect.ts need the same context-based creds pattern. this is duplication but acceptable until generic solution exists.

### ConstraintError catch-all

to catch all ConstraintError may catch errors not related to credentials. unclear if this is desired. for now, assume yes — any ConstraintError should surface its message.

## deliverables

1. **upgrade rhachet@latest** — formally shipped support for two-level credential strategy
2. **install rhachet-brains-fireworksai@latest** — get the new two-level credential strategy
3. **review.ts** — remove `isXaiBrain` conditional, add fireworks keyrack context + catch ConstraintError
4. **reflect.ts** — same changes as review.ts
5. **default brain** — change default from `xai/grok-code-fast-1` to `fireworks/deepseek/v4-flash` (better, cheaper, faster)
6. **keyrack.yml** — replace `XAI_API_KEY` with `FIREWORKS_API_KEY` as required key for reviewer role
7. **acceptance tests** — prove fireworks reviewer works end-to-end
   - snapshot stdout when creds absent (verify error hint format)
   - snapshot stdout when creds present (verify review completes)
   - verify exit codes (2 for constraint, 0 for success)

**already done:** github issue in ehmpathy/rhachet for generic credential discovery

**dropped:** xai brain support — fireworks is the new default, xai conditionals removed

**note:** no `getFireworksCredsFromKeyrack.ts` — the brains repo handles keyrack lookup internally via the new two-level credential strategy. bhrain just passes the keyrack config via context.
