re-read vision line 149-150:
> exit code 2 → "blocked by constraints" (reviewer/judge outputs `passed: false` and `reason:` in stdout)
> exit code != 2 → "blocked by malfunction" (capture whatever is there)

wait. this says "exit code != 2" but earlier table says "other". is exit 0 included in "!= 2"?

**no.** exit 0 denotes passed. the condition is for NON-ZERO exit codes:
- exit 0 → passed
- exit 2 → constraint
- exit != 0 && != 2 → malfunction

blueprint correctly classifies:

├─ code === 2 → 'constraint'
└─ else → 'malfunction'

**no misinterpretation.**

### what if exitCode is undefined?

could execa return undefined exitCode in some cases?

**analysis:** execa with `reject: false` should always return an exitCode. if the process crashed, exitCode would be the signal number. if execa itself failed (e.g., command not found), it would throw unless we catch it.

does blueprint handle execa failure?

re-read blueprint runStoneGuardReviews codepath:
```
├─ [○] execa command with reject: false
├─ [+] capture exitCode as number
```

`reject: false` means execa won't throw on non-zero exit. but if the command doesn't exist, execa throws ENOENT.

**potential gap:** should blueprint handle ENOENT case?

**analysis:** if command doesn't exist, execa throws. this is caught by whoever calls runStoneGuardReviews. the error should propagate up as a malfunction.

wait - this is an edge case from the vision table (line 225: "command doesn't exist"). vision says artifact should show "command not found".

does blueprint handle this? the blueprint captures stdout/stderr from execa result. if execa throws, there's no result to capture.

**actual gap found:** blueprint doesn't explicitly handle the case where execa throws (command not found).

**fix needed?** let me think...

the vision says the artifact should contain "command not found". but if execa throws, we're in a try/catch scenario. the catch block would need to:
1. detect ENOENT error
2. create an artifact with error message as stderr

but the blueprint codepath shows:
```
├─ [○] execa command with reject: false
├─ [+] capture exitCode as number
├─ [+] classify via getExitCodeClass
```

this assumes execa returns successfully (even with non-zero exit). if execa throws, this flow breaks.

**is this a deviation from vision?**

vision edge case table says "command doesn't exist → clear 'command not found' in artifact"

blueprint doesn't address this case.

**verdict:** minor gap. the blueprint handles the common case (command runs but fails). edge case (command doesn't exist) is not addressed.

should I fix this?

**analysis:** this is an edge case, not the core bug. the core bug was opaque error output. the blueprint fixes that for commands that run. command-not-found is a different failure mode.

**action:** note as future work, not blocker for this behavior.
