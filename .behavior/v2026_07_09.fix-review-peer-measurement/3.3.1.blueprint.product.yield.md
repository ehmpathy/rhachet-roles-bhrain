# 3.3.1 blueprint — fix review peer measurement

## summary

add a **fallback cascade** to the guard's per-review measurement: try the deterministic count parse
first; when it misses on an exit-0 review, fall back to a cheap sub-brain
(`fireworks/deepseek/v4-flash`) that extracts the tally from the reviewer's own review text.
a review with no readable verdict still fails loud as a malfunction. when the fallback fires,
the guard's treestruct shows a `tallied by reviewer@$brain` branch.

the change decomposes into **three domain operations** (one transformer, one communicator, one
orchestrator), a **discriminated-union** return type that makes the failhide a compile error, a
**brain context threaded** down the extant 5-hop guard chain, and one **observability branch**
on the reviewer tree render.

### grain decomposition

| operation | grain | purity | test scope |
|-----------|-------|--------|------------|
| `getReviewCountsViaRegex` | transformer | pure | unit |
| `getReviewCountsViaBrain` | communicator | brain i/o | integration (real brain) |
| `getReviewCounts` | orchestrator | composes the two | integration |

**tactic-family names (both leaves named by MECHANISM, not by input).** the extant deterministic
parser is `getReviewCountsFromContent` — named by its *input* (content). this blueprint renames it to
`getReviewCountsViaRegex`, named by its *mechanism* (regex), so it reads as the symmetric peer of
`getReviewCountsViaBrain` under the `getReviewCounts` orchestrator. both leaves take `{ content }`, so
a name by input would be redundant across the pair; the mechanism is the differentiator. the rename
carries the file too (`getReviewCountsFromContent.ts` → `getReviewCountsViaRegex.ts`) and its test, per
`rule.require.sync-filename-opname`; it rides along the reshape+narrow edits already planned for that
function and its 4 callers, so it adds no new touch sites — only a `git mv`. research citations below
that say "`getReviewCountsFromContent`" refer to this same function pre-rename.

### the return contract (discriminated union)

```ts
// leaf tactics return this — no `tactic`, a tactic does not know it is in a cascade
export type ReviewCounts =
  | { detected: false }
  | { detected: true; blockers: number; nitpicks: number };

// the orchestrator stamps which tactic won, only on the true branch
export type ReviewCountsResolved =
  | { detected: false }
  | { detected: true; blockers: number; nitpicks: number; tactic: 'deterministic' | 'probabilistic' };
```

the `{ detected: false }` branch carries no count fields, so a caller cannot read a fake `0/0`
— `rule.forbid.failhide` is enforced by the type system.

### research-backed prompt design (for `getReviewCountsViaBrain`)

the sub-brain prompt applies four mitigations. the first three trace to
`3.1.1.research.external.product.flagged._.yield.md`; the fourth traces to a live in-repo observation
(the L3 reviewers on this very stone):

1. **count from the reviewer's own labels/summary**, not the raw transcript. per that yield
   [5][6][7] (convergence): the count operation is a documented, architectural LLM weakness that
   worsens with input length ([7]: *"as the length of the string increases from 50 to 100, the
   accuracy of the LLMs decreases further"*).
2. **explicitly license "no verdict"** — if no verdict is discernible, the prompt asks for
   `detected: false`. per that yield [8]: models *"exhibit a reluctance to express uncertainties
   ... even when their responses are incorrect"* and are *"over-confident"*; [8] also finds
   *"instruction hints ... can enhance model performance on abstention"*, and [9] finds
   abstention *"improves safety and reduces hallucinations"*.
3. **quote the verdict text counted** — the brain's internal schema includes an `evidence`
   string; a fabricated `0/0` on empty input has no quote to supply, so the communicator treats
   empty/absent evidence as `{ detected: false }`. per that yield [10] (*"typed evidence
   grounding, and extractive quote verification"*) — the named, research-backed solution to
   *"unverifiable score attribution"* in LLM scoring.
   **flagged for wisher (conscious addition, not drift):** the `evidence` field is a design
   addition beyond the vision's schema snippet (which had only `detected/blockers/nitpicks`). it is
   an *internal* schema field, dropped before the returned `ReviewCounts`, so it changes no caller
   contract — but it is a real safety-mitigation decision, so it is flagged here for the wisher the
   same way the `exitCode`-in-orchestrator deviation is, rather than assumed silently.
4. **interpret each item's severity from HOW IT IS WRITTEN, not from its section heading (r12 obs #5
   — the headline trigger).** real enrolled reviewers — including the two L3 `enroll-*` reviewers on
   THIS stone — do not write "N blockers, N nitpicks". they emit **headed markdown sections** from the
   architect rubric: `## Blockers`, `## Maintenance hazards`, `## Scope leaks`, `## Worth surfaced`,
   often with inline severity marks (`🔴`/`🟠`/`🟡`, `[confirmed, unresolved]`, `blocker-grade`,
   `minor`, `consider`). the vision's rule named only two buckets, so the fix could ship and STILL not
   rescue the very reviewers that motivated it.
   **the mapping rule is per-item, by expressed severity — the heading is only a weak hint.** a
   "maintenance hazard" or "scope leak" is a **blocker** ONLY when the reviewer wrote it as must-fix /
   would-block / unresolved (e.g. "MUST bound this", "regresses an extant guarantee", `🔴`,
   `[confirmed, unresolved]`); the SAME heading is a **nitpick** when written as optional / advisory
   ("consider", "worth surfaced", "minor", `🟡`). the prompt is therefore explicit: *"an issue's
   severity comes from how the reviewer describes IT, not from the section title it sits under. count
   each reported issue as a **blocker** if the reviewer frames it as must-fix / would-block /
   unresolved / high-severity, or a **nitpick** if the reviewer frames it as optional / a suggestion /
   advisory / low-severity. read the item's own wording and severity marks; the heading (Blockers,
   Maintenance hazards, Scope leaks, Worth-surfaced, etc.) is a hint, never a decree. do not require
   the literal words 'blocker' or 'nitpick'."* this keeps the sub-brain brief-less (mitigation #1's
   zero-briefs rule holds) while it teaches the *severity-reading* it will actually meet in production.

### two-severity contract — solve-at-cause, so reviewers conform in the first place

the sub-brain fallback is the **defense** (read a tally out of a review that broke the contract). the
**offense** is to stop reviewers breaking it: the reviewer role's own brief must state, unmistakably,
that **exactly two severities exist — blocker or nitpick — and nothing else is a counting bucket.**
this is `rule.require.solve-at-cause`: the L3 `enroll-*` reviewers malfunctioned because the architect
rubric taught them to emit `## Maintenance hazards` / `## Scope leaks` / `## Worth surfaced` as if they
were their own severities; the contract brief never said "collapse everything to one of the two."

- **the edit** ([~] `src/domain.roles/reviewer/briefs/contract.reviewer-output.md`): add a
  `.two severities only` section — *"every issue you report is EXACTLY one of two severities: **blocker**
  (must-fix / would-block) or **nitpick** (optional / advisory). there is no third bucket. section
  headings like 'hazard', 'scope leak', 'worth surfaced' are fine as PROSE, but each item under them
  still resolves to blocker OR nitpick, and the two numeric summary lines (`N blockers` / `N nitpicks`)
  count EVERY item by its severity. do not report a count for any other category."* this makes the
  reviewer collapse its own taxonomy up front, so the deterministic parser succeeds and the fallback
  is never needed.
- **it is booted onto the role — verified.** `getReviewerRole.ts` declares
  `briefs: { dirs: [{ uri: __dirname + '/briefs' }] }` (lines 18-20), and an `onBrain.onBoot` hook runs
  `rhachet roles boot --role reviewer` (line 26). the strengthened brief lives in exactly that `briefs`
  dir, so it boots onto every reviewer brain — the two-severity rule reaches the reviewer at run time,
  not just in docs. (this is why the offense actually lands: the brief is wired into the role's boot.)
- **defense + offense are complementary, not redundant.** the offense reduces how OFTEN the fallback
  fires (well-briefed reviewers emit numeric counts → deterministic path, zero cost); the defense
  guarantees correctness when a reviewer STILL drifts (raw brain clis a human enrolls without the
  brief, older reviewers, one-off prose). both ship together: the wish's whole premise is that some
  reviewers will not conform, so the defense stays even as the offense shrinks the gap.

**concrete: the internal `.choice.ask` schema ≠ the returned `ReviewCounts`.** the brain's schema
adds `evidence` on the true branch; the communicator uses it to gate, then maps to the clean
`ReviewCounts` (evidence dropped — it is a validation artifact, not part of the contract). execution
MUST NOT copy the vision's earlier schema snippet verbatim (it omitted `evidence`):

```ts
// getReviewCountsViaBrain.ts — the INTERNAL brain schema (note the extra evidence field)
const schema = {
  output: z.discriminatedUnion('detected', [
    z.object({ detected: z.literal(false) }),
    z.object({
      detected: z.literal(true),
      blockers: z.number().int().min(0),
      nitpicks: z.number().int().min(0),
      evidence: z.string(),   // the quoted verdict text — mitigation #3
    }),
  ]),
};
// build the brain lazily — ONLY here, on the fallback path (r11 behavior observation)
const brain = await context.getReviewBrain();
const { output } = await brain.brain.choice.ask({ role: { briefs: [] }, prompt, schema });

// gate on evidence, then map INTERNAL → ReviewCounts (drop evidence)
if (!output.detected) return { detected: false };
if (output.evidence.trim() === '') return { detected: false };   // empty quote = fabricated → no verdict
return { detected: true, blockers: output.blockers, nitpicks: output.nitpicks };
```

### data flow for the observability marker (the tactic field's journey)

the `tactic` discriminant must travel the same path the counts already travel (the brain name is
NOT stored — it is derived from the fixed model constant at render time). because the flat display
shape is redeclared **5 times**, `tactic` must be threaded through **every** site or the marker
silently never reaches stdout (the r11 observation — verified across both files):

```
getReviewCounts → ReviewCountsResolved.tactic
  → runOneStoneGuardReview writes `tactic` onto RouteStoneGuardReviewPeerArtifact (+tactic only)
    → setStoneAsPassed.ts:812  [site 1] inline builder maps artifact[] → flat shape: add `tactic: r.tactic`
      → formatRouteStoneEmit.ts:120  [site 2] FormatInput.guard.reviews inline shape: add `tactic`
      → formatGuardTree.ts:276  [site 3] input.guard.reviews inline shape: add `tactic`
      → formatGuardTree.ts:466  [site 4] hand-built map into RouteFormatTreeGuardReviewRecord: add `tactic`
      → formatGuardTree.ts:40   [site 5] RouteFormatTreeGuardReviewRecord.artifact Pick<>: add `'tactic'`
        → formatGuardTree.ts:226 asReviewerTreeStateFromMeter reads `review?.artifact.tactic` (fallback: 'deterministic')
          → builds ReviewerTreeState.finished { ..., tactic }
            → formatGuardReviewerTree renders `tallied by reviewer@${FIXED_FALLBACK_BRAIN}` when tactic='probabilistic'
```

- **`setStoneAsPassed` calls `formatRouteStoneEmit`, not `formatGuardTree` directly** on the pass path
  (`setStoneAsPassed.ts:114` et al.). the prior draft named only `formatGuardTree.ts:226` and omitted
  `formatRouteStoneEmit.ts` entirely — so the marker would have rendered in `formatGuardTree`'s own
  tests yet never in the real pass-path stdout. both render fns are now in the filediff.

- **YAGNI prune:** the brain name is NOT stored per-artifact — the model is a **fixed constant**
  (wisher: not configurable), so the render derives the display name from that constant. only the
  `tactic` discriminant travels on the artifact. if the model ever became configurable, *then*
  store the slug — not before.

- **critical (5-site plumbing):** the field must be threaded through all 5 sites above; a miss at any
  one drops the marker while isolated `formatGuardReviewerTree` unit tests stay green. site 5 is the
  read-out (`asReviewerTreeStateFromMeter` at `formatGuardTree.ts:226`, where `blockers/nitpicks` are
  already sourced via `review?.artifact.* ?? meter.*`); sites 1-4 are the write-through shapes that
  must carry `tactic` for the read-out to find it populated.
- **cached reviews (corrected):** the artifact *object* holds `tactic` in-memory, but only the
  reviewer's raw stdout/stderr is persisted to the file — so `tactic` does NOT survive a cache hit
  for free. it is recovered on reconstruction from the persisted `tallied` footer (see "cache
  survival" above): the `tallied by reviewer@…` line → `tactic: 'probabilistic'`, its absence →
  `'deterministic'`. `getAllStoneGuardArtifactsByHash` / `getLatestReviewArtifactForIndex` set the
  reconstructed artifact's `tactic` from that footer. the `meter` fallback need not carry `tactic`.
- **decouple render copy from parse contract (r11 maintenance observation):** the footer's
  `tallied by reviewer@…` prefix is written by the render AND regex-matched by the two cache-recovery
  parsers. to stop a copy tweak (e.g. `counted by`) from silently breaking tactic-recovery, the prefix
  is a **single shared exported constant** `TALLIED_FOOTER_PREFIX` referenced by the writer
  (`runOneStoneGuardReview`), both renders (`formatGuardReviewerTree`, `formatRouteStoneEmit`), and both
  cache parsers. one constant, one contract — a copy change moves the parse in lockstep, so drift is
  impossible (`rule.require.single-source-of-truth-for-render`).

**legend applied:** `[+]` create, `[~]` update, `[○]` retain, `[←]` reuse. no `[-]` / `[→]`.

```ts
export const getReviewCounts = async (
  input: { content: string; exitCode: number },
  context: ContextReviewBrainSupply,   // carries a LAZY brain supplier, not a resolved brain
): Promise<ReviewCountsResolved> => {
  // deterministic first — free, exact
  const countsViaRegex = getReviewCountsViaRegex({ content: input.content });
  if (countsViaRegex.detected) return { ...countsViaRegex, tactic: 'deterministic' };

  // only rescue a review that RAN (exit 0) but phrased its verdict oddly
  if (input.exitCode !== 0) return { detected: false };

  // probabilistic fallback — cheap sub-brain reads the small review text
  // .note = getReviewCountsViaBrain calls context.getReviewBrain() internally; the brain
  //         builds lazily HERE, only when a review actually needs the fallback
  const countsViaBrain = await getReviewCountsViaBrain({ content: input.content }, context);
  if (countsViaBrain.detected) return { ...countsViaBrain, tactic: 'probabilistic' };

  // neither tactic found a verdict — the false branch carries no counts to fake
  return { detected: false };
};
```

- **exit-0 gate** lives in the orchestrator (per wisher: only rescue `exitCode === 0`). the
  deterministic tactic still runs for non-zero exits (a numeric verdict is still read verbatim),
  but the *brain* fallback is skipped — a crashed reviewer is never rescued by a brain.
- **deliberate deviation from the vision signature (documented, not drift):** the vision declared
  `getReviewCounts(input: { content: string }, context)` — no `exitCode` — and left the exit-0
  decision implicit at the call site. this blueprint adds `exitCode: number` to the orchestrator
  input and places the gate **between** the two tactics. rationale: the vision's own narrative ran
  both tactics unconditionally, which contradicts its exit-0 rule; the coherent resolution is to let
  the orchestrator own the *complete* tactic-selection logic (deterministic always, brain only on
  exit 0), so the call site stays a single `getReviewCounts({ content, exitCode })` call rather than
  a scattered gate. behavior is identical to the vision's intent; only the gate's *home* moved into
  the orchestrator (single-responsibility). flagged here so a reviewer sees this as a conscious
  choice, not an accidental signature drift.

### brain supply — a lazy memoized supplier, not an eager context (addresses r11 behavior observation)

the guard threads a **brain supplier**, not an already-built brain context. this preserves blackbox
usecase.1 ("no extra brain cost is incurred for a conforming reviewer") at the *construction* level,
not just the *call* level.

- **the type** (threaded down the 5-hop chain in place of a bare `ContextBrain`):

  ```ts
  export interface ContextReviewBrainSupply {
    // lazy + memoize-on-success: the first fallback builds the brain; later reuse it
    getReviewBrain: () => Promise<ContextBrain<BrainChoice>>;
  }
  ```

- **the supplier is a hand-written closure, NOT a fictional `withSimpleCache`** (r11 obs #6 + r12 obs
  #1 — verified: `withSimpleCache` exists nowhere in this repo or its deps). memoize **only on
  success**, so a transient build failure does not permanently disable the fallback:

  ```ts
  // genReviewBrainSupply.ts — the pass-path composition root builds ONE of these
  export const genReviewBrainSupply = (
    input: { choice: BrainChoice; creds: KeyrackCreds },
  ): ContextReviewBrainSupply => {
    let cached: ContextBrain<BrainChoice> | null = null;   // memoize the RESOLVED value, not the promise
    const getReviewBrain = async (): Promise<ContextBrain<BrainChoice>> => {
      if (cached) return cached;
      const built = await genContextBrain(input);   // a throw propagates — no value is memoized on throw
      cached = built;                               // reached ONLY on success
      return cached;
    };
    return { getReviewBrain };
  };
  ```
  because the memoize writes `cached` only after `genContextBrain` returns, a throw leaves `cached`
  null and the next prose-reviewer re-attempts — the "one hiccup poisons the process" failure mode
  r11 named cannot occur. (the resolved value is cached, not the in-flight promise, so a concurrent
  double-build is at worst one redundant construction, never a cached rejection.)

- **why lazy:** `genContextBrain({ choice })` in discovery mode does async package discovery + choice
  validation **at construction**, throwing if the brain package is not discoverable (this repo just
  removed one such package — commit `ef68d79`). building it eagerly at `route.ts` on *every*
  `--as passed` would make a brain-package problem break **100% of stone-passes**, including
  fully-deterministic ones — the opposite of "zero cost for a numeric-only reviewer". the supplier
  defers the build to the *first fallback that actually runs*, so a stone whose reviewers all emit
  numeric counts never touches the brain machinery at all.
- **why a supplier, not construction inside the communicator:** the supplier injected via `context`
  preserves `rule.require.dependency-injection` — tests inject a test-brain supplier
  (`{ getReviewBrain: async () => genTestBrainContext(...) }`); a hardcoded in-communicator
  `genContextBrain` would be untestable and non-injectable. this also honors the wisher's confirmed
  "in-process, not subprocess" decision: the supplier runs in-process, merely lazily.
- **blast-radius narrowing:** the vision conceded "wider blast radius than a leaf change ... accepted"
  for threading a brain. the supplier keeps the *threading* (5 one-line context-type widens) but moves
  the *failure surface* to the one leaf that needs it — `rule.prefer.most-common-denominator` in
  practice: the dependency lives where it is used, not where it is built.

### fallback timeout — the brain call must not unbound a bounded measurement (r11 obs #5 / r12 obs #4)

`runOneStoneGuardReview` already timeboxes the reviewer subprocess (`input.timeout` →
`getReviewTimeoutMs`, `runStoneGuardReviews.ts:167`). the sub-brain fallback must inherit an
equivalent bound, else a hung `brain.choice.ask` makes a bounded reviewer's *measurement* step
unbounded — a regression of an extant guarantee for a reason unrelated to the reviewer.

- **bound:** the fallback brain call is wrapped in a timeout (a `FALLBACK_BRAIN_TIMEOUT` constant,
  default `PT30S` — generous vs the vision's "sub-second" expectation, tight enough to fail loud). on
  timeout the call **throws** (a `MalfunctionError` / timeout error), which the per-reviewer seam
  converts to a `malfunction` — same path as any other brain fault, never a silent pass.
- **where:** the timeout lives in `getReviewCountsViaBrain` (the communicator owns its own i/o bound),
  applied around `brain.choice.ask`. it does NOT reuse the reviewer subprocess timeout (that bounds a
  child process; this bounds an in-process async call) — a separate, explicitly-named constant.
- **test:** an integration case injects a supplier whose brain hangs past the bound → the call throws
  a timeout, the reviewer becomes a `malfunction`, the stone blocks loud (added to the
  `getReviewCountsViaBrain` boundary rows + the `runStoneGuardReviews` e2e malfunction-cause set).

### brain-error response — a failed fallback is a malfunction, never a silent pass

the vision's edgecases require: "brain itself errors / times out → surfaces as malfunction, never
silent-passes." the brain call is now **in-process** in the guard, so a brain throw is an in-process
exception, not a subprocess exit — the blueprint must say where it lands.

- **`getReviewCountsViaBrain` does NOT swallow the error.** it must never `try/catch → { detected:
  false }` — that would be a `rule.forbid.failhide` (a brain crash masquerades as "no verdict" and
  could pass). the communicator lets the error propagate, wrapped for context via
  `UnexpectedCodePathError.wrap` / `HelpfulError.wrap` (per `rule.prefer.helpful-error-wrap`) so the
  stone gets a diagnosable message.
- **`getReviewCounts` (orchestrator) does NOT catch it** either — its contract is "return
  `ReviewCountsResolved` or throw"; a brain fault is exceptional, so it throws.
- **`runOneStoneGuardReview` (the per-reviewer boundary) converts it to a per-reviewer malfunction.**
  an allowlisted catch at this seam marks *that* reviewer `exitClass: 'malfunction'` (fail loud, the
  stone blocks) while other reviewers still run — the error is not re-thrown to abort the whole
  guard, and it is not swallowed into an approval. the caught error's message is written to the
  reviewer's `stderr`/artifact so a human sees *why* (brain error), distinct from the "no verdict"
  malfunction (`{ detected: false }`). this keeps the two malfunction *causes* legible, per the
  wish's concern that a single failure signal does double duty.

net: a brain error and a genuine no-verdict both **block loud** as malfunctions (never a silent
pass), but their stderr messages differ, so the human can tell an odd-phrase miss from a brain fault.

### cache survival — persist the resolved tally into the artifact file

`getReviewCountsViaRegex` has **four** non-test callers, not one. besides the fresh-run call at
`runStoneGuardReviews.ts:212`, three others re-parse **stored** review files:
`computeReviewTotalsFromFiles.ts:17` (threshold aggregation), `getAllStoneGuardArtifactsByHash.ts:141`
and `getLatestReviewArtifactForIndex.ts:83` (both **cache reconstruction**). all four blind-access
`counts.blockers/nitpicks`, so the union forces each to narrow on `detected` (undetected → 0
contribution / 0-0 artifact, to preserve today's `?? 0` behavior for the genuinely-absent case).

but that narrow alone does not suffice — it exposes a deeper hazard. today the guard persists only
the reviewer's raw `stdout`/`stderr` into the artifact file (`runStoneGuardReviews.ts:234-250`) — the
extracted counts live only in-memory. on a **cache hit**, reconstruction re-parses that stored prose
with the *deterministic* parser. for a review originally rescued by the sub-brain (prose, no
numbers), the re-parse returns `detected: false` → **the probabilistic counts are lost**, and a
cached reviewer silently regresses to 0/0. the marker would vanish on the second run
(`rule.forbid.surprises`).

**fix (centralized, one write makes all re-parsers correct):** when the orchestrator resolves a
tally, the guard composes a numeric `tallied` footer **into** the artifact content that
`runOneStoneGuardReview` authors, then persists the whole via the extant single
`fs.writeFile(stdoutPath, artifactContent)` (`runStoneGuardReviews.ts:253`). this is an **upsert**,
not an append: the footer is one segment of the single authored write that replaces the file
wholesale, so a rerun re-authors the same content and **cannot stack footers**
(`rule.forbid.nonidempotent-mutations`). a cache *hit* re-runs no review, so it performs no write —
the footer authored on the fresh run is the durable source. the footer shape:

```
└─ tallied
   ├─ 0 blockers
   ├─ 1 nitpick
   └─ tallied by reviewer@fireworks/deepseek/v4-flash     # only when tactic = probabilistic; matches the render phrasing
```

because `getReviewCountsViaRegex` takes the **last** numeric match per dimension, this footer is
what every downstream re-parse recovers — so cache reconstruction and totals aggregation both
recover the *resolved* counts deterministically, with **no** brain call on the cache path. the
`tactic` is recovered from the `tallied by reviewer@…` line (present → probabilistic; absent →
deterministic). this is why the three reconstruction/aggregation callers can stay pure and cheap:
the expensive resolution happened once, at fresh-run time, and was written down.

- deterministic reviewers already carry their numbers in `stdout`; the footer is still written for
  uniformity, and its last-match position is harmless (same numbers).
- this makes the artifact file **self-descriptive**: the persisted tally is the source of truth on
  re-read, not a re-derivation. it is what makes the journey `[t1] == [t2]` cache assertion pass.

### single source of truth — counts standardized once at ingest (addresses r1 repo-rules blockers)

two peer-review blockers (repo-rules) flagged this design against `rule.require.judge-derived-counts`
and `rule.require.review-standardization-at-parse`. both resolve to one principle, which this
blueprint now states explicitly and adheres to: **a review's blocker/nitpick tally is resolved
exactly once, at the ingest boundary, and every downstream consumer reads that one standardized
value — never a parallel re-derivation.**

- **the ingest boundary is `runOneStoneGuardReview`** (`runStoneGuardReviews.ts:212` + `:253`). it is
  already the sole place a review is first executed, its counts extracted, and its artifact authored
  to file. the cascade (deterministic → sub-brain) runs *there and only there*. the resolved tally is
  written into the artifact as the standardized `tallied` footer (see "cache survival"), so every
  persisted review file — deterministic or brain-tallied — carries the same numeric footer shape.

- **addresses `review-standardization-at-parse`:** the footer makes standardization happen at the
  parse/ingest boundary, not scattered across consumers. downstream code does not branch on
  "structured vs unstructured" or "was this brain-tallied": the three re-parse callers
  (`computeReviewTotalsFromFiles`, `getAllStoneGuardArtifactsByHash`, `getLatestReviewArtifactForIndex`)
  read the one standardized footer with the *same* deterministic parser, uniformly. the sub-brain is
  an ingest-time detail of *how* the footer's number was obtained; by the time any consumer reads the
  artifact, the shape is singular. this is the pit-of-success the rule wants — one parse, one shape.

- **addresses `judge-derived-counts`:** the invariant the rule protects is "one extraction, no drift,
  identical across verdict states." this design honors it: after ingest there is exactly one count
  source (the persisted footer), which the `reviewed?` verdict operation
  (`computeReviewPeerVerdict`) consumes unchanged. we do **not** add a second live extraction path —
  the cascade collapses to a single persisted number before any verdict or formatter reads it. note
  the extant guard already separates count-extraction (`getReviewCountsViaRegex`) from
  threshold-application (`computeReviewPeerVerdict`, the judge); this change does not widen that
  separation — it *narrows* drift risk by collapsing all reads to one standardized source. if the
  rule is read literally as "route extraction through a `runReviewedJudge` wrapper," that wrapper is
  not the extant guard's shape (the split predates this wish); adopting it is a separate refactor,
  out of scope here, and flagged for the wisher rather than silently assumed.

### maintainability & readability

- each operation is single-responsibility, `(input, context)`, arrow-only, get-prefixed.
- the orchestrator reads as narrative (`rule.require.orchestrators-as-narrative`); no
  decode-friction inline.
- the union return removes the `?? 0` fiction at the source — fewer ways to misuse.
- the fix reuses 4 proven prod patterns (brain maker, schema-ask, tree mechanics, verdict
  computer); it invents no new infra.
- **single model constant (no magic string):** the model `'fireworks/deepseek/v4-flash'` is declared
  ONCE as an exported `FIXED_FALLBACK_BRAIN` constant and referenced by every site that needs it —
  the cli brain-construction (`genContextBrain({ choice: FIXED_FALLBACK_BRAIN })`), the render
  (`tallied by reviewer@${FIXED_FALLBACK_BRAIN}`), and the persisted footer. no raw literal is
  scattered across files, per `rule.forbid.magic-values`. (the test brain uses the same string via
  `genTestBrainContext`, the extant test util.)

### journey acceptance test

`blackbox/driver.route.peer-measurement.journey.acceptance.test.ts` [+] — one realistic,
multi-timestep journey that drives the whole cascade + observability + **cache** path end-to-end
(real brain, wrapped in `when.repeatably`). placed in `blackbox/` under the `driver.route.*` name to
match the extant journey-test convention (`blackbox/driver.route.journey.acceptance.test.ts` et al.). isolated cases prove each branch; this journey proves they compose across time —
and is the ONLY test that exercises the cache re-run, the one path where a lost `tactic` would
silently regress the marker (`howto.journey-tests-reveal-cache-bugs`).

- **explicit blackbox guarantee (r12 observation #1 sub-point):** this journey drives through the real
  cli → `setStoneAsPassed` → `formatRouteStoneEmit` render path (per `rule.require.acceptance.blackbox`),
  NOT `formatGuardTree` in isolation. this is what makes it the end-to-end catch for the 5-site marker
  plumbing: if `tactic` is dropped at ANY of the 5 write-through sites, the marker fails to appear in
  this journey's pass-path snapshot even though the isolated `formatGuardReviewerTree` unit test stays
  green. stated here as a guarantee, not left implicit on the blackbox convention.

- **[t0] preconditions** — a stone arrives for peer review with two enrolled raw-brain reviewers:
  one that emits numeric counts, one that writes prose. no artifacts cached yet. → assert clean start.
- **[t1] fresh run** — guard measures both. the numeric reviewer reads verbatim (no marker); the
  prose reviewer falls to the sub-brain (marker shown). → snapshot the full guard treestruct.
- **[t2] cache re-run** — guard runs again; artifacts persist, so `tactic` must survive on the
  cached artifact. the prose reviewer STILL shows `tallied by reviewer@$brain`; the numeric reviewer
  still shows none. → snapshot; it MUST equal [t1] (this is the cache-drops-`tactic` regression guard).
- **[t3] blocked path** — a third reviewer emits garbage (exit 0, no verdict). both tactics miss →
  malfunction, the stone blocks. → snapshot the malfunction render + the blocked stone state.
- **[t4] crash path** — a fourth reviewer exits non-zero. the brain fallback is skipped; the reviewer
  is treated as failed, never rescued. → snapshot.
- **[tN] final state** — the stone's verdict reflects the blocking malfunction; the human sees
  exactly which reviewers the sub-brain tallied and which read verbatim. → final snapshot.

this is the acceptance-level proof of blackbox usecases 1–5 as a single lived journey, with a
checkpoint snapshot at every timestep and a final-outcome assertion.

## must surface to the wisher (open questions, not closed decisions)

- **single-source-of-truth reading (r1 + r12 observation #6).** the "single source of truth" section argues
  this design already satisfies `rule.require.judge-derived-counts` and
  `rule.require.review-standardization-at-parse` in *intent* (one ingest-time resolution, one persisted
  footer all consumers read). but if those rules are read *literally* — "route every count read through

  refactor (a larger, separate scope). this is the crux the repo-rules peer blockers raised.

  are raw brain clis that write prose verdicts and are promoted to `💥 malfunction` by the guard's
  numeric-count contract — a **live instance of the exact bug this wish fixes** (r12 said so directly).
  until this fix ships, those reviewers structurally cannot pass the guard. the wisher must decide the
  disposition for *this* stone: overrule the L3 malfunctions, or approve the stone despite them.
- **collapse the 5-site display shape now, or defer? (r11 obs #3).** r11 argues that since the wish
  already hand-edits all 5 redeclarations of the review-display shape to add `tactic`, one collapse

  diff into shared render types — wisher's call to fold in or defer.
