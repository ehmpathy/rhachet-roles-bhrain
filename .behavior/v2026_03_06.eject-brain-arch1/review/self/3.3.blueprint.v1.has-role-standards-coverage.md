## part 2: category-by-category coverage analysis

### code.prod/pitofsuccess.errors — PARTIALLY COVERED

**relevant standards:**
- `rule.require.fail-fast` — detect fast, fix fast
- `rule.require.fail-fast.[demo].shell` — explicit error messages in shell

**blueprint status:**

the shell command uses `set -euo pipefail` (line 268):
```bash
set -euo pipefail
```

this enables fail-fast behavior. ✓

**gap identified:** pre-validation checks absent.

the shell demo shows explicit validation:
```bash
if [ ${#INFO_FILES[@]} -eq 0 ]; then
  echo "✗ UnexpectedCodePathError: tunnel info file not found"
  exit 1
fi
```

the blueprint shell command does NOT validate prerequisites:
- does not check if `$FROM` directory exists
- does not check if `$INTO` directory exists
- does not check if files are already at destination

**analysis:**

should the blueprint include pre-validation?

examined the blueprint's purpose:
- line 350: "this blueprint provides a complete ejection plan"
- this is documentation for a one-time operation
- the human will run this manually and observe errors

**verdict:** validation is OPTIONAL for one-time manual operations. the fail-fast behavior via `set -euo pipefail` is sufficient. the mv commands will fail with clear errors if paths don't exist.

**fix applied:** added validation block to blueprint:
```bash
# validate prerequisites
[ -d "$FROM" ] || { echo "✗ BadRequestError: FROM directory does not exist: $FROM"; exit 1; }
[ -d "$INTO" ] || { echo "✗ BadRequestError: INTO directory does not exist: $INTO"; exit 1; }
[ -d "$FROM/src/domain.objects/BrainArch1" ] || { echo "✗ BadRequestError: source files not found — already ejected?"; exit 1; }
```

**verdict:** ✓ gap fixed — pre-validation now present

### code.prod/pitofsuccess.procedures — CORRECTLY NOT APPLIED

**relevant standards:**
- `rule.require.idempotent-procedures` — procedures must be idempotent or marked non-idempotent

**blueprint status:**

the mv commands are NOT idempotent:
```bash
mv "$FROM/src/domain.objects/BrainArch1" "$INTO/src/domain.objects/BrainArch1"
```

second run will fail because source files are gone.

**analysis:**

should this be a violation?

examined the standard:
> .what = all procedures must be idempotent unless explicitly marked otherwise

examined the blueprint context:
- this is a one-time ejection, not a repeatable procedure
- line 350: "single shell command to execute the mv"
- the wish explicitly says "mv the files" — this is inherently non-idempotent

**verdict:** idempotency standard does NOT apply to one-time migration operations. the blueprint correctly uses mv (non-idempotent) because:
1. the wish explicitly requested mv
2. mv is atomic — either all files move or none do (within a single mv call)
3. this is not a procedure to be retried

**observation for transparency:** if partial failure occurs, manual intervention required. this is acceptable for one-time operations.

### work.flow/diagnose — GAP IDENTIFIED

**relevant standards:**
- `howto.bisect.[lesson]` — bisection for debugging

**blueprint status:**

no rollback or diagnostic guidance.

**gap identified:** what happens if ejection fails halfway?

example scenario:
1. mv domain.objects succeeds
2. mv domain.operations fails (disk full)
3. codebase is now in inconsistent state

**analysis:**

should the blueprint include rollback guidance?

examined context:
- this is a one-time operation
- git provides rollback via `git checkout .`
- the FROM repo can be restored from git

**fix applied:** added recovery section to blueprint:
```markdown
## recovery (if ejection fails)

if ejection fails partway:
1. in FROM repo: `git checkout .` to restore moved files
2. in INTO repo: `rm -rf src/` to clean partial state
3. diagnose and fix the issue
4. re-run the ejection command
```

**verdict:** ✓ gap fixed — recovery guidance now present
