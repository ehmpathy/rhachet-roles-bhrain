# 3.3.1 product blueprint — peer-review conversation

blueprints the code changes to deliver the wish. grounded in `3.1.3...prod` and `3.1.3...test` research (pattern numbers below reference those yields). follows extant repo patterns: one operation per file, arrow fns, `(input, context)`, `get/set/gen` verbs, inline i/o types, pure line-builder formatters.

---

## summary

five separable boundaries (per `2.2` matrices), all additive over extant patterns (no `[REPLACE]`):

1. **the `.taken.by_self` key + path transformers** — pure functions that compute the taken/given paths from a review key, so the given path (currently inline in `runOneStoneGuardReview`) and the new taken path share one source of truth.
2. **`enumRouteGuardReviewConversationFiles`** — one shared expander that unions a stone's `.given.by_peer.*` + `.taken.by_self.*` globs. feeds BOTH the guard `$conversation` var and the `review --conversation` flag (R4 anti-drift guardrail).
3. **the contemplation gate** — a peer analogue of the self-review promise flow: `--as contemplated --that <slug>` requires a `.taken` for every current-iteration `.given` before a stone may progress. renders a crystal-clear reply-prompt on the stophook, `--as arrived`, and `--as passed` (one shared formatter, three surfaces). the guard's standing per-reviewer tree (`formatGuardReviewerTree`, already emits `review: <given path>`) also gains a paired `taken: <taken path>` line, so both the live stdout AND the mirrored `{stone}.stamp` show the two-sided conversation per reviewer — given next to taken.
4. **`$conversation` substitution + `review --conversation`** — opt-in per reviewer; a reviewer without the token is unchanged.
5. **self-review absent-file parity** — carry the symmetric `contemplate from` / `articulate into` + "guard assigns the round (rN), copy verbatim" treatment to the extant self-review `challenge:absent` message.

plus **two co-located refactors** (folded in per human direction, not deferred):

**refactor A — de-duplicate the blocker-branch shape.** the contemplation gate touches the two files that already carry the worst blocker-branch duplication. verified counts (i8-r11-N1): `setStoneAsPassed.ts` = 4 `stampGuardReport`-wrapped blocked/passed shapes (`:417-427,524-535,599-610,663-671`) + a 5th divergent `review.self` shape (`:169-186`, no stamp) = 5 shapes; `stepRouteDrive.ts` duplicates TWO checks (approval + exhausted) across 3 hook surfaces = 6 near-identical blocks. rather than add a 6th shape + more surface-copies, the blueprint extracts `genStoneGuardBlockedEmit` (shared persist+return tail) and `getRouteDriveBlockerMessage` (one stophook dispatcher) and migrates the extant blocks onto them.

**refactor B — cluster `guard/` by sub-noun (structure-of-knowledge observability).** the feature adds several review/peer operations; today they would land in a flat `guard/` of ~40 loose files whose only extant cluster is `reviewPeerMeter/`. per `rule.require.group-by-noun-not-verb` + `rule.prefer.most-common-denominator`, the extant `guard/` files are reorganized into sub-noun directories (`review/`, `review/peer/`, `review/peer/meter/`, `review/self/`, `stamp/`, `tree/`, `judge/`, `artifact/`, plus leaf utils) so the whole peer-review subdomain reads as one place — see the **guard/ sub-noun cluster** section below for the full target tree + move-map. the feature's own new files then land natively by noun. NOTE: this is an EXECUTION-phase move — the route bouncer locks `src/` until this blueprint passes, so the physical relocation happens during build, not here; this blueprint only specifies the target structure.

**response to the i8 arch-defects review (B1/B2 — split the refactors out):** the reviewer recommends refactors A and B ship as their own prior, behavior-free stones/PRs. the wisher has directed otherwise: refactors A and B stay in-scope AND ship in **ONE atomic commit** with the feature (explicit wisher direction — "all in one commit"). the reviewer's bisect concern is acknowledged but consciously traded away by the wisher: a single self-contained commit is preferred over split-commit bisectability here, because the refactors and the contemplation gate touch the same two files (`setStoneAsPassed.ts`, `stepRouteDrive.ts`) and a split would create an artificial intermediate state that never ships on its own. the behavior-preservation SAFETY the reviewer wants is still honored without a split: the extant acceptance suite must pass with a ZERO snapshot diff for the untouched behaviors, and `--what types` + `--what unit` must stay green across the guard/ moves — these are execution gates ON the single commit, not a reason to fragment it.

the single most pivotal new primitive is `enumRouteGuardReviewConversationFiles`; the largest surface is the contemplation gate.

## guard/ sub-noun cluster (refactor B — execution-phase move)

**why**: `guard/` is a flat ~40-file namespace whose only extant cluster is `reviewPeerMeter/`. that lone nested subdomain is the signal — the review knowledge wants a `guard/review/peer/` home. cluster by **sub-noun** (the domain subject); the verb stays in the filename (`rule.require.group-by-noun-not-verb`, `rule.require.treestruct`). the payoff is structure-of-knowledge observability: a reader lands on `guard/review/peer/` and sees the whole peer-review subdomain — extant ops AND this feature's new ops — in one place.

**target tree**: the filediff tree above IS the authoritative clustered structure — every extant guard file appears there at its post-cluster sub-noun path, marked `[»] move` / `[»~] move + update`, beside this feature's `[+]` new files. this section adds only the move-map + rationale + execution notes; it does not restate the tree.

(no `leaf/` bucket — "leaf" is a structural category, not a sub-noun; a forced bucket would violate group-by-noun. the leaf utils stay at the most-common-denominator `guard/` root per `rule.prefer.most-common-denominator`.)

**move-map** (extant file → sub-noun dir):

| sub-noun dir | extant files moved in |
|--------------|-----------------------|
| `review/` | computeReviewThresholdVerdict, computeReviewTotalsFromFiles, computeStoneReviewInputHash, getReviewCountsFromContent, getStoneReviewLevelState, runStoneGuardReviews |
| `review/peer/` | enumRouteGuardReviewPeerFiles, getLatestReviewArtifactForIndex, getLatestReviewFilesPerIndex |
| `review/peer/meter/` | the 10 ops formerly in `reviewPeerMeter/` (dir renamed, verb-in-name `reviewPeerMeter` → noun path) |
| `review/self/` | getSelfReviewArticulationPath |
| `stamp/` | asStampContent, setStoneGuardStamp |
| `tree/` | formatGuardTree, formatGuardReviewerTree, formatTreeBucket, formatLetsReflect, formatWalkTheWay, formatPatienceFriend, formatWhatsTheRush, formatWhatHaveYouSeen |
| `judge/` | enumRouteGuardJudgeFiles |
| `artifact/` | getAllStoneGuardArtifactsByHash, getMaxStoneGuardIteration, getGitBlobHashes, enumRouteFiles |
| `guard/` root (unmoved) | parseStoneGuard, genContextCliEmit, getExitCodeClass, getDurationMsFromContent, getRepoRootWithFallback, isENOENT — these leaf utils share NO domain noun, so per `rule.prefer.most-common-denominator` they STAY at the `guard/` root rather than forced into an artificial `leaf/` bucket ("leaf" is a structural category, not a sub-noun) |

**execution notes**:
- each `.ts` moves with its `.test`/`.integration.test` sibling.
- import rewrite has two shapes: cross-boundary refs (contain `guard/…`, incl. `@src/…guard/…`) — one sub-dir insert per unique basename; and intra-`guard/` relative refs (`./x`, `../x`) — per-file depth fix. tsc drives the fix loop (unresolved-module errors are the exhaustive checklist).
- verify green: `rhx git.repo.test --what types` then `--what unit` after the moves.
- **the filediff tree above is already clustered** (post-refactor-B sub-noun paths). the **test tree** further below, however, still lists its files at their flat PRE-cluster `guard/` paths for readability against today's tree; after refactor B those test files land beside their `.ts` at the same clustered paths (e.g. `getRouteGuardReviewPeerPathTaken.test` → `guard/review/peer/`, `formatRouteGuardReviewPeerContemplatePrompt.test` + `formatGuardReviewerTree.test` → `guard/tree/`, `enumRouteGuardReviewConversationFiles.test` → `guard/review/`, `enumRouteGuardReviewPeerFiles.test` → `guard/review/peer/`).

**design-note B1 — `--as contemplated` is the acknowledgement affordance, the `.taken` presence is the gate (a deliberate wisher decision, not an oversight):** the gate (`getRouteGuardReviewPeerContemplationStatus`) derives readiness by a scan of paired `.taken` files, so a driver *could* write the `.taken` and go straight to `--as passed` without a call to `--as contemplated`. this is intentional per the vision ("the articulation IS the proof of contemplation … no separate record — the `.taken` IS the record"). `--as contemplated --that <slug>` is NOT the gate; it is the **feedback affordance** — it (a) validates the named reviewer's `.taken` is present and (b) emits the crystal-clear absent-file guidance when it is not, a per-reviewer confirmation loop. we deliberately do NOT add a durable contemplation record (that would contradict the wisher's "no separate record" and duplicate the `.taken`). the tradeoff — `--as contemplated` is skippable — is accepted: the `.taken` file the wisher chose as the proof is what actually gates passage. (the skip path is now an explicit acceptance case, item 3.)

**design-note B6 — admin overrides bypass the contemplation gate (item 7):** `--as forced` and `--as overruled` are extant admin-escape statuses that bypass the extant review gates entirely; the contemplation gate lives inside `setStoneAsPassed` (and the stophook), NOT in the `forced`/`overruled` handlers (`stones/setStoneAsForced.ts`, `stones/setStoneAsOverruled.ts`). so they bypass contemplation by construction — consistent with how they already bypass the peer-review blocker gate. this is the intended behavior (an admin escape must not be re-gated by a driver-facing requirement); it is now asserted by `driver.route.peer-contemplation-override.acceptance.test.ts`.

**design-note B7 — `--as contemplated` on a clean (0/0) reviewer is a no-op acknowledgement (item 8):** a clean reviewer has no `.given` critique to answer and needs no `.taken` (usecase 5). if a driver nonetheless signals `--as contemplated --that <clean-slug>`, it is NOT an error — the gate reports the reviewer already satisfied (no critique to contemplate) and acknowledges. this keeps the pit-of-success: a driver who over-signals is never punished with a confusing error. asserted as an edge case in `driver.route.peer-contemplation.acceptance.test.ts`.

**design-note B8 — the nitpick-only quadrant: 0 blockers + N nitpicks does NOT require a `.taken` (i8-r12-blocker-1, the "most important" boundary):** the vision states the skip rule only for the 0/0 clean case, leaving the fourth quadrant (0 blockers, N>0 nitpicks) undefined. this blueprint DECIDES it: **contemplation is required iff blockers > 0.** a nitpick-only review does NOT gate passage and needs no `.taken`. rationale: nitpicks are informational everywhere else in this codebase (the `reviewed?` judge gates on blockers, not nitpicks — `--allow-nitpicks` is routinely > 0), so a gate that forced a response to a non-gating suggestion would be disproportionate and would punish a driver for a reviewer's optional polish note. the contemplation requirement mirrors the blocker gate: you must respond to what would block you; you MAY respond to a suggestion. (`getAllRouteGuardReviewPeersUncontemplated` therefore tags a reviewer uncontemplated only when its `.given` carries blockers > 0; clean AND nitpick-only are both "satisfied".) `[wisher]`-confirmable: if the wisher wants nitpick-only to also gate, flip the one threshold in `getAllRouteGuardReviewPeersUncontemplated` — the test matrix already isolates this quadrant so the flip is a one-line change + one snapshot.

**design-note B9 — in-flight route migration is graceful BY CONSTRUCTION, no shim needed (i8-r12-blocker-2):** a stone mid-flight when this ships may hold `.given.by_peer` files from a peer review run under the old rules (no `.taken` expected). the gate does NOT hard-fail on these: contemplation is HASH-SCOPED (usecase 8) — it only requires a `.taken` for givens at the CURRENT `{iteration,hash}`. a pre-existing given sits at a PRIOR hash, so on the driver's next `--as arrived`/`--as passed` the gate sees it as prior-generation (not current) and does not gate on it; the guard re-runs the reviewers at the current hash, producing fresh givens the driver answers normally. worst case (a pre-existing given that happens to match the current hash) degrades to the standard ABSENT path — a normal crystal-clear reply-prompt, never a crash. so the "no backcompat shim" stance is safe, not risky. asserted by a migration smoke case (below).

**design-note B10 — concurrent-blocker precedence is fixed and total (i8-r12-blocker-3):** `RouteStoneGuardBlockerReport.blocker` holds ONE value, but a stone can be simultaneously (e.g.) budget-exhausted for reviewer A and uncontemplated for reviewer B. the precedence, highest-first, is: **malfunction > constraint > exhausted > contemplation.** rationale: a malfunction means a reviewer's verdict is unreadable (fix the reviewer first — you cannot contemplate an unparseable critique, per B5/item4); a constraint is a hard guard-config stop; exhaustion means the reviewer is out of budget (a human-gated state) and outranks contemplation (a driver-fixable state) so the driver is pointed at the human-blocking issue first. contemplation is LAST because it is the most volatile and most cheaply cleared. `setStoneAsPassed` evaluates the gates in this order (the extant malfunction/constraint/exhausted short-circuits already run before the new contemplation gate, so the ordering falls out of gate placement — B5/i5-item3 anchor); the stophook dispatcher (`getRouteDriveBlockerMessage`) renders whichever single blocker was persisted, and its live-recompute skips a stale one so a cleared higher-precedence blocker cannot mask a live lower one. tested with both blockers live at once (below).

**decompose/eject notes**:
- `enumRouteGuardReviewConversationFiles` is the single anti-drift source the vision's R4 demands — both the guard var and the review flag consume it.

**deletables questioned (this review's outcome)** — three components cut after a delete-before-optimize pass:
- ✂️ `getReviewPeerGivenPath` — the given path has only ONE call site (`runOneStoneGuardReview` writes it). per `rule.prefer.wet-over-dry`, one usage stays inline `[○]`. only `getRouteGuardReviewPeerPathTaken` earns a transformer — it has THREE call sites (the gate's taken-path derivation, `formatRouteGuardReviewPeerContemplatePrompt`, and `formatGuardReviewerTree`'s paired `taken:` line), well past the rule-of-three.
- ✂️ `RouteStoneGuardReviewTakenArtifact` — the `.taken` markdown file the driver writes IS the durable record; a wrapper domain object adds ceremony with no consumer. deleted.
- ✂️ `formatContemplateAbsent` — the absent-file guidance is the reply-prompt scoped to the absent reviewer(s); `formatRouteGuardReviewPeerContemplatePrompt` renders both. one formatter, not two.

```
given('[case1] a stone gated on two peer reviewers, one with --conversation')

  when('[t0] driver arrives, guard runs both reviewers')
    then('two .given.by_peer files exist; architect raises 1 blocker, mechanic clean 0/0')
    then('the arrive stdout snapshot shows the reply-prompt for architect only (mechanic clean → skipped)')

  when('[t1] driver attempts --as passed with no .taken')

  when('[t2] driver signals --as contemplated --that architect before the .taken exists')
    then('blocked; crystal-clear absent-file snapshot names the exact .taken path + why')

  when('[t3] driver writes the .taken, signals --as contemplated --that architect')
    then('architect recorded contemplated; snapshot confirms')

  when('[t4] driver attempts --as passed')
    then('progresses — mechanic clean needs no .taken, architect paired; snapshot of progress')

  when('[t5] next iteration: stone artifact changes, guard re-runs architect at a new hash with --conversation')
    then('the reviewer receives the prior .given + .taken as reference context')
    then('the claimed fix is present → architect drops the blocker (0/0); snapshot of the converged verdict')

  when('[t6] driver attempts --as passed with the now-clean architect and no new .taken')
    then('progresses — the iteration converged without a fresh contemplation (clean review); snapshot')
```
