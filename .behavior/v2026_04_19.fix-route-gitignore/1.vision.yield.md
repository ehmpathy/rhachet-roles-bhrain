# vision: gitignore self-reviews in route directories

## the outcome world

### before

```
git diff --stat main
...
159 files changed, 21209 insertions(+)
```

scroll through endless self-review files. every PR shows hundreds of lines of AI-generated review artifacts nobody reads post-merge. the signal (actual code changes) drowns in noise (review ceremony).

### after

```
git diff --stat main
...
12 files changed, 847 insertions(+)
```

diffs show what matters: stones, yields, code. self-reviews exist locally, served their purpose within the route, then vanish from git history.

### the "aha" moment

open a PR review and see only the artifacts humans care about. no more "oh right, collapse those 150 review files" muscle memory.

## user experience

### the contract

**input**: `.gitignore` findserted into each `$route/review/self/` directory

**output**: self-review files excluded from git track

**mechanism**: when guard creates `review/self/` dir, findsert `.gitignore` with `*` pattern (ignore all except the gitignore itself)

### timeline

1. driver starts route, works through stones
2. guard generates self-reviews into `review/self/`
3. self-reviews influence driver decisions (their purpose)
4. route completes, PR opens
5. self-reviews stay local, never pollute history

### what users leverage

- local self-reviews still exist, readable within active route
- `git status` cleaner, shows only intentional changes
- PR diffs focused on deliverables, not process artifacts

## mental model

### how users would describe it

> "self-reviews are work notes, not deliverables. we gitignore them like we gitignore `.env` files."

### analogies

| concept | analogy |
|---------|---------|
| self-review | scratch paper in exam |
| tracked artifact | the submitted answer |
| gitignore | recycle bin for scratch paper |

### terms

| user says | we mean |
|-----------|---------|
| "the reviews" | `review/self/*.md` files |
| "actual artifacts" | yields, stones, code |
| "noise" | tracked files nobody reads post-route |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| reduce git diff noise | excellent | 1296 fewer files tracked |
| preserve review value | good | reviews still exist locally within route |
| zero behavior change | excellent | gitignore is passive |

### pros

- **consistent**: same pattern as `.route/.gitignore` — findsert `*` with whitelist
- **self-contained**: ignore rule lives where ignored files live
- **passive**: requires no driver awareness once applied
- **reversible**: delete the `.gitignore` to re-track

### cons

- **one-time migration**: must run `git rm --cached` to untrack 1296 files already in git
- **archaeology lost**: can't `git log` to see historical self-reviews
- **no cross-machine sync**: if route spans machines, reviews don't follow
- **implicit**: new users may not realize reviews are local-only

### implementation steps

1. modify guard to findsert `.gitignore` into `$route/review/self/` when dir is created
2. `.gitignore` content: `*` and `!.gitignore` (ignore all, track the gitignore)
3. for migration: run `git rm --cached` on 1296 tracked self-review files
4. commit with explanatory message (PR will show "deletions" — from git only, not disk)

### edgecases & pit of success

| edgecase | pit of success |
|----------|----------------|
| user wants to commit a review | they can `git add -f` explicitly |
| user wants review history | they keep reviews in a separate notes repo |
| route spans machines | user syncs manually or accepts fresh reviews |

## open questions & assumptions

### assumptions

1. **self-reviews serve their purpose within the route, not after** — the wish states this explicitly
2. **nobody audits merged self-reviews** — if compliance needs them, this breaks
3. **local existence is sufficient** — no need for cloud backup of reviews

### resolved questions

1. **compliance/audit needs?** — wisher answered: "nah, no audit needs"
2. **gitignore human reviews?** — wisher answered: "only $route/review/self for now"
3. **global vs per-route gitignore?** — wisher answered: per-route `.gitignore` findserted into `review/self/` dir (consistent with `.route/` pattern)

### external research needed

none — this is a straightforward gitignore pattern

## what is awkward?

### what feels off

- **asymmetry**: human reviews tracked, self-reviews not — but this may be intentional (human reviews are the audit trail)
- **precedent set**: if we gitignore self-reviews, what else "deserves" gitignore?

### where design fights mental model

users might expect "all in `.behavior/` is tracked" — but `review/self/.gitignore` makes the exception explicit and local. no global rules, no hidden behavior.

### uncomfortable tradeoffs

| tradeoff | discomfort level |
|----------|------------------|
| lose history vs reduce noise | low — history wasn't used anyway |
| local-only vs synced | low — routes rarely span machines |
| implicit vs explicit | medium — could surprise users |
