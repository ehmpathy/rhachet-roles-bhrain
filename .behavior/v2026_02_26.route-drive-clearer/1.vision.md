# vision: route-drive-clearer

## the outcome world

### day-in-the-life

**before**: an agent that works on a route encounters a guard that requires human approval. the hook tells them to "ask a human to run `--as approved`". the agent asks the human, the human runs the command... and then what? the agent is confused. they try to run `--as approved` themselves (which they shouldn't be able to). they don't know that their next step is to run `--as passed` again.

**after**: the same agent encounters the same guard. the hook says:
```
   └─ halted, human approval required
      ├─ please ask a human to
      │  └─ rhx route.stone.set --stone 1.vision --as approved
      │
      └─ once they do, run
         └─ rhx route.stone.set --stone 1.vision --as passed
```

the agent knows exactly what to do. they message the human, wait for confirmation, then run `--as passed` to continue.

if an agent tries to run `--as approved` directly, they get:
```
🗿 route.stone.set
   └─ stone = 1.vision
      ├─ ✗ only humans can approve
      └─ please ask a human to run this command
```

### before/after contrast

| before | after |
|--------|-------|
| agents can run `--as approved` | `--as approved` blocked for agents |
| agents confused about next step | clear guidance: "once they do, run `--as passed`" |
| agents sometimes loop to run `--as approved` | agents wait for human, then `--as passed` |

### "aha" moment

the value clicks when an agent seamlessly:
1. sees they're blocked on approval
2. asks the human (with the exact command to run)
3. sees the human approved
4. runs `--as passed` to continue
5. progresses to the next stone

no confusion, no wasted cycles, no permission errors.

## user experience

### usecases

**for agents**:
- get clear error when they attempt `--as approved`
- get clear next-step when blocked on approval
- know exactly what command to run after human approves

**for humans**:
- approve stones via `--as approved` when requested by agents
- continue to use all commands normally

### contract inputs & outputs

**input**: `rhx route.stone.set --stone X --as approved`

**output (human)**:
```
🗿 route.stone.set
   └─ stone = 1.vision
      └─ ✓ approved
```

**output (agent)**:
```
🗿 route.stone.set
   └─ stone = 1.vision
      ├─ ✗ only humans can approve
      └─ please ask a human to run this command
```

### timeline

1. agent runs `--as passed`, guard blocks, needs approval
2. agent sees guidance: "ask human to run X, once they do run Y"
3. agent messages human with the approval command
4. human runs approval command
5. human confirms to agent
6. agent runs `--as passed`
7. stone passes, route continues

## mental model

### how users describe this

> "when i'm stuck, awaited for approval, the system tells me exactly what to ask the human AND what to do after they approve. no more guesswork."

### analogies

- **traffic light**: red means stop and wait. when it turns green, you know to go. the system makes the "green light" explicit ("run `--as passed`").
- **relay race**: the baton handoff is explicit. human approves (hands off baton), agent runs `--as passed` (receives baton and continues).

### terms

| user says | system says |
|-----------|-------------|
| "it's blocked" | "halted, human approval required" |
| "the human approved it" | approval artifact created |
| "i can continue now" | "run `--as passed`" |
| "it won't let me approve" | "only humans can approve" |

## evaluation

### how well does it solve the goals?

**goal 1: prevent agents from run of `--as approved`**
- ✅ detect agent vs human and block agents
- ✅ clear error message explains why

**goal 2: clarify next step for agents**
- ✅ explicit "once they do, run X" guidance
- ✅ removes ambiguity about what comes after approval

### pros

- pit of success: agents can't do the wrong action
- clear guidance: agents know exactly what to do next
- minimal change: just add agent check + tweak output

### cons

- adds a bit more text to the hook output

### edgecases

**edge: how do we detect agent vs human?**
- use TTY detection (process.stdout.isTTY)
- agents run in non-interactive sessions → no TTY
- humans run in terminals → TTY present
- not perfect but covers the common case

### pit of success

- agents cannot accidentally approve their own work
- agents always know what command to run next
- humans have clear, simple approval flow
- no loops, no confusion, no wasted cycles
