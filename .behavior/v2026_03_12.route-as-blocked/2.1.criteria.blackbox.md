# blackbox criteria: route --as blocked

## usecase.1 = robot signals block (two-step flow)

### episode: first attempt triggers nudge

```
given a robot on stone 3.blueprint
  when robot runs `rhx route.stone.set --stone 3.blueprint --as blocked`
    and no articulation file exists at $route/.route/blocker/3.blueprint.md
    then stdout shows nudge with "failure is only the opportunity to begin again"
      sothat robot is reminded to try first
    then stdout shows "passage = unchanged"
      sothat robot knows block did not succeed
    then stdout shows articulation guidance (what blocks you, what you tried, what you need)
      sothat robot knows what to articulate
    then stdout shows path to articulate into
      sothat robot knows where to write
    then system creates $route/.route/$stone.blocked.triggered file
      sothat system can verify robot saw the nudge
    then passage.jsonl is NOT updated
      sothat blocked status requires articulation
```

### episode: second attempt with articulation succeeds

```
given a robot on stone 3.blueprint
  and .triggered file exists
  when robot creates articulation file at $route/.route/blocker/3.blueprint.md
    and robot runs `rhx route.stone.set --stone 3.blueprint --as blocked`
    then stdout shows "passage = blocked"
      sothat robot knows block succeeded
    then stdout shows reason with path to articulation file
      sothat human knows where to read
    then passage.jsonl records { stone: "3.blueprint", status: "blocked" }
      sothat hooks can detect blocked state
```

### episode: articulation without trigger fails

```
given a robot on stone 3.blueprint
  and NO .triggered file exists
  when robot creates articulation file at $route/.route/blocker/3.blueprint.md
    and robot runs `rhx route.stone.set --stone 3.blueprint --as blocked`
    then stdout shows nudge (same as first attempt)
      sothat robot cannot skip the nudge
    then passage.jsonl is NOT updated
      sothat robot must see nudge first
```

## usecase.2 = human sees blocked state

### episode: route.drive shows blocked stone

```
given stone 3.blueprint has status "blocked" in passage.jsonl
  when human runs `rhx route.drive`
    then stdout shows stone = 3.blueprint
    then stdout shows status = blocked
    then stdout shows reason with path to articulation file
      sothat human knows where to read the blocker details
```

### episode: human reads articulation

```
given stone 3.blueprint is blocked
  when human reads $route/.route/blocker/3.blueprint.md
    then file contains "what blocks me" section
    then file contains "what i tried" section
    then file contains "what i need" section
      sothat human has full context to help
```

## usecase.3 = robot recovers from blocked

### episode: robot unblocks via --as passed

```
given stone 3.blueprint has status "blocked"
  when human provides guidance
    and robot addresses the blocker
    and robot runs `rhx route.stone.set --stone 3.blueprint --as passed`
    then passage.jsonl records { stone: "3.blueprint", status: "passed" }
      sothat robot can continue the route
```

## usecase.4 = route.drive suggests escalation

### episode: blockers > 5 shows --as blocked option

```
given robot is on stone 3.blueprint
  and review blockers count > 5
  when route.drive hook runs
    then stdout shows "are we there yet?" with --as passed option
    then stdout shows "are you blocked?" with --as blocked option
      sothat robot considers escalation when stuck
```
