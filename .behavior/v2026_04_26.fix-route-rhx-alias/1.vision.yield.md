# vision: $rhx and $rhachet aliases with npx failfast

## the outcome world

### before

guard files use `npx rhachet` everywhere:

```yaml
reviews:
  peer:
    - npx rhachet run --repo bhrain --skill review --rules '...' --output '$output'

judges:
  - npx rhachet run --repo bhrain --skill route.stone.judge --stone $stone --route $route
```

problems:
- `npx` adds 500-2000ms latency per command (must lookup package)
- `npx` has cross-platform issues (npm's typical path expansion bugs on windows)
- authors don't know the best way to invoke rhachet from guards

### after

guard files use `$rhx` alias:

```yaml
reviews:
  peer:
    - $rhx review --rules '...' --output '$output'

judges:
  - $rhx route.stone.judge --stone $stone --route $route
```

benefits:
- zero lookup latency (direct binary invocation)
- cross-platform reliability (no npm path expansion)
- obvious best practice (the alias screams "use me")

### the "aha" moment

when an author writes `npx rhachet` in a guard:

```
error: npx causes latency and cross-platform issues

use $rhx instead:
  - $rhx review --rules '...'

$rhx expands to ./node_modules/.bin/rhx
```

they fix it immediately because the error shows exactly what to do.

## user experience

### use cases

| who | goal | contract |
|-----|------|----------|
| guard author | invoke rhachet skill | `$rhx skill-name --args` |
| guard author | invoke full rhachet | `$rhachet run --skill ...` |
| route driver | run guard | automatic variable expansion |

### contract inputs & outputs

**variables available in guard commands:**

| variable | expands to | extant? |
|----------|------------|---------|
| `$stone` | stone name | yes |
| `$route` | route path | yes |
| `$hash` | input hash | yes |
| `$output` | output path | yes |
| `$rhx` | `./node_modules/.bin/rhx` | **new** |
| `$rhachet` | `./node_modules/.bin/rhachet` | **new** |

**failfast validation:**

| pattern | error |
|---------|-------|
| `npx rhachet` | "npx causes latency; use $rhachet instead" |
| `npx rhx` | "npx causes latency; use $rhx instead" |

### what it looks like

**writing a guard (peer review):**

```yaml
reviews:
  peer:
    - $rhx review --rules '.agent/**/rules/*.md' --paths '$route/*.md' --output '$output' --mode hard
```

**writing a guard (judge):**

```yaml
judges:
  - $rhx route.stone.judge --mechanism reviewed? --stone $stone --route $route
```

**error on npx usage:**

```
$ rhx route.drive

error: guard uses npx which causes latency

.behavior/v2026_04_26.my-feature/5.1.execution.guard:12
  npx rhachet run --skill review ...
  ^^^ npx causes 500-2000ms latency per invocation

fix: use $rhx alias instead
  - $rhx review --rules '...'

$rhx expands to ./node_modules/.bin/rhx
```

### timeline

1. author writes guard with `$rhx` (or `$rhachet` for full command)
2. driver runs `rhx route.drive`
3. guard parser validates no `npx` patterns
4. guard parser expands `$rhx` and `$rhachet` variables
5. command executes with direct binary path

## mental model

### how users would describe this

> "use $rhx in guards — it's faster than npx and just works"

> "if you try npx rhachet, it yells at you and shows the right way"

### analogies

- **npm scripts**: like how npm scripts find `node_modules/.bin` automatically
- **PATH variable**: like adding a directory to PATH, but scoped to guards
- **shell aliases**: like `alias rhx='./node_modules/.bin/rhx'`

### terms

| user says | we call it |
|-----------|------------|
| "the rhx shortcut" | $rhx alias |
| "the full command" | $rhachet alias |
| "npx blocker" | npx failfast validation |

## evaluation

### how well does it solve the goals?

| goal | solution | grade |
|------|----------|-------|
| reduce latency | direct binary = 0ms lookup | A |
| avoid cross-platform issues | no npm path expansion | A |
| make best practice obvious | failfast + error message | A |

### pros

- zero learning curve (same command, just `$rhx` prefix)
- immediate feedback on mistakes (failfast)
- extant guards can migrate incrementally
- consistent with extant variable pattern ($stone, $route, etc)

### cons

- another variable to learn (minor)
- extant guards need migration (one-time effort)
- error message must be clear enough (implementation detail)

### edgecases and pit-of-success

| edgecase | handling |
|----------|----------|
| `npx rhachet` anywhere in command | failfast with fix suggestion |
| `npx rhx` anywhere in command | failfast with fix suggestion |
| `$rhx` when node_modules missing | error: "rhx not found; run npm install" |
| command without $rhx/$rhachet | allowed (may use other tools) |

## open questions & assumptions

### assumptions

1. `./node_modules/.bin/rhx` always found after `npm install`
2. users prefer immediate feedback (failfast) over silent degradation
3. the variable syntax (`$rhx`) is intuitive for guard authors

### questions for wisher

1. should we support `$npx` as an escape hatch for edge cases?
   - recommendation: no, keep it simple
2. should the failfast apply to judges only, reviews only, or both?
   - recommendation: both
3. should we auto-migrate extant guards or just failfast on them?
   - recommendation: failfast only, let authors migrate

**contracts: substituteVars functions**

both review and judge execution use `substituteVars`:

- `src/domain.operations/route/guard/runStoneGuardReviews.ts:206-214`
- `src/domain.operations/route/judges/runStoneGuardJudges.ts:292-300`

```ts
const substituteVars = (
  cmd: string,
  vars: { stone: string; route: string; hash: string; output: string },
): string => {
  return cmd
    .replace(/\$stone/g, vars.stone)
    .replace(/\$route/g, vars.route)
    .replace(/\$hash/g, vars.hash)
    .replace(/\$output/g, vars.output);
};
```

**observation**: judges already add `node_modules/.bin` to PATH (line 53-66 in runStoneGuardJudges.ts), but reviews do not. this is an inconsistency that the `$rhx` alias will help address.

- `$route` — route path
- `$hash` — input hash
- `$output` — output file path

## what is awkward?

### variable expansion order

if someone writes `$rhx $stone`, does `$stone` get expanded before or after `$rhx`? currently all variables expand in one pass, so order doesn't matter. but if we ever have variables that reference other variables, this could get confusing.

**mitigation**: `$rhx` expands to a static path, never references other variables.

### the $rhachet vs $rhx distinction

`$rhx` is the shorthand (`rhx skill-name`)
`$rhachet` is the full command (`rhachet run --skill ...`)

most users only need `$rhx`. having both could cause confusion.

**mitigation**: documentation should emphasize `$rhx` as the primary choice.

### bash -c wrapping

some extant guards wrap commands in `bash -c "..."`:

```yaml
- bash -c ". .agent/.../use.apikeys.sh && npx rhachet run ..."
```

the npx failfast should still trigger inside bash -c strings.

**mitigation**: pattern match on content, not just command start.
