## final reflection: what makes conventions matter

### why I almost chose "classifyExitCode"

the verb "classify" felt natural because the function takes an exit code and returns a class. but reviewing extant code, I see:
- get* is for retrieval and simple computation
- compute* is for complex derivation
- the exit code → class lookup is a simple map, not classification

the extant patterns favor clarity over cleverness. `getExitCodeClass` reads as "give me the class for this exit code" - direct and obvious.

### patterns I now understand better

| pattern | when used | why |
|---------|-----------|-----|
| get* | simple retrieval or lookup | reads as "give me X" |
| compute* | derivation with logic | reads as "figure out X" |
| format* | string output | reads as "render X as string" |
| run* | execute external process | reads as "run X" |
| step* | workflow stage | reads as "do this step" |
| parse* | text to typed shape | reads as "interpret X" |
| gen* | find-or-create | reads as "ensure X exists" |

### how this review improved the blueprint

1. **classifyExitCode → getExitCodeClass** - aligns with get* pattern
2. **confirmed formatTreeBucket** - format* is the right pattern
3. **verified directory structure** - judges/ exists as blueprint states
4. **checked type/test conventions** - all match extant patterns

the review caught a real divergence and fixed it. the blueprint is now more consistent with the codebase.
