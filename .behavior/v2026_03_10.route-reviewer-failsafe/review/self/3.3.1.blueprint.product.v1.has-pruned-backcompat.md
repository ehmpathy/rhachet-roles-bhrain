## deeper reflection: why no backwards compat is needed

### the nature of this change

this is a **bug fix**, not a feature addition. the wisher's core complaint:
> "we didn't get a clear output"
> "the hook never triggered again"

bug fixes often require breaking the broken behavior. backwards compat with broken behavior is not a virtue.

### what would backwards compat look like here?

if we tried to maintain backwards compat, we might:
1. keep the old `---stderr---` format alongside tree buckets (dual formats)
2. add a flag `--legacy-output` to use old format
3. maintain the empty judge artifact format for "compatibility"

**all of these would be wrong:**
- the old format is the bug
- maintaining it perpetuates confusion
- adds complexity for no user benefit

### the tell-tale signs we avoided

common "to be safe" patterns that sneak in:

1. **"support both formats"** - NOT here. we're replacing the broken format.
2. **"fallback if tree bucket fails"** - NOT here. tree buckets are simple strings.
3. **"config flag to disable"** - NOT here. no one wants the old broken output.
4. **"preserve for consumers"** - NOT here. artifacts are for humans, not code.

### why this holds

the blueprint correctly:
- replaces broken behavior outright
- does not hedge with fallbacks
- does not add configuration for "old mode"
- treats this as what it is: a fix, not a migration
