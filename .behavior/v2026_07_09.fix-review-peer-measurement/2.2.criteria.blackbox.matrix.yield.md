# 2.2 blackbox criteria — coverage matrix

> distills `2.1.criteria.blackbox.md` into coverage matrices. one matrix per related usecase
> set. independent variables = the input conditions that vary; dependent variables = the
> observable outcomes.

---

## matrix A — per-review measurement (the exchange)

the atomic contract: the guard measures ONE reviewer's output. the two things that vary are
**exit code** and **the form of the verdict in the review text**. everything the human/guard
observes is a function of those two.

### dimensions

- **ind — exit code**: `0` (reviewer ran, exited clean) vs `non-zero` (reviewer failed)
- **ind — verdict form**: how the review text carries its blocker/nitpick verdict
  - `numeric` — states counts as numbers (`0 blockers`, `2 nitpicks`)
  - `prose` — reports findings only in prose, no numbers
  - `clean (numeric)` — explicit numeric clean pass (`0 blockers`, `0 nitpicks`)
  - `clean (prose)` — clear clean pass phrased in prose ("no issues")
  - `absent` — no discernible verdict (empty / garbage)

- **dep — tactic**: how the tally was obtained — `verbatim` / `fallback` / `none`
- **dep — tally**: `counts (N/M)` / `declared 0/0` / `no verdict`
- **dep — marker**: is the `tallied by reviewer@$brain` branch shown? `yes` / `no`
- **dep — classification**: `proceed-or-block` (on counts) / `approved` / `malfunction` / `failed`

### matrix

| ind: exit code | ind: verdict form | dep: tactic | dep: tally | dep: marker | dep: classification | usecase |
|----------------|-------------------|-------------|------------|-------------|---------------------|---------|
| `0` | `numeric` | verbatim | counts (N/M) | no | proceed-or-block on counts | uc.1, uc.6 |
| `0` | `prose` | fallback | counts (N/M) | **yes** | proceed-or-block on counts | uc.2, uc.7 |
| `0` | `clean (numeric)` | verbatim | declared 0/0 | no | approved | uc.8 |
| `0` | `clean (prose)` | fallback | declared 0/0 | **yes** | approved | uc.2, uc.8 |
| `0` | `absent` | none | no verdict | no | **malfunction** | uc.3, uc.9 |
| `non-zero` | `numeric` | none (skipped) | — | no | **failed** | uc.4 |
| `non-zero` | `prose` | none (skipped) | — | no | **failed** | uc.4 |
| `non-zero` | `clean (numeric)` | none (skipped) | — | no | **failed** | uc.4 |
| `non-zero` | `clean (prose)` | none (skipped) | — | no | **failed** | uc.4 |
| `non-zero` | `absent` | none (skipped) | — | no | **failed** | uc.4 |

### reading the matrix

- **the marker (`tallied by ...`) appears iff tactic = fallback** — i.e. exactly when exit is
  `0` AND the verdict was `prose` (or `clean (prose)`). that is the whole observability signal.
- **the `non-zero` block collapses across verdict form** — a failed reviewer is failed
  regardless of what its text says; the fallback never runs (uc.4). this is why 5 rows share
  one outcome. (see decomposition note below — this is expected, not a gap.)
- **`absent` + exit 0 is the only path to malfunction** — the failloud boundary. no counts are
  returned, so no fake 0/0 can leak (uc.9).
- **`clean (…)` returns a DECLARED 0/0**, structurally distinct from `absent`'s `no verdict`
  (uc.8) — an honest clean pass is never confused with an unreadable review.

## matrix B — provenance & trust (boundary conditions)

these rows are not new exit/form combinations; they refine the `fallback` and `absent` paths
against adversarial or trust-sensitive inputs. independent variable = the *nature of the review
text* the fallback confronts; dependent variable = the promised correct outcome.

### dimensions

- **ind — text nature**: what makes this text tricky for extraction
- **dep — correct tally**: what the exchange must return to be correct
- **dep — hazard guarded**: which failure mode the outcome prevents

### matrix

| ind: text nature | dep: correct tally | dep: hazard guarded | usecase |
|------------------|--------------------|--------------------|---------|
| prose mentions earlier issues, then declares all cleared (final 0/0) | declared 0/0 (final wins) | mentioned-in-passing misread as unresolved blockers | uc.10 |
| prose reports genuine unresolved findings | counts (N/M) matching the prose | real issues dropped / undercounted | uc.7 |
| empty / garbage (no verdict) | `no verdict` (never a fabricated 0/0) | over-confident fallback fakes a clean pass | uc.9 |
| a small review about a large subject | counts from the review text only | fallback reads the huge subject repo (cost blowup) | uc.11 |
| fallback brain errors / times out | `no verdict` → malfunction | silent pass on a failed extraction | uc.3 (extended) |

### reading the matrix

- rows 1–2 pin the **accuracy** promise: the fallback must reflect the reviewer's *final,
  actual* verdict — not the first number it sees, not an undercount.
- row 3 pins the **failloud** promise at the probabilistic boundary: garbage must yield
  `no verdict`, never a confident fake zero. this is the single most safety-critical row.
- row 4 pins the **cheapness** promise: input is the small review, never the large subject.
- row 5 pins the **failloud** promise for fallback infrastructure failure.

## matrix C — multi-reviewer episode (observability across reviewers)

when a stone has several enrolled reviewers, the marker must let a human tell each path apart
at a glance.

### dimensions

- **ind — reviewer mix**: the set of verdict forms across the enrolled reviewers
- **dep — per-line markers**: which reviewer lines show `tallied by ...`
- **dep — render stability**: is the treestruct visually distinct & stable across runs?

### matrix

| ind: reviewer mix | dep: per-line markers | dep: render stability | usecase |
|-------------------|-----------------------|-----------------------|---------|
| all numeric | no marker on any line | stable | uc.1, uc.5 |
| all prose | marker on every line | stable | uc.2, uc.5 |
| mixed (numeric + prose) | marker only on the prose lines | stable | uc.5 |

### reading the matrix

- the marker's **presence is the signal**; its **absence means "read verbatim."** the mixed
  row is the proof case — a human sees exactly which counts came from a fallback.

## gaps & decomposition

### gaps flagged

- **no gap in matrix A** — all 10 exit×form combinations map to a specified outcome. the 5
  `non-zero` rows intentionally share one outcome (uc.4).
- **partial-numeric form** (only ONE dimension numeric, e.g. `0 blockers` but nitpicks in
  prose) is **not** an explicit blackbox usecase. under the "both dimensions required" rule this
  is a non-numeric verdict overall → it flows to the `prose`/`fallback` row. **flagged**: the
  blueprint should confirm a partial-numeric review routes to the fallback (not a half-verbatim
  read) and add a boundary test for it. *(candidate new usecase — surfaced by the matrix.)*

### decomposition signal

- matrix A has **2 independent dimensions** (exit, form) → well within the 4+ explosion
  threshold. no decomposition needed.
- the `non-zero` block collapse to one outcome is a **healthy sign**, not bundled concerns:
  exit-code handling is a prior guard behavior; the fix only adds the `exit 0` fallback path.
  the matrix confirms the fix's surface is narrow.
- matrices B and C are refinements of matrix A's `fallback`/`absent` cells, split out to keep
  each matrix ≤ 3 dimensions and single-concern (measurement / trust / observability).
