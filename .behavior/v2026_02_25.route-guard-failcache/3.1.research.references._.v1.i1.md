# research: references

## summary

industry best practices support **cache successes only** approach. transient errors should never be cached.

---

## key references

### [1] HTTP cache semantics

**source:** RFC 7234 - Hypertext Transfer Protocol (HTTP/1.1): Caching

> "A cache MUST NOT store a response to any request, unless... the response status code is cacheable by default; or the response status code is defined as cacheable by the method definition"

**relevance:** HTTP caches only store successful responses (2xx) by default. errors (5xx) are explicitly excluded.

### [2] Google Cloud Apigee anti-pattern

**source:** https://docs.cloud.google.com/apigee/docs/api-platform/antipatterns/caching-error

> "Apigee does not recommend caching 5xx responses from backend servers."

> "If error responses are cached, then you may continue to send error responses to users even though the problem with the backend server has been fixed."

**relevance:** directly supports our approach — error cache causes stale failures to persist.

### [3] AWS CloudFront behavior

**source:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/HTTPStatusCodes.html

> "By default, error responses with HTTP codes 500, 502, 503, and 504 are not cached unless you enable this behavior."

**relevance:** industry-standard CDN explicitly excludes server errors from cache by default.

### [4] circuit breaker pattern

**source:** Microsoft Azure Architecture Patterns

> "Circuit breaker objects work like electrical circuit breakers that automatically interrupt the current when there is an abnormality... half-open (one probe request is allowed to decide whether to close the circuit again)."

**relevance:** failures are assumed to be recoverable. the system retries to detect recovery.

### [5] memoization constraints

**source:** Higher-Order Perl, Chapter 3: Cache and Memoization

> "Functions whose return value indicates a success or failure of some sort should generally not be memoized to return the same value every time it is called."

**relevance:** directly supports our approach — failure results should not be memoized.

## concepts established

### transient fault

> "In a distributed environment, calls to remote resources and services can fail due to transient faults, such as slow network connections and timeouts, or if resources are responding slowly or are temporarily unavailable. These faults typically correct themselves after a short time."
> — [6] Microsoft Azure Architecture Center

### negative cache

> "Negative cache is a cache strategy where the system caches failed or negative responses (e.g., 'not found' or 'error') instead of successfully fetched data."
> — [7] AWS Builders Library

**note:** negative cache applies to deterministic failures (404 not found), not transient failures (500 server error).

---

## conclusion

the "cache successes only" approach aligns with:
- HTTP cache semantics
- CDN default behaviors
- resilience patterns (circuit breaker)
- memoization best practices

all sources agree: **do not cache transient errors**.
