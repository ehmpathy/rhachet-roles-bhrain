# vision: fix-achiever

> no ask forgotten, no goal abandoned

---

## the outcome world

### before: goals drift into the void

the brain receives asks from the human. it does things. context compacts. asks vanish. goals evaporate. the brain "forgets" what it was supposed to do.

**a typical broken session:**
```
human: hey can you fix the login bug and also update the readme?
brain: sure! *fixes login bug*
brain: *drifts to another question*
[compaction happens]
brain: *has no memory of the readme ask*
human: did you update the readme?
brain: ...what readme?
```

the human feels unheard. the brain feels confused. trust erodes.

### after: every ask is a promise

the brain receives asks. it immediately triages them into goals. goals persist across compaction. hooks remind the brain of unfinished work. the brain cannot "forget" because the system remembers for it.

**a session with goals that work:**
```
human: hey can you fix the login bug and also update the readme?
brain: *triages asks into goals*
brain: *fixes login bug, marks goal fulfilled*
brain: *starts to drift*
[onStop hook fires]
> 🦉 patience, friend. you have unfinished goals:
> - update-readme (inflight): update the readme with new api docs
brain: right! *completes readme, marks fulfilled*
brain: all goals fulfilled, ready to rest
```

the human feels heard. the brain feels grounded. trust deepens.

### the "aha" moment

the aha clicks when the brain realizes: **goals are not overhead, they are freedom.**

without goals, the brain carries cognitive load: "what was i supposed to do?" "did i forget an item?"

with goals, the system carries the load. the brain can focus on the work. it trusts that hooks will remind it of unfinished items.

## user experience

### primary usecases

| usecase | who | what happens |
|---------|-----|--------------|
| ask capture | brain | human says an item, brain triages it into a goal |
| work focus | brain | brain picks an inflight goal, does the work, marks fulfilled |
| session end | brain | onStop hook reminds of unfinished goals |
| context refresh | brain | onBoot hook reloads goal state after compaction |
| accountability | human | human can see all asks and their coverage |

### contract inputs & outputs

**goal.memory.set** — create or update a goal
```bash
# flags one-by-one (recommended — increases focus on each component)
rhx goal.memory.set \
  --slug fix-login-bug \
  --why.ask "fix the login bug" \
  --why.purpose "users cannot access the app" \
  --why.benefit "users can log in again" \
  --what.outcome "login works without errors" \
  --how.task "debug auth flow, fix the issue" \
  --how.gate "login test passes" \
  --status inflight \
  --status.reason "start now" \
  --source peer:human

# status update
rhx goal.memory.set --slug fix-login-bug --status fulfilled --status.reason "fixed in commit abc123"

# full goal via stdin (allowed but not recommended)
cat <<'EOF' | rhx goal.memory.set
slug: fix-login-bug
why:
  ask: fix the login bug
  ...
EOF
```

**goal.triage.infer** — see what needs attention
```bash
rhx goal.triage.infer
# shows: uncovered asks, incomplete goals, suggested actions
```

**goal.triage.next** — get the next goal to work on
```bash
rhx goal.triage.next
# shows: highest priority unfinished goal
```

### timelines

1. **ask received** (t=0): human says an item
2. **triage** (promptly): brain creates goal that covers the ask before proceed to other work
3. **work** (t+1m to t+Nm): brain does the work
4. **fulfill** (t+N): brain marks goal fulfilled
5. **verify** (t+N+1): onStop hook confirms no unfinished goals

## evaluation

### how well does it solve the goals?

| wish | solution | status |
|------|----------|--------|
| 1. achiever briefs to teach goal usage | boot.yml already boots briefs; add onBoot hook to re-emit after compaction | planned |
| 2. discourage --scope repo | fail-fast if `--scope repo` while bound to route; scope is automatic | planned |
| 3. clearer skill headers/help | rewrite headers with examples and best practices | planned |
| 4. clearer "do the work" message | escalate blockers.json style in onStop hooks | planned |
| 5. onBoot hook for goal.triage.next | add `--when hook.onBoot` mode | planned |
| 6. forbid unknown args | add strict arg validation with helpful errors; also validate status enum (incomplete, blocked, enqueued, inflight, fulfilled) | planned |
| 7. better --help output | comprehensive help with yaml examples | planned |

### pros

- **persistence**: goals survive compaction
- **accountability**: every ask is tracked
- **focus**: brain knows exactly what to do next
- **trust**: human sees their asks are remembered
- **escalation**: repeated blocks get louder reminders

### cons

- **overhead**: brain must spend tokens to triage
- **rigidity**: structured goals may feel heavy for quick asks
- **learn curve**: brain must learn the goal workflow

### edgecases and pit-of-success

| edgecase | pit-of-success design |
|----------|----------------------|
| brain forgets to triage | onStop hook catches uncovered asks |
| brain marks fulfilled too early | human can verify via goal.triage.infer |
| context compacts mid-work | onBoot hook reloads goal state |
| unknown flag passed | fail-fast with helpful error |
| invalid status value | fail-fast; must be one of: incomplete, blocked, enqueued, inflight, fulfilled |
| brain tries to edit .goals/ directly | goal.guard blocks with skill suggestion |

## what is awkward?

### scope flag behavior

the `--scope` flag exists but scope should be automatic based on route bind state.

**decision**: if bound to a route, `--scope repo` should fail-fast with a helpful error. scope is automatic — no flag needed.

### goal completeness strictness

all six fields (why.ask, why.purpose, why.benefit, what.outcome, how.task, how.gate) are required for a goal to be "complete". this is strict.

**tradeoff**: strictness ensures thoughtful goals but may feel heavy for simple asks like "fix the typo".

**resolution**: accept the overhead as the cost of accountability. even trivial asks deserve structured thought.

### onBoot vs boot.yml

boot.yml already boots briefs. an onBoot hook that re-emits goal state feels redundant.

**resolution**: onBoot hook is specifically for post-compaction refresh. boot.yml is for session start. they serve different purposes.

### escalation messages

route.drive has blockers.json that tracks repeated blocks and escalates messages. goals don't have this yet.

**proposed**: add `.goals/$branch/.blockers.latest.json` that tracks:
- how many times onStop fired with unfinished goals
- which goals have been "reminded" repeatedly
- escalate message intensity based on count

**future consideration**: research shows that messages alone have weak effect (d = 0.05). if escalation proves insufficient, consider behavioral scaffold:
- force articulation: require brain to state what it will do next
- implementation intention: "when you see this goal, i will..."

for now: match route.drive's pattern first, then evaluate if more is needed.

## summary

the achiever role transforms the brain from a forgetful assistant into an accountable partner.

**key changes:**
1. auto-scope detection (no --scope flag needed)
2. onBoot hook to refresh goals after compaction
3. escalate onStop hooks that get louder with repeated reminders
4. strict arg validation with helpful errors
5. comprehensive --help with yaml examples

**the mantra:**
> to forget an ask is to break a promise. remember. 🦉
