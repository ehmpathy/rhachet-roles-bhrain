# 1.vision ‚Äî fix-guard-stamp

> ü¶â a stamp at the border gate ‚Äî proof of what the guards saw, frozen for later eyes.

## the wish, restated

whenever a guard runs **peer reviews**, the final reviewer report it prints to stdout vanishes into terminal scrollback. we want that same report **echoed into a `.stamp` file** next to the stone's `.guard`, `.stone`, and `.yield` files ‚Äî written each time the guards finish peer reviews + judgement.

the stamp is a **permanent record of the last thing that happened at the border gate**: latest peer review statuses, paths, blockers/nitpicks, and judge verdicts.

---

## the outcome world

### day-in-the-life

a driver runs `rhx route.stone.set --stone 5.1.execution --as passed`. the guard runs its peer reviewers and judges, prints the familiar `üóø route.stone.set` tree to stdout, and the driver moves on (or gets blocked).

**before:** that tree lives only in terminal scrollback. close the terminal, come back tomorrow, or hand off to a teammate ‚Äî the report is gone. to see "what's blocking" you must re-run the guard (spending budget, time, tokens) or hunt through `.reviews/peer/*.md` and reconstruct the verdict yourself.

**after:** a `5.1.execution.stamp` file sits right next to `5.1.execution.guard`. open it and you see the exact final report ‚Äî who reviewed, what they found, which paths, what the judge decided, whether passage was allowed or blocked. no re-run required.

### before/after contrast

| | before | after |
|---|---|---|
| see last guard outcome | re-run guard or read terminal scrollback | `cat 5.1.execution.stamp` |
| see current blocker | re-run guard | open the stamp |
| hand off to teammate | "trust me, it was blocked on X" | point them at the stamp |
| audit what happened | gone | persisted next to the stone |

### the "aha" moment

the value clicks the first time someone returns to a route after a break ‚Äî or picks up a teammate's route ‚Äî and asks *"where are we? what's blocking?"* ‚Äî and the answer is just **a file sitting right there**, not a re-run that costs another reviewer budget round.

## user experience

### usecases & goals

- **review handoff** ‚Äî a human or driver inspects the latest border-gate result without re-running guards
- **blocker triage** ‚Äî see *which* reviewer blocked and *why*, with paths to the full review artifacts
- **audit trail** ‚Äî a durable record of the last passage attempt, co-located with the stone
- **budget thrift** ‚Äî avoid re-running expensive peer reviewers just to re-read their verdict

### contract ‚Äî inputs & outputs

this is **not a new command**. it's a **side-effect** of the current `route.stone.set --as passed` flow.

- **input (implicit):** a stone whose guard declares peer reviews, being set as passed
- **output (new):** a `{stoneName}.stamp` file in the route directory, containing the same `emit.stdout` already printed (the guard tree from `formatRouteStoneEmit`) **plus `emit.stderr` appended when present** (the blocker/malfunction detail) ‚Äî per wisher Q1

what it looks like to leverage:

```
$ rhx route.stone.set --stone 5.1.execution --as passed
üóø route.stone.set
   ‚îú‚îÄ stone = 5.1.execution
   ‚îú‚îÄ passage = blocked
   ‚îú‚îÄ reason = ...
   ‚îî‚îÄ guard
      ‚îú‚îÄ artifacts
      ‚îú‚îÄ reviews
      ‚îÇ   ‚îî‚îÄ r1: mech-decode-friction (l1, 1/2) ...
      ‚îî‚îÄ judges
          ‚îî‚îÄ j1: ... ‚úó

# and now, also on disk:
$ cat .behavior/v.../5.1.execution.stamp
üóø route.stone.set
   ‚îú‚îÄ stone = 5.1.execution
   ‚îú‚îÄ passage = blocked
   ... (the stdout tree above)

ü™∂ stderr
ü™∂ judge 1
   ... (the blocker detail from stderr, appended below the tree)
```

### timeline

1. driver sets stone as passed
2. guard runs peer reviews + judges (current behavior)
3. final report computed (current `formatRouteStoneEmit`)
4. **new:** report written to `{stoneName}.stamp` (overwrites prior stamp)
5. report emitted to stdout (current behavior)

the stamp always reflects the **most recent** border-gate event ‚Äî it is a findsert/upsert, not an append log.

## mental model

### how users describe it

> "it's the receipt the guard prints ‚Äî but it also keeps a copy stapled to the stone."

### analogies

- **passport stamp** ‚Äî proof you passed (or were turned away) at the border gate, kept in your book
- **inspection sticker** ‚Äî the latest result, posted where the thing lives
- **receipt / paper trail** ‚Äî the last transaction, on file

### terms ‚Äî theirs vs ours

| they say | we say |
|---|---|
| "the guard's report" | `emit.stdout` / final guard tree |
| "the receipt / record" | the **stamp** (`{stone}.stamp`) |
| "the border gate" | the guard's peer review + judge run |
| "what's blocking" | passage = blocked + reason in the stamp |

`stamp` is the new domain term. it fits the border-gate metaphor already used in route briefs ("here's the passage stamp, issued when there were peer reviews").

## evaluation

### how well it solves the goals

- ‚úÖ **durable record** ‚Äî file persists next to the stone, survives terminal close & handoff
- ‚úÖ **zero re-run cost** ‚Äî reading the stamp spends no reviewer budget
- ‚úÖ **no new surface to learn** ‚Äî it's a passive side-effect, no new flags or commands
- ‚úÖ **single source** ‚Äî reuses the exact `emit.stdout` already produced; no second formatter (honors `rule.forbid.duplicate-format-tree-operations`)

### pros

- minimal contract change ‚Äî write the string we already compute
- co-located with `.guard`/`.stone`/`.yield` ‚Üí discoverable by convention
- upsert semantics ‚Üí always shows the latest, no stale-log confusion

### cons / tradeoffs

- one more file per guarded stone in the route dir (slightly more clutter)
- the stamp can go stale relative to artifacts if artifacts change but the guard is not re-run (mitigation: stamp reflects *last guard run*, which is exactly its purpose ‚Äî it's a record, not a live view)

### edgecases & pit-of-success

| edgecase | behavior |
|---|---|
| stone has **no guard** | no stamp (no gate was passed) |
| guard has **no peer reviews** (judges only / artifacts only) | **no stamp** ‚Äî wish scopes this to "whenever the guard has peer reviews" |
| guard **blocked** by reviewer/exhaustion/constraint/malfunction | stamp written (records the block) |
| guard **passed** | stamp written (records the pass) |
| re-run after a fix | stamp **overwritten** with the new result |
| review.self required (blocked before peer reviews run) | **no stamp** ‚Äî peer reviews never ran |

the pit of success: the stamp is present **iff** peer reviews actually ran at the gate, so its presence is itself a signal.

## open questions & assumptions

### assumptions made

1. **"final reviewer report" = `emit.stdout`** from `formatRouteStoneEmit({ ..., guard })` in `setStoneAsPassed.ts`. this is the `üóø route.stone.set` tree that embeds `formatGuardTree`. *(needs wisher confirm ‚Äî but strongly implied)*
2. **file name = `{stoneName}.stamp`**, e.g. `5.1.execution.stamp`, matching the `{stoneName}.guard` / `.stone` / `.yield.md` sibling convention.
3. **upsert, not append** ‚Äî "the latest" / "what happened last" implies overwrite. one stamp per stone, always current.
4. **scope = peer reviews only** ‚Äî wish says "whenever the guard has **peer reviews**". judges-only or artifacts-only guards do **not** get a stamp.
5. **written in all post-peer-review branches** of `setStoneAsPassed`: allowed (passed), blocked (judge reject), blocked (exhaustion), blocked (constraint), malfunction.

### triaged open items

each item tagged: **[decided]** (wisher ruled) ¬∑ **[answered]** (settled via code/logic) ¬∑ **[research]** (defer to research phase).

- **Q1 [decided]:** stamp = `emit.stdout` **+ `emit.stderr`**. the wisher chose to include stderr, so the stamp captures both the guard tree *and* the blocker/malfunction detail ("what currently blocks the driver"). when stderr is present, it is appended to the stamp; when absent, the stamp is stdout alone.
- **Q2 [decided]:** filename = **plain `{stoneName}.stamp`** (e.g. `1.vision.stamp`), no `.md`. matches `.guard`/`.stone` since it's terminal output.
- **Q3 [decided]:** **leave stale stamps.** if a later re-run no longer has peer reviews, the prior stamp stays as a record of what last happened. no delete logic.
- **A6 [answered]:** concurrent `route.stone.set` race on the stamp write? ‚Üí last-writer-wins on a record file; a driver issues one set at a time; blast radius tiny. no lock/guard warranted.
- **A7 [answered]:** is the stamp path sealed by `route.mutate.guard`? ‚Üí **no.** the sealed region is the `.route/` subdir (holds `passage.jsonl`); the stamp sits at the route **root** next to `.guard`/`.stone`/`.yield`, which is unsealed and writable. only residual: git-track stance (a blueprint convention choice, not a blocker).

### all questions resolved

every open item is now **[decided]** (wisher) or **[answered]** (code/logic). no open question blocks the blueprint.

## what is awkward?

- **scope boundary feels slightly arbitrary at first** ‚Äî "only when there are peer reviews" means a judges-only guard leaves no stamp. this is faithful to the wish, but a future reader might expect a stamp for *any* guard outcome. the metaphor saves it: a stamp records a **peer review** event specifically. worth noting in the blueprint so it's a deliberate decision, not an accident.
- **stdout vs stderr split** ‚Äî the most useful "what's blocking" detail (failed judge bodies, malfunction traces) lives in `emit.stderr`, separate from the `emit.stdout` tree. a stamp of stdout-only would show *that* it's blocked but not the full *why*. this is the sharpest design question (Q1) and the place the design could fight the user's "I want to see what blocks me" mental model.
- **upsert timing** ‚Äî the stamp must be written on **every** post-peer-review branch (5 of them). that's repetition across return paths in `setStoneAsPassed`; a single choke-point (write once, just before each guard-bearing return ‚Äî or wrap the emit) would be cleaner than five scattered writes. a decode-friction / narrative concern to settle in blueprint, not vision.
- **`.stamp` extension has no precedent** in this repo ‚Äî `.guard`/`.stone` are extensionless-ish, `.yield.md` is markdown. picking `.stamp` is a small new convention; fine, but it's a choice we're introducing.
