# playtest: enbrief route creation

## prerequisites

- repo cloned and dependencies installed (`npm ci`)
- rhachet cli available (`npx rhachet --help`)

## sandbox

all file operations in this playtest target the repo as-is (read-only verification of extant files).

---

## happy paths

### 1. verify brief exists and contains expected sections

**action:**
```bash
cat src/domain.roles/driver/briefs/howto.create-routes.\[ref\].md
```

**expected outcome:**
- file exists
- contains `## .what` and `## .why` headers
- contains `## concepts` section with `### route`, `### stone`, `### guard` subsections
- contains `## commands` table with route.bind.set, route.bind.get, route.bind.del, route.drive, route.stone.set
- contains `## example` section with 5-step walkthrough
- contains `## parallel stones` section
- contains `## guard syntax` section with yaml example
- contains `## advanced: nested routes` section
- contains `## advanced: variants` section

---

### 2. verify boot.yml exists and references the brief

**action:**
```bash
cat src/domain.roles/driver/boot.yml
```

**expected outcome:**
- file exists
- contains `always.briefs.ref` list
- list includes `briefs/howto.create-routes.[ref].md`
- list includes `briefs/im_a.bhrain_owl.md`
- list includes `briefs/define.routes-are-gardened.[philosophy].md`
- all briefs are in `ref` mode (not `say`)

---

### 3. verify getDriverRole.ts includes boot property

**action:**
```bash
grep "boot:" src/domain.roles/driver/getDriverRole.ts
```

**expected outcome:**
- output shows `boot: { uri: __dirname + '/boot.yml' },`

---

### 4. verify unit tests pass

**action:**
```bash
npm run test:unit -- getDriverRole
```

**expected outcome:**
- test suite passes
- 2 tests pass: [t0] every brief in briefs/ is declared in boot.yml, [t1] all declared briefs exist in briefs/

---

### 5. verify brief is readable and useful

**action:**
read through `src/domain.roles/driver/briefs/howto.create-routes.[ref].md` as a human would

**expected outcome:**
- concepts section clearly explains what stones, guards, and routes are
- commands table is comprehensive and copy-pasteable
- example section is followable without prior context
- guard syntax example shows full yaml structure
- no jargon or unexplained terms

---

## edgey paths

### edge 1: absent brief detection

**action:**
```bash
# temporarily rename a brief and run tests
mv src/domain.roles/driver/briefs/howto.create-routes.\[ref\].md src/domain.roles/driver/briefs/howto.create-routes.\[ref\].md.bak
npm run test:unit -- getDriverRole
mv src/domain.roles/driver/briefs/howto.create-routes.\[ref\].md.bak src/domain.roles/driver/briefs/howto.create-routes.\[ref\].md
```

**expected outcome:**
- test fails with message about undeclared brief
- brief is restored after test

### edge 2: phantom declaration detection

**action:**
```bash
# add a fake brief to boot.yml and run tests
# (manual: edit boot.yml to add "briefs/fake.md" then run tests)
```

**expected outcome:**
- test [t1] fails because declared brief does not exist

---

## pass/fail criteria

### pass if
- [ ] brief file exists at expected path
- [ ] brief contains all required sections (concepts, commands, example, parallel stones, guard syntax, advanced)
- [ ] boot.yml exists and declares all briefs as ref
- [ ] getDriverRole.ts includes boot property
- [ ] unit tests pass (2/2)
- [ ] brief content is clear and followable by someone new to routes

### fail if
- brief file absent
- brief lacks required sections
- boot.yml declares brief as `say` instead of `ref`
- getDriverRole.ts lacks boot property
- unit tests fail
- brief content unclear or uses undefined jargon
