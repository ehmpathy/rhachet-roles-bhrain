# factory opportunities analysis

## the constraint

### what is THE bottleneck?
**opaque failure output.** when a reviewer or judge fails, the current artifact shows only `exit code: 1` without the stdout/stderr that would explain why.

### if you could fix one problem, what would unblock the most?
clear error visibility. the wish describes time spent to debug failures that should have been immediately obvious:
- "what happened?"
- "why was the error not clear?"
- "why did it fail without clear explanations?"

---

## flow and queues

### what queues exist in verification?
1. test execution queue (jest runs tests)
2. shell process spawn queue (execa executes commands)
3. artifact write queue (fs writes)

all are fast and automated.

### what must you wait for that you can't control?
1. shell process execution (~1-5s per command)
2. jest test discovery (~1s)

### what's the work-in-progress limit before you get stuck?
unlimited - tests can run in parallel

---

## waste

### what handoffs slow you down?
the handoff from "test failed" to "understand why" is slow because:
- must read artifact file
- artifact shows `exit code: 1` but not why
- must re-run command manually to see output

### what setup is repeated that could be cached or shared?
- temp directory creation (already fast via `genTempDir`)
- fixture copy (already fast, ~10ms)

### what partial work sits idle?
none - tests are atomic

### what do you do manually that's error-prone?
manual observation of CLI output to understand failures. this is exactly what the behavior aims to fix via captured stdout/stderr in tree bucket format.

---

## small batches

### can you verify incrementally, or must you wait for completion?
incrementally:
1. unit test for `formatTreeBucket` helper (pure function)
2. unit test for exit code classification (pure function)
3. integration test for capture behavior
4. integration test for route.drive malfunction behavior

### what's the smallest verifiable unit?
`formatTreeBucket` - a pure function that formats stdout/stderr into tree bucket format

### can you get feedback before the whole is done?
yes - unit tests for formatters can pass before integration tests are written

## systems view

### would this improvement help just this behavior, or all future work?
**all future work.** clear error output benefits every route.drive session:
- faster debug cycles
- less human intervention
- better agent self-correction

### are you in a local optimum at the cost of the whole?
no - the tree bucket format is a global improvement:
- consistent output format across all guards
- same format for reviewers and judges
- same format for constraint and malfunction failures

---

## opportunity identified

| aspect | current | improved |
|--------|---------|----------|
| failure output | `exit code: 1` | stdout/stderr in tree buckets |
| malfunction detection | none | exit code classification |
| hook continuation | blocked | continues on constraint, halts on malfunction |

**primary opportunity:** capture and display stdout/stderr clearly so failures are immediately debuggable without manual re-execution.
