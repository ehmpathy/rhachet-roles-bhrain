# self-review: has-divergence-addressed (r3)

## question

deeper scrutiny: are the divergence backups truly valid?

## verification: setStoneAsBlocked.test.ts

examined actual git diff:

```diff
-        const blockerDir = path.join(routeDir, 'blocker');
+        const blockerDir = path.join(tempDir, 'blocker');
```

### what changed

- before: `routeDir` = the `.route/` subdirectory = `tempDir/.route/`
- before: blockerDir = `tempDir/.route/blocker/`
- after: blockerDir = `tempDir/blocker/`

### why this was necessary

the production code changed blocker location from `$route/.route/blocker/` to `$route/blocker/`.

in the test:
- `tempDir` represents the route directory (`$route`)
- `routeDir` represents the `.route/` subdirectory (`$route/.route/`)

the test fixture was correctly updated to place blockers in the new location.

### was this laziness?

no. this test:
1. creates a blocker file at the expected location
2. runs setStoneAsBlocked
3. asserts the result

if the blocker file is in the wrong location, the test would fail. the change was mandatory.

## verification: snapshot files

snapshot files are generated by jest via `toMatchSnapshot()`. they are not hand-written. the blueprint does not enumerate generated artifacts because:
1. their content depends on test execution
2. they change automatically when tests run with RESNAP=true
3. to enumerate them would create false precision

## conclusion

both divergences are valid backups:
1. setStoneAsBlocked.test.ts - necessary consistency fix (verified via diff)
2. snapshot files - generated artifacts (standard practice)

no divergence represents avoidance of required work.
