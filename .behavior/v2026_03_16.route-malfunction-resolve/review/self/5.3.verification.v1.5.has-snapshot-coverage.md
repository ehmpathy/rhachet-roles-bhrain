# self-review: has-snapshot-coverage (iteration 5)

## snapshot review

---

### does this feature need snapshots?

the repros artifact (3.2.distill.repros.experience._.v1.i1.md) explicitly stated:

```
## snapshot coverage plan

- [ ] case1 t0 malfunction output → not needed (exit code check sufficient)
- [ ] case1 t1 passage.jsonl state → inline assertion
- [ ] case2 t0 malfunction review artifact → content check
- [ ] case2 t1 pass review artifact → content check

note: snapshots not required — assertions on exit codes and file contents are sufficient.
```

---

### why snapshots are not needed here

**what callers see:**

1. **exit codes** — callers see exit code 0, 2, or 3
   - verified via `expect(result.code).toEqual(X)`
   - no visual output to snapshot

2. **passage.jsonl state** — internal state file
   - verified via inline assertions on line count and content
   - not user-visible output

3. **review artifacts** — internal artifacts
   - verified via `expect(content).toContain(X)`
   - not user-visible output

**contrast with other route tests:**

tests like `driver.route.journey.acceptance.test.ts` use snapshots because:
- they verify CLI output (what user sees in terminal)
- the output format is part of the contract

this feature verifies:
- internal state transitions
- exit codes
- artifact content

these are behavioral assertions, not output format assertions.

---

### verification

checked the test file for snapshot usage:
- no `.toMatchSnapshot()` calls
- no `__snapshots__` directory created

this aligns with the repros plan.

---

## conclusion

snapshots are intentionally absent because:
1. repros explicitly planned "snapshots not required"
2. feature verifies internal state, not visual output
3. assertions on exit codes and file contents are sufficient

this is a deliberate design choice, not an oversight.
