# vision: fix reflect.snapshot capture ENOBUFS

## the outcome world

### before

```
$ rhx reflect.snapshot capture

Error: spawnSync /bin/sh ENOBUFS
    at execSync (node:child_process:983:15)
    at setSavepoint (.../setSavepoint.js:63:54)
```

users with large staged diffs (>1MB) hit a wall. the snapshot capture crashes before it can even begin. this is particularly frustrating during behavior routes where the staged changes grow large — exactly when you most need to capture progress.

### after

```
$ rhx reflect.snapshot capture

🦉 know thyself

🪨 reflect.snapshot capture
   ├─ savepoint: 2026-03-26T12:34:56Z
   └─ staged: 847 files, 1.2MB
```

large diffs just work. no buffer limits to worry about. users capture snapshots confidently regardless of how much work is staged.

### the "aha" moment

the value clicks when a user realizes they can stage an entire refactor — hundreds of files, megabytes of changes — and still capture a snapshot without thinking about it. the tool gets out of the way.

## user experience

### usecases

| goal | experience |
|------|------------|
| capture mid-work snapshot | `rhx reflect.snapshot capture` — succeeds regardless of diff size |
| resume from snapshot | `rhx reflect.snapshot restore` — works with large snapshots |
| review staged changes | snapshot includes full diff for later review |

### contract

**input**: git working directory with staged changes (any size)
**output**: savepoint file with metadata and diff content

the contract doesn't change — we just remove the artificial buffer limit that was causing failures.

### timeline

1. user stages large changes (100+ files)
2. user runs `rhx reflect.snapshot capture`
3. tool captures diff, creates savepoint
4. user continues work or shares snapshot

## evaluation

### how well does it solve the goal?

**completely** — the fix eliminates the buffer entirely. diff content flows directly to file via shell, never enters node.

### pros

- eliminates constraint (no buffer limit, not just higher limit)
- lower memory (diff never buffered in node)
- shell does what shell does best (redirect, sha256sum)
- no API changes
- backwards compatible

### cons

- depends on `sha256sum` (standard unix tool)

### pit of success

users don't need to think about buffer sizes. the tool handles it. if a diff is truly enormous (system memory limits), node will throw a clear memory error rather than the cryptic ENOBUFS.

## open questions & assumptions

### assumptions

- `sha256sum` is available (standard on linux/mac)
- shell redirect handles large output without limits

### questions to validate

1. [answered] should we stream large diffs instead of buffer? — yes! use shell redirect so diff never enters node's buffer.
2. [answered] what's a reasonable upper bound? — no longer relevant. no buffer involved.
3. [answered] where are the execSync calls that need this fix? — `setSavepoint.ts` lines 100 and 106 (`git diff --staged` and `git diff`).
4. [answered] do we need atomicity for file + hash? — no. hash is metadata only, not in filename. write then hash is fine.

## what is awkward?

### no major concerns

the approach is simple:
- shell redirect handles write-to-file (no node buffer)
- `sha256sum` handles hash (on complete file)
- node only orchestrates

no awkwardness — we solve the root cause (buffer in node) not a symptom (buffer too small).

### tradeoffs

| tradeoff | resolution |
|----------|------------|
| shell vs node | shell redirect is the right tool for large output to file |
| node hash vs shell hash | shell hash follows from shell write — content is in file, not node |

## summary

move write and hash from node to shell.

### before (node buffer — ENOBUFS)

```typescript
const patch = execSync('git diff --staged', { encoding: 'utf-8' });  // ← buffers in node
fs.writeFileSync(path, patch);
const hash = computeHash(patch);  // ← hash in node
```

### after (shell — no buffer)

```typescript
execSync(`git diff --staged > "${path}"`, { cwd });  // ← direct to file
const hash = execSync(`sha256sum "${path}" | cut -d' ' -f1`, { encoding: 'utf-8' }).trim();  // ← hash via shell
const size = fs.statSync(path).size;
```

diff content never enters node. that's the fix.
