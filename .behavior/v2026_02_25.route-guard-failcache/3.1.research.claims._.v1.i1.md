# research: claims

## claims inventory

### [FACT] 5xx errors represent transient failures

> "In a distributed environment, calls to remote resources and services can fail due to transient faults, such as slow network connections and timeouts, or if resources are responding slowly or are temporarily unavailable. These faults typically correct themselves after a short time."
> — [1] Microsoft Azure Architecture Center

### [FACT] caching 5xx errors is an antipattern

> "Apigee does not recommend caching 5xx responses from backend servers."
> — [2] Google Cloud Apigee Documentation

> "If error responses are cached, then you may continue to send error responses to users even though the problem with the backend server has been fixed."
> — [2] Google Cloud Apigee Documentation

### [FACT] industry default excludes 5xx from cache

> "By default, error responses with HTTP codes 500, 502, 503, and 504 are not cached unless you enable this behavior."
> — [3] Amazon CloudFront Documentation

### [FACT] negative caching exists but is selective

> "Negative caching is a caching strategy where the system caches failed or negative responses (e.g., 'not found' or 'error') instead of successfully fetched data."
> — [4] AWS Builders Library

note: negative caching applies to deterministic failures (404 not found), not transient failures (500 server error).

### [SUMP] retry is appropriate for transient faults

> "The Retry Pattern is used in software development to handle transient faults by automatically retrying failed operations. It provides a mechanism to repeat an operation that failed temporarily, with the expectation that a later attempt will succeed."
> — [5] Microsoft Azure Circuit Breaker Pattern

assumption: transient faults will clear on their own, so retry is a valid strategy.

### [SUMP] circuit breaker pattern assumes failures are recoverable

> "Circuit breaker objects work like electrical circuit breakers that automatically interrupt the current when there is an abnormality in the circuit. [...] half-open (one probe request is allowed to decide whether to close the circuit again)."
> — [5] Microsoft Azure Circuit Breaker Pattern

assumption: after some time, the failed service will recover and requests should flow again.

### [OPIN] lazy caching is foundational

> "Laziness should serve as the foundation of any good caching strategy. The basic idea is to populate the cache only when an object is actually requested by the application."
> — [4] AWS Builders Library

opinion: cache-aside (lazy) is preferred over eager caching.

### [KHUE] when should failures be cached?

> "The choice between caching failures and success-only depends on your use case, with negative caching being particularly valuable for scenarios where avoiding repeated failed requests significantly reduces backend load."
> — [4] AWS Builders Library

open question: should we cache deterministic failures (like exit code 2 = explicit block)?

our answer: no. even exit code 2 may become stale (e.g., approval granted). cache only exit code 0.

### [KHUE] what about cache stampede on failure?

> "To remedy the thundering herd issue, use request coalescing, where the servers or external cache ensure that only one pending request is out for uncached resources."
> — [4] AWS Builders Library

open question: if errors aren't cached, will retries cause stampede?

our answer: judges are local operations, not distributed. stampede not a concern for single-user CLI.

## conclusion

industry consensus strongly supports our approach:
- **cache successes** (equivalent to 2xx responses)
- **do not cache errors** (equivalent to 5xx responses)
- **retry transient failures** (let them clear on their own)

the "cache successes only" strategy aligns with HTTP cache best practices and resilience patterns.
