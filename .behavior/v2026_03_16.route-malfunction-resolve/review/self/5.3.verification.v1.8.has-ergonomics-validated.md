### journey 1: malfunction recovery - traced end-to-end

**what the repros sketched:**

```
t0: mechanic runs route, reviewer breaks
    → passage.jsonl: {"stone":"X","status":"malfunction"}
    → route exits with code 3
    → user sees: "halted, guard malfunction"

t1: human fixes reviewer, mechanic retries
    → passage.jsonl: {"stone":"X","status":"passed"} appended
    → route.drive --mode hook sees latest = passed
    → hook exits 0 (allows stop)
```

**what we implemented:**

the test file `driver.route.malfunction.acceptance.test.ts` exercises this exact flow:

1. **[t0] preconditions**: test creates `.route/passage.jsonl` with malfunction entry
2. **[t0] verification**: invokes `route.drive --mode hook`, asserts exit code 3
3. **[t1] setup**: appends passed entry to passage.jsonl
4. **[t1] verification**: invokes `route.drive --mode hook`, asserts exit code 0

the implementation in `stepRouteDrive.ts` was modified to:
- collect all passage entries
- group by stone
- check only the LATEST status per stone
- halt only if latest === 'malfunction'

**ergonomics match?**

| aspect | repros planned | implementation |
|--------|----------------|----------------|
| input command | `route.drive --mode hook` | same |
| output on malfunction | exit 3 + message | exit 3 + "malfunction" in output |
| output after fix | exit 0 (silent) | exit 0 |
| user action to fix | append passed entry | tested via manual passage.jsonl edit |

the fix pathway in production would be `route.stone.set --as passed`, but for acceptance test isolation we directly manipulate passage.jsonl. this is acceptable because we test the contract (exit codes) not the CLI surface.

### journey 2: reviewer cache exclusion - traced end-to-end

**what the repros sketched:**

```
t0: reviewer malfunctions
    → review artifact created with malfunction indicator
    → NOT cached (only successes get cached)

t1: reviewer fixed, retry same hash
    → fresh review runs (cache miss)
    → new artifact created
```

**what we implemented:**

the test file exercises this via:

1. **[t0] setup**: sets malfunction flag in test fixture
2. **[t0] verification**: invokes `route.stone.set`, asserts non-zero exit
3. **[t1] setup**: removes malfunction flag, sets pass flag
4. **[t1] verification**: invokes `route.stone.set`, asserts exit 0, checks new artifact

the implementation in `runStoneGuardReviews.ts` was modified to:
- filter cached reviews by `exitClass !== 'malfunction'`
- only reuse reviews that passed

**ergonomics match?**

| aspect | repros planned | implementation |
|--------|----------------|----------------|
| cache behavior | malfunction excluded | `exitClass !== 'malfunction'` filter |
| retry behavior | fresh review | cache miss triggers new review |
| artifact creation | new on retry | asserted via file count |

### friction analysis

**potential friction points identified in repros:**

1. "user does not need to edit files manually" - the route system handles passage.jsonl via commands
2. "user does not need to know internal format" - commands abstract the jsonl structure
3. "same command works on retry" - `route.stone.set` is idempotent

**implementation delivers on these:**

1. in production, `route.stone.set --as passed` appends to passage.jsonl
2. user never sees raw jsonl, only command output
3. retry of same command triggers fresh review (cache exclusion works)
