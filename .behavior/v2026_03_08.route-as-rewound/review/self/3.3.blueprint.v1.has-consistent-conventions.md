# self-review: has-consistent-conventions

## reflection

I enumerated the operation names in `domain.operations/route/` to identify extant conventions, then verified the blueprint aligns.

---

## extant name conventions

### setStoneAs* pattern

operations that change a stone's status:
- setStoneAsPassed
- setStoneAsApproved
- setStoneAsPromised

**blueprint proposes**: setStoneAsRewound — follows this pattern exactly ✓

### del* pattern

deletion operations use `del` prefix:
- delStone
- delRouteBind
- delDriveBlockerState
- delStoneGuardBlockerReport

**blueprint proposes**: delStoneGuardArtifacts — follows this pattern ✓

### StoneGuard* namespace

operations on guard-related state:
- setStoneGuardApproval
- getOneStoneGuardApproval
- setStoneGuardBlockerReport
- getStoneGuardBlockerReport
- delStoneGuardBlockerReport
- getAllStoneGuardArtifactsByHash

**blueprint proposes**: delStoneGuardArtifacts — follows StoneGuard namespace ✓

### plural vs singular

- `getAllStones` — plural for enumeration
- `getAllPassageReports` — plural for enumeration
- `delStone` — singular for single entity
- `delStoneGuardBlockerReport` — singular for single report type

**blueprint proposes**: delStoneGuardArtifacts — plural because it deletes multiple artifact types (reviews, judges, promises, triggers). this matches the scope semantically.
