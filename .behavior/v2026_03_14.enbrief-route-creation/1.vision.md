# vision: enbrief route creation

## the outcome world

### before

today, someone who wants to create a new thought route must:
- reverse-engineer extant routes (`.route/v$date.$slug/`)
- guess at the file name conventions (`.stone`, `.guard`)
- trial-and-error the route.bind mechanism
- wonder which variant to use (`.route/` default vs `.$variant/` for special routes)

knowledge is scattered across code, examples, and tribal memory. new route authors stumble through discovery.

### after

a reference brief in the driver role explains:
- **what stones are** — milestones on a thought journey
- **what guards are** — validation gates that require passage before one proceeds
- **how to invoke route.bind** — `rhx route.bind.set --route .route/v$isodate.$slug`
- **where to emit routes** — `.route/v$isodate.$slug/` by default, or `.$variant/v$isodate.$slug/` for special routes

the brief is a `ref` (not `say`) — it only loads when someone explicitly seeks guidance on route creation. this keeps context lean for the 90% of sessions that don't involve route creation.

### the "aha" moment

a new driver reads the brief, sees the anatomy of a route, and realizes: "oh, it's just files with conventions. the `.stone` describes the work, the `.guard` describes the validation, and `route.bind` links my branch to the route. I can create a route in 5 minutes."

## user experience

### usecases

| goal | contract | invocation |
|------|----------|------------|
| create a new route | pick a slug, create `.route/v$isodate.$slug/` | `mkdir .route/v2026_03_15.my-feature` |
| define a stone | create `N.name.stone` file with instructions | write markdown that describes what to do |
| add validation gate | create matched `N.name.guard` file | yaml with `judges:` and optional `reviews:` |
| bind route to branch | `route.bind.set --route $path` | `rhx route.bind.set --route .route/v2026_03_15.my-feature` |
| check current bind | `route.bind.get` | `rhx route.bind.get` |
| remove bind | `route.bind.del` | `rhx route.bind.del` |

### timeline

1. **author decides to create a route**
   - reads the brief (via ref lookup or explicit fetch)
   - understands stone/guard anatomy

2. **author creates the route directory**
   - `mkdir .route/v$isodate.$slug` (or `.$variant/` for special routes)
   - creates `0.wish.md` (optional but recommended)
   - creates `1.vision.stone` (first milestone)

3. **author adds validation if needed**
   - creates `1.vision.guard` with judges and/or reviews
   - guard requires passage before stone is marked complete

4. **author binds the route to their branch**
   - `rhx route.bind.set --route .route/v2026_03_15.my-feature`
   - now `route.drive` will guide them through the stones

5. **author walks the route**
   - `route.drive` shows next stone
   - author completes work, marks as passed
   - guards validate passage
   - repeat until route complete

## mental model

> "a route is like a checklist with guardrails — each stone is a step, each guard is a quality gate."

### analogies

| concept | analogy |
|---------|---------|
| route | trail with waypoints |
| stone | trail marker — "you are here, go this way" |
| guard | checkpoint — "prove you're ready before you proceed" |
| route.bind | GPS destination set |
| passage | trail passport stamp |

### user terms vs our terms

| user might say | we call it |
|----------------|-----------|
| "checklist" | route |
| "step" or "milestone" | stone |
| "validation" or "gate" | guard |
| "activate the route" | route.bind.set |
| "mark as done" | stone.set --as passed |

## evaluation

### pros

- **discoverable** — brief explains the entire contract in one place
- **low overhead** — ref mode means it doesn't load unless requested
- **self-documented** — the brief itself is a route artifact (meta!)
- **progressive disclosure** — basic concepts first, then guards, then advanced patterns

### cons

- **maintenance** — brief must be updated when route mechanics evolve
- **single source risk** — if the brief drifts from reality, it misleads
- **ref accessibility** — users must know to ask for it

### edgecases

| edgecase | pit-of-success |
|----------|---------------|
| user creates route without wish | works, but recommended to add 0.wish.md for context |
| user forgets to bind | route.drive will prompt "no route bound" |
| user creates guard without stone | fails fast — guard requires matched stone |
| user creates malformed guard yaml | fails fast with parse error |
| user creates duplicate stone numbers | fails fast — uniqueness required per prefix group |

## what is awkward?

### the `.$variant/` convention

routes live in `.$variant/v$isodate.$slug/`:
- `.route/` — default variant for most routes
- `.behavior/` — special variant for behavior-driven routes (like this one)
- other variants as needed

within each route, execution artifacts go in `.$variant/v$date.$slug/.route/` (e.g., passage.jsonl, reviews).

**resolved**: wisher clarified `.route/` is the default, with `.$variant/` for special cases.

### boot.yml for driver role

the architect role uses boot.yml to distinguish say vs ref. the driver role doesn't have boot.yml yet. we'll need to add it.

**tradeoff**: adds a new file to the driver role, but enables proper ref boot.

### "ref" semantics

users may not understand what "ref" means in the boot context. the brief should probably not use that term — just say "this documentation loads when you need it, not by default."
