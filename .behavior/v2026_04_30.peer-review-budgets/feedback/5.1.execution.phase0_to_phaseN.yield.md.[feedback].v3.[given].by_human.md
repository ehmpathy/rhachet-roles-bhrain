## 🔴 blockers

### 1. snapshot shows incorrect verdict: "approved" when blockers exist

**location**: `blackbox/__snapshots__/driver.route.peer-budget-multilevel.journey.acceptance.test.ts.snap` (lines 17-23)

```
├─ r1: linter (l1, 1/2, approved)
│   ├─ finished [TIME] ✓
...
└─ reason: blockers exceed threshold (2 > 0)
```

the linter mock (`mock-review-linter.sh`) produces blockers by default (no `linter-should-pass` flag exists at [t0]). yet snapshot shows verdict `approved` and judge shows 2 blockers. 

per `computeReviewPeerVerdict.ts`: if `blockers > 0`, verdict should be `rejected`, not `approved`. either:
- mock output format isn't being parsed for blocker count
- verdict computation ignores actual blocker count from review output
- the mock produces 0 blockers unexpectedly

**impact**: false confidence — snapshots appear correct but behavior may be wrong.

## 🟡 nitpicks

### 4. `route.guard.budget` CLI signature drift from vision

vision shows:
```sh
rhx route.guard.budget --for review --add 2
```

actual implementation:
```sh
rhx route.guard.budget --add 2 --peer r1 --stone 1.execute --route .
```

- `--for review` flag doesn't exist
- must specify `--stone` and `--route` explicitly

**ergonomic friction**: users following vision docs will hit "unknown flag" errors.

### 5. approval unification scenario lacks combined snapshot

criteria usecase.8 says: "one approval covers both" (exhaustion AND explicit approved? judge)

the `3.release` journey tests explicit approval separately. the `1.execute` journey tests exhaustion approval separately. no test shows BOTH present simultaneously and demonstrates single approval satisfies both.

```typescript
then('primo did NOT run yet (level 2 waits)', () => {
```

could pass even if "awaits" appears in unrelated context. should assert `awaits l1` specifically.

unclear behavior when `--peer` slug doesn't match any declared reviewer. no test coverage.
