# 1.vision — statusline shows progress toward the stone

## .what (one breath)

today the statusline shows **which stone** the driver is on (`🗿 1.vision`). it says
zero about **where inside that stone** the driver sits — does it still yield the
artifact? mid self-review? deep in peer review? does it await a human to approve? is it
blocked?

this wish paves the next glance: a **suffix** on the stone line that names the guard
**phase** and its **progress**, tipped with an **attention-color emoji** so an observer
knows — at a glance, by color alone — whether the driver needs them.

rendered as (a test-drive against this very route's stones):

```
🗿 1.vision, yield 🌾
🗿 1.vision, review.self, r7/r10 🔍
🗿 5.1.execution.from_vision, review.peer, l3@i002 🔍
🗿 5.3.verification, judge, approved? 👋
🗿 5.1.execution.from_vision, review.peer, l3@i002, exhausted 👋
🗿 5.1.execution.from_vision, review.peer, l3@i002, blocked ✋
🗿 5.1.execution.from_vision, review.peer, l3@i002, malfunction 💥
🗿 route complete 🌴🤙
```

the emoji is the signal. its **color** answers one question — *does this need me?* — and it
tracks exactly one truth: whether the route still **drives itself** (`push`, calm) or has
**halted** and needs a human (`halt`, 👋/✋).

| suffix | emoji | disposition | attention |
|--------|-------|-------------|-----------|
| `yield` | 🌾 grain | push | none — it grows |
| `review.self` / `review.peer` | 🔍 magnifier | push | none — machine at work |
| `judge, approved?` | 👋 wave | halt (approval) | **soft** — a human must approve |
| `…, exhausted` | 👋 wave | halt (exhausted) | **soft** — a human must approve or extend |
| `…, blocked` | ✋ stophand | halt (wall) | **hard** — stop, act now |
| `…, malfunction` | 💥 collision | halt (malfunction) | **hard** — a human must fix |
| `route complete` | 🌴🤙 palmtree + shaka | — | **soft** — done, hang loose (close it out, or rewind) |

> the palette walks calm → warm-ask → red-stop. **push** is always calm (🌾/🔍); **halt** is
> always an ask — 👋 for a routine approve, ✋ for a hard wall, 💥 for a malfunction. an
> observer reads the *color temperature* before they read a single word.

## the outcome world

### day-in-the-life

a human sets a driver loose on a route and walks away. they glance at the bottom of
the screen every so often. the line tells a **story** without a single command:

```
🗿 1.vision, yield 🌾              ← still yields the vision. calm. leave it be.
🗿 1.vision, review.self, r3/r10 🔍 ← self-review, 3 of 10. calm. machine's at work.
🗿 1.vision, review.self, r10/r10 🔍
🗿 1.vision, review.peer, l1@i001 🔍 ← peers now review. calm.
🗿 1.vision, review.peer, l3@i002 🔍 ← escalated to level 3, second round. still calm.
🗿 5.1.execution.from_vision, yield 🌾 ← stone passed! moved on. calm.
   ...
🗿 5.3.verification, judge, approved? 👋 ← 👋 waves. THIS needs a human. approve it.
🗿 route complete 🌴🤙              ← 🌴🤙 done. hang loose — close the route out, or rewind.
```

the human never typed a word. the **grain and magnifier** told them "keep on the
tea" 🍵. the **wave** told them "put the tea down, i need you." the **claps** told them
"we're done — wrap it up."

### before / after

| | before | after |
|---|--------|-------|
| "which stone?" | `🗿 1.vision` | `🗿 1.vision` *(unchanged)* |
| "where **in** the stone?" | invisible — run `rhx route.drive`, read the tree | glance: `, review.peer, l3@i002 🔍` |
| "does it need me?" | invisible until you re-ask or it stops | the emoji **color** answers at once |
| self-review progress | scroll the guard output | `r7/r10` in the suffix |
| peer-review depth | run the guard, read the meter tree | `l3@i002` in the suffix |
| await approval | the session stalls silent on Stop | `judge, approved? 👋` — an explicit wave |

| route done | `🎉` — festive but unclear ("do i celebrate or act?") | `🌴🤙` — "done, hang loose — close it out or rewind" |

### the "aha"

the value clicks the first time an observer **does zero** because the emoji is calm —
and then **looks up** the moment it turns to a 👋 or ✋. progress stops as a state you
*query* and becomes a state you *feel by color*. the statusline graduates from "which
step" to "which step, how far, and do you need me" — the whole GPS, pinned.

## user experience

### who & what goal

- **the observer / co-pilot** — wants to know, without interrupts, *whether the driver
  needs them*. the emoji color is the entire answer.
- **the operator** — wants to confirm the driver makes forward progress inside a long
  stone (peer review can churn for many rounds) rather than a silent stall.
- **the reviewer/approver human** — wants an unmissable, ambient cue that an
  `approved?` gate awaits their signature.

### the contract (what we extend)

this wish **extends the extant renderer** and adds **one shared disposition**. the same
`route.status.line` command emits a richer line; the same `route.drive` onStop hook reads the
same disposition — so the pinned emoji and the autonomous-drive behavior become one truth.

#### the three orthogonal concepts

the design rests on three concepts, each with exactly one job:

- **phase** — *where* the work is: `yield | review.self | review.peer | judge`. pure context;
  it never carries "stopped".
- **status** — the passage marker in `passage.jsonl` (latest-entry-wins):
  `passed | blocked | exhausted | promised | contemplated | malfunction | …`. a halt marker
  (`blocked`, `exhausted`) is **ephemeral** — the next `--as <status>` supersedes it, so a
  stale halt never shows.
- **disposition** — *whether* the route self-drives: **`push | halt`**. one derived value,
  computed from `(status, blocker)`, read by **both** the onStop hook and the statusline. it
  replaces the old `blocked` phase overlay.

> there is no "blocked phase". "blocked" was the informal name for `disposition = halt`. a
> "driven blocked" is unrepresentable by construction — which is the point.

> **the hard rule — any forward motion clears the blocker.** the ephemeral-halt guarantee
> rests on one invariant: *every* `--as <verb>` writes a passage entry, so the latest entry
> always reflects the driver's most recent move. mark a stone `--as xyz` and you are over
> the blocker. the two review sub-steps — `--as promised` (a self-review) and
> `--as contemplated` (a peer reply) — were the exception: they wrote only an artifact, no
> passage entry, so a stale escalation lingered through them. they now write a `promised` /
> `contemplated` status (disposition **push** — the machine's own review work), so the rule
> holds with no exception. see `rule.require.forward-motion-clears-blocker`.

#### the one derivation both surfaces read

`asStoneDisposition({ status, blocker })` → `push | halt` (with the halt reason):

| status / blocker | disposition | onStop | statusline glyph |
|------------------|-------------|--------|------------------|
| reviews still open · review.peer active · uncontemplated · non-approval judge | **push** | block the stop (exit 2) | 🌾 / 🔍 |
| `blocker: 'approval'` | **halt** (approval) | allow the stop (exit 0) | 👋 |
| `status: 'exhausted'` | **halt** (exhausted) | allow the stop | 👋 |
| `status: 'blocked'`, no blocker (driver wall) | **halt** (blocked) | allow the stop (exit 0) | ✋ |
| `status: 'malfunction'` | **halt** (malfunction) | allow the stop, but escalate (exit 1) | 💥 |

- **push** → the route self-drives; the glyph is the phase's calm glyph (🌾 yield, 🔍 review/judge).
- **halt** → the route stopped; the glyph tells the human *how* to help: 👋 for a routine approve
  (approval or exhausted), ✋ for a hard wall (driver `--as blocked`), 💥 for a malfunction.
- onStop reads `push | halt`; the statusline reads the same value — they can never disagree.

#### the data model

**layer 1 — `asStatusLine` (pure transformer, the single source of render truth)**

the render state carries both a **phase** and a **disposition**:

```ts
type StatusLineState =
  | { kind: 'blank' }
  | {
      kind: 'stone';
      stone: string;
      phase: StatusLinePhase;
      disposition: StatusLineDisposition;
    }
  | { kind: 'complete' };

// WHERE the work is — pure context, never "stopped"
type StatusLinePhase =
  | { of: 'yield' }
  | { of: 'review.self'; done: number; total: number }   // r7/r10
  | { of: 'review.peer'; level: number; rounds: number }  // l3@i002
  | { of: 'judge' };

// WHETHER the route self-drives — the one bit onStop reads; the glyph derives from it
type StatusLineDisposition =
  | { of: 'push' }
  | { of: 'halt'; why: 'approval' | 'exhausted' | 'blocked' | 'malfunction' };
```

rendered (the full test-drive matrix, `phase` + `disposition`):

| phase + disposition | output |
|---------------------|--------|
| `yield` + `push` | `🗿 1.vision, yield 🌾` |
| `review.self{done:7,total:10}` + `push` | `🗿 1.vision, review.self, r7/r10 🔍` |
| `review.peer{level:3,rounds:2}` + `push` | `🗿 5.1…, review.peer, l3@i002 🔍` |
| `judge` + `halt{why:'approval'}` | `🗿 5.3.verification, judge, approved? 👋` |
| `review.peer{level:3,rounds:2}` + `halt{why:'exhausted'}` | `🗿 5.1…, review.peer, l3@i002, exhausted 👋` |
| `review.peer{level:3,rounds:2}` + `halt{why:'blocked'}` | `🗿 5.1…, review.peer, l3@i002, blocked ✋` |
| `review.peer{level:3,rounds:2}` + `halt{why:'malfunction'}` | `🗿 5.1…, review.peer, l3@i002, malfunction 💥` |
| `{ kind:'complete' }` | `🗿 route complete 🌴🤙` |
| `{ kind:'blank' }` | `` (empty) |

the suffix shape is uniform: `<phase-text>[, <halt-word>] <glyph>`. **push** shows the phase
and its calm glyph; **halt** appends a halt-word (`approved?` / `exhausted` / `blocked` /
`malfunction`) and the attention glyph.

**layer 2 — derivation in `stepRouteStatusLine`**

`getRouteStatusLineState` already derives `blank | stone | complete`. for a `stone` it now
derives two orthogonal values from extant guard + passage state:

- **phase** (where the work is) — from the guard/review state, unchanged from the base design:
  peer meters → `review.peer, l{level}@i{rounds}`; self promises → `review.self, r{done}/{total}`;
  a judge context → `judge`; else → `yield`.
- **disposition** (`push | halt`) — from `asStoneDisposition({ status, blocker })` over the
  latest `PassageReport` for the stone: the same op the onStop hook calls.

phase and disposition are orthogonal. a driver wall mid-peer-review is
`phase = review.peer` + `disposition = halt(blocked)` → `review.peer, l3@i002, blocked ✋`;
an exhausted peer review is `phase = review.peer` + `disposition = halt(exhausted)` →
`review.peer, l3@i002, exhausted 👋`.

**the new `exhausted` status**

`exhausted` graduates from a blocker sub-type (`review.peer.exhausted`) to a first-class
passage **status**. why: by phase an exhausted peer review looks like ordinary `review.peer`
(a calm 🔍 push), but it is truly a human-wait (a 👋 halt). as its own status it maps direct —
no phase-inference, no ambiguity. `approval` stays a blocker (`blocker: 'approval'` is already
unambiguous). `blocked` is left to mean only the hard driver wall.

### what it looks like to leverage

nobody invokes this by hand in the happy path — same as today. the observer watches the
color. to debug, the same direct invocation still works:

```bash
node -e "import('rhachet-roles-bhrain/cli/route').then(m => m.routeStatusLine())"
# -> 🗿 1.vision, review.self, r7/r10 🔍
```

**layer 3 — onStop parity in `stepRouteDrive`**

the onStop hook computes the **same** `asStoneDisposition` and honors it:

- `push` → block the stop (exit 2). the route keeps its own momentum — even when a passage
  attempt just failed on an agent-fixable blocker, the driver converges without a human.
- `halt` → allow the stop. the route will not self-drive; a human is needed (approve,
  extend, or clear a wall). the statusline's 👋/✋ already says which. the exit code is 0 for
  the routine halts (approval, exhausted, driver wall) and 1 for a malfunction — both allow
  the stop (never exit 2), but malfunction uses the exit-1 escalation code so the broken
  reviewer/judge reaches a human, per `rule.require.exit-code-semantics` (0 ok, 1
  malfunction, 2 constraint). onBoot never blocks boot, so it shows the same message with no
  exit stderr (always exit 0).

this is the fix for the reported defect: a `--as blocked` escalation, once superseded by a
re-arrival, no longer halts; and a re-arrival that fails on an agent-fixable blocker **pushes
forward** instead of a halt on a stale "blocked".

### timeline

```
statusLine (debounced ~300ms)                       onStop → route.drive
  └─ stepRouteStatusLine                              └─ stepRouteDrive
       ├─ derive current stone                             ├─ derive current stone
       ├─ phase  (yield/review.self/.peer/judge)           ├─ disposition = asStoneDisposition(status, blocker)
       ├─ disposition = asStoneDisposition(status,blocker) └─ push → block stop (exit 2)
       └─ asStatusLine({ phase, disposition })                halt → allow stop (exit 0)
            → "🗿 <stone>, <phase>[, <halt-word>] <glyph>"
```

## evaluation

### how well it solves the goal

directly. the wish's stated core — *"the suffix emojis are visual indicators of whether
attention is required via their colors, and it's always clear what part of the progress
the route is in"* — is exactly the two things this delivers: (1) a **phase + progress**
suffix, (2) an **attention-color emoji**. it rides the surface that already works (the
pinned statusline) and reuses every extant guard operation for the numbers.

### pros

- **attention-by-color** — the whole point: an observer triages by emoji temperature
  before a word is read.
- **progress-by-glance** — `r7/r10` and `l3@i002` expose depth that today needs a full
  `rhx route.drive` read.
- **reuses domain truth** — no new phase model invented; `RouteStoneGuardBlockerType`,
  the self-promise count, and the peer meter already exist.
- **one render source** — the format stays in `asStatusLine` (honors
  `rule.require.single-source-of-truth-for-render`); the suffix is not re-derived
  elsewhere.
- **graceful superset** — `blank`/`complete` unchanged in shape; the `stone` case gains a
  `phase` + `disposition`, both derived from extant state.
- **one disposition, two surfaces** — `asStoneDisposition` is the single source both the
  statusline glyph and the onStop push/halt read, so the pinned emoji and the autonomous
  drive can never disagree.

### cons / costs

- **heavier per-refresh work.** today's line is a cheap stone lookup. a read of the
  **peer meter** (`getAllReviewPeerMeterStatuses`) reads artifacts and computes a hash —
  materially more than a glob of the route dir. it runs every ~turn (debounced 300ms) and
  **blocks the line while it runs**. if slow, the line flickers/blanks. this is the
  primary build risk (see awkward #1).
- **more emoji vocabulary** to keep coherent with the repo's owl/turtle palette.
- **suffix length.** a halted peer review yields a long line
  (`🗿 5.1.execution.from_vision, review.peer, l3@i002, blocked ✋`) that may wrap on a
  narrow terminal (see edge cases).
- **the `exhausted` write-path.** the one core-state-machine touch: the guard must record
  `status: 'exhausted'` where it recorded `blocked` + `blocker: 'review.peer.exhausted'`, and
  readers of that blocker type must move to the status. isolated as its own build step.

### edge cases & pit-of-success

| edge case | contract keeps you safe by |
|-----------|---------------------------|
| stone has **no guard** | no blocker, no promises, no meters → `yield 🌾` (calm default) |
| peer meter derivation **slow** | v1: accept the cost, keep the read cheap; if too slow, fall back to `yield`/plain stone rather than a blocked line (never error) |
| **parallel** stones (same prefix) | show the first stone's phase, per `@next-one` — the same rule `route.drive` uses; a group is a known under-tell (deferred) |
| driver ran `--as blocked` mid-peer-review | `phase = review.peer` + `disposition = halt(blocked)` → `…, review.peer, l3@i002, blocked ✋`; onStop halts too |
| driver `--as blocked`, then re-arrives | the re-arrival supersedes the escalation (latest-entry-wins) → disposition recomputes; a stale `blocked` never lingers on the line or in onStop |
| re-arrival fails on an agent-fixable blocker | `disposition = push` → onStop **blocks the stop** (exit 2), the route keeps its own momentum; the line shows the calm phase glyph (🔍), not ✋ |
| `exhausted` status | `disposition = halt(exhausted)` → `…, exhausted 👋`; onStop allows the stop — a human must approve or extend |
| **narrow terminal** | phase suffix may wrap; keep tokens terse (`r7/r10`, `l3@i002`), never spell words out |
| a **stone-lookup** fault | strict: log to stderr, non-zero exit → harness blanks the line (fail-loud per `rule.forbid.failhide`, never a swallowed catch) |
| a **phase-derivation** fault | best-effort: log the phase fault to stderr, then degrade to the plain stone line (`🗿 <stone>`) — the fault is **surfaced** (not swallowed → not failhide), and the base feature never regresses to a blank line just because the *suffix* failed. only a stone-lookup fault blanks the line |
| route **complete** | swap `🎉` → `🌴🤙` (palmtree + shaka — the wisher's refinement of the wish's original `👏`): "done, hang loose — close it out or rewind" |

## open questions & assumptions

### assumptions made

1. **the suffix separator is a comma**, per the wish literally (`$stone, yield`).
   `🗿 1.vision, review.self, r7/r10 🔍`. (a middot `·` is the alternative — cosmetic.)
2. **emoji ↔ label map** from the wish's literal tokens: `:grain:`→🌾,
   `:magnifier:`→🔍, `:wave:`→👋, `:stophand:`→✋. malfunction gets `💥` (collision), the
   wisher's addition — a hard halt distinct from the `✋` driver wall (broken vs walled). the
   complete state = `:tada:`→🌴🤙 (palmtree + shaka), the wisher's refinement of the wish's
   original `:claphands:`/👏 — see assumption 6 for the tradeoff.
3. **`i{rounds}` means peer `rounds` at the highest active level**, read from the meter.
   the wish says "*@ i002 = rounds at level*" — so the number after `i` is `rounds`, even
   though the `i` glyph reads like "iteration". i keep the `i` glyph (the wish's choice)
   but source it from `rounds`. **flagged as a vocab tension below.**
4. **`yield` is the default phase** when the current stone has no blocker report and no
   promises yet — the stone still yields its artifact.
5. **phase is singular per stone** — one phase, the "furthest" active one. a stone that
   sits mid-self and mid-peer at once shows peer (the later phase). i assume the later
   phase is the right one to surface.
6. **`route complete` stays a single line** — only the final emoji changes 🎉→🌴🤙; the
   words "route complete" are unchanged. **wisher decision + tradeoff:** the wish's own
   text said swap 🎉→👏 (claphands, an "act — close or rewind" nudge); the wisher refined
   this to 🌴🤙 (palmtree + shaka, "done, hang loose"). the tradeoff, surfaced and
   accepted: 🌴🤙 reads calmer/"relax" where 👏 read more "act now" — the wisher chose the
   island vibe, since a complete route is a low-urgency, close-it-out-when-ready state, not
   a red-alert. the ✋/👋 states still carry the sharp attention signals; complete does not
   need to.
7. **the derivation reuses extant ops** (`getStoneGuardBlockerReport`,
   `computePromisedReviewCount`, `getAllReviewPeerMeterStatuses`) — no new guard truth.

### open questions (triaged)

`[answered]` decided now · `[research]` verify in blueprint/research · `[wisher]` needs the wisher

1. **`i002` = rounds or iterations?** `[wisher]` the wish's caption says "rounds at
   level" but the glyph is `i` (iteration). the meter exposes `rounds` (budget consumed)
   *and* artifacts carry a separate `iteration`. i assume **`i{rounds}`** (per the
   caption). confirm the intended number and whether the `i` glyph should become `r`
   (e.g. `l3@r002`) for term-honesty. **the one real vocab decision.**
2. **`review.peer.exhausted` — 🔍 or 👋?** `[answered]` **👋.** exhausted graduates to its own
   passage **status** (`exhausted`), which maps to `disposition = halt(exhausted)` → 👋. the
   machine is out of budget; a human must approve or extend. as a status it never masks as a
   calm `review.peer` push.
3. **magnifier glyph 🔍 vs 🔎.** `[answered]` use 🔍 (left-glyph), the common default.
   cosmetic.
4. **does `blocked ✋` overlay or replace the phase?** `[answered]` **neither — they are
   orthogonal.** `phase` (where the work is) and `disposition` (`push | halt`) are separate
   axes. a halt renders the phase text + a halt-word + the ✋/👋 glyph; there is no `blocked`
   phase to overlay. "blocked" was `disposition = halt`, now modeled directly.
5. **judge phase when a non-approval judge fails.** `[answered]` the blocker-type
   contract settles it: `RouteStoneGuardBlockerReport.ts:10-15` documents `'judge'` as
   "non-approval judges failed (**agent can fix**)" vs `'approval'` as "only approval judge
   blocks (**agent must wait for human**)". so a failed non-approval judge is the `judge`
   blocker — the *driver* fixes it, the observer human is not needed — render it 🔍 (calm,
   machine's turn), NOT the 👋 wave. reserve `judge, approved? 👋` strictly for the
   `approval` blocker (the one state that truly needs a human). the emoji thus tracks the
   contract's own "who fixes this" axis, which is exactly the wish's "is attention
   required" question.
6. **separator + width budget.** `[research]` confirm the longest realistic line
   (`blocked` over `review.peer` on a deep stone name) fits a typical terminal, or accept
   a wrap. no truncation planned for v1.
7. **are the numeric counters worth their cost?** `[wisher]` the wish asks for `r7/r10`
   and `l3@i002`. but the wish's *core* — "emoji color signals whether attention is
   required" + "clear what part of progress" — is satisfied by the **phase name + emoji
   alone** (`review.peer 🔍`). the counters are the one requirement that forces the
   expensive per-refresh peer-meter read (the primary build risk). a cheaper split is
   viable: **v1 = phase-name + emoji** (cheap: only the blocker report + self-promise
   count, both light), **v1.1 = numeric counters** (adds the peer-meter read). i hold the
   full requirement (the wish is explicit), but flag the split as the wisher's call if the
   peer-meter read proves too slow to run every ~turn.

## what is awkward?

1. **the peer-meter read on every refresh.** the base line was a cheap stone-glob; a
   derivation of `l3@i002` reads artifacts + hashes on every ~turn, and the harness
   blocks the line while it runs. this is the honest cost of "show the depth." the
   pit-of-success escape: if the derivation is slow, **degrade to `yield` / plain stone
   rather than a blocked or errored line** — the line stays alive, just less detailed.
   worth a small cache keyed on stone + artifact mtime later; not worth an over-build in
   v1.
2. **`i` for rounds.** the wish's `l3@i002` labels `rounds` with an `i` (which reads as
   "iteration"). fidelity to the wish's glyph collides with the repo's `ubiqlang` instinct
   (`rounds` is the meter's word). it's a one-character tension — surfaced as Q1 for the
   wisher: keep `i` (fidelity to the wish) or switch to `r`/`@r` (term-honesty).
3. **color depends on the reader's font/platform.** the whole thesis is "attention by
   emoji **color**" — but 🌾/🔍/👋/✋/🌴🤙 render with platform-specific palettes. the *shape*
   (a hand that waves vs a stop hand vs a magnifier) carries the signal even where color
   drifts, so the design leans on shape-plus-color, not color alone. still, "by color" is
   a slightly softer promise than it sounds.
4. **a halt suffix can run long.** `🗿 5.1.execution.from_vision, review.peer, l3@i002,
   blocked ✋` is a lot of line — the phase text plus a halt-word plus the glyph. it trades
   brevity for completeness; terse tokens keep it as short as honesty allows; no truncation
   in v1.
5. **the `exhausted` promotion ripples.** to pull `exhausted` out of the
   `review.peer.exhausted` blocker into its own passage **status** is the one change that
   touches the guard's write path + every reader of that blocker type. it removes the
   phase-masking ambiguity (exhausted no longer looks like a calm `review.peer`), but the
   migration is real work — isolated as its own build step, with the blocker→status readers
   swept together.
6. **onStop momentum vs the 21-attempt cap.** once a failed but agent-fixable arrival
   **pushes forward**, the route drives itself harder than before. the extant 21-consecutive-
   block cap (then escalate to a human) still bounds it, so it cannot spin forever — but the
   cap is now the safety net for the new momentum, and worth a conscious eye.
