# audience research: enbrief route creation

## who is the audience?

### primary: route authors

**who**: developers, architects, or team leads who want to create new thought routes for their brains to follow.

**context**: they've seen routes work (or heard about them) and want to create their own. they may be:
- new to rhachet and routes entirely
- familiar with rhachet but never authored a route
- familiar with other workflow tools (github actions, makefiles) but new to this paradigm

**constraints**:
- limited time to learn (want to create a route in <10 minutes)
- may not have access to examples beyond what's in the repo
- may be solo developers without team support

### secondary: route maintainers

**who**: developers who inherit or evolve extant routes.

**context**: they need to understand route structure to modify it — add stones, adjust guards, fix issues.

**constraints**:
- may not have original author available
- need to understand conventions to make safe changes

### tertiary: tool builders

**who**: developers who build on rhachet infrastructure.

**context**: they need to understand route mechanics to build tools that interact with routes.

**constraints**:
- need detailed specification, not just practical guidance

## what do they care about?

### route authors (primary)

**success criteria**:
- understand what a stone is in <1 minute
- understand what a guard is in <1 minute
- create first route in <10 minutes
- feel confident the route works as expected

**tradeoffs accept**:
- brief is reference, not tutorial (load on demand)
- assumes basic shell/cli familiarity
- doesn't cover every edge case

**tradeoffs reject**:
- incomplete core contract (must cover stone, guard, bind)
- broken examples that don't work
- jargon without explanation

**delight vs disappointment**:
- delighted: "I read the brief and created a route in 5 minutes. It just works."
- disappointed: "I still don't understand why my guard isn't run. The brief didn't explain."

## how will they experience it?

### route author journey

1. **discover**: brain (LLM) references the brief when route creation is relevant
   - touchpoint: `rhachet roles boot --repo bhrain --role driver` loads brief as ref
   - friction: must know to ask for it (ref mode, not say mode)

2. **read**: author reads the brief
   - touchpoint: markdown content in context
   - friction: must scroll through if long

3. **act**: author creates route directory, stones, optional guards
   - touchpoint: file system, shell commands
   - friction: typos in file names, yaml syntax errors

4. **bind**: author runs `rhx route.bind.set --route $path`
   - touchpoint: CLI output
   - friction: wrong path, typos

5. **verify**: author runs `rhx route.drive` to confirm
   - touchpoint: CLI output shows next stone
   - friction: none if prior steps correct

### smooth experience

- brief provides clear anatomy diagram
- brief includes minimal copy-paste example
- brief links to skill help (--help flag)
- errors are actionable ("guard not found" points to solution)

## summary

| audience | needs | brief should |
|----------|-------|--------------|
| route authors | quick start, clear contracts | explain stone, guard, bind; provide example |
| route maintainers | safe evolution | explain conventions, file patterns |
| tool builders | specification | cover edge cases, error behaviors |

primary focus: **route authors who want to create their first route in <10 minutes**.
