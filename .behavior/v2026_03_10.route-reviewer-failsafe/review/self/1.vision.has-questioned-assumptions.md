# self-review: has-questioned-assumptions

## assumption 1: the review skill crashed

- **what we assumed:** review crashed before it could produce output
- **evidence:** artifact shows API key load but no review content
- **what if opposite:** review completed successfully but with no work to do

**deeper analysis:**

the guard command uses `--join intersect` which means:
- `--rules` glob matches rule files
- `--diffs since-main` matches changed files
- `--paths-with '$route/3.3.blueprint.*.md'` matches blueprint files
- INTERSECTION of all three determines what gets reviewed

if the blueprint files weren't modified in the diff, the intersection is EMPTY. the review would:
1. load API keys (observed)
2. find 0 files in the intersection
3. complete "successfully" with zero files to review
4. produce no meaningful output

this is NOT a crash - it's a valid edge case where the review has zero files to review.

**counterexample that changes the hypothesis:**

the review skill might be correct. the REAL bug is:
1. the skill should fail-fast when intersection = 0 files
2. OR the judge should recognize "empty review" as a distinct failure mode
3. OR the guard should validate that paths-with files exist in diffs before invoke review

**verdict:** assumption is LIKELY WRONG - this is probably not a crash but an empty-intersection edge case

## assumption 2: hook stopped due to a bug

- **what we assumed:** hook has a bug that prevents re-fire
- **evidence:** output shows "blocked by constraints" message
- **what if opposite:** "blocked by constraints" might be CORRECT behavior (permission denied, not a bug)
- **did wisher say?** wisher asked "why was the hook never fired again?" which implies unexpected behavior

**deeper analysis:**

the "blocked by constraints" message comes from the pretooluse hook system. this means:
1. the hook DID fire (route.drive ran)
2. but a CONSTRAINT blocked the skill invocation (route.stone.set)

the hook is NOT broken. the hook is correctly block the skill from execution.

the question becomes: WHY is the constraint block the skill? possible causes:
1. permission config in `.claude/settings.local.json` or `.claude/settings.json`
2. a pretooluse hook that denies repeated calls
3. rate limit or cooldown logic in the constraint system

**verdict:** assumption is CONFIRMED WRONG - the hook works correctly, a constraint blocks the skill

## assumption 3: the judge failure is opaque

- **what we assumed:** judge output is unhelpfully empty
- **evidence:** judge artifact shows only `exit code: 1`
- **what if opposite:** maybe the capture logic works but there's no stderr to capture
- **counterexamples:** if the subprocess never started or crashed immediately, there may be no output

**deeper analysis:**

the `runOneStoneGuardJudge` function in `runStoneGuardJudges.ts` does:
1. lines 66-81: capture stdout/stderr from exec, even on failure
2. lines 86-96: write output if no file was written by command
3. lines 99-106: fallback message if file doesn't exist

the capture logic is correct. so why is the artifact empty?

possible causes:
1. the judge command never started (e.g., command not found, PATH issue)
2. the judge crashed before any output (e.g., import error in node)
3. node exec returned empty stdout AND empty stderr

if `errObj.stdout` and `errObj.stderr` are both empty strings, the artifact gets only the metadata.

this suggests the judge command DID start and DID exit, but produced zero output. looking at `judgeReviewed`:
- it calls `enumFilesFromGlob` to find review files
- if no files found, it outputs "no review files found for hash XXX"

so the judge SHOULD have output. unless... the judge itself crashed before that line.

**new hypothesis:** the judge crashed during import or setup, before any console.log executed.

**verdict:** assumption is VALID but incomplete - the judge failed early, before it could output a reason
