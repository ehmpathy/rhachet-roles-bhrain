2. **assumption:** fast-glob expands symlinks and then applies ignore patterns to the expanded path
3. **assumption:** the blocker count inconsistency is caused by wrong review file references — the second tree shows `r6.md` for r1-r6 entries instead of `r1.md`-`r6.md` respectively

## groundwork

### external research

**fast-glob symlink handling:**
- `followSymbolicLinks` defaults to `true` — symlinks are followed
- however, the `ignore: ['**/node_modules/**']` pattern excludes `node_modules` paths
- unclear: does ignore apply to the user-visible path (`.agent/...`) or the expanded path (`node_modules/...`)?

**citation:** fast-glob README section on `followSymbolicLinks`:
> Indicates whether to traverse descendants of symbolic link directories when expanding `**` patterns.

### internal research

**current enumFilesFromGlob implementation:**

file: `src/domain.operations/review/enumFilesFromGlob.ts:80-85`
```ts
const files = await fg(positivePatterns, {
  cwd,
  ignore: ['**/node_modules/**', ...negativePatterns],
  onlyFiles: true,
  dot: true,
});
```

the `ignore: ['**/node_modules/**']` is always applied. if fast-glob applies ignore patterns to expanded symlink paths, this would filter out files accessed via symlinks that point into `node_modules`.

**symlink structure in ghlitch repo:**

file: tree output of `/home/vlad/git/ehmpathy/_worktrees/rhachet-roles-ghlitch.vlad.operator/.agent`
```
├── repo=ehmpathy
│   └── role=mechanic
│       ├── briefs -> ../../../node_modules/.pnpm/rhachet-roles-ehmpathy@.../briefs
```

the `briefs` directory is a symlink to a path inside `node_modules`.

**blocker count inconsistency:**

observed in wish output — r1 shows "1 blocker" in first tree, "0 blockers" in second tree. on deeper inspection: the first tree references `r1.md` but the second tree references `r6.md` for the r1 entry. this is a data/reference bug, not a display bug. all r1-r6 entries in the second tree reference the same `r6.md` file. need to trace where the guard summary tree gets its review file references.

## important distinction (from wisher)

**rules/refs vs targets:**
- rules and refs CAN come from `node_modules` via symlinks — this is the standard pattern for package-distributed roles
- target files (files under review) should NEVER come from `node_modules`

**implication for fix:** the `node_modules` ignore should remain for target file globs, but be relaxed for rules/refs globs. this may mean separate ignore patterns for different glob contexts.

---

## verification: acceptance test coverage

**critical:** the fix must include thorough acceptance tests that prove these cases cannot regress:

1. **symlink traversal test:** rules accessed via symlink to `node_modules` must be found
2. **target file protection test:** target files in `node_modules` must still be ignored
3. **blocker count consistency test:** both output trees must show identical counts for the same reviewer

these tests are the primary deliverable — they encode the contract and prevent future regression.

---

## what is awkward?

1. **the `node_modules` ignore is a sledgehammer** — we ignore ALL paths that include `node_modules`, but symlinks from `.agent/` that lead to `node_modules` are intentional and desired for rules/refs

2. **two separate issues bundled** — symlink traversal and blocker count display are unrelated bugs; fixing one doesn't fix the other

3. **unclear fast-glob behavior** — need empirical testing to confirm whether static patterns work through symlinks and whether ignore applies to expanded paths

4. **potential performance regression** — if we loosen the `node_modules` ignore, we might accidentally traverse actual `node_modules` directories and cause OOM

## proposed investigation steps

1. **defect 1 (symlink traversal):**
   - write integration test that reproduces the issue (rule path through symlink to node_modules)
   - verify whether fast-glob is the culprit or something else
   - if ignore pattern is the issue, consider:
     - option A: don't ignore `node_modules` when path starts with `.agent/`
     - option B: use a different ignore pattern that only ignores direct `node_modules/` traversal
     - option C: expand symlinks before passing to fast-glob

2. **defect 2 (blocker count inconsistency):**
   - trace where the guard summary tree gets its review file references
   - the bug is that r1-r6 in the second tree all reference `r6.md` instead of `r1.md`-`r6.md`
   - find the code that populates the guard.reviews array and fix the reference logic
