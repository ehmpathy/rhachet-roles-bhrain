## gaps identified

**gap 1: exit code 2 (constraint) behavior**

the wish focuses on malfunction (exit 1), but what about constraint (exit 2)?

- exit 0 = passed → cached
- exit 1 = malfunction → NOT cached
- exit 2 = constraint → ??? (not in wish)

**is this a real gap?**

the wish says "only successes get cached" which implies:
- success (exit 0) → cached
- non-success (exit != 0) → not cached

the implementation uses `exitClass !== 'malfunction'` which means:
- passed → cached
- constraint → cached (exit 2)
- malfunction → not cached

**wait:** this might be wrong. let me check the wish again.

> "if a reviewer malfunctions, we dont cache its broken state. only successes get cached"

this says "only successes" which means constraint should NOT be cached either. but the current logic only excludes malfunction.

**however:** constraint (exit 2) means "blocker found" which is a VALID review result, not a broken state. the reviewer worked correctly, it just found issues.

**verdict:** not a gap. constraint is a valid review outcome, malfunction is a broken reviewer. the cache semantics are correct.
